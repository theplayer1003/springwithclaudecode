# 솔트(Salt)가 레인보우 테이블을 무력화하는 원리

## Q. DB가 탈취되면 솔트도 노출되는데, 왜 여전히 효과적인가?

**A:** 솔트의 방어 원리는 "솔트를 숨기는 것"이 아니라 **레인보우 테이블의 재사용을 불가능하게 만드는 것**이다.

---

## 레인보우 테이블 공격 시나리오

### 솔트가 없는 경우

공격자는 DB 탈취 **전에** 미리 테이블을 만들어둔다:

```
레인보우 테이블 (수십억 개의 평문-해시 쌍, 수개월~수년에 걸쳐 계산)
password123  →  ef92b778...
qwerty       →  65e84be3...
iloveyou     →  f25a2fc7...
abc123       →  6ca13d52...
... (수십억 개)
```

DB를 탈취하면:

```
user_A: ef92b778...  → 테이블에서 조회 → password123 (즉시)
user_B: 65e84be3...  → 테이블에서 조회 → qwerty (즉시)
user_C: 6ca13d52...  → 테이블에서 조회 → abc123 (즉시)
```

**10만 명의 비밀번호를 몇 초 만에 전부 알아낼 수 있다.** 테이블은 한 번만 만들면 어디서든 재사용 가능하다.

### 솔트가 있는 경우

DB를 탈취하면 이런 데이터가 보인다:

```
user_A: salt=x7k2  hash=a1b2c3d4...
user_B: salt=m9p3  hash=e5f6g7h8...
user_C: salt=q4w8  hash=i9j0k1l2...
```

기존 레인보우 테이블은 `hash(password123)` = `ef92b778...`로 계산되어 있다.
하지만 솔트가 붙으면 `hash(x7k2 + password123)` = `a1b2c3d4...`가 된다.
**완전히 다른 해시값**이므로 기존 테이블에서 일치하는 것이 하나도 없다.

---

## Q. 솔트를 알면 그 솔트에 맞는 새 테이블을 만들면 되지 않나?

**A:** 만들 수 있다. 하지만 그것이 바로 솔트가 노출되어도 효과적인 이유다.

```
user_A용 테이블 (salt=x7k2): 수십억 개를 새로 계산 → 수개월 소요
user_B용 테이블 (salt=m9p3): 수십억 개를 새로 계산 → 수개월 소요
user_C용 테이블 (salt=q4w8): 수십억 개를 새로 계산 → 수개월 소요
```

사용자마다 솔트가 다르기 때문에 **사용자 1명당 테이블 1개를 새로 만들어야** 한다.
10만 명이면 10만 개의 테이블을 각각 수개월씩 계산해야 하므로, 이것은 사실상 **한 명씩 무차별 대입(brute force)하는 것과 동일한 비용**이다.

레인보우 테이블의 핵심 장점은 "한 번 만들면 모든 사용자에게 재사용"인데, 솔트가 이 재사용을 차단한다.

---

## BCrypt의 이중 방어

솔트로 레인보우 테이블을 무력화한 뒤, BCrypt의 **cost factor(느린 연산)**가 brute force마저 비현실적으로 만든다.

한 번 해싱하는 데 0.1초가 걸리면, 비밀번호 후보 10억 개를 시도하려면 약 3년이 걸린다.

---

## 정리

| 상황 | 공격 비용 |
|------|----------|
| 솔트 없음 | 테이블 1개로 전체 사용자 공격 가능 (효율적) |
| 솔트 있음 | 사용자 1명당 테이블 1개 필요 = brute force와 동일 (비효율적) |
| 솔트 + BCrypt cost | brute force 자체도 수년 소요 (비현실적) |

**솔트는 숨기기 위한 것이 아니라, 사전 계산의 재사용을 막기 위한 것이다.**
