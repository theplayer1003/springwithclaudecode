# 총정리 Part 2: 개념 퀴즈

시나리오에서 다루지 못했거나 깊이가 부족했던 개념들을 주제별로 정리한다.
각 질문에 직접 답변을 작성한 후 리뷰를 받는다.

## A. 스프링 핵심 개념 (Phase 1)

### A-1. IoC와 DI

1. "의존성이 있다"는 것은 어떤 상태를 말하는가?

A 클래스가 실행되기 위해 B 클래스가 필요할 경우 A 클래스가 B 클래스에 의존성이 있다고 말합니다.


2. 직접 생성(`new`)으로 의존성을 해결하면 어떤 문제가 생기는가? (2가지)

new 로 생성된 객체가 다른 객체에게 강하게 결합됩니다.
추후 요구사항의 변경으로 new 로 생성한 객체가 변경되면 그 변경의 여파가 직접적으로 반영되어 유지 보수가 어려워집니다.
또 단위 테스트를 어렵게 만듭니다. 해당 클래스의 테스트를 위해 new 로 생성된 객체의 특정 조작된 결과가 필요할 경우 이를 모사하기가 어려워집니다.

```java
class PostService {
    private PostRepository repo = new PostRepository();
}
```
직접 생성 시 변경에 취약해집니다. 만약 PostRepository 의 생성자가 변경된다면 이 곳의 코드도 같이 변경되어야 합니다.
테스트가 어려워집니다. 테스트에서 DB 없이 가짜 Repository 를 넣고 싶어도 이미 new PostRepository() 로 고정되어 있어 교체가 불가능합니다.


3. DI(의존성 주입)란 무엇이며, 직접 생성의 문제를 어떻게 해결하는가?

의존성 주입이란 클래스 간의 강한 결합을 막기 위해 클래스 내부에서 직접 생성하는 대신 생성자를 통해 외부에서 만들어진 객체를 주입시켜주는 방식입니다.
인터페이스 등의 다형성과 결합해 작성되는 클래스는 외부 클래스의 변경 등을 알 필요가 없어져 유지보수, 테스트 등이 용이해집니다.


4. DI 방식 3가지를 나열하고, 실무에서 권장되는 방식과 그 이유는?

첫 번째는 생성자를 통한 의존성 주입입니다. 이 형태가 실무에서 기본으로 권장되는 방식입니다.
private final 키워드를 사용 가능해 불변성을 구현할 수 있습니다.
객체 생성 시점에 모든 의존성이 주입되도록 하기 때문에 누락 방지의 이점이 있습니다.
Spring 4.3+ 부터 생성자가 1개일 경우 @Autowired 어노테이션 생략이 가능합니다.
만약 Lombok 라이브러리를 사용한다면 @RequiredArgsConstructor 어노테이션을 사용해 더 깔끔한 코드 작성이 가능합니다.
이 어노테이션은 final 이 붙은 필드를 모아 생성자를 자동으로 선언해줍니다.

두 번째는 setter 주입입니다. 객체 생성 이후 setter 호출로 필요한 의존성을 주입해주는 방식입니다.
이는 객체의 불변성을 위해 final 을 선언할 수 없습니다. 객체가 생성된 뒤에 주입해줘야 하기 때문입니다.

세 번째는 필드 주입 입니다. 필드에 @Autowired 를 직접 선언하는 방식입니다.
코드가 짧지만 final 이 불가능하며 테스트 시 리플렉션을 써야 주입이 가능합니다.


5. IoC(제어의 역전)란 무엇인가? DI와 어떤 관계인가?

제어의 역전이란 개발자가 직접 의존성을 주입해주는게 아닌 객체의 생성과 연결을 프레임워크에게 맡기는 걸 의미합니다.
Spring 프레임워크를 사용하게 되면 적절한 구현체를 Spring 이 자동으로 외부에서 생성해 주입해주게 됩니다.
DI 는 IoC 를 실천하는 방식 중 하나입니다.



### A-2. Bean

1. Bean이란 무엇인가?

Bean 이란 스프링이 관리하는 객체를 의미합니다.
@Component, @Controller, @RestController, @Service, @Repository
어노테이션 등이 붙으면 Spring 이 해당 클래스를 Bean 으로 등록합니다.
이후 애플리케이션이 실행되면 Spring 이 자동으로 이를 생성하고(싱글톤) 필요한 경우 주입해서 사용할 수 있도록 해줍니다.


2. Bean을 등록하는 어노테이션 4가지를 나열하라. 각각 어떤 계층에서 사용하는가?

@Component : 특정 계층에 속하지 않는 일반 빈에 선언되는 어노테이션입니다. 아래 세 개가 특별하게 지정되어 생성되는 bean 이라면 컴포넌트는 일반적인 클래스에 선언됩니다.
@Controller, @RestController : Controller 계층에서 사용되는 어노테이션입니다. 웹 요청을 받아 적절한 Service 객체에게 전달해주는 역할을 하는 클래스에 선언합니다.
@Service : Service 계층에서 사용되는 어노테이션입니다. Controller 가 보내주는 요청을 받아 Repository 클래스와 협력해 로직을 수행합니다.
@Repository : Repository 계층에서 사용되는 어노테이션입니다. 데이터의 영속성을 처리하는 클래스로 Service 클래스의 요청에 따라 DB 와 통신해 데이터를 가져옵니다.

   
3. Bean은 기본적으로 몇 개가 생성되는가? 이를 무엇이라 하며, 어떤 이점이 있는가?

Bean 은 단 한개 생성됩니다. 이를 싱글톤이라고 하며 프로그램 실행과 끝 까지 하나만 생성되어
사용되므로 메모리를 절약할 수 있고 객체의 생성과 소멸에 드는 비용을 제거할 수 있습니다.

빈의 생명주기는 다음과 같습니다.
애플리케이션이 시작되고 스프링이 먼저 @Component, @Service 등 어노테이션이 붙은 클래스를 자동으로 탐색합니다.
이를 컴포넌트 스캔 과정이라고 합니다.
해당 클래스들을 찾아 미리 생성해 스프링 컨테이너에 저장해둡니다.
빈이 필요해지면 외부에서 Spring 이 자동으로 적절한 구현체를 찾아 주입해줍니다.
애플리케이션이 실행되는 동안은 계속 빈을 재사용하며 애플리케이션이 종료 되면 빈은 소멸 됩니다.


### A-3. @SpringBootApplication

1. 이 어노테이션은 내부적으로 3가지 어노테이션을 포함한다. 각각 무엇이며 어떤 역할을 하는가?

@SpringBootConfiguration
@EnableAutoConfiguration
@ComponentScan(excludeFilters = { @Filter(type = FilterType.CUSTOM, classes = TypeExcludeFilter.class),
@Filter(type = FilterType.CUSTOM, classes = AutoConfigurationExcludeFilter.class) })

이렇게 세가지 어노테이션을 포함합니다.
스프링 부트는 스프링에서 한 단계 더 나아가 스프링의 복잡한 설정 피로도를 줄이고자 자동으로 웹 개발에 필요한 설정을 처리해주는 프레임워크입니다.

@SpringBootConfiguration 
어노테이션은 이 클래스가 스프링 부트의 설정 클래스임을 표시합니다. @Configuration 의 부트 버전으로, 빈 정의를 포함할 수 있다는 의미 입니다.

@EnableAutoConfiguration
어노테이션은 프로젝트에 추가된 라이브러리를 보고 자동으로 설정을 하도록 합니다.
예를 들어 spring-boot-starter-web 이 있으면 톰캣, DispatcherServlet 을 자동 구성 하는 식 입니다.
spring-boot-starter-data-jpa 가 있으면 EntityManager 를 자동 구성 하는 식 입니다.

@ComponentScan
어노테이션은 있는 클래스의 패키지와 그 하위 패키지에서 @Component 계열 어노테이션을 탐색하여 빈으로 등록하게 합니다.
즉, SpringBootApplication 은 항상 최상위 패키지에 두기 때문에 하위 모든 패키지가 스캔 대상이 됩니다.


2. `@ComponentScan`은 어디서부터 빈을 탐색하는가?

앞서 언급한대로 선언된 클래스의 하위 패키지 모두가 대상이 됩니다. 이 어노테이션은 @SpringBootApplication 에 포함되어 있고
해당 어노테이션은 Application 진입점 클래스에 선언되기 때문에 메인 클래스 아래의 모든 패키지가 스캔 됩니다.


### A-4. REST API

1. REST에서 URI는 무엇을 나타내고, HTTP 메서드는 무엇을 나타내는가?

Uniform Resource Identifier, 통합 자원 식별자란 뜻으로 인터넷에 있는 항목을 구분하기 위한 모든 이름을 뜻합니다.
REST API 에서는 엔드포인트 주소를 의미하게 되며 각 Controller 의 메서드는 자신에게 대응되는 HTTP 메서드와 주소를 가져
HTTP 메서드 + 주소 조합으로 구별됩니다.
HTTP 메서드는 HTTP 프로토콜에 정의된 통신 규약 종류를 말하며 GET, POST, PUT, DELETE 등 각 동작에 대응되는 메서드를 클라이언트가
함께 보내 필요한 요청을 수행하도록 합니다.


2. `@Controller`와 `@RestController`의 차이는?
@Controller 는 전통적인 웹 페이지 제작에 사용되던 어노테이션이었습니다.
사용자의 요청에 대해 반환한 문자열과 일치하는 View를 찾아 화면을 보여주도록 했습니다. 여기에 JSP, Thymeleaf 등이 사용되었습니다.
@RestController 는 Representational State Transfer 의 약자 입니다.
Representational 이란 자원을 이름으로 표현한단 뜻이고 State Transfer 는 상태를 주고 받는단 뜻입니다.
REST 는 웹의 기존 인프라인 HTTP 프로토콜을 그대로 활용하기 위해 만들어진 규칙으로 이 규칙을 지키는 서비스를 RESTful 하다고 합니다.
REST 의 핵심은 3가지 구성 요소 입니다.
Resource, 내가 다룰 데이터를 URI 로 표현합니다. /posts, /auth 가 예시입니다.
Verb, 내가 할 행위를 HTTP 메서드로 표현합니다.
Representation, 데이터를 어떤 형식으로 보낼지 결정합니다. JSON, XML 등이 있습니다.

전통적인 @Controller 는 URL 을 페이지를 여는 주소로 썼지만, @RestController 는 URL 을
데이터 자원을 가리키는 고유 ID 로 사용합니다.
예를 들어 전통 방식, 비REST 에서는 
`/getUserInfo?id=1` : ID 가 1인 유저 정보를 가져와서 화면을 보여달라
RESTful 에선
`GET /users/1` : ID가 1인 유저 데이터를 보내달라
`DELETE /users/1` : ID가 1인 유저 데이터를 삭제해달라

즉, 전자는 '어떤 화면' 을 요청합니다.
후자는 '어떤 데이터' 를 요청합니다.
비 REST 는 주소를 '기능을 수행하는 특정 페이지의 위치' 로 취급하며
REST 는 주소를 '서버에 존재하는 데이터 그 자체' 로 취급하는 차이가 있습니다.

REST 는 페이지를 그려주는 행위(View)와 결합되지 않고 데이터를 보내주기만 하면 되기 때문에
더 유연하게 다양한 환경에 대응할 수 있게 됩니다.(웹 브라우저, 아이폰 앱, 안드로이드 앱 등등)



3. `@RestController`는 내부적으로 어떤 두 어노테이션이 결합된 것인가?

@RestController = @Controller + @ResponseBody 입니다.
@ResponseBody 는 자바 객체를 HTTP 응답 본문으로 통째로 바꿔서 보내라고 지시하는 어노테이션입니다.
전통 웹 페이지 구조에서는 기본적으로 View 를 통해 화면을 그려서 주고 필요할때 이 어노테이션을 붙여 데이터로 보내줬습니다.
이제 REST 구조를 채택하게 되면 @ResponseBody 는 필수 어노테이션이 되기 때문에 번거롭게 모든 코드에 이 어노테이션을 붙이지 않아도 되도록
두 어노테이션이 합쳐져 @RestController 어노테이션이 되었습니다.

### A-5. 요청-응답 흐름

1. 클라이언트의 HTTP 요청이 Controller 메서드에 도달하기까지의 흐름을 순서대로 나열하라.
   (톰캣, DispatcherServlet, Filter 등을 포함)

클라이언트가 요청을 보내오면 가장 먼저 톰캣 서버에 도착하게 됩니다.
톰캣은 무수한 클라이언트 요청을 알맞은 애플리케이션에 분배해주는 역할을 합니다.
또 Http 요청을 Java 가 응답 가능한 HttpServletRequest 객체로 변환하는 역할을 합니다.

톰캣이 분배해준 요청이 애플리케이션에 도착하면 먼저 SpringSecurity 가 작동 됩니다.
SpringSecurity 는 요청의 헤더에 존재하는 토큰을 가져옵니다. 불필요한 prefix 를 제거하고 순수한 토큰 문자열만을 획득합니다.
해당 토큰이 유효한지 확인하기 위해 filter 를 거칩니다. filter 속에서 토큰이 유효한지 확인이 되면
필요한 정보를 담은 객체를 생성하고 이를 SecurityContext 에 등록합니다.
인증에서 문제가 생길 경우 예외가 발생하며 이를 filter 에서 잡아 HttpServletResponse 를 작성해 인증, 인가에 실패했음을 알릴 수 있는
대답을 작성해 반환합니다.
문제 없이 통과했다면 DispatcherServlet 으로 전달됩니다. 여기서 URL 에 맞는 Controller 를 찾고
비로소 클라이언트의 요청이 Controller 에 도착해 로직이 수행됩니다.


2. DispatcherServlet의 역할은 무엇인가?

DispatcherServlet 은 요청을 Controller 로 분배해주는 역할입니다.
과거에는 모든 URL마다 서블릿을 따로 만들어야 했지만 이제는 모든 요청을 그냥 DeispatcherServlet 이 다 받습니다. 이를 프런트 컨트롤러 패턴이라고도 합니다.
HTTP 요청은 전부 받은 다음 적절한 컨트롤러 탐색을 위해 HandlerMapping 에게 물어봅니다. 이렇게 찾아낸 컨트롤러의 메서드를 호출하고
응답을 받아 결과물을 JSON 으로 보낼지, HTML 화면을 띄울지 결정해 클라이언트에게 응답합니다.

DispatcherServlet 이 작동하는 타이밍은 어떤 인증, 인가 방식을 사용하느냐에 따라 약간 차이가 있습니다.
인증, 인가 방식은 크게 우리가 구현한 Filter 방식과 Interceptor 방식이 있습니다.
전자는 Spring MVC 계층에 진입하기 전에 선제적으로 작동되는 방식입니다.
이 방식에서는 요청이 들어오면 먼저 Filter 를 거치고 여길 통과해야 DispatcherServlet 이 작동하고 Controller 에게 요청이 들어갑니다.
후자는 요청이 곧바로 DispatcherServlet 에 들어가게 됩니다. 그 다음 컨트롤러에게 보내지기 전에 Interceptor 가 작동해 인증, 인가 처리를 합니다.

두 방식의 핵심 차이점은 Spring Context 안으로 들어왔느냐 아니냐 입니다.
Filter 방식은 스프링이 존재하기 전부터 존재하던 자바 표준 기술 입니다. Spring 과 무관하게 웹 애플리케이션 전체에 적용하는 단위입니다.
Interceptor 는 스프링 컨테이너가 수행합니다. 즉, 스프링의 모든 빈과 컨트롤러 정보에 접근 가능하며 filter 방식과 다르게 GlobalExceptionHandler 에서
예외를 잡아 처리할 수 있게 됩니다.

filter 방식은 주로 애플리케이션 전체 보안, 로깅, 이미지 변환, 인코딩 등 웹 애플리케이션 전 범위에 적용해야하는 작업에 이용됩니다.
Interceptor 방식은 로그인 체크, 세부 권한 확인, 공통 데이터 처리 등 컨트롤러의 정보를 모두 알아 정밀한 제어가 필요한 작업에 이용됩니다.


3. Controller의 반환값이 클라이언트에게 JSON으로 도착하기까지 어떤 과정을 거치는가?

Controller 가 ResponseEntity 객체에 Http 응답과 요청 결과에 대한 DTO 를 담아 반환합니다.
이 객체는 DispatcherServlet 으로 전달되어 HTTP 응답을 위해 가공됩니다.
먼저 ReturnValueHandler 를 호출해 응답 타입을 확인 시키고 적절한 처리기로 처리할 수 있도록 합니다.
ReturnValueHandler 는 자바 객체를 JSON 으로 변환하기 위해 HttpMessageConverter 를 선택합니다.
HttpMessageConverter 는 자바 객체를 JSON 문자열로 직렬화 해줍니다.
이 값이 DispatcherServlet 으로 돌아와 ResponseEntity 에 같이 온 Http 응답 코드 등과 조합해 최종 응답을 완성합니다.
응답 역시 그냥 나가지 않고 Filter 를 거칩니다. 응답 데이터를 암호화하거나 공통적인 응답 로그를 남기거나, 응답 헤더에 보안 설정을 추가해야할 수도 있기 때문입니다.
Filter 를 거쳐 Tomcat 에 도착합니다.
Tomcat 은 스프링에서 조립된 HTTP 응답 메시지를 바이트 스트림으로 변환해 네트워크망을 통해 클라이언트에게 가도록 전송합니다.
클라이언트가 최종적으로 응답을 수신합니다.

## B. 데이터베이스와 JPA (Phase 2)

### B-1. ORM과 JPA

1. SQL을 직접 작성하면 생기는 3가지 문제는?

SQL 을 직접 작성한다는 것은 JDBC 를 사용해 코드에 `String sql = "...";` 가 선언된단 뜻 입니다.
JDBC 는 PreparedStatement 를 사용해 쿼리 결과를 반복문을 순회하며 꺼내야하는데 이 때문에 코드가 매우 길어지고 유지 보수 비용 또한 늘어나게 됩니다.
또 DB 의 경우 RDBMS 를 사용하게 되면 관계로 데이터를 다루게 되는데 이 형태는 Java 의 객체 패러다임과 일치하지 않아 관리가 어렵습니다.
마지막으로 특정 DB 가 쓰는 특화 문법들이 존재하는데 이를 코드에 하드코딩하게 되면 DB 종속성이 생겨 유연한 프로그램을 작성할 수 없습니다.


2. ORM이란 무엇이며, 어떤 문제를 해결하는가?

ORM 이란 Object Relational Mapping 의 약자로 객체와 RDBMS 의 관계를 맵핑해주는 기술을 뜻합니다.
앞서 말한대로 객체지향과 관계형 데이터 관리는 패러다임이 일치하지 않습니다.
Post 와 Comment 를 예로 들어보겠습니다.
관계형 데이터 베이스에서는 Post 테이블이 존재할 것이고 Comment 테이블이 따로 존재합니다.
이 둘을 이어주기 위해 Comment 테이블이 Post 테이블의 PK 를 참조하는 FK 를 가집니다.
이 연결고리(관계)를 통해 두 테이블을 조인해 데이터를 완성시킬 수 있습니다.

반면 Post 와 Comment 를 자바로 구현한다면 Post 객체에는 자신 게시글에 속한 List<Comment> 참조가 존재해야하고
Comment 는 자기가 속한 Post 를 참조를 가져야합니다.

이 차이로 인해 RDBMS 에서 꺼내온 데이터를 객체로 만드는데 복잡성이 생기게 되고
이를 쉽게 해결해주는 기술이 ORM 입니다. 객체를 작성할때 규칙을 지어주면 ORM 이 자동으로 RDBMS 테이블의 형태에 맞게 매핑해
엔티티를 저장해줍니다.


3. JPA, Hibernate, Spring Data JPA의 관계를 설명하라. (인터페이스 → 구현체 → 추상화 계층)

JPA 는 ORM 기술을 실현하는 Java 의 명세 규약(인터페이스)이고, 그 명세를 실제로 알맞게 구현한 구현체가 바로 Hibernate 입니다.
Spring Data JPA 는 Spring 에서 Hibernate 를 쉽게 사용하기 위해 한번 감싸 편의성을 제공하는 라이브러리(Spring Data 프로젝트의 하위 모듈) 입니다.


### B-2. Entity

1. Entity란 무엇인가?

엔티티란 DB 테이블과 1:1 매핑되는 자바 클래스입니다. @Entity 어노테이션을 붙여 해당 클래스를 JPA 가 테이블로 인식하도록 하고
클래스의 각 필드가 테이블의 컬럼이 됩니다. DB 테이블의 한 행을 자바 객체로 표현한 것이 엔티티 입니다.


2. `@Id`와 `@GeneratedValue`는 각각 무엇을 하는가?

`@Id` 는 해당 엔티티의 키 값을 의미하며 `@GeneratedValue` 는 키 생성전략입니다. 자동 생성으로 엔티티가 저장될 때마다 1 씩 증가하며
JPA 가 자동으로 입력해줍니다.


`@Id` 는 해당 필드가 엔티티의 기본키(PK) 임을 선언하는 어노테이션이며 `@GeneratedValue` 는 기본키 값의 생성 전략을 지정하는 어노테이션 입니다.
생성 전략에는
`IDENTITY` : DB에 INSERT 할 때 DB 가 자동으로 키를 생성(MySQL 의 AUTO_INCREMENT)
`SEQUENCE` : DB의 시퀸스 객체를 사용 (PostgreSQL, Oracle)
`TABLE` : 별도의 키 생성 테이블 사용
`AUTO` : DB 에 맞게 JPA 가 자동으로 선택
이 있습니다.
우리가 사용한 `@GeneratedValue` 는 `IDENTITY` 에 속하는 전략으로 DB가 INSERT 시점에 자동으로 키를 생성합니다. 


3. JPA Entity에 기본 생성자가 필수인 이유는?

앞서 이야기한 키값처럼 JPA 가 자동으로 세팅해주는 작업이 필요하기 때문입니다.
테이블에서 데이터를 꺼내와 객체를 생성할 때는 비어있는 객체인채로 생성하며 객체를 생성하고 setter 를 통해 알맞은 자료를 set 해줍니다.
따라서 protected 접근제어자로 외부에서 직접 호출하는 것을 방지하며 JPA 가 접근 가능하도록 하는 비어있는 기본 생성자가 필수 입니다.

Spring 프레임워크가 JpaRepository 를 상속하면 미리 CRUD 기본 기능이 작성된 구현체를 주입시켜주기 때문에

### B-4. 영속성 컨텍스트

1. 영속성 컨텍스트란 무엇인가?

JPA 의 1차 캐시 기능입니다. 데이터를 조회할 때 항상 외부에 존재하는 DB 에 쿼리를 날려 통신하게 되면 많은 자원을 소모하게 됩니다.
이를 방지하고자 JPA 는 이번 트랜잭션에서 이미 조회된 엔티티를 잊어버리지 않도록 영속성 컨텍스트에 미리 저장해둡니다. 
이제 조회가 필요할 때 JPA 는 무조건 쿼리를 날리지 않고 먼저 영속성 컨텍스트를 확인합니다.
여기에 데이터가 있다면 곧바로 리턴에 자원을 절약할 수 있고 그렇지 않다면 그 때 외부로 쿼리를 날려 데이터를 가져오고 해당 데이터를 영속성 컨텍스트에 저장하게 됩니다.


2. 1차 캐시는 어떤 구조이며, 어떤 종류의 조회에서 캐시 히트가 발생하는가?

1차 캐시는 영속성 컨텍스트를 의미합니다. 영속성 컨텍스트란 엔티티 매니저가 관리하는 저장소 입니다. 우리는 Spring Data JPA 를 사용하기 때문에 아직 엔티티 매니저를 직접 다룰 일이 없었지만
엔티티 매니저란 하나의 트랜잭션 안에서 일어나는 DB에 관련된 처리를 도맡아 하는 객체를 말합니다.
엔티티 매니저는 하나의 트랜잭션 안에서 생명 주기를 가집니다. 요청이 들어오면 엔티티 매니저 팩토리가 엔티티 매니저를 생성해 해당 요청을 처리하도록 배치합니다.
이 엔티티 매니저가 가진게 1차 캐시, 영속성 컨텍스트 입니다.

트랜잭션 안에서 어떤 데이터의 조회 요청이 실행되면 엔티티 매니저가 이를 처리하는데 곧바로 외부 DB에 쿼리를 날리지 않습니다.
엔티티 매니저는 먼저 자기가 가진 1차 캐시를 확인합니다. 여기에 원하는 데이터가 있다면 곧바로 반환하여 값 비싼 외부 DB 와의 통신을 생략할 수 있습니다.
만약 없다면 그때 외부 DB 에 쿼리를 날려 데이터를 받아오고 이 데이터를 1차 캐시에 저장해둔 다음 반환합니다.
이제 이 트랜잭션 안에서 또 같은 데이터가 필요하다면 외부 DB 요청 없이 곧바로 1차 캐시에 저장된 데이터를 반환할 수 있습니다.
하지만 모든 요청에 대해 캐시 작업이 일어나고 캐시 히트가 발생할 수는 없습니다.
엔티티 매니저는 Id 값을 Key 로 하는 Map 을 가집니다. 이게 1차 캐시 자료구조 입니다. 즉, Id 를 기반으로 하는 조회에서만 캐싱 기능을 사용할 수 있습니다.
나머지 요청은 그냥 캐싱과 상관없이 곧바로 외부 DB 에 쿼리를 보내 처리합니다.
왜 그럴까요? Id 기반이 아닌 다른 조회에서도 캐싱을 활용할 수 있으면 성능이 올라갈텐데요.
이유는 여러가지가 있지만 제일 결정적인 부분은 데이터의 무결성 때문입니다. 무결성, 최신 상태를 보장하기 어렵단 뜻 입니다.
예를 들어 나이가 20살인 회원을 조회했고 이를 저장해두는 캐싱을 구현했다고 가정하겠습니다.
해당 트랜잭션이 진행되는 동안 다른 곳에서도 작업이 일어납니다. 만약 다른 로직에서 회원 C 의 나이를 20살로 업데이트 했다면?
이는 아직 DB 에 반영이 안됬을 수 있습니다. 앞선 트랜잭션 안에서 또 다시 20살의 회원을 조회할 일이 있다면 이 데이터는 캐시에서 반환되어
새롭게 등장한 C 회원을 확인할 수 없게 됩니다. 단순히 Id 를 기반으로 단건의 데이터를 조회하는 것에 비해 넓은 범위의 데이터 조회 작업에서는 이 같은 무결성 문제가 생길 수 있습니다.
그 밖에 Map 이라는 자료 구조에서 오는 한계가 있습니다. Map 은 <ID, Entity>로 고정되어 있습니다. findByUsername 같은 요청이 들어왔다고 해서
이 Map 을 대상으로 조건 검사를 해 객체를 꺼내오는 것은 DB에서 인덱스를 타고 조회한느 것보다 훨씬 느리고 비효율적인 작업이 됩니다.
추가로, ID 기반이 아닌 경우 1차 캐시를 무시하고 곧바로 쿼리를 수행한다고 했는데 디테일하게 보면 과정이 하나 있습니다.
쿼리를 수행해 데이터를 가져온 후 엔티티의 ID 를 기반으로 1차 캐시를 검사합니다. 만약 캐시에 해당 데이터가 있었음이 확인되면 방금 쿼리로 가져온 데이터는 버리고
1차 캐시에 있던 기존 객체를 반환하게 됩니다. 
이유는 이미 1차 캐시로 불러와진 객체를 대상으로 새로운 데이터가 씌워졌을 수 있기 때문입니다. 아직 트랜잭션 진행 중이니까요. DB 에서 가져온 데이터는 구버전일 가능성이 있습니다. 
당연히 없다면 캐시에 저장하고 반환 합니다.

1차 캐시라는 이름에서 추측할 수 있듯 캐시는 2차 캐시도 존재합니다.
2차 캐시는 따로 라이브러리를 불러와 설정해주지 않으면 기본적으로 사용되지 않습니다. 보통은 1차 캐시만으로 작업이 진행됩니다.
2차 캐시는 1차 캐시가 하나의 트랜잭션에서 생명주기를 가지는 것과 다르게 애플리케이션 전체를 생명주기로 가집니다.
애플리케이션이 실행되고 2차 캐시가 생성되며 이는 프로그램 종료까지 유지 됩니다.
따라서 2차 캐시는 하나의 트랜잭션 단위에 구애받지 않습니다.
2차 캐시가 활성화 되면 1차 캐시 사용 시나리오에 한 단계가 추가됩니다.
앞서 1차 캐시에서 DB 조회 작업이 일어나면 1차 캐시를 확인하고 외부 DB에 쿼리를 날린다고 했습니다.
이 부분에 2차 캐시가 추가됩니다.
1차 캐시를 확인한 후 없으면 엔티티 매니저는 2차 캐시를 가지고 관리하는 엔티티 매니저 팩토리에게 해당 데이터의 확인을 요청합니다.
만약 2차 캐시에 데이터가 있다면 이를 복사본으로 받아 1차 캐시에 저장하고 반환합니다.
없다면 그때 외부 DB 에 쿼리를 날려 데이터를 받습니다. 받은 데이터는 2차 캐시에 저장되며 이 데이터의 복사본이 1차 캐시에 저장됩니다.
복사본이 오는 이유는 동시성 제어를 위해서 입니다. 원본을 그대로 저장하면 여러 곳에서 객체를 동시에 수정할 수 있기 때문입니다. 락을 걸기에는 비용이 너무 큽니다.
2차 캐시는 트랜잭션의 생명 주기와 무관하기 때문에 1차 캐시와 다르게 다른 두 개의 트랜잭션에서 똑같은 조회 요청이 들어왔을 때 2차 캐시에서 히트하게 되면
쿼리 없이 바로 작업이 진행될 수 있습니다.


3. `findAll()`은 매번 DB에 쿼리를 보내는가? 그 이유는?

findAll() 은 매번 DB에 쿼리를 보냅니다. 앞서 캐시에서 언급한대로 캐시는 Id 기반 Map 으로 동작하기 때문입니다.
추가로, findAll() 로 1차 캐시를 무시하고 DB 에서 데이터를 조회 해왔지만 반환 하기 전 1차 캐시를 확인합니다.
여기서 가져온 데이터가 1차 캐시에 있었다면 가져온 데이터를 버리고 1차 캐시 데이터로 채워넣습니다. 이유는 앞서 다룬대로 트랜잭션이 진행 중이기 때문입니다.


4. 엔티티의 상태 3가지(비영속, 영속, 준영속)를 설명하라.

3가지에 하나 더해 4가지, 비영속(New / Transient), 영속(Managed), 준영속(Detached), 삭제(Removed) 를 다루겠습니다.
비영속은 영속성 컨텍스트나 DB 와 아무런 관련 없는 순수 자바 객체 상태를 말합니다.
이제 막 생성 되어(new 키워드) 메모리에만 올라가 있는 순간을 의미합니다.

영속은 엔티티 매니저를 통해 영속성 컨텍스트에 저장되어 관리되는 상태를 말합니다.
`em.persist(member);` 가 실행되거나 DB에서 조회된 직후엔 영속성 컨텍스트에 저장되어 있기 때문에 이 때를 영속 상태라고 합니다.
1차 캐시에 저장되어 있으며, 더티 체킹 대상이고, 트랜잭션 커밋 시점에 DB 에 반영됩니다.

준영속은 영속성 컨텍스트에 저장되었다가 분리된 상태입니다.
`em.detach(member);`, `em.clear();`(캐시 전체 비우기), `em.close():`(매니저 종료) 수행 시 준영속 상태가 됩니다.
1차 캐시에서 제거된 상태이며 영속성 컨텍스트가 제공하는 기능 더티 체킹 등을 사용할 수 없습니다.
값 자체는 그대로 가지고 있지만, DB와의 연결고리가 끊어진 상태입니다.

삭제는 엔티티를 영속성 컨텍스트와 DB 에서 삭제하기로 한 상태입니다.
`em.remove(member);` 코드로 실제 DB 삭제는 트랜잭션 커밋 시점에 DELETE 쿼리가 나가면서 일어나지만 해당 메서드 호출 시점부터
해당 엔티티는 삭제 상태를 가지게 됩니다.

이 4가지 상태가 중요한 이유는 엔티티의 상태에 따라 JPA 의 동작이 완전히 달리지기 때문입니다.

|상태|JPA 동작|
|---|---|
|비영속|JPA가 모름. save() 필요|
|영속|Dirty Checking 작동. save() 불필요|
|준영속|Dirty Checking 안 됨. save() 필요 (merge)|
|삭제|커밋 시 DELETE SQL|

즉, `save()` 를 호출해야 하는가? 라는 판단은 이 4가지 엔티티 상태에 따라 결정됩니다. 


### B-5. 트랜잭션

1. 트랜잭션이란 무엇인가? 왜 필요한가?

트랜잭션이란 하나의 요청이 여러 작업으로 나뉠 때 모든 작업이 완료되어야 비로소 하나의 요청이 완료되는 것으로 취급하는 단위 입니다.
여러 작업 중 중간에 실패가 일어난다면 그 트랜잭션 안의 모든 작업은 실패로 취급되며 상태가 원래대로 돌아가야합니다.
예를 들어 은행 프로그램의 송금 과정을 들 수 있습니다.
A 사용자가 B 사용자에게 10만원을 이체한다. 는 작업은 문장으로는 얼핏 하나의 작업처럼 보입니다.
하지만 이를 분리해보면,
1. A 사용자의 잔액을 10만원 감소시킨다.
2. B 사용자의 잔액을 10만원 증가시킨다.
로 나눌 수 있습니다.
만약 1번 작업이 성공한 뒤 2번 작업이 모종의 사유로 인해 수행되지 못한다면 1번 작업 또한 취소되어야 합니다.
그렇지 않으면 A 의 돈만 증발하게 되니까요.
이처럼 개념적으로 반드시 다 같이 성공하거나 다 실패해야하는 작업 묶음의 단위가 트랜잭션 입니다.


2. 트랜잭션 내에서 일부 작업이 실패하면 어떻게 되는가? 이를 무엇이라 하는가?

앞서 언급한대로 일부 작업이 실패한다면 반드시 이전에 실행되었던 작업들도 모두 실패로 돌아가야합니다.
이를 롤백이라고 합니다.


3. `@Transactional`을 메서드에 붙이면 어떤 일이 일어나는가?

@Transactional 어노테이션이 붙으면 해당 메서드 내에서 일어나는 모든 작업을 트랜잭션 묶음으로 취급합니다.
모든 작업은 다 같이 성공하거나 다 같이 실패해야합니다.

@Transactional 어노테이션을 붙이면 Spring 이 내부적으로 프록시 객체를 만들어 메서드 호출을 감쌉니다.
실제 동작은 다음과 같습니다.

1)메서드 시작 전 -> 트랜잭션 시작 (BEGIN)
2)메서드 실행 (비즈니스 로직)
3-1)정상 완료시 -> COMMIT -> 이 시점에 Dirty Checking, INSERT/UPDATE SQL 실행
3-2)예외 발생 -> ROLLBACK -> 모든 변경을 취소

즉, 어노테이션이 붙으면 메서드의 시작과 끝에 트랜잭션 관리 코드를 자동으로 끼워 넣는 것 입니다.
덕분에 Dirty Checking, 예외 발생 시 자동 롤백, readOnly = true 등의 옵션이 구현 가능해집니다.


### B-6. 연관관계

1. 외래키(FK)란 무엇이며, 왜 필요한가?

외래키는 다른 테이블과 자신 테이블의 연관 관계가 있음을 의미합니다.
RDBMS 는 각 엔티티를 테이블로 만들어 저장하고 둘 간의 관계를 정의해야하는데 이 때 FK 를 사용합니다.
예를 들어 Post 와 Comment 의 관계는 댓글이 게시글에 속하는 형태기 때문에 Comment 테이블에 Post 테이블의 PK 값을 FK 로 가지게 됩니다.
이제 두 테이블에 연결고리가 생겼으니 JOIN 연산을 통해 어떤 게시글에 어떤 댓글이 있는지 조회할 수 있게 됩니다.


2. 관계의 4가지 종류를 나열하고, 게시판 프로젝트에서 사용한 관계를 예시와 함께 설명하라.

일대일 관계, 일대다 관계, 다대일 관계, 다대다 관계가 있습니다.
어노테이션으로는 `@OneToOne`, `@OneToMany`, `@ManyToOne`, `@ManyToMany` 으로 표현됩니다.
게시판에서의 사용 예시로는 Post - Comment 관계를 살펴볼 수 있겠습니다.
게시글 과 댓글은 서로 연관된 엔티티 입니다. 댓글이 게시글에 속하는 구조 입니다.
게시글은 여러 개의 댓글이 달릴 수 있습니다. 댓글들은 각자 하나의 게시글에 속할 겁니다.
따라서 게시글 입장에서 게시글과 댓글의 관계는 @OneToMany 입니다. 모든 게시글들은 자기 자신은 하나지만 댓글 엔티티가 여러개 속할 수 있단 뜻 입니다.
댓글 입장에서는 @ManyToOne 관계가 됩니다. 각 댓글들은 모두 하나의 게시글을 가져야 합니다.

@ManyToOne 에서 fetch = FetchType.LAZY 를 명시하는 이유
fetch 전략인 기본적으로 EAGER 입니다. 이 옵션은 엔티티를 로드 했을 때 연관 있는 엔티티를 모두 불러오는 옵션입니다.
따라서 불필요한 조회가 발생하는걸 방지하기 위해 LAZY 옵션을 선택할 수 있습니다.

@JoinColumn(name = "post_id") 이 실제 DB에 어떤 컬럼을 만드는지
@JoinColumn 은 외래 키를 매핑할 때 사용하는 어노테이션 입니다. 객체 관계에서 어떤 컬럼을 통해 상대방 테이블과 연결할 것인지를 지정하는 역할입니다.
즉, @JoinColumn(name = "post_id") 는 이 엔티티가 관리하는 테이블에 post_id 라는 컬럼을 만들고 그걸 외래 키로 사용하자는 의미입니다.
우리 프로젝트에서는 Comment 클래스에 선언되어 게시글 id 를 외래 키로 참조하도록 쓰였습니다.

우리 프로젝트에서 Post-Member 관계는 단방향으로만 설정한 이유

Post-Member 관계는 단방향 관계로 설정되었습니다. 양방향이 아니게끔 판단한 기준은
"비주인 쪽에서 주인 쪽 목록을 자주 조회할 필요가 있는가?" 입니다.
Post, Comment 의 관계는 게시글 상세 페이지에서 이 게시글의 댓글 목록을 자주 보여줘야 할 필요가 있기 때문에 양방향입니다.
Post, Member 의 관계는 게시글에서 작성자를 보여주는 건 보통 필요하지만 반대로 멤버의 입장에서 이 회원의 모든 게시글을 조회할 일은 거의 없습ㄴ디ㅏ.
자바 코드로 치면 Member 클래스를 통해 모든 게시글(`List<Post>`)을 조회할 필요가 있는가? 인데 이런 작업은 보통 
`PostRepository` 에서 Member 를 조건으로 검색하는게 효율적입니다.
따라서 Post, Member 관계에서 양방향 설정은 불필요한 관리 복잡도만 높아지기 때문에 단방향으로 설정됩니다.


3. 객체의 참조와 DB의 외래키는 방향성에서 어떤 차이가 있는가?

객체는 엔티티가 서로를 알려면 서로를 필드로 참조 해야하지만 RDBMS 에서는 어느 한 쪽 테이블에 외래키만 등록되면 양방향이 서로 참조가 가능해집니다.(JOIN)
이 패러다임의 불일치 때문에 JPA 에서는 관계를 어노테이션으로 설정해주고 주인이 누구인지 가르쳐줘야합니다.


4. 양방향 관계에서 "관계의 주인"은 누구이며, 왜 주인을 정해야 하는가?

양방향 관계 문제는 DB 는 외래키가 하나 뿐인데 자바 객체는 양쪽 모두 서로를 참조해야해서 발생하는 패러다임 불일치 문제입니다.
`comment.setPost(post);`
`post.getComments().add(comment);`
자바에서 구현되는 게시글, 댓글 엔티티는 이렇게 서로를 참조하도록 작성 됩니다.
이제 프로그램이 실행되면 둘 중 어느 하나의 메서드가 호출될텐데 이 때 JPA 는 어느 쪽을 DB 에 반영해야하는 알 수 없습니다.
그래서 한쪽만 DB에 반영되도록 주인을 정해줍니다.
외래키가 있는 쪽이 관계의 주인이되며 이제 변화 감지 대상은 주인으로 한정됩니다.
Post 엔티티는 읽기 전용이 되어 여기서 변화가 일어나도 DB에 반영이 되지 않습니다.
`comment.setPost(post);`가 호출되어야 DB에 반영이 됩니다.

(근데 그러면 양방향 관계 설정이 필요한 이유가 뭔가요? Post-Member 처럼 한 쪽에만 관계를 설정해주면 해결되는 문제 아닌가요?)
양방향 관계로 설정된 이유는 객체 그래프 탐색의 편의성 때문입니다.
양방향으로 설정되어 있을 경우 postRepository 에서 post 객체를 꺼내면 덩달아 post.getComments() 로 바로 접근이 가능해집니다.
만약 단방향이었다면 postRepository 에서 post 객체를 꺼내고 댓글은 따로 commentRepository 에서 꺼내주는 비효율이 발생합니다.
게시글과 댓글은 서로 생명주기도 비슷하며 쓰임새에서 같이 움직이는 경우가 많기 때문에 양방향으로 설정해주는게 좋습니다.

반면 Post-Member 관계는 member 조회를 통해 게시글을 꺼낼일이 많지 않다는 점이 다릅니다.
post의 입장에서는 게시글을 조회하면 member 정보도 같이 나와야하기 때문에 한방향 연결만 설정하게 됩니다.


5. `mappedBy`와 `@JoinColumn`은 각각 어느 쪽에 사용하며, 어떤 의미인가?

@JoinColumn 은 주인쪽에, mappedBy 는 비주인 쪽에 선언됩니다.
주인 쪽인 Comment 에는 JoinColumn, FK 가 post_id 임을 표시하는걸로 사용되며
비주인 쪽인 Post 는 자신이 읽기 전용이며 주인 필드가 post 임을 표시합니다.


6. 즉시 로딩(EAGER)과 지연 로딩(LAZY)의 차이는? 실무에서 권장되는 전략은?

즉시 로딩은 한 엔티티를 로딩 했을 때 연관 있는 엔티티를 전부 불러오는 전략입니다.
게시글을 로딩하면 참조하고 있는 댓글 리스트도 다 불러와지는게 즉시 로딩입니다.
지연 로딩은 즉시 로딩과 반대로 해당 옵션의 대상이 호출될 때 비로소 로딩해오는 전략입니다.
게시글을 로딩했지만 즉각 댓글을 불러오지 않습니다. '이 게시글에 달린 댓글을 보여줘' 같은 요청이 들어온 경우에만
그제서야 댓글을 로딩해옵니다.

실무에서는 지연 로딩을 주로 권장합니다.
N+1 문제를 방지 할 수 있단 점이 가장 큰 이유입니다.
만약 EAGER 전략으로 게시글을 10개 조회한다고 하면,
1)게시글이 10개 조회(SELECT 1회 수행)
2)각 게시글의 댓글을 즉시 로딩(게시글 마다 SELECT 10번 수행)
총 11번으로 10개의 게시글 + 1, N+1 문제가 발생하게 됩니다. 게시글 10개를 조회하는데 반드시 모든 댓글이 로딩되어야 할 필요는 없습니다.

필요할때 각 댓글을 불러올 수 있도록 LAZY 전략을 사용해 성능을 개선시킬 수 있습니다.


7. `cascade = CascadeType.ALL`은 어떤 동작을 하는가?

해당 옵션은 부모 엔티티의 작업이 자식에게 연쇄 적용되는 것입니다.
CascadeType.ALL = PERSIST + MERGE + REMOVE + REFRESH + DETACH
게시글을 저장하면 댓글도 함께 저장되며 게시글을 삭제하면 댓글도 함께 삭제 됩니다.


8. `orphanRemoval = true`는 cascade와 어떻게 다른가?

orphanRemoval 은 댓글 목록에서 제거된 댓글은 DB 에서도 지워지도록 하는 옵션입니다.

만약 Post 엔티티에 cascade 옵션 REMOVE 만 존재한다면 게시글이 삭제되면 댓글도 함께 삭제되지만,
post.getComments().remove(comment); 의 코드의 경우 자바 리스트에서만 comment가 삭제되고 DB 에는 반영이 되지 않습니다.

orphanRemoval = true 옵션으이 경우
cascade REMOVE 와 동일하게 게시글 삭제시 댓글이 삭제되며,
post.getComments().remove(comment); 코드도 부모와의 관계가 끊어진 고아 객체로 판단하고 자동으로 DELETE 해 줍니다.

## C. 계층 구조와 예외 처리 (Phase 3)

### C-1. 3계층 구조

1. Controller, Service, Repository 각 계층의 역할을 한 줄로 설명하라.

Controller 는 클라이언트의 요청을 받아 알맞은 Service 메서드를 호출해 결과를 받고 이를 돌려주는 역할을 합니다.
Service 는 Controller 에서 넘어온 요청을 애플리케이션 도메인 로직에 맞게 실행시키고 결과를 받아 반환하는 역할을 합니다.
Repository 는 Service 로부터 엔티티 조회 요청이 들어오면 DB 와 통신해 알맞은 정보를 반환하는 역할을 합니다.


2. 모든 로직을 Controller 하나에 넣으면 어떤 문제가 생기는가? (4가지)

Controller 클래스가 비대해져 유지보수가 어려워집니다.
코드가 길어지기 때문에 가독성이 나빠지고 어떤 코드가 무슨 일을 하는지 쉽게 파악하기가 어렵습니다.
여러 성격의 코드가 한 군데에 모여있기 때문에 테스트를 하기가 어려워집니다.
어떤 작업에 대한 로직이 한 메서드에 모여있기 때문에 코드의 중복이 발생합니다. 여기서 이뤄진 작업이 저기서도 비슷하게 이뤄져 똑같이 작성되는
코드 중복이 늘어납니다. 이것을 분리해 재사용성을 높여야합니다.
여러 성격의 코드가 모이기 때문에 각 성격의 요구사항 변경이 다른 코드에 영향을 미칩니다.
분리되어 있는 상태라면 해당 클래스만 변경이 가능하지만 컨트롤러 하나에 모든 코드가 모여있으면 변경의 여파가 커집니다.


### C-2. DTO

1. Response DTO를 사용하는 이유 3가지는?

불필요한 값을 외부로 노출시키지 않기 위해서 입니다.
어떤 요청에 의해 객체가 만들어졌는데 이걸 통째로 그냥 반환한다면 내부에 클라이언트가 구지 알 필요 없는 정보가 섞여 들어가게 되어 보안이 취약해집니다.

Presentation 계층(UI)과 Service(/Entity, 도메인 계층)의 결합도를 낮추기 위해서 입니다.
UI 계층이 서비스 계층이 넘겨준 요청 응답 객체를 그대로 사용해 클라이언트에게 정보를 전달한다면 추후 백엔드에서 어떤 이유로
반환하는 객체가 변경되면 이를 넘겨받아 해석하는 계층에도 영향이 갑니다. 유지보수를 위해 필요한 만큼의 정보만 응답 객체 DTO를 사용해 전달해야 합니다.

순환 참조를 방지하기 위해서입니다. 애플리케이션이 응답을 하면 Spring이 전송을 위해 JSON 객체로 직렬화 하는 과정이 필요합니다.
이 때 엔티티가 그대로 반환되게 되면 이를 직렬화 하면서 순환 참조 방지가 일어날 수 있습니다.
Post-Comment 엔티티 구조를 예시로 살펴보겠습니다.
Post 엔티티에 관한 처리를 요청 받아 처리하고 해당 Post 를 그대로 반환한다고 해보겠습니다.
Spring 은 Post 객체를 직렬화 하려고 합니다. 직렬화를 위해 Post 엔티티의 필드들을 살펴보는데 List<Comment> 필드가 나타납니다.
직렬화를 위해 해당 객체에 담긴 내용을 살핍니다. Comment 객체들 안에 방금 참조한 Post 객체가 등장합니다.
Spring 은 다시 이 Post 객체를 직렬화하기 위해 담긴 내용을 살핍니다. 마찬가지로 List<Comment> 필드가 나타납니다.
...
이를 무한 반복하는게 순환 참조 현상입니다.
이를 방지하기 위해 요청에 대한 객체를 그대로 응답에 싣지 않고 응답에 필요한 내용만 DTO 로 만들어 반환해 문제를 원천 차단합니다.


2. Request DTO를 사용하는 이유 3가지는? (Entity로 직접 요청을 받으면 안 되는 이유)

클라이언트가 조작해서는 안 되는 필드를 받을 수 있습니다.
Entity 로 직접 요청을 받게 되면 Jackson 라이브러리가 JSON 의 모든 키를 Entity 필드에 매핑하려고 합니다.
클라이언트가 id, createdAt, role 등 조작해서는 안되는 값을 실어 보내면 값이 그대로 반영될 위험이 있습니다.
따라서 요청 DTO 를 사용해 필요한 값만 담도록 해 문제를 예방합니다.

검증 규칙의 분리 때문입니다. 상황별로 다른 검증이 필요할 수 있습니다.
게시글 생성 요청은 제목과 내용이 모두 필수지만, 게시글 수정 요청에는 제목과 내용 중 일부분만 바꾸는 경우도 있을 수 있습니다.
하나의 Entity 로 이 두 가지 시나리오를 표현할 수가 없습니다. 따라서 각 요청 마다 필요한 값을 정의해 DTO 로 처리해야 상황별 검증이 가능해집니다.

결합도 때문입니다. 엔티티가 변경되면 API 에 직접 영향이 생깁니다.
앞서 응답 DTO 를 사용해야했던 이유 처럼, 반대도 성립합니다.
만약 클라이언트가 어떤 엔티티를 통해 요청을 하고 이를 받아 처리하는 구조라면, 나중에 요청 처리 구조가 바뀌면(Entity 에 필드를 추가하거나 이름을 바꾸거나)
클라이언트가 보낼 JSON 형식 역시 따라 바뀌어야 합니다. DTO 가 중간에 있다면 Entity의 변경이 일어나도 DTO 를 수정하면 되기 때문에 파급 효과가 밖으로 세어나가지 않습니다.


3. DTO의 검증과 Entity의 검증은 어떻게 다른가?

DTO 는 사용자의 입력값을 검증합니다.
이는 클라이언트의 입력이 형식적으로 올바른지에 대해 검사합니다. 예를 들어 null 인지?, 빈 문자열인지?, 사이즈 제약에 맞는지 안 맞는지? 등을 검사합니다.

Entity 의 검증은 비즈니스 규칙에 대한 검사입니다.
예를 들어 게시글 상태가 DRAFT(초안) 상태라면 이 데이터는 외부에서 조회될 상태가 아닙니다. 만약 조회 요청이 잘못해서 들어오면 이에 대한 예외 처리를 해야합니다.
이런 부분들이 Entity 의 검증입니다. 아직 현재 프로젝트는 복잡한 처리가 없기 때문에 Entity 규칙이 드러나지 않았지만 서비스가 복잡해지면 Entity 내부에서
규칙들을 정해 검증을 하게 됩니다.


4. `PostResponse.from(post)` 같은 정적 팩토리 메서드의 역할은?

요청에 대한 결과로 만들어진 객체를 파라미터로 받아 클라이언트에게 전송해야할 정보만 따로 담아 적절한 타입을 설정하고 반환할 수 있도록
DTO 객체를 만드는 정적 팩토리 메서드 입니다.
정적 팩토리 메서드의 핵심 장점은 변환 로직이 DTO 안에 캡슐화된다는 점입니다.
Service 가 new PostResponse(post.getId(), post.getTitle(), ...) 이렇게 일일이 꺼내지 않고, PostResponse.from(post) 한 줄로 호출하며,
변환 방법이 바뀌어도 Service 에 변경 여파가 끼치지 않고 팩토리 메서드만 수정해주면 됩니다.


### C-3. 예외 처리

1. 커스텀 예외는 왜 만드는가? `RuntimeException`을 상속하는 이유는?

자바에서 제공하는 기초 예외보다 더 도메인에서 발생한 예외에 대해 상세하게 설명할 수 있도록 커스텀 예외를 작성합니다.
`RuntimeException` 을 상속하는 이유는 다음과 같습니다.

먼저 try-catch 문을 강제하지 않아도 됩니다. Java 의 예외는 두 가지로 나뉘며 다른 하나는 `CheckedException` 입니다.
이 예외는 컴파일러가 해당 예외를 처리하는 코드를 작성하도록 강제합니다. try-catch 문과 throws 문 중 하나를 반드시 작성해야합니다.
만약 `CheckException` 을 상속하면 Service -> Controller -> Handler 까지 가는 모든 경로에 throws 가 붙어야합니다.

다음으로 `@Transcational` 어노테이션의 롤백 때문입니다. 해당 어노테이션은 RuntimeException 만 자동으로 롤백하며 CheckedException 은
기본적으로 롤백하지 않습니다. 따라서 RuntimeException 을 상속 받아야합니다.


2. `@RestControllerAdvice`는 어떤 범위의 예외를 잡을 수 있는가?

`@RestControllerAdvice` 가 붙은 클래스는 스프링 시큐리티를 지나 Spring MVC 계층에서부터 예외를 잡을 수 있습니다.

SpringFramework 는 여러 모듈의 집합으로 만들어진 프레임워크입니다.
핵심 모듈은 다음과 같습니다.
Spring Core: IoC/DI 컨테이너, 빈관리, @Component, @Autowired
Spring MVC: 웹 요청 처리, @Controller, DispatcherServlet, @ExceptionHandler
Spring AOP: 관점 지향 프로그래밍, @Transactional
Spring Data JPA: DB 접근, Repository, JPA 통합

또 SpringFramework 위에 얹는 모듈로 별도 프로젝트가 있습니다.
Spring Boot: 자동 설정, 내장 톰캣, 빠른 시작
Spring Security: 인증, 인가, Filter Chain 기반, Spring MVC 와 별도 계층이 됨
Spring Batch: 대량 데이터 배치 처리
Spring Cloud: 마이크로서비스, 분산 시스템

이제 요청이 처리되는 과정에서 모듈 간의 경계가 어떻게 되는지 알 수 있습니다.
HTTP 요청 -> 톰캣 -> SpringSecurity Filter Chain(별도 모듈) -> Spring MVC (DispatcherServlet -> Controller 흐름)(별도 모듈)
따라서 @RestControllerAdvice 는 Spring MVC 모듈에 존재하는 어노테이션이기 때문에 다른 단계에서의 예외를 잡지 못합니다.


3. 예외 처리에서 "관심사의 분리"란 무엇인가? Service와 Handler의 역할을 구분하라.

관심사의 분리란 처리하고자 하는 작업의 성격에 따라 코드를 분리하는 원칙입니다.
기존의 콘솔 프로그램을 작성할때는 Service 에서 호출하는 Domain 에서 예외를 던지고 Service 에서, 혹은 Controller 에서 이를 잡아
처리를 하는 식으로 프로그램을 작성했습니다.
하지만 실제 웹 애플리케이션에서 Service 는 '무엇이 잘못되었는지' 판단하고 예외를 던지기만 합니다. 예를 들어 '게시글을 찾을 수 없다'면
Service 는 적절한 예외는 ResourceNotFoundException 을 던지기만 하고 끝내야 합니다.
던져진 예외는 Handler 가 잡아서 해당 예외를 어떻게 응답할지 결정해 응답합니다.
이렇게 예외를 던지기만 하고 처리하는 책임이 분리되면
Service 는 HTTP 상태 코드나 응답 형식을 알 필요가 없고,
Handler 는 비즈니스 로직을 알 필요가 없습니다.
각자의 자기 관심사만 담당해 처리하는 이것이 관심사의 분리입니다.
만약 분리하지 않는다면 Service 에 response.setStatus(404) 같은 HTTP 코드가 섞여 들어가 같은 예외를 다른 곳(예를 들어 메시지 큐)에서
재사용할 때 문제가 발생할 수 있습니다.

메시지 큐 시나리오의 핵심은 요청이 지금처럼 HTTP 요청으로 들어오지 않을 수도 있다는 점입니다.
Kafka 메시지 -> Consumer -> PostService.create() -> 응답은 Kafka 메시지
이 경우 관심사의 분리 없이 Service 안에 response.setStatus(404) 같은 코드가 있을 경우 해당 시나리오에서는 response 객체 자체가 없어서
에러가 발생합니다. 그렇게 때문에 분리해서 Service의 재사용성을 늘려야 합니다.



### C-4. 입력값 검증

1. `@NotNull`, `@NotBlank`, `@NotEmpty`의 차이는?

`@NotNull`: 해당 값은 비어있을 수 있지만(""," ") null 이면 안됩니다.
`@NotBlank`: 해당 값은 null 과 "", " " 모두 허용하지 않습니다.
`@NotEmpty`: 해당 값은 비어있을 수 없고 null 이면 안됩니다. " " 는 허용됩니다.


2. `@Valid`를 붙이지 않으면 검증 어노테이션이 동작하는가?

동작하지 않습니다. @Valid 어노테이션이 컨트롤러의 파라미터 앞에 붙으면 해당 DTO 를 Spring 이 생성할 때 조건을 검사하게 됩니다.


3. 검증 실패 시 어떤 예외가 발생하며, 이를 어디서 처리하는가?

검증이 실패한다면 MethodArgumentNotValidException 이 발생합니다. 이 예외는 Spring MVC 내부에서 발생하므로
GlobalExcepitonHandler 가 잡아서 처리합니다.
@Valid 검증은 이미 Spring Security(Filter 앞단) 를 통과해 요청이 들어오고 요청의 JSON 을 역직렬화 하는 과정에서 발생하기 때문에 Spring MVC 내부입니다.
따라서 Handler 가 이 예외를 잡을 수 있습니다.




### C-5. JPA 콜백

1. `@PrePersist`와 `@PreUpdate`는 각각 언제 호출되는가?

`@PrePersist` 는 엔티티가 처음 생성되었을 때 호출될 메서드들에 붙는 어노테이션이며
`@PreUpdate` 는 엔티티가 수정될 때 자동으로 생성될 메서드들에 붙는 어노테이션 입니다.

좀 더 정확히는 `@PrePersist` 어노테이션은 INSERT 직전, 새 엔티티가 DB에 처음 저장되기 직전에 발생합니다.
이를 이용해 createdAt, updatedAt 등의 필드를 초기화 합니다.

`@PreUpdate` 는 UPDATE 직전 입니다. 이미 저장된 엔티티가 변경이 감지되어 DB 에 반영되기 직전에 발생합니다.
updatedAt 필드의 갱신 등에 사용됩니다.

콜백 메서드란 개발자가 직접 호출하지 않고, 특정 이벤트가 발생하면 프레임워크가 대신 호출해주는 메서드를 말합니다.
`@Pre..` 어노테이션이 콜백 메서드들입니다.


2. `persist()`와 `@PrePersist`의 관계는?

`persist()` 는 엔티티 매니저가 엔티티를 1차 캐시, 영속성 컨텍스트에 등록하는 메서드 입니다.
이 작업이 일어났을 때 `@PrePersist` 가 붙은 메서드가 자동으로 호출됩니다.
이후 트랜잭션이 모두 끝나 최종적으로 엔티티를 데이터베이스에 저장하게 됩니다.


3. 콜백 메서드를 Service에서 직접 호출하면 안 되는 이유는?

콜백 메서드를 Service 에서 직접 호출하게 되면 2번 실행되는 문제점이 있습니다. JPA 가 persist/update 시점에 자동으로 한 번 호출하는데,
Service 에서도 수동으로 한 번 호출하면 createdAt 이 두 번 세팅되는 등의 문제가 생깁니다.
`post.onCreate();` 이렇게 Service 에서 직접 호출하면 1번 실행되고,
`postRepository.save(post);` 가 일어나면 JPA 가 @PrePersist 콜백 자동 호출로 2번 실행됩니다.

따라서 JPA 를 사용할 경우 개발자는 수동으로 직접 호출 할 생각을 할 필요가 없습니다. JPA 에게 위임한 자동화 로직이기 때문에
save 메서드 호출로 JPA 가 알아서 메서드를 호출하다록 해야합니다.

## D. 인증과 인가 (Phase 4)

### D-1. 인증 vs 인가

1. 인증(Authentication)과 인가(Authorization)의 차이를 설명하라.

인증과 인가는 같은 보안에 관한 단어지만 그 의미에 약간 차이가 있습니다.
인증은 클라이언트가 어떤 사람인지를 구분할 수 있게 해주는 장치입니다.
인증을 받게 되면 클라이언트는 이제 서버에 자기가 그 클라이언트 임을 입증할 수 있게 됩니다.

인가는 인증 이후에 그 클라이언트가 어디까지 권한을 가지고 있는지 구분하는 장치입니다.
클라이언트들은 각기 다른 인가를 가질 수 있습니다. 어떤 클라이언트는 자신이 작성한 글만 삭제할 수 있고 어떤 클라이언트는
자신이 작성한 글이 아니어도 삭제 가능한 권한이 있도록 구분하기 위해 알맞은 인가를 만들어 구분을 할 수 있게 합니다.


2. 인증은 통과했지만 인가에 실패하는 상황의 예시를 들어라.

Service 레벨 인가 예시
일반 사용자 클라이언트가 인증을 통해 자신의 회원임을 인증했지만,
해당 사용자가 자신이 작성하지 않은 글에 삭제 요청을 보낼 경우 거부할 수 있습니다.
이것이 인가 에 실패한 상황입니다.

Role 기반 인가 예시
일반 사용자가 관리자 전용 API(/admin/**) 에 접근하는 경우 인가에 실패합니다.


### D-2. 세션 vs 토큰

1. HTTP의 무상태(Stateless) 특성이란 무엇이며, 인증에 어떤 문제를 야기하는가?

HTTP 의 무상태란 서버가 클라이언트의 정보를 기억하지 않는 상태를 이야기합니다.
따라서 같은 사용자가 요청을 보내고 있는건지 서버 입장에서는 알 수 가 없습니다.
HTTP 요청마다 독립적인 요청이 되기 때문에 사용자 입장에서는 매 요청 마다 인증도 같이 진행해야하는 불편함이 있습니다.
이 문제를 해결하기 위해 토큰 방식 또는 세션 방식을 도입할 수 있습니다.


2. 세션 방식의 동작 원리, 장점, 단점을 설명하라.

세션은 무상태 방식이 아닌 상태 유지 방식입니다.
클라이언트가 인증 요청을 보내면 서버는 세션이란 객체를 생성합니다. 그리고 이 세션에 대응되는 id 를 반환하게 됩니다.
이제 id 를 받은 클라이언트가 다음 요청에 id 를 함께 보내면 서버는 해당 id 를 통해 세션 상태를 확인하고 요청에 대한 인증, 인가를 처리할 수 있게 됩니다.

단점으로는 서버가 상태를 기억해야한다는 점 입니다.
단순하게 클라이언트가 늘어나면 늘어날 수록 서버가 기억해야할 세션에 대한 메모리 부담이 커집니다.
또 상태를 가지게 되면 서버 확장에 불리해집니다. 서버가 늘어나게 되면 요청을 분산해서 처리해야하는데
첫번째 요청을 A 서버에서 처리해 세션을 보유하고 있었는데 다음 두번째 요청이 B 서버로 가게 되면 세션 조회가 이루어지지 않기 때문에
세션을 공유하는 기능이 추가되어야 합니다. 이처럼 서버 증설에 어려움이 있습니다.

장점으로는 서버가 통제권을 가진다는 점 입니다.
세션을 기억하는 주체가 서버이기 때문에 서버는 필요하면 해당 세션을 삭제해서 클라이언트의 접근을 곧바로 제어 가능 합니다.
만약 클라이언트의 요청이 공격으로 탈취된 점이 확인되면 서버가 곧바로 대응할 수 있게 됩니다.
또 데이터 변경이 즉각 반영될 수 있습니다. 세션에 저장된 정보를 서버에서 바로 수정 가능하기 때문입니다.
예를 들어 권한 변경 등에 사용될 수 있습니다.


3. 토큰 방식의 동작 원리, 장점, 단점을 설명하라.

토큰은 무상태 방식입니다.
클라이언트가 인증 요청을 보내면 서버는 토큰을 발급합니다. 토큰은 문자열 객체 입니다.
이후 클라이언트가 다음 요청의 헤더에 토큰을 함께 보내면 서버는 요청에서 이를 분리해내 토큰을 얻고
토큰이 유효한지 검증합니다. 유효하다면 해당 요청을 통과시켜 Controller 에 도달하도록 하며
유효하지 않다면 예외를 발생시켜 처리합니다.

토큰 방식의 단점은 인증의 주도권이 클라이언트에게 있다는 점입니다.
요청에 토큰이 있기만 하면 서버는 인증해주기 때문에 토큰이 탈취된 토큰인지 알 수 없으며
세션은 서버 측에 자료가 있기 때문에 필요할 경우 세션을 지우는 등 강제력을 행사할 수 있지만
토큰은 서버 측에서 해당 인증에 대한 강제력을 구사할 수 없습니다.

토큰 방식의 장점은 서버가 상태를 저장하지 않는다는 점 입니다.
서버는 메모리에 부담 없이 인증 인가 시스템을 구성할 수 있습니다.
또 스케일 아웃에 유리해집니다. 어떤 서버로 요청이 가도 토큰만 검증하면 되기 때문에 세션에서 언급한 것 처럼 세션 공유가 필요 없게 됩니다.
마지막으로 다른 서비스, 도메인에서도 그대로 사용이 가능합니다. 모바일 앱, 마이크로서비스 간 인증 등에 만들어둔 토큰이 그대로 활용될 수 있습니다.


4. 서버 확장(스케일 아웃) 관점에서 세션과 토큰 중 어느 것이 유리한가? 그 이유는?

스케일 아웃은 동일한 서버의 대 수를 늘려 성능을 업그레이드 하는 방법을 말합니다.
이 관점에서는 세션보다 토큰이 유리합니다.
만약 세션 방식을 채택하면 서버 여러대에 걸쳐 세션 정보에 대한 공유 기능이 필요해지기 때문입니다.
토큰 방식은 무상태 서버의 특징을 가지기 때문에 스케일 아웃과 관계가 없습니다.


### D-3. JWT

1. JWT의 3가지 구성 요소와 각각의 역할은?

헤더: 헤더에는 토큰에 관한 메타 정보가 담깁니다. 서명에 사용할 알고리즘이 뭔지, 토큰의 타입은 무엇인지 등이 기록됩니다.
페이로드: 실제 데이터가 담기는 부분입니다. Claims 라고도 합니다. 사용자 식별자, 만료 시간, 인가 등에 관한 정보가 기록됩니다.
주의할 점은 토큰은 암호화 데이터가 아니란 점입니다. 인코딩된 자료기 때문에 페이로드에 민감한 정보를 담을 수 없습니다.
시그니처: 토큰이 위변조 되지 않았음을 표시하는 서명이 들어가는 부분입니다. 서버만이 아는 비밀키로 서명하면 해당 자료가 서버가 발급한 토큰이 맞음을
검사할 수 있게 됩니다.


2. JWT는 암호화인가, 인코딩인가? 이것이 의미하는 보안상 주의점은?

JWT 는 인코딩 입니다. 인코딩은 특정 알고리즘에 의해 A 값을 B 값으로 바꾸는 것이지 암호화처럼 키 같은 보안 장치가 없습니다.
따라서 민감한 정보는 토큰에 담기면 안됩니다.


3. Claims란 무엇인가?

JWT 의 페이로드를 의미합니다. 실질적인 토큰 내용이 담기는 부분으로 크게 3종류, 사용자 식별자, 만료 시간, 인가 등의 정보를 담을 수 있습니다.
3종류와 별개로 더 많은 정보를 자유롭게 추가 가능합니다.
Registerd Claims, JWT 표준에서 정의한 예약된 키로 sub(주체), exp(만료), iat(발급시간), iss(발급자) 등의 정보를 말합니다.
Public Claims, 충돌 방지를 위해 등록된 공개 키로 IANA 에 등록된 이름들을 말합니다.
Private Claims, 서버와 클라이언트가 합의한 커스텀 키로 username, role 등 우리가 직접 넣는 정보들을 말합니다. 


### D-4. 비밀번호 암호화

1. 비밀번호를 DB에 평문으로 저장하면 안 되는 이유는?

만약 DB 가 탈취당했을 때 모든 비밀번호가 노출이 될텐데 다른 서비스에서 똑같은 비밀번호를 쓰는 클라이언트가 있다면 피해가 확산되기 때문입니다.
또 내부자의 위협으로부터도 시스템을 보호할 수 있습니다. DB 접근 권한이 있는 자가 공격자가 되면 DB 에 비밀번호가 평문 그대로 저장되면 탈취가 쉬워집니다.


2. BCrypt의 "단방향 해시"란 무엇인가? 왜 복호화가 불가능한가?

단방향 해시란 수학적으로 해시 값을 보고 원문을 알아낼 수 없도록 설계된 해시 함수 입니다.
복호화가 불가능한 이유는 해시 함수가 정보를 압축하고 섞는 과정이기 때문입니다.
예를 들어 'password123' 라는 비밀번호가 있다고 하면, 이 10글자를 해싱해서 고정된 길이의 값이 나오는데
이는 원본의 정보 일부가 사라진다는 뜻 입니다. 수학적으로 여러 다른 입력이 같은 해시 값을 만들 수 있기 때문에 이 특징으로 인해 해시값만 보고
원래 어떤 값이었는지 역추적(복호화) 가 불가능해집니다.

그래서 로그인 시 비밀번호 확인은 다음과 같이 이루어집니다.
password123 입력 -> 같은 해시 함수 적용 -> 결과를 DB의 해시값과 비교
즉, 원문을 대조해서 인증하는게 아니라 같은 과정을 거친 결과가 같은지 비교하는게 비밀번호 확인입니다.


3. "솔트(Salt)"란 무엇이며, 왜 필요한가?

솔트란 비밀번호를 해싱할 때 서버 측에서 첨가하는 값입니다.
매번 새로운 랜덤값을 선택해서 클라이언트가 입력한 비밀번호화 함께 해싱 됩니다.
이러한 기법이 필요한 이유는 공격자의 레인보우 테이블의 존재 때문입니다.
우리가 사용하는 알고리즘은 기법 그 자체가 이미 모두 공개되어 있습니다. 보안은 알고리즘 자체를 숨기는게 아닙니다.
레인보우 테이블은 미리 비밀번호가 될만한 평문을 아주 많이 수집하고 이를 해싱하면 어떤 값이 나오는지 모두 계산되어 있는 테이블입니다.
만약 솔트 없이 DB 에 해싱해 저장되어 있다면 공격자는 이 테이블을 이용해 비밀번호를 알 수 있게 됩니다.
따라서 서버에서 임의로 솔트 값을 첨가해 이 테이블을 무력화 시킬 수 있습니다.
솔트 값은 매번 해싱할 때마다 완전히 새로운 랜덤 값을 생성하는데 이는 거의 무한대에 가깝습니다.
서버가 솔트 값을 무한대로 제공하면 나뉘어지는 경우의 수도 무한대에 가깝게 됩니다.
또 서버에서는 임의로 연산 속도 자체를 조절해 이러한 전사 공격 시도를 실질적으로 성공하지 못하게 막습니다.


4. 같은 비밀번호를 가진 두 사용자의 해시값이 다른 이유는?

솔트 값의 존재 때문입니다. 매 사용자마다 새로운 랜덤 솔트가 생성되므로, 같은 비밀번호여도 다른 솔트가 붙어 해시값이 달라지게 됩니다.
사용자가 입력한 비밀번호는 서버가 만들어준 솔트와 함께 해싱되어 알고리즘 + 솔트값 + 해시값 의 문자열로 저장됩니다.(BCrypt)
이후 로그인 요청이 들어오면 DB 에서 데이터를 꺼내 솔트를 추출하고 요청 시 입력한 비밀번호화 추출 솔트로 해싱해 결과가 일치한지 확인하는 것으로
인증 할 수 있습니다.




### D-5. Spring Security

1. Spring Security의 핵심 구조는 무엇인가?

스프링 시큐리티의 핵심 구조는 filter chain 입니다.

Spring Security 는 여러 개의 Filter 를 체인으로 연결해서 요청을 순차적으로 처리합니다.
요청이 들어오면 -> Security Filter Chain
                  ㄴ JwtAuthenticationFilter (우리가 만든 것)
                  ㄴ UsernamePasswordAuthenticationFilter
                  ㄴ ExceptionTranslationFilter
                  ㄴ AuthorizationFilter
                  ㄴ 이하 약 15개의 기본 필터를 지나
            -> SecurityContext : Authentication 객체 저장, Principal, Credentials, Authorities 정보가 담김

정리하면,
Filter Chain : 여러 필터가 순차적으로 요청을 검사한다
SecurityContext : 인증된 사용자 정보를 저장하는 공간(요청 스레드별로 하나)
Authentication 객체 : 인증된 사용자의 정보를 담는 객체 


2. `OncePerRequestFilter`를 상속하는 이유는? 일반 `Filter`와의 차이는?

해당 인터페이스를 상속하면 해당 필터는 모든 요청에 대해 딱 한번만 작업을 처리하게 됩니다.

톰캣이 요청을 처리하는 과정에서 내부적으로 forward(다른 서블릿으로 전달) 가 발생할 수 있습니다.
이때 일반 Filter 는 forward 된 요청에 대해 다시 한번 검사를 진행합니다. 같은 요청에 같은 Filter 가 2번 이상 실행될 수 있습니다.
이에 따라 토큰 검증도 다시 하고 로깅도 다시하는 문제가 발생할 수 있습니다.

이와 다르게 OncePerRequestFilter 는 forward 같은 상황에서도 HTTP 요청당 딱 한번만의 실행을 보장하기 때문에 인증 필터처럼
1번만 실행되어도 되는 작업에 등록합니다.


3. 인증/인가 실패가 `GlobalExceptionHandler`로 처리되지 않는 이유는?

GlobalExceptionHandler 는 탐색 범위가 Spring MVC 에 국한되기 때문에 그 앞에서 실행되는 Spring Security 에서는 영향력이 닿지 않습니다.


4. `AuthenticationEntryPoint`와 `AccessDeniedHandler`는 각각 어떤 상황을 처리하는가?

두 키워드는 `SecurityConfig` 에서 등장한 키워드 입니다.
각각
`AuthenticationEntryPoint` 는 인증 실패, 토큰이 없거나 유효하지 않은 요청인 401 Unauthorized 를 다루는 핸들러이고,
`AccessDeniedHandler` 는 인가 실패, 인증은 성공했지만 권한이 부족한 요청인 403 Forbidden 을 다루는 핸들러 입니다.

`.exceptionHanding(ex -> ex
   .authenticationEntryPoint(...)
   .accessDeniedHandler(...)`

Filter 에서 발생하는 예외는 `GlobalExceptionHandler` 가 처리하지 못하기 때문에 이 두 핸들러를 설정해서 401, 403 에 대응하도록 합니다.


5. 인증 실패(401)와 인가 실패(403)를 구분하려면 어떤 설정이 필요한가?

전자는 인증이 실패한 에러 코드며 후자는 인증은 성공했지만 인가에서 실패한 에러 코드 입니다.
이를 핸들링하기 위해 4번에서 언급한것 처럼 SecurityConfig 에서 예외핸들링에 핸들러 옵션을 설정해줘야합니다.
이 설정이 없으면 Spring Security 는 기본 동작으로 로그인 페이지 리다이렉트 등을 시도하게 됩니다.
REST API 에서는 JSON 응답을 내려줘야 하므로, 이 두 핸들러를 직접 구현해서 등록한 것 입니다.



### D-6. SecurityConfig

1. JWT 기반 REST API에서 `csrf`를 비활성화하는 이유는?

csrf 는 세션 관련 공격을 방지하는 옵션입니다. 우리 프로젝트에서는 현재 토큰만 사용하고 있기 때문에 옵션을 비활성화 합니다.
csrf란 Cross-Site Request Forgery 로 사이트 간 요청 위조 공격 입니다.
시나리오는 다음과 같습니다.
사용자가 취약한 은행 사이트에 로그인했다고 가정합니다. 여기서 세션 쿠키가 발급됩니다. 
공격자는 악성 사이트를 만들어 사용자를 유인합니다.
이 악성 사이트의 스크립트가 사용자 브라우저가 쿠키를 자동으로 전송하는 점을 이용해 은행에 송금 요청을 보내도록 합니다.
은행 측에선 이 요청을 정상 요청으로 판단해버립니다.

이 공격은 브라우저가 쿠키를 자동으로 보내는 특성을 악용하는 기법입니다. JWT 는 쿠키가 아니라 Authorization 헤더에 직접 넣어서
보내기 때문에 이 공격이 성립하지 않기 때문에 해당 옵션을 비활성화합니다.
토큰 사용 외에 csrf 를 방어하는 방법에는 csrf 토큰 사용, SameSite 쿠키 속성, Referer/Origin 검증 등이 있습니다.


2. `sessionManagement(STATELESS)`를 설정하는 이유는?

Spring Security 는 기본적으로 인증 정보를 세션에 저장합니다. STATELESS 로 설정해야 이 서버가 세션을 만들거나 사용하지 않는다는 것을
Spring 이 알고 해당 로직을 수행하지 않게 됩니다.


3. `addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)`는 어떤 의미인가?

해당 코드는 `UsernamePasswordAuthenticationFilter` 앞에 우리가 만든 `jwtFilter` 를 끼워넣어달라는 뜻입니다.
Spring Security Filter Chain 에는 이미 여러 기본 필터가 순서대로 배치되어 있습니다.
`UsernamePasswordAuthenticationFilter`는 폼 로그인을 처리하는 기본 필터인데 우리는 JWT 방식이니까 해당 필터가 실행되기 전에
먼저 JWT 를 검증하겠다는 의미 입니다.


4. `authorizeHttpRequests`에서 규칙의 순서가 중요한가? `anyRequest().authenticated()`는 왜 맨 마지막에 와야 하는가?

인가 규칙이 위에서부터 순차적으로 처리되기 때문입니다.
따라서 작은 범위의 인가 규칙이 제일 먼저오고 그 뒤로 점파 큰 범위의 규칙이 와야 개발자가 의도한대로 규칙에 의해 요청이 판단됩니다.
예를들어 제일 아래에 오는 `anyRequest().authenticated()` 규칙은 모든 요청에 대해 인증이 필요하단 규칙인게
이게 제일 먼저 오게되면 인증 없이 사용가능하도록하는 규칙을 만들 수 없게 됩니다.
따라서 작성 시 범위의 순서에 대해 유의해야합니다.


### D-7. 기타 Phase 4 개념

1. `CommandLineRunner`란 무엇이며, 어떤 용도로 사용했는가?

CommandLineRunner 는 Spring Boot 애플리케이션이 완전히 시작된 직후 자동으로 실행되는 코드를 정의하는 인터페이스 입니다.
예를 들어 초기 관리자 계정을 자동 생성하는 등의 작업에 활용할 수 있습니다.
매번 수동으로 DB 에 데이터를 넣지 않아도 서버 시작만 하면 테스트용 계정이 준비되도록 할 수 있습니다.


2. `@Value`는 어떤 역할을 하는가? 어디에서 값을 읽어오는가?

`@Value` 어노테이션은 설정값을 코드에 하드코딩하지 않고 외부 설정 파일에서 주입받기 위해 선언하는 어노테이션입니다.
외부 설정 파일로 application.properties 파일에서 맵핑된 값을 읽어오게 됩니다.


3. 라이브러리 의존성에서 `implementation`과 `runtimeOnly`의 차이는?

`implementation` 은 상시 필요한 라이브러리를 불러 올때 사용하고
`runtimeOnly` 는 애플리케이션이 실행될 때만 필요한 라이브러리 입니다.

`implementation` 은 컴파일 시 필요하며, 실행 시 필요합니다. Spring Boot, JPA 등 코드에서 직접 import 하는 것들이 그 예시입니다.
`runtimeOnly` 는 컴파일 시 불필요하며, 실행 시 피룡합니다. MySQL 드라이버 처럼 코드에 직접 import 하지 않고 JPA가 내부적으로 사용하는게 그 예시입니다.

### E-4. save()가 필요한 경우 구분

1. 다음 상황에서 `save()` 호출이 필요한지 판단하고 이유를 설명하라:
   - 새 게시글을 생성할 때
   - 영속 상태인 게시글의 제목을 변경할 때
   - 트랜잭션이 끝난 후 가져온 엔티티를 다른 트랜잭션에서 수정할 때

새 게시글을 작성할 경우 호출이 필요합니다. 아직 영속성 컨텍스트에 등록되지 않은 새 엔티티기 때문에
save() -> persist() -> INSERT 과정을 거칩니다.

영속 상태의 게시글 제목을 변경할 때는 호출이 불필요합니다. 이미 영속 상태이므로 Dirty Checking 이 자동으로 UPDATE 처리를 해줍니다.
개발자가 save() 호출을 하게 되면 내부적으로 merge() 가 호출되지만 이미 영속 상태이므로 실질적으로 아무 일도 일어나지 않습니다.
최종적으로 Dirty Checking 으로 UPDATE 가 1번 실행되기 때문에 save() 호출은 불필요한 호출이 됩니다.

트랜잭션이 끝난 이후는 영속성 컨텍스트가 비워져 준영속 상태입니다.
이 상태에서 엔티티를 다른 트랜잭션에서 수행할 경우 save() 호출이 필요합니다.
save() -> merge() -> SELECT + UPDATE 과정을 거칩니다.
