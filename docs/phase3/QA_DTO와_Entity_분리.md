# DTO와 Entity 분리

## Q1. Response DTO에도 검증 어노테이션이 필요한가?

**A:** 필요 없다. 검증은 시스템 경계(외부 입력이 들어오는 지점)에서만 하면 된다.
- Request DTO — 클라이언트가 보내는 데이터 → 신뢰할 수 없음 → 검증 필요
- Response DTO — 서버가 만들어서 보내는 데이터 → 이미 검증된 데이터 → 검증 불필요

## Q2. 요청도 DTO로 받아야 하는 이유는?

**A:** Entity로 요청을 직접 받으면 세 가지 문제가 생긴다.

1. **보안** — 클라이언트가 id, createdAt 같은 조작되면 안 되는 필드까지 보낼 수 있음. DTO에는 필요한 필드만 정의하므로 나머지는 무시됨.
2. **검증 규칙 분리** — 생성과 수정의 검증 규칙이 다를 수 있는데(생성 시 author 필수, 수정 시 변경 불가), Entity 하나로는 구분 불가. DTO를 분리하면 상황별 검증 가능.
3. **결합도** — Entity 구조 변경이 API에 직접 영향. DTO가 중간에 있으면 Entity가 바뀌어도 DTO만 수정하면 됨.

## Q3. Entity는 검증을 아예 안 하는가?

**A:** DTO와 Entity의 검증은 역할이 다르다.
- **DTO의 검증** — 클라이언트 입력이 형식적으로 올바른가? (빈 문자열인지, 길이 초과인지)
- **Entity의 검증** — 비즈니스 규칙을 만족하는가? (예: 게시글 상태가 DRAFT인데 공개 요청이 왔는가)

콘솔 프로그램에서 도메인 객체가 했던 검증이 사라진 것이 아니라, 입력 검증과 비즈니스 검증으로 분리된 것이다.

## Q4. delete 작업에도 DTO가 필요한가?

**A:** 필요 없다. DTO를 만드는 이유는 여러 필드를 묶어서 전달하거나 검증이 필요할 때다. `Long id` 하나를 위해 DTO를 만들면 복잡도만 늘어나고 얻는 것이 없다. URL 경로에서 오는 단순 값(`@PathVariable Long id`)까지 DTO로 감싸라는 뜻은 아니다. 파라미터가 2개(`Long postId, Long commentId`)여도 마찬가지.
