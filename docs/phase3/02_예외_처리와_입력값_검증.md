# 예외 처리와 입력값 검증

---

## 1. 현재 코드의 문제점

현재 존재하지 않는 게시글을 조회하면 어떻게 될까요?

```java
// PostController
@GetMapping("/{id}")
public ResponseEntity<Post> getPost(@PathVariable Long id) {
    return postRepository.findById(id)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());  // 단순히 404만 반환
}
```

클라이언트가 받는 응답:

```
HTTP 404
(본문 없음)
```

문제점:

- **왜 404인지 알 수 없음** — 게시글이 없는 건지, URL이 틀린 건지 구분 불가
- **일관성 없음** — 각 컨트롤러 메서드마다 에러 처리를 개별적으로 해야 함
- **입력값 검증 없음** — 제목 없이 게시글을 생성해도 그대로 저장됨

---

## 2. 커스텀 예외 클래스

자바의 예외 메커니즘을 활용해 "게시글을 찾을 수 없음" 같은 비즈니스 예외를 만들 수 있습니다.

```java
// 리소스를 찾을 수 없을 때 사용하는 커스텀 예외
public class ResourceNotFoundException extends RuntimeException {

    public ResourceNotFoundException(String message) {
        super(message);  // 예: "게시글을 찾을 수 없습니다. id: 999"
    }
}
```

`RuntimeException`을 상속합니다. 자바의 예외에는 두 종류가 있는데:

- **Checked Exception** (`Exception` 상속) — 반드시 `try-catch`로 처리해야 함
- **Unchecked Exception** (`RuntimeException` 상속) — 처리를 강제하지 않음. 스프링에서는 이쪽을 주로 사용

### Service에서 예외 던지기

```java

@Service
public class PostService {

    public Post getPost(Long id) {
        return postRepository.findById(id)
                .orElseThrow(() -> new ResourceNotFoundException("게시글을 찾을 수 없습니다. id: " + id));
    }
}
```

`orElseThrow()` — Optional이 비어있으면 예외를 던집니다. Controller에서 `if (post == null)` 체크를 할 필요가 없어집니다.

## 3. 글로벌 예외 처리 (@RestControllerAdvice)

커스텀 예외를 던지면, 스프링이 기본적으로 500 에러를 반환합니다. 이것을 적절한 응답으로 변환해야 합니다.

**`@RestControllerAdvice`**는 모든 컨트롤러에서 발생하는 예외를 한 곳에서 처리하는 글로벌 예외 처리기입니다.

```java

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(ResourceNotFoundException e) {
        ErrorResponse error = new ErrorResponse("NOT_FOUND", e.getMessage());
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }
}
```

### ErrorResponse — 에러 응답 형식 통일

```java
public class ErrorResponse {
    private String code;     // 에러 코드
    private String message;  // 에러 메시지

    public ErrorResponse(String code, String message) {
        this.code = code;
        this.message = message;
    }
    // getter
}
```

이제 클라이언트가 받는 응답:

```json
{
  "code": "NOT_FOUND",
  "message": "게시글을 찾을 수 없습니다. id: 999"
}
```

### 동작 흐름

```
클라이언트 → Controller → Service → orElseThrow() → ResourceNotFoundException 발생!
                                                          ↓
클라이언트 ← JSON 응답 ← GlobalExceptionHandler가 가로채서 ErrorResponse로 변환
```

모든 컨트롤러에서 발생하는 `ResourceNotFoundException`을 한 곳에서 처리하므로, 각 컨트롤러 메서드에서 개별적으로 404 처리를 할 필요가 없어집니다.

## 4. 입력값 검증 (Bean Validation)

클라이언트가 보내는 데이터를 무조건 신뢰하면 안 됩니다. 제목 없이 게시글을 생성하거나, 1만 자짜리 내용을 보내는 경우를 막아야 합니다.

### Bean Validation 어노테이션

DTO 클래스의 필드에 검증 어노테이션을 붙이면 됩니다:

```java
public class PostCreateRequest {

    @NotBlank(message = "제목은 필수입니다")       // null, "", "   " 모두 불허
    @Size(max = 100, message = "제목은 100자 이하여야 합니다")
    private String title;

    @NotBlank(message = "내용은 필수입니다")
    private String content;

    // getter, setter
}
```

### 주요 검증 어노테이션

| 어노테이션              | 대상      | 설명                        |
|--------------------|---------|---------------------------|
| `@NotNull`         | 모든 타입   | null 불허                   |
| `@NotBlank`        | 문자열     | null, "", 공백만 있는 문자열 불허   |
| `@NotEmpty`        | 문자열/컬렉션 | null, 빈 값 불허 (공백 문자열은 허용) |
| `@Size(min, max)`  | 문자열/컬렉션 | 길이/크기 제한                  |
| `@Min`, `@Max`     | 숫자      | 최소/최대값                    |
| `@Email`           | 문자열     | 이메일 형식 검증                 |
| `@Pattern(regexp)` | 문자열     | 정규표현식 패턴 검증               |

### 컨트롤러에서 @Valid 적용

```java

@PostMapping
public ResponseEntity<PostResponse> createPost(@Valid @RequestBody PostCreateRequest request) {
    // @Valid가 붙으면 요청 본문이 DTO로 변환될 때 검증 실행
    // 검증 실패 시 MethodArgumentNotValidException이 자동 발생
}
```

`@Valid` 하나만 붙이면 DTO의 모든 검증 어노테이션이 실행됩니다.

### 검증 실패 시 예외 처리

검증이 실패하면 `MethodArgumentNotValidException`이 발생합니다. 이것도 글로벌 예외 처리기에서 처리합니다:

```java

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(ResourceNotFoundException e) {
        ErrorResponse error = new ErrorResponse("NOT_FOUND", e.getMessage());
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException e) {
        String message = e.getBindingResult()
                .getFieldErrors()
                .stream()
                .map(error -> error.getField() + ": " + error.getDefaultMessage())
                .collect(Collectors.joining(", "));
        ErrorResponse error = new ErrorResponse("INVALID_INPUT", message);
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(error);
    }
}
```

클라이언트가 제목 없이 요청하면:

```json
{
  "code": "INVALID_INPUT",
  "message": "title: 제목은 필수입니다"
}
```

## 5. 전체 흐름 정리

### 정상 요청

```
클라이언트 → @Valid 검증 통과 → Controller → Service → Repository → DB
         ← PostResponse(DTO) ← Controller ← Service ←
```

### 검증 실패

```
클라이언트 → @Valid 검증 실패! → MethodArgumentNotValidException
         ← ErrorResponse (400) ← GlobalExceptionHandler
```

### 리소스 없음

```
클라이언트 → Controller → Service → orElseThrow() → ResourceNotFoundException
         ← ErrorResponse (404) ← GlobalExceptionHandler
```

## 6. 핵심 정리

| 개념                        | 설명                                        |
|---------------------------|-------------------------------------------|
| **커스텀 예외**                | 비즈니스 상황에 맞는 예외 클래스. `RuntimeException` 상속 |
| **@RestControllerAdvice** | 모든 컨트롤러의 예외를 한 곳에서 처리                     |
| **@ExceptionHandler**     | 특정 예외 타입을 처리하는 메서드를 지정                    |
| **ErrorResponse**         | 에러 응답 형식을 통일하는 DTO                        |
| **Bean Validation**       | DTO 필드에 어노테이션으로 검증 규칙 선언                  |
| **@Valid**                | 컨트롤러 파라미터에 붙여서 검증을 실행                     |
| **@NotBlank, @Size 등**    | 구체적인 검증 규칙 어노테이션                          |
