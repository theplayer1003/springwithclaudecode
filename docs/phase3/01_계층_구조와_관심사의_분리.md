# 계층 구조와 관심사의 분리

---

## 1. 현재 코드의 문제점

Phase 2까지 작성한 `PostController`를 살펴보면, 하나의 클래스가 너무 많은 일을 하고 있습니다.

```java

@RestController
@RequestMapping("/posts")
public class PostController {

    private final PostRepository postRepository;

    // HTTP 요청을 받는 것 (컨트롤러의 본래 역할)
    @PostMapping
    public ResponseEntity<Post> createPost(@RequestBody Post request) {
        // 비즈니스 로직 (게시글 생성 규칙)
        Post post = new Post(request.getTitle(), request.getContent());
        // 데이터베이스 접근
        Post saved = postRepository.save(post);
        return ResponseEntity.status(HttpStatus.CREATED).body(saved);
    }
}
```

지금은 단순 CRUD라서 괜찮아 보이지만, 기능이 추가되면 어떻게 될까요?

```java
@PostMapping
public ResponseEntity<Post> createPost(@RequestBody Post request) {
    // 입력값 검증
    if (request.getTitle() == null || request.getTitle().isBlank()) {
        throw new IllegalArgumentException("제목은 필수입니다");
    }
    if (request.getTitle().length() > 100) {
        throw new IllegalArgumentException("제목은 100자 이하여야 합니다");
    }
    // 금칙어 필터링
    if (containsBannedWords(request.getTitle())) {
        throw new IllegalArgumentException("금칙어가 포함되어 있습니다");
    }
    // 게시글 생성
    Post post = new Post(request.getTitle(), request.getContent());
    post.setViewCount(0);
    post.setLikeCount(0);
    // 작성자 정보 설정
    User currentUser = getCurrentUser();
    post.setAuthor(currentUser);
    // 저장
    Post saved = postRepository.save(post);
    // 알림 발송
    notificationService.notifyFollowers(currentUser, saved);
    // 검색 인덱스 업데이트
    searchService.index(saved);
    return ResponseEntity.status(HttpStatus.CREATED).body(saved);
}
```

하나의 메서드에 검증, 비즈니스 로직, DB 접근, 외부 서비스 호출이 전부 섞여있습니다.

### 이렇게 되면 생기는 문제

1. **가독성** — 메서드가 길어져서 무슨 일을 하는지 한눈에 파악하기 어려움
2. **테스트** — 컨트롤러를 테스트하려면 DB, 외부 서비스 등 모든 것을 준비해야 함. 비즈니스 로직만 단독으로 테스트할 수 없음
3. **재사용** — 같은 게시글 생성 로직을 다른 곳(배치 작업, 관리자 API 등)에서도 써야 하면? 컨트롤러에 박혀있으니 재사용 불가
4. **유지보수** — 금칙어 규칙이 바뀌면 컨트롤러를 수정해야 함. DB 접근 방식이 바뀌어도 컨트롤러를 수정해야 함. 모든 변경이 한 곳에 집중됨

## 2. 관심사의 분리 (Separation of Concerns)

해결 방법은 **역할별로 클래스를 나누는 것**입니다.

> 각 클래스는 하나의 관심사(역할)만 담당해야 한다.

이것을 **관심사의 분리**라고 합니다.

---

## 3. 3계층 아키텍처 (Layered Architecture)

스프링에서는 관심사를 **3개의 계층**으로 나누는 것이 표준입니다.

```
┌─────────────────────────────────┐
│  Presentation Layer (표현 계층)  │  ← Controller
│  HTTP 요청/응답 처리             │
├─────────────────────────────────┤
│  Business Layer (비즈니스 계층)   │  ← Service
│  비즈니스 로직, 규칙 처리         │
├─────────────────────────────────┤
│  Data Access Layer (데이터 계층)  │  ← Repository
│  데이터베이스 접근               │
└─────────────────────────────────┘
```

### Controller — 표현 계층

- HTTP 요청을 받고 응답을 반환하는 역할만 담당
- 비즈니스 로직을 직접 수행하지 않고, Service에게 위임
- `@RestController`

```java

@RestController
@RequestMapping("/posts")
public class PostController {
    private final PostService postService;  // Service에게 위임

    @PostMapping
    public ResponseEntity<Post> createPost(@RequestBody PostCreateRequest request) {
        Post post = postService.createPost(request);  // 로직은 Service가 처리
        return ResponseEntity.status(HttpStatus.CREATED).body(post);
    }
}
```

### Service — 비즈니스 계층

- 비즈니스 로직을 담당 (검증, 규칙, 판단)
- 트랜잭션 관리 (`@Transactional`)
- Controller와 Repository 사이에서 중간 역할
- `@Service`

```java

@Service
public class PostService {
    private final PostRepository postRepository;

    @Transactional
    public Post createPost(PostCreateRequest request) {
        // 비즈니스 로직: 검증, 생성 규칙 등
        Post post = new Post(request.getTitle(), request.getContent());
        return postRepository.save(post);
    }
}
```

### Repository — 데이터 계층

- 데이터베이스 접근만 담당
- 이미 Phase 2에서 만들어놓음 (`JpaRepository` 상속)

```java
public interface PostRepository extends JpaRepository<Post, Long> {
}
```

### 호출 흐름

```
클라이언트 → Controller → Service → Repository → DB
              (HTTP)      (로직)     (데이터)
```

각 계층은 **바로 아래 계층만** 호출합니다:

- Controller → Service ✅
- Controller → Repository ❌ (Service를 건너뛰면 안 됨)
- Service → Repository ✅

## 4. DTO — 계층 간 데이터 전달 객체

현재 코드에서는 Entity(`Post`)를 그대로 요청/응답에 사용하고 있습니다. 이것에는 문제가 있습니다.

### Entity를 직접 노출하면 생기는 문제

1. **보안** — 클라이언트가 `id`, `createdAt` 등 서버가 관리해야 할 필드를 조작할 수 있음
2. **결합** — API 응답 형태를 바꾸려면 Entity를 수정해야 함. Entity를 수정하면 DB 테이블이 바뀜
3. **과다 노출** — Entity의 모든 필드가 JSON에 노출됨. 비밀번호, 내부 상태 등도 포함될 수 있음

### DTO (Data Transfer Object)

**DTO**는 계층 간에 데이터를 전달하기 위한 전용 객체입니다. Entity와 분리하여 사용합니다.

```
클라이언트 ←→ [요청 DTO] ←→ Controller ←→ Service ←→ Repository ←→ [Entity] ←→ DB
             [응답 DTO]
```

### 요청 DTO 예시

```java
// 게시글 생성 요청 — 클라이언트가 보내는 데이터만 포함
public class PostCreateRequest {
    private String title;
    private String content;
    // getter, setter
}

// 게시글 수정 요청 — 수정 가능한 필드만 포함
public class PostUpdateRequest {
    private String title;
    private String content;
    // getter, setter
}
```

### 응답 DTO 예시

```java
// 게시글 응답 — 클라이언트에게 보여줄 데이터만 포함
public class PostResponse {
    private Long id;
    private String title;
    private String content;
    private LocalDateTime createdAt;
    // getter

    // Entity → DTO 변환 메서드
    public static PostResponse from(Post post) {
        PostResponse response = new PostResponse();
        response.id = post.getId();
        response.title = post.getTitle();
        response.content = post.getContent();
        response.createdAt = post.getCreatedAt();
        return response;
    }
}
```

### 변환 흐름

```
[요청]
클라이언트 → JSON → PostCreateRequest(DTO) → Controller → Service → Post(Entity) → DB

[응답]
DB → Post(Entity) → Service → Controller → PostResponse(DTO) → JSON → 클라이언트
```

## 5. 패키지 구조

계층별로 패키지를 나누어 코드를 정리합니다.

```
com.study.board/
├── controller/          ← 표현 계층
│   ├── PostController.java
│   └── CommentController.java
├── service/             ← 비즈니스 계층
│   ├── PostService.java
│   └── CommentService.java
├── repository/          ← 데이터 계층
│   ├── PostRepository.java
│   └── CommentRepository.java
├── entity/              ← Entity 클래스
│   ├── Post.java
│   └── Comment.java
├── dto/                 ← DTO 클래스
│   ├── PostCreateRequest.java
│   ├── PostUpdateRequest.java
│   ├── PostResponse.java
│   ├── CommentCreateRequest.java
│   └── CommentResponse.java
└── BoardApplication.java
```

현재는 모든 클래스가 `com.study.board` 패키지 하나에 몰려있습니다. 이것을 역할별로 나누는 것이 첫 번째 작업입니다.

## 6. 핵심 정리

| 개념             | 설명                                                   |
|----------------|------------------------------------------------------|
| **관심사의 분리**    | 각 클래스가 하나의 역할만 담당하도록 나누는 원칙                          |
| **Controller** | HTTP 요청/응답만 처리. 로직은 Service에 위임                      |
| **Service**    | 비즈니스 로직 담당. 트랜잭션 관리                                  |
| **Repository** | 데이터베이스 접근만 담당                                        |
| **DTO**        | 계층 간 데이터 전달 전용 객체. Entity를 직접 노출하지 않기 위함             |
| **호출 방향**      | Controller → Service → Repository (한 방향, 계층 건너뛰기 금지) |
