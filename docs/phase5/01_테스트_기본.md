# 테스트 기본

## 1. 테스트 코드가 필요한 이유

### 회귀 방지 (Regression Prevention)

기존에 정상 작동하던 기능이 새로운 코드 변경으로 인해 깨지는 것을 자동으로 감지한다. 시스템이 고도화될수록 기능 간 의존관계가 복잡해지고, 한 곳의 변경이 예상치 못한 곳에 영향을 줄 수 있다. 미리 요구사항을
테스트 코드로 정의해두면 새 코드 추가 시 기존 기능의 오작동 여부를 즉시 확인할 수 있다.

### 반복 검증 자동화

API가 10개, 50개로 늘어나도 매번 수동으로 확인할 필요 없이 명령어 하나로 전체를 검증한다. 수동 테스트는 누락과 실수가 발생하기 쉽지만, 자동화된 테스트는 항상 동일한 기준으로 검증한다.

## 2. JUnit 5 주요 어노테이션

### 테스트 정의

- `@Test` — 메서드가 하나의 테스트임을 표시
- `@DisplayName("설명")` — 테스트 실행 결과에 메서드명 대신 한글 설명을 표시

### 생명주기

- `@BeforeEach` — **각 테스트 메서드 실행 전**에 호출. 공통 준비 작업에 사용. 매 테스트마다 새로 실행되므로 테스트 간 격리를 보장
- `@AfterEach` — **각 테스트 메서드 실행 후**에 호출. 테스트 후 정리 작업(DB 데이터 삭제 등)에 사용
- `@BeforeAll` — **클래스 전체에서 딱 한 번** 실행. 비용이 큰 초기화 작업에 사용. `static` 메서드여야 함
- `@AfterAll` — **클래스 전체에서 딱 한 번**, 모든 테스트 종료 후 실행. `static` 메서드여야 함

```
@BeforeAll (1회)
  ├── @BeforeEach → @Test (1번 테스트) → @AfterEach
  ├── @BeforeEach → @Test (2번 테스트) → @AfterEach
  └── @BeforeEach → @Test (3번 테스트) → @AfterEach
@AfterAll (1회)
```

### 검증 (AssertJ)

JUnit 기본 `Assertions`보다 가독성이 좋은 AssertJ를 사용한다.

```java
// 값 검증
assertThat(result.title()).

isEqualTo("제목");

// 예외 검증
assertThatThrownBy(() ->service.

getPost(999L))
        .

isInstanceOf(ResourceNotFoundException .class)
        .

hasMessageContaining("게시글을 찾을 수 없습니다");
```

## 3. 테스트의 종류

### 단위 테스트 (Unit Test)

테스트 대상의 로직**만** 격리해서 검증하는 테스트. 의존하는 외부 요소(DB, 외부 API 등)는 가짜 객체(Mock)로 대체한다. Spring 컨텍스트를 띄우지 않아 빠르다.

```java

@ExtendWith(MockitoExtension.class)  // Spring 없이 Mockito만 사용
class PostServiceTest {
    @Mock
    private PostRepository postRepository;       // 가짜 객체
    @InjectMocks
    private PostService postService;      // 테스트 대상 (진짜)
}
```

### 슬라이스 테스트 (Slice Test)

전체 애플리케이션이 아닌 **한 계층만** 로드하여 테스트한다. `@WebMvcTest`가 대표적이며, 컨트롤러 계층 + Spring MVC 설정만 띄운다.

```java

@WebMvcTest(PostController.class)    // 컨트롤러 계층만 로드
class PostControllerTest {
    @MockitoBean
    private PostService postService;  // Spring 컨텍스트에 Mock 빈 등록
    @Autowired
    private MockMvc mockMvc;            // HTTP 요청 시뮬레이션 도구
}
```

### 통합 테스트 (Integration Test)

**실제 애플리케이션 컨텍스트**를 띄워서 Mock 없이 여러 계층이 함께 동작하는지 검증한다. Service → Repository → DB 전체 흐름을 실제로 실행한다.

```java

@Transactional       // 각 테스트 후 자동 롤백
@SpringBootTest      // 전체 애플리케이션 컨텍스트 로드
class PostIntegrationTest {
    @Autowired
    private PostService postService;          // 진짜 Service
    @Autowired
    private MemberRepository memberRepository; // 진짜 Repository
}
```

### 비교

```
단위 테스트       — Spring 없음, Mock 사용, 빠름, 한 클래스만 검증
슬라이스 테스트   — Spring 일부 로드, 한 계층만 검증 (Controller 등)
통합 테스트       — Spring 전체 로드, Mock 없음, 실제 DB 사용, 느리지만 확실함
```

## 4. 단위 테스트와 Mock

### Mock이 필요한 이유

`PostService`를 테스트하려면 `PostRepository`가 필요하고, Repository는 실제 DB에 연결되어 있다. 매 테스트마다 DB를 띄우면 커넥션 생성, 스키마 초기화, 데이터 준비/정리 등
비용이 크다. Mock을 사용하면 테스트 대상의 로직에만 집중하고, 의존성은 "이렇게 호출하면 이것을 반환해라"고 미리 지정한 가짜 객체로 대체한다.

### Mockito 주요 어노테이션

- `@ExtendWith(MockitoExtension.class)` — 클래스 레벨에 선언. Mockito 어노테이션을 사용하겠다는 선언
- `@Mock` — 해당 필드를 가짜 객체로 생성. 모든 메서드는 기본적으로 null(또는 0, false)을 반환
- `@InjectMocks` — `@Mock`으로 선언된 객체들을 찾아 생성자에 주입하여 테스트 대상 객체를 생성

### when().thenReturn() — Mock의 행동 정의

```java
// "findById(1L)이 호출되면 이 Post를 반환해라"
when(postRepository.findById(1L)).

thenReturn(Optional.of(post));

// "findById(999L)이 호출되면 빈 값을 반환해라"
when(postRepository.findById(999L)).

thenReturn(Optional.empty());
```

핵심 원칙: **Mock의 행동을 정의하고, 테스트 대상은 진짜로 실행한다.**

```
postService.getPost(1L)            ← 진짜 실행 (테스트 대상)
  → postRepository.findById(1L)   ← Mock이 대체 (지정한 값 반환)
  → PostResponse.from(post)       ← 진짜 실행
```

### verify() — Mock의 호출 검증

`when`은 "호출되면 이것을 반환해라"이고, `verify`는 "이것이 호출되었는지 확인해라"이다. void 메서드처럼 반환값으로 검증할 수 없는 경우에 사용한다.

```java
// deletePost는 void — 반환값이 없으므로 assertThat으로 검증할 수 없다
postService.deletePost(1L,"username");

// "postRepository.deleteById(1L)이 호출되었는가"를 검증
verify(postRepository).

deleteById(1L);
```

주의: 단위 테스트에서 Mock은 실제로 데이터를 저장하거나 삭제하지 않는다. 삭제 후 `getAllPosts()`를 호출해도 "삭제되었기 때문에 빈 리스트"가 아니라 "Mock의 기본 반환값인 빈 리스트"가 나올
뿐이다. 삭제 동작을 검증하려면 `verify`를 사용해야 한다.

### @Mock vs @MockitoBean

|       | @Mock              | @MockitoBean                  |
|-------|--------------------|-------------------------------|
| 제공    | Mockito            | Spring                        |
| 사용 환경 | Spring 없이 (단위 테스트) | Spring 컨텍스트 안에서 (슬라이스/통합 테스트) |
| 동작    | Mockito가 가짜 객체 생성  | Spring 컨텍스트에 가짜 빈을 등록         |
| 짝꿍    | `@InjectMocks`     | `@Autowired`                  |

## 5. 슬라이스 테스트 (@WebMvcTest)

### 컨트롤러 테스트의 목적

컨트롤러는 HTTP 요청을 받아 Service에 전달하고 응답을 반환하는 역할이다. 단순히 자바 메서드를 호출하는 것으로는 검증할 수 없는 것들이 있다:

- URL 매핑 (`GET /posts/1` → `getPost()` 호출)
- JSON 요청/응답 변환
- `@Valid` 입력값 검증
- HTTP 상태코드

### MockMvc

실제 서버를 띄우지 않고 Spring MVC의 요청 처리 과정을 시뮬레이션하는 도구. HTTP 요청을 만들어 **진짜 컨트롤러**에 전달하고, 응답을 검증한다.

```java
mockMvc.perform(get("/posts/1"))              // GET /posts/1 요청 시뮬레이션
        .

andExpect(status().

isOk())            // 상태코드 200 검증
        .

andExpect(jsonPath("$.title").

value("제목")); // 응답 JSON 검증
```

### jsonPath 문법

JSON 응답 구조를 탐색하는 경로 표현식.

| 표현식          | 의미                            |
|--------------|-------------------------------|
| `$.필드명`      | 객체의 필드 (`$.title`)            |
| `$.length()` | 배열의 크기                        |
| `$[0]`       | 배열의 첫 번째 요소                   |
| `$[0].필드명`   | 배열 첫 번째 요소의 필드 (`$[0].title`) |

### Security 충돌 해결

`@WebMvcTest`는 Spring Security 필터를 함께 로드한다. JWT 필터 등 커스텀 Security 컴포넌트가 있으면 의존성 문제로 에러가 발생할 수 있다. Security를 제외하고 순수
컨트롤러만 테스트하려면:

```java
@WebMvcTest(
        controllers = PostController.class,
        excludeAutoConfiguration = SecurityAutoConfiguration.class,
        excludeFilters = @ComponentScan.Filter(
                type = FilterType.ASSIGNABLE_TYPE,
                classes = {JwtAuthenticationFilter.class, SecurityConfig.class}
        )
)
```

## 6. 통합 테스트 (@SpringBootTest)

### 통합 테스트의 목적

단위 테스트와 슬라이스 테스트는 각 계층을 격리해서 검증한다. 하지만 Mock은 "이렇게 동작할 것이다"라는 **가정**이지, 실제 동작을 보장하지 않는다. 통합 테스트는 Mock 없이 실제 계층을 연결하여 다음을
검증한다:

- JPA 엔티티 매핑이 올바른가 (필드 ↔ 컬럼)
- 쿼리 메서드가 정상 동작하는가
- 연관관계 설정이 올바른가
- 트랜잭션 안에서 저장과 조회가 일관성 있게 동작하는가

### @Transactional의 테스트 동작

프로덕션과 테스트에서 동작이 다르다:

```
프로덕션: @Transactional → 성공 시 커밋, 실패 시 롤백
테스트:   @Transactional → 무조건 롤백
```

테스트 클래스에 `@Transactional`을 붙이면 `@BeforeEach` → `@Test` → `@AfterEach`가 하나의 트랜잭션으로 묶여서 전부 롤백된다. `@AfterEach`에서 직접 정리할 필요가
없다.

단, **auto-increment ID는 롤백되지 않는다.** DB 엔진의 특성이므로 테스트에서 ID를 하드코딩하면 안 된다. 생성 결과에서 ID를 받아 사용한다.

### 통합 테스트의 데이터 준비

Mock 테스트에서는 `when()`으로 원하는 값을 바로 반환시켰지만, 통합 테스트에서는 실제 데이터 의존 관계를 모두 준비해야 한다. 예를 들어 게시글을 생성하려면 먼저 회원이 DB에 존재해야 한다.

```java

@BeforeEach
void setup() {
    memberRepository.save(new Member("username", "password", "USER"));  // 회원 먼저
    postResponse = postService.createPost(request, "username");          // 그 다음 게시글
}
```

## 7. 테스트 구조: Given-When-Then

테스트 메서드를 세 단계로 구성하는 패턴.

- **Given** (준비) — 테스트에 필요한 데이터와 Mock 행동을 설정
- **When** (실행) — 테스트 대상 메서드를 호출
- **Then** (검증) — 결과가 기대한 값과 일치하는지 확인

```java

@Test
void getPost_existsEntity_returnCorrectResponse() {
    // given — Mock 행동 정의
    Post post = new Post("제목", "내용", member);
    when(postRepository.findById(1L)).thenReturn(Optional.of(post));

    // when — 테스트 대상 실행
    PostResponse response = postService.getPost(1L);

    // then — 결과 검증
    assertThat(response.title()).isEqualTo("제목");
    assertThat(response.content()).isEqualTo("내용");
}
```

## 8. 테스트 커버리지 (JaCoCo)

### 커버리지란

테스트에 의해 실행된 코드의 비율. "테스트를 충분히 작성했는가"를 수치로 확인한다.

### JaCoCo 설정

`build.gradle.kts`에 플러그인 추가:

```kotlin
plugins {
    jacoco
}
```

실행: `./gradlew test jacocoTestReport`
리포트: `build/reports/jacoco/test/html/index.html`

### 주요 지표

| 지표       | 의미                                      |
|----------|-----------------------------------------|
| Lines    | 소스코드 라인 단위 커버리지                         |
| Branches | 분기(if/else) 커버리지 — 모든 경로를 테스트했는가        |
| Methods  | 메서드 단위 커버리지                             |
| Cxty     | Cyclomatic Complexity — 분기가 많을수록 높은 복잡도 |

소스코드에서 초록색(실행됨), 빨간색(실행 안 됨), 노란색(분기 일부만 실행)으로 표시된다.

### IntelliJ 커버리지 vs JaCoCo

IntelliJ 커버리지는 IDE에서 바로 확인 가능하지만 개인 환경에 한정된다. JaCoCo는 리포트 파일을 생성하므로 CI/CD에서 자동 측정, 팀 공유, "커버리지 N% 미만이면 빌드 실패" 같은 규칙 설정이
가능하다.

### 단위 테스트와 통합 테스트의 역할 분담

모든 테스트 케이스를 양쪽에 중복 작성할 필요는 없다.

- **단위 테스트** — 비즈니스 로직의 모든 분기를 촘촘하게 검증. 빠르므로 케이스를 많이 작성
- **통합 테스트** — 핵심 시나리오의 정상 흐름 위주. 실제 DB 연동이 잘 되는지 확인

## 테스트 메서드 네이밍 관례

`메서드명_조건_기대결과` 형태를 사용한다. 조건은 상태/상황을 설명하고, 메서드명과 중복하지 않는다.

```java
void getPost_ExistingId_ReturnsPost()

void getPost_NonExistingId_ThrowsResourceNotFoundException()

void updatePost_ByAuthor_UpdatesSuccessfully()

void updatePost_ByNonAuthor_ThrowsUnauthorizedAccessException()
```
