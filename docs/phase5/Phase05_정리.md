# Phase 5 정리

## 1. 테스트의 필요성

**Q1.** 테스트 코드가 필요한 이유 두 가지를 설명하세요.

**A1.**

첫번째는 회귀 방지를 위함입니다.
회귀 방지란 개발을 해나가면서 나중에 추가된 코드로 인해 앞서 작성한 코드가 영향을 받아도 의도대로 동작하고 있음을 보장해줘야한다는 것 입니다.
테스트 코드로 해당 코드의 요구사항을 면밀히 정의해두면 새로운 코드가 작성되도 테스트 실행을 통해 곧바로 앞서 작성한 코드가 정상적으로 동작하고 있음을 알 수 있습니다.

두번째는 자동화 입니다.
프로그램이 개발되고 기능이 많아질수록 이를 일일히 수동으로 호출해 의도적으로 동작하는지 확인하기 어렵습니다.
이를 방지하기 위해 테스트 코드를 잘 작성해둬 전체 테스트 실행 하나로 프로그램의 정상 작동 여부를 곧바로 확인할 수 있습니다.

**Q2.** 수동 테스트 대비 자동화된 테스트의 장점은 무엇인가요?

**A2.**

앞서 말한대로 코드가 새로운 코드의 영향을 받는 것에 대해 감지할 수 있고 많아지는 API 를 수동 호출 하지 않고도 코드가 의도대로 동작하는지 확인할 수 있습니다.
실행 속도 측면에서는 Postman 같은 툴로 하나하나 확인하는 것 보다 수백배 빨라집니다.
재현성 측면에서 누가 돌려도 같은 결과를 보장합니다.
문서 역할로는 작성된 테스트 코드 자체가 해당 클래스의 동작이 이렇게 되어야한다는 명세서가 됩니다.

## 2. JUnit 5

**Q3.** `@Test`, `@BeforeEach`, `@AfterEach`, `@BeforeAll`, `@AfterAll`의 역할과 실행 순서를 설명하세요.

**A3.**

`@Test` 어노테이션은 메서드 레벨에 붙으며 해당 메서드가 하나의 테스트임을 나타내는 단위 입니다.
이 개별 테스트 메서드들이 모여 하나의 테스트 클래스가 됩니다.
`@BeforeEach` 어노테이션은 개별 테스트들이 실행되기 전에 선행되는 작업을 정의합니다.
테스트에 공통적으로 필요한 부분들을 묶어 미리 세팅할 수 있습니다.
`@AfterEach` 어노테이션은 개별 테스트 실행 이후에 실행되는 작업입니다.
테스트가 끝난 후 다음 테스트를 실행하기 전에 상태를 조정해야할 때 활용하여 테스트 환경을 일관되게 유지할 수 있습니다.
`@BeforeAll` 어노테이션은 테스트 실행 전 한 번만 수행되는 작업을 정의합니다.
테스트 시작 전에 수행되어야할 세팅 중에 무거운 작업들은 테스트 개별적으로 일일히 실행하면 낭비가 발생할 수 있습니다.
이를 여기에 정의해 모든 테스트 수행 전 한번만 실행할 수 있도록 합니다.
`@AfterAll` 어노테이션은 테스트 실행 후 한 번만 수행하는 작업을 정의합니다.
모든 테스트가 끝나고 자원을 정리해야할 때가 있습니다. 이 때 해당 어노테이션을 달아 리소스 종료 코드를 작성합니다.

**Q4.** `@BeforeEach`가 각 테스트마다 새로 실행되는 이유는 무엇인가요? 한 번만 실행하면 안 되는 이유를 설명하세요.

**A4.**

모든 테스트 별로 독립되고 일관된 컨디션을 유지하기 위해서입니다.
테스트를 진행하다보면 어떤 상태를 변화시킬 수 있는데 이 상태가 다음 테스트에 이어져 영향을 주면 의도한 테스트 환경이 아닐 수 있습니다.
이를 방지하기 위해 항상 테스트 전에 개별로 `@BeforeEach` 를 실행시켜 테스트 환경을 정돈합니다.

**Q5.** `@SpringBootTest`가 붙은 테스트 클래스에서 `contextLoads()` 메서드가 비어있어도 의미가 있는 이유를 설명하세요.

**A5.**

해당 테스트는 SpringBoot 가 정상적으로 실행되는지, 구성요소들이 정상 작동해 서버 실행에 문제가 없는지 확인하는 테스트입니다.
따라서 별다른 작업 정의 없이 비어있는 채로 실행 가능 여부만 테스트해줍니다.

## 3. 테스트의 종류

**Q6.** 단위 테스트, 슬라이스 테스트, 통합 테스트의 차이를 Spring 컨텍스트 로드 여부, Mock 사용 여부, 검증 범위 측면에서 비교하세요.

**A6.**

단위 테스트는 스프링 컨텍스트 로드 없이 Mockito 라이브러리만 사용해서 수행하는 테스트 입니다.
해당 클래스의 비즈니스 로직만을 테스트하는게 목적으로 클래스 레벨에 `@ExtendWith(MockitoExtenstion.class)` 가 붙습니다.
스프링 컨텍스트를 로드하지 않기 때문에 `@InjectMocks`, `@Mock` 으로 테스트 대상과 대상에 주입할 의존성을 설정해 테스트합니다.

슬라이스 테스트는 하나의 계층을 대상으로 하는 테스트로 스프링 컨텍스트를 일부 로드합니다.
스프링 컨텍스트 로드를 위해 `@MokitoBean`, `@Autowired` 로 테스트 구성요소를 지정하고
클래스 레벨에 `@WebMvcTest` 어노테이션이 붙습니다. 슬라이스 테스트기 때문에 테스트 대상 계층 밖의 구성요소(스프링 시큐리티 필터)를 사용하지 않도록 설정할 수 있습니다.
@MockitoBean 으로 Spring 컨텍스트에 등록된 PostService 빈을 가짜로 교체하고 컨트롤러가 주입받는 PostService 가 Mock 이 됩니다.
@Autowired 로 Spring 이 만들어준 MockMvc 객체를 주입 받습니다.
우리가 작성한 슬라이스 테스트의 테스트 대상은 PostController 입니다. 단위 테스트와 다른 검증 방식이 필요합니다.
단위 테스트에서는 자바 메서드를 직접 호출해 리턴값을 검증합니다.
슬라이스 테스트에서는 HTTP 요청을 보내서 URL 매핑, 상태코드, JSON 응답 검증 등을 합니다.

통합 테스트는 모든 계층을 대상으로 mock 없이 실제 객체로 수행하는 테스트입니다. 스프링 컨텍스트를 전부 로드합니다.
`@SpringBootTest` 어노테이션이 붙는 테스트로 애플리케이션을 구성하는 여러 계층이 실제로 협력해 기능이 동작하는지 확인하는 테스트입니다.

**Q7.** 세 종류의 테스트 중 실행 속도가 가장 빠른 것과 가장 느린 것은 무엇이고, 그 이유는 무엇인가요?

**A7.**

단위 테스트가 가장 빠릅니다. 단위테스트는 스프링 컨텍스트 로드 없이 해당 클래스의 비즈니스 로직만 테스트하며 외부 DB 등은 Mock 으로 대신하기 때문에 가장 빠릅니다.
통합 테스트가 가장 느립니다. 통합 테스트는 모든 스프링 컨텍스트를 로드하며 실제 애플리케이션을 구성하는 객체들을 직접 이용해서 테스트하기 때문에 가장 느립니다.

## 4. Mock과 Mockito

**Q8.** Mock이 필요한 이유를 PostService 테스트를 예로 들어 설명하세요.

**A8.**

PostService 의 경우 PostRepository 와 MemberRepository 와 협력해야 자신의 로직을 수행할 수 있습니다.
이를 테스트할때 PostService 단일 대상으로만 테스트를 하고 싶은 경우 실제 Repository 를 생성해서 주입하지 않고 Mock 객체를 활용할 수 있습니다.
Mock 은 가짜 객체로 비어있는 두 Repository 를 생성해 줍니다. 기본적으로 내부의 모든 가짜 메서드가 0, null, false 등을 반환하는 상태이며
여기에 PostService 로직 수행에 필요한 부분만 임의로 어떻게 동작할지 정의해서 PostService 만을 대상으로 테스트할 수 있게 됩니다.

**Q9.** `@Mock`과 `@InjectMocks`의 역할을 각각 설명하고, 두 어노테이션이 함께 동작하는 흐름을 설명하세요.

**A9.**

@Mock 은 가짜 객체의 대상에 선언하는 어노테이션입니다.
@InjectMocks 앞서 만든 @Mock 달린 가짜 객체들을 주입할 대상에 선언하는 어노테이션입니다.

Mokito 는 @Mock 이 달린 객체들을 보고, @InjectMocks 대상의 생성자 파라미터 타입과 매칭하여 자동으로 주입하게 됩니다.

**Q10.** `when().thenReturn()`과 `verify()`의 차이를 설명하세요. 각각 어떤 상황에서 사용하나요?

**A10.**

`when().thenReturn()` 은 mock 객체의 동작 중 어떤 값을 리턴할 필요가 있을 때 사용하는 구문입니다.

`verify()` 는 미리 mock 객체의 행동을 미리 정의하는 `when()` 절과 다르게 mock 객체가 특정 메서드가 실제로 호출되었는지를 검증하는 메서드 입니다.
delete 테스트에서 verify() 를 사용한 이유는 리턴값이 void 라 assertThat 으로 검증이 어려워 verify() 로 실제 호출여부를 통해 테스트 했습니다.

**Q11.** `@Mock`과 `@MockitoBean`은 어떤 차이가 있나요? 각각 어떤 테스트에서 사용하나요?

**A11.**

`@Mock` 어노테이션은 Mockito 가 만드는 순수 가짜 객체 입니다. Spring 컨텍스트와 무관하며 단위 테스트에 사용됩니다.
`@MockitoBean` 어노테이션은 Spring 컨텍스트에 등록된 실제 빈을 Mock 으로 교체해서 동작합니다. Spring 컨텍스트를 직접 로드하는 슬라이스 테스트에 사용합니다.

## 5. 슬라이스 테스트 (@WebMvcTest)

**Q12.** 컨트롤러를 테스트할 때 자바 메서드를 직접 호출하지 않고 MockMvc를 사용하는 이유는 무엇인가요?

**A12.**

컨트롤러는 HTTP 요청을 받아 처리하는 계층입니다.
컨트롤러 코드의 테스트 대상은 URL 매핑이 올바르게 되어있는지, 응답 상태코드는 올바른지, 응답 JSON 형식이 올바른지 등 입니다.
이러한 부분은 자바 메서드를 직접 호출하면 이런 HTTP 계층의 동작을 검증할 수 없습니다.
예를 들어 테스트 코드에서 PostController 의 객체를 만들고 controller.getPost() 호출을 했다고 가정해보겠습니다.
이 테스트로는 리턴값인 PostResponse 에 대한 확인만 가능합니다.
하지만 실제 Controller 에서 일어나는 동작은
GET HTTP 메서드가 getPost() 메서드에 매핑
URL 파라미터가 바인딩
리턴된 Response 가 JSON 으로 변환
HTTP 상태 코드를 리턴
등의 다양한 동작이 수반됩니다.
이 동작들은 Spring MVC 가 처리하는 동작이기 때문에 단순히 메서드를 호출하고 결과를 본다고 해서 제대로 수행되는지 체크할 수 없습니다.

MockMvc 는 이 전체 과정을 시뮬레이션 해서 검증할 수 있게 해주는 도구 입니다.

**Q13.** `@WebMvcTest`에서 Spring Security 관련 에러가 발생했던 원인과 해결 방법을 설명하세요.

**A13.**

`@WebMvcTest` 는 슬라이스 테스트에 선언되는 어노테이션입니다.
컨트롤러 계층의 슬라이스 테스트기 때문에 외부에서 들어오는 요청을 시뮬레이션 해야 하고 여기에는 스프링 시큐리티 코드 또한 포함됩니다.
앞서 슬라이스 테스트는 스프링 컨텍스트를 일부 실제로 로드한다고 했습니다. 따라서 컴포넌트 스캔이 일어나게 되는데 우리가 작성한 시큐리티 코드에
`@Component` 가 달려 있기 때문에 같이 스캔됩니다. 하지만 그 중 일부 필요한 빈들이 로드가 되지 않아 오류가 발생했습니다.
따라서 옵션을 조절해 스프링 시큐리티를 제외해줬습니다. 시큐리티 동작까지 포함시키기엔 테스트 목적이 컨트롤러의 요청 매핑과 응답 형식 검증이라
적합하지 않았으며 시큐리티 자체는 따로 테스트가 가능하기 때문에 분리를 결정했습니다.

추가로 `@WithMockUser` 어노테이션으로 인증된 사용자 시뮬레이션도 가능합니다.
여기서 로드되는 스프링 컨텍스트란, 즉 `@WebMvcTest` 어노테이션이 로드하는 부분은
`@Controller`, `@RestController`, `@ControllerAdvice`, `@JsonComponent`, `Filter`, `WebMvcConfigurer` 등
웹 계층 관련 빈 입니다.
`@Service`, `@Repository`, `@Component` 등 비즈니스, 데이터 계층 빈은 로드 되지 않습니다.

**Q14.** `jsonPath("$[0].title").value("제목")`이 의미하는 바를 설명하세요.

**A14.**

jsonPath 는 JSON 의 응답구조를 탐색하는 경로 표현식입니다.
JSON 은 하나의 오브젝트 일 수도 있고 여러 오브젝트일 수도 있습니다. 각 오브젝트에는 키-값 쌍이 존재합니다.
여러 오브젝트가 있는 경우 이를 배열로 처리할 수 있습니다.
문법의 제일 앞에 오는 $ 는 JSON 의 최상위 경로를 의미합니다. 그 다음 [0], 배열이 왔기 때문에 이 JSON 은 여러 오브젝트로 이루어져있을 것 입니다.
배열에서 0번 인덱스는 제일 첫번째 원소를 의미하며 0번 째 오브젝트의 title 키 값을 의미하게 됩니다.
키-값 쌍이므로 뒤에 오는 .value("제목") 의 제목은 title 필드에 해당하는 값이 될 것 입니다.

## 6. 통합 테스트 (@SpringBootTest)

**Q15.** 통합 테스트에서 `@Transactional`을 붙이면 프로덕션 코드와 다르게 동작합니다. 어떻게 다르고, 왜 그렇게 설계되었나요?

**A15.**

프로덕션 코드에서 트랜잭션은 해당 메서드 동작을 트랜잭션 취급하도록 만듭니다.
테스트 코드에서도 마찬가지인데 한가지 다른 점은 트랜잭션의 모든 동작이 끝나면 값을 롤백하도록 되어 있습니다.
여러 테스트에 걸쳐서 일관된 테스트 환경을 유지하기 위해 하나의 테스트 실행 후 변경된 상태를 원래대로 돌리기 위해 설계되었습니다.

**Q16.** 통합 테스트에서 게시글을 생성하려면 먼저 회원을 저장해야 했습니다. 단위 테스트에서는 이 과정이 필요 없었는데 왜 그런가요?

**A16.**

통합 테스트는 모든 애플리케이션의 구성요소를 모두 사용해 테스트하기 때문에 실제 애플리케이션 동작 처럼 회원 저장의 과정도 필요했습니다.
단위 테스트는 대상 되는 클래스의 로직만 테스트하기 때문에 mock 을 사용해 테스트하고자 하는 로직 중 수행되는 협력 객체의 기대되는 동작을 정의하기만 하면 결과를 얻을 수 있었습니다.
springsecurity 등의 다른 모듈을 로드하지 않기 때문에 회원 저장 없이 곧바로 게시글 생성에 관한 테스트를 수행 가능했습니다.

**Q17.** 통합 테스트에서 ID를 하드코딩하면 안 되는 이유를 설명하세요.

**A17.**

ID 의 생성전략이 DB 쪽에 있기 때문입니다. ID 를 Auto increment 로 처리하고 있기 때문에 id 는 애플리케이션에서 처리되고 있지 않습니다.
테스트를 진행하면 상태 변화를 초기화 시킬 필요가 있는데 DB 는 해당 사실을 알지 못한채 엔티티가 생성될 때마다 id 값을 올리게 됩니다.
따라서 통합 테스트에서 ID 를 하드코딩하지 않고 테스트에 쓰이는 엔티티에서 직접 id 를 뽑아서 테스트를 수행해야 정상적인 테스트가 진행됩니다.

## 7. 커버리지와 테스트 전략

**Q18.** 테스트 커버리지란 무엇이고, JaCoCo와 IntelliJ 커버리지의 차이는 무엇인가요?

**A18.**

테스트 커버리지는 테스트가 해당 코드의 동작을 빠짐없이 다 점검하고 있는지에 관한 지표입니다.

JaCoCo 는 Gradle 플러그인으로 실행됩니다. 테스트 커버리지를 측정하면 이를 HTML 리포트 파일로 생성해주기 때문에 이 파일을 활용해
CI/CD 파이프라인에서 자동 검증을 하거나 팀에게 공유하기가 용이합니다. 빌드 도구만 있으면 어디서든 실행이 가능한 점도 장점입니다.

인텔리제이, IDE 가 기본제공하는 커버리지 측정 모듈도 있습니다. JaCoCo 와 다르게 따로 리포트 파일을 생성해주지 않고 IDE 에디터 상에서
표시해줍니다. 개발 중간에 즉시 확인하는데는 용이하지만 인텔리제이라는 환경에 종속되게 됩니다.

**Q19.** 단위 테스트와 통합 테스트에 모든 케이스를 중복 작성할 필요가 없는 이유를 설명하세요. 각각 어떤 케이스에 집중해야 하나요?

**A19.**

단위 테스트와 통합 테스트는 검증 목적이 서로 다릅니다.
단위 테스트는 비즈니스 로직의 모든 분기를 철저히 검증하는게 목적이며 통합 테스트는 계층 간 협력이 실제로 동작하는지 주요 흐름을 테스트 합니다.
따라서 단위 테스트에서 이미 검증한 모든 예외 케이스를 통합 테스트에서 반복할 필요가 없습니다.
각자의 목적에 맞는 테스트 로직만 가지면 됩니다.

## 8. 기타 개념

**Q20.** `ReflectionTestUtils.setField()`가 private 필드에 값을 넣을 수 있는 원리를 설명하세요. 프로덕션 코드에서 사용하지 않는 이유도 함께 설명하세요.

**A20.**

리플렉션은 자바에서 제공하는 기능으로 런타임에 클래스의 구조를 조사하고 조작하는 기능입니다.
리플렉션 코드를 실행시키면 리플렉션은 해당 객체의 필드를 탐색하고 접근 제한자를 해제해 필드를 조작할 수 있게 해줍니다.
getDeclaredField() 메서드로 원하는 필드를 탐색하며
setAccessible(true) 메서드로 접근제어자를 변경하고
set() 메서드로 값을 바꿉니다.
이는 java의 설계의도를 고의적으로 무시하는 행위기 때문에 프로덕션 코드에서는 캡슐화를 위해 금지하며, 테스트 코드 등 아주 제한된 일부 상황에서만
편의를 위해 사용하는 기능입니다.
본래 private 으로 선언된 필드는 외부에서 접근할 수가 없는데 리플렉션은 이 규칙을 우회할 수 있습니다.

리플렉션에 대해 좀 더 구체적으로 보자면
자바에는 두 가지 레벨의 규칙이 있습니다. 하나는 컴파일 타임 규칙이며 하나는 런타임 기능 입니다.
private 같은 접근제어자는 컴파일 타임 규칙입니다. 따라서 IDE 등에서 private 에 대한 규칙에 맞지 않은 코드를 작성하면 컴파일러 에러, 경고 처리를 해줄 수 있는겁니다.
리플렉션의 경우 컴파일 타임 규칙이 아니라 런타임 기능에 속합니다.
JVM 이 실행 중에 클래스 구조를 들여다보는 별도의 API 가 리플렉션 입니다. setAccessible(true) 는 private 라는 규칙을 알고 있지만 필요에 의해 이 규칙을 우회하겠다는 의미가 됩니다.

이런 런타임 우회 기능이 필요한 이유는 프레임워크 개발에 있습니다. Spring 의 @Autowired, JPA 필드 주입 등 모두 내부적으로 Reflection 을 활용하여 구현 됩니다.

**Q21.** `deprecated`란 무엇이고, deprecated된 기능을 바로 삭제하지 않는 이유는 무엇인가요?

**A21.**

`deprecated` 는 더 이상 지원하지 않는다는 의미입니다.
프레임워크가 발전하면서 과거에 있던 기능들이 개선되거나 삭제될 수 있습니다. 이 과정에서 더 발전된 기능이 등장하거나 보안 상의 이유로 사용되지
말아야할 기능들이 있으면 이를 deprecated 로 분류해 IDE 가 코드 작성 시 해당 기능을 사용하지 않도록 권합니다.
곧바로 기능을 삭제하지 않는 이유는 이미 deprecated 되기 이전에 해당 기능을 사용한 프로그램들과의 호환성 때문입니다.
보통 deprecated 로 분류되고 몇번의 버전 업그레이드를 거쳐 시간이 충분히 지나면 제거됩니다.
