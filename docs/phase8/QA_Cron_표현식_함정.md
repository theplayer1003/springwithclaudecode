# Cron 표현식의 함정 — 6필드와 폭주의 원인

## 질문 배경

Phase 8 의 BatchScheduler 작성 후 *1분 간격 실행* 을 의도하고 작성:

```java
@Scheduled(cron = "* */1 * * * *")  // ← 1분 간격이라고 의도
public void runArchiveBatchJob() { ... }
```

실행 결과: **매 초마다 폭주**. SQL 로그가 *1초 간격으로 무한 반복*.

원인 분석에서 **Cron 표현식의 6필드 구조** 와 **초 필드의 함정** 이 드러남.

---

## 1. Spring 의 Cron 은 *6필드*

표준 cron (Unix crontab) 은 5필드:

```
분  시  일  월  요일
```

그러나 **Spring 의 cron 은 6필드** — 앞에 *초* 가 추가됨:

```
초   분   시   일   월   요일
*    *    *    *    *    *
```

### 각 필드의 범위

| 필드 | 범위 | 비고 |
|--|--|--|
| 초 | 0-59 | Spring 특유 |
| 분 | 0-59 | |
| 시 | 0-23 | |
| 일 | 1-31 | |
| 월 | 1-12 | |
| 요일 | 0-7 | 0과 7 모두 일요일 |

### 특수 문자

- `*` — *모든 값*
- `?` — *어떤 값이든* (보통 일/요일 중 하나에만 사용)
- `,` — *여러 값* (`1,15`)
- `-` — *범위* (`1-5`)
- `/` — *간격* (`*/5` = 5 단위마다)

---

## 2. 학습자가 본 폭주의 원인

학습자의 cron:

```
* */1 * * * *
```

각 필드 해석:

| 필드 | 값 | 의미 |
|--|--|--|
| 초 | `*` | **매 초마다** |
| 분 | `*/1` | 매 1분마다 (= 매 분) |
| 시 | `*` | 매 시간 |
| 일 | `*` | 매일 |
| 월 | `*` | 매월 |
| 요일 | `*` | 매 요일 |

→ **모든 조건을 만족하는 모든 시점에 실행** = *매 초마다 실행*.

### 함정의 본질

학습자가 *"매 1분마다"* 라는 의도로 `*/1` 을 적었는데, 이건 *분 필드만* 의 의미. **초 필드를 `*` 로 두면 *그 분 안의 매 초마다* 실행**.

```
14:30:00 ← 실행 (초=0)
14:30:01 ← 실행 (초=1)
14:30:02 ← 실행 (초=2)
...
14:30:59 ← 실행 (초=59)
14:31:00 ← 실행 (초=0, 분=31)
...
```

→ *모든 초가 매칭됨* → *매 초마다 실행*.

---

## 3. 올바른 표현식 — *초 필드를 명시*

### 매 1분마다 실행하려면

```java
// 옵션 1: 매 분의 0초에 실행 (가장 흔한 표현)
@Scheduled(cron = "0 * * * * *")

// 옵션 2: 명시적으로 "1분 간격"
@Scheduled(cron = "0 */1 * * * *")
```

→ **초 필드를 `0` 으로 고정**. *매 분의 정각 1번만* 실행.

### 매 30초마다

```java
@Scheduled(cron = "*/30 * * * * *")
```

→ 초 필드 `*/30` = *0초, 30초 시점*.

---

## 4. 흔한 실수 표

| 의도 | 잘못된 표현식 | 실제 동작 | 정확한 표현식 |
|--|--|--|--|
| 매일 새벽 3시 | `* 0 3 * * *` | 3시대의 *매 초마다* 3600번 실행 | `0 0 3 * * *` |
| 매 시간 정각 | `0 0 * * * *` | 의도대로 ✅ | (그대로) |
| 매 1분마다 | `* */1 * * * *` ← 학습자 케이스 | *매 초마다* | `0 */1 * * * *` |
| 매 5분마다 | `* */5 * * * *` | *5분 단위의 분 안의 매 초마다* | `0 */5 * * * *` |
| 매주 일요일 자정 | `* 0 0 * * 0` | 일요일 자정 *1분 동안 매 초* | `0 0 0 * * 0` |

### 패턴 정리

> ***"매 시점에 한 번만"* 을 원하면 *그 단위보다 작은 모든 필드를 명시적 값으로 고정*.**

- 매 1분 → 초 = 0
- 매 1시간 → 초 = 0, 분 = 0
- 매일 → 초 = 0, 분 = 0, 시 = N (원하는 시각)

## 5. Spring 의 매크로 표현 — *안전한 대안*

Spring 은 자주 쓰는 패턴을 *매크로 표현* 으로 제공:

```java
@Scheduled(cron = "@yearly")    // 0 0 0 1 1 *  (매년 1월 1일 자정)
@Scheduled(cron = "@monthly")   // 0 0 0 1 * *  (매월 1일 자정)
@Scheduled(cron = "@weekly")    // 0 0 0 * * 0  (매주 일요일 자정)
@Scheduled(cron = "@daily")     // 0 0 0 * * *  (매일 자정)
@Scheduled(cron = "@hourly")    // 0 0 * * * *  (매 시간 정각)
```

표준적 사용에는 매크로가 *오류 가능성을 줄여줌*. *학습/실험* 용으로 *비표준 간격* 이 필요할 때만 직접 cron 작성.

---

## 6. 다른 어노테이션 옵션 — `fixedRate`, `fixedDelay`

`@Scheduled` 는 cron 외에도 *간격 기반 스케줄링* 지원:

### fixedRate — *고정 간격*

```java
@Scheduled(fixedRate = 60_000)  // 60초마다 (실행 시작 시점 기준)
public void task() { ... }
```

- 이전 실행이 끝나기 전에 다음 실행 시작 가능 (동시 실행)
- 시계 시각 기반이 아닌 *상대 간격*

### fixedDelay — *완료 후 대기*

```java
@Scheduled(fixedDelay = 60_000)  // 이전 실행 완료 후 60초 대기
public void task() { ... }
```

- 이전 실행 *완료 시점부터* 다음 대기 시작
- 동시 실행 없음

### 어느 걸 쓸까

- **시각 기반** (매일 새벽 3시 같은): `cron`
- **간격 기반, 동시 실행 가능**: `fixedRate`
- **간격 기반, 순차 실행만**: `fixedDelay`

배치 처리는 보통 *시각 기반* — cron 적합.

## 7. 검증 도구

### 외부 cron 시뮬레이터

웹 도구 (예: crontab.guru) 로 cron 표현식의 *다음 실행 시점* 을 미리 확인. 단, **이런 도구는 보통 5필드 cron 기준**. Spring 의 6필드는 *초 필드를 제외* 하고 검증해야 함.

### Spring 의 cron 파싱 검증

코드로 직접 확인:

```java
CronTrigger trigger = new CronTrigger("0 0 3 * * *");
SimpleTriggerContext context = new SimpleTriggerContext();
Date next = trigger.nextExecutionTime(context);
System.out.println("다음 실행: " + next);
```

배포 전 cron 표현식의 *다음 실행 시점이 의도와 맞는지* 검증하는 패턴.

### 로그로 검증

학습 단계에서는 *cron 을 짧은 간격으로 변경 후 실행 횟수 확인* 도 유효. 단, 변경 후 *원상 복구* 필수.

---

## 8. 메타 통찰

### Cron 의 함정 — *한 글자 차이로 수천 배의 실행 빈도*

```
"매일 새벽 3시"            : 0 0 3 * * *     (하루 1회)
"3시대의 매 초마다"         : * 0 3 * * *     (하루 3600회)
                              ↑
                              차이는 *단 한 글자*
```

이런 *작은 문법 차이* 가 *큰 동작 차이* 로 이어지는 함정. 학습자가 *실제로 마주친* 것 — *직접 코드를 쳐보는 경험* 의 가치를 보여주는 사례.

### 실수의 가시성

학습자가 *잘못된 cron 으로 폭주를 직접 봄* → *그 함정을 평생 잊지 않게 됨*. *개념만 짚고 넘어간 부분이 실제 코드 작성 시 실수로 드러난다* 는 학습 자세의 정확함이 확인된 순간.

### *학습용 vs 실전* 의 의식적 분리

학습 단계의 패턴:
1. 실전 cron 을 *먼저 작성* (`0 0 3 * * *`)
2. *주석 또는 변경* 으로 학습용 빠른 트리거 (`*/30 * * * * *`)
3. 검증 완료 후 *원상복구*
4. 커밋

이 패턴은 *cron 만의 이야기가 아님*. 실무에서 *디버그 코드, 로깅, 테스트 우회 코드* 등을 임시로 추가 후 원상복구 하는 모든 경우에 적용. **원상복구를 잊지 않는 것** 이 핵심.

### 도구는 도구다 — *프레임워크의 의도 의식*

> **`@Scheduled` 의 cron 은 *시각 기반 스케줄링*. 매 초 폭주는 *프레임워크의 버그가 아니라 사용자의 표현 실수*.** 프레임워크는 *시키는 대로 정확히 실행* 했을 뿐.

도구를 비난하기 전에 *내가 의도한 표현이 정확한가* 를 점검. *Cron 같은 작은 문법* 도 *읽을 수 있어야* 진단 가능.
