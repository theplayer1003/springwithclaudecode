# 아카이빙의 본질 — 스냅샷이라는 개념

## 질문 배경

Phase 8 의 실전 배치로 *"매일 새벽 3시에 1년 이상 된 게시글을 ArchivedPost 테이블로 옮긴다"* 시나리오를 진행하던 중, `ArchivedPost` 엔티티 설계에서 *Post 의 데이터를 토씨 하나 바뀌지 않고 그대로 옮겨야 한다* 라는 직관 때문에 `comments`, `member` 연관관계까지 그대로 복제하려 했음.

이 직관이 *왜 부분적으로만 맞고 어디서 어긋나는가* 를 풀어가며, 아카이빙이라는 행위의 *목적* 과 *스냅샷의 본질* 을 정리한 문서.

---

## 1. 아카이빙이란 무엇인가

> **메인 서비스에서 사용되지 않는 데이터임이 확실해지면, 삭제하기는 그렇고 그대로 들고 있기도 그런 데이터를 치워두는 행위.**

핵심은 *"치워둔다"* 의 동기. 한 가지가 아니다.

### 활용 시나리오 (5가지)

| # | 시나리오 | 빈도 | 특징 |
|--|--|--|--|
| 1 | **법적/규제적 보관 (compliance, audit)** | 거의 안 꺼냄 | 일정 기간 보관 의무. *있었다는 사실* 만 증명되면 됨 |
| 2 | **사용자 복구 요청** | 가끔 | "1년 전에 쓴 글 다시 보고 싶어요". 본문 + 작성 시각 필요 |
| 3 | **데이터 분석 / 통계** | 정기적 (배치) | 본문보다는 메타데이터 (작성일, 카테고리, 작성자 ID) 가 주 |
| 4 | **운영 핫 테이블 가볍게 유지** | **항상** | 활성 데이터만 빠르게 다루기 위한 *시스템 살리기* |
| 5 | **소송/분쟁 대응** | 드물지만 결정적 | 본문 + 작성 시점의 작성자 정보 (시점 박제 필요) |

### 무엇이 *주된 동기* 인가

학습 흐름에서 도달한 통찰:

> *목적들이 다 부가적인, 어떤 상황에 일어난다는 가정에 대한 대비 용도 같다.*

학습자 관점에서는 맞는데, **규모 차원에서는 4번 (운영 테이블 가볍게 유지) 이 본질적 동기에 가깝다**.

- 데이터 1만 건일 때: 아카이빙은 *부가적*. 안 해도 됨.
- 데이터 1억 건일 때: 인덱스 크기, 쿼리 시간, 백업 시간, 메모리 캐시 효율 모두 무너짐. 아카이빙이 *생사의 문제*.

→ *부가적이냐 본질적이냐* 는 **시스템 규모에 따라 바뀐다**. 학습 프로젝트에서는 작아서 부가적으로 느껴질 뿐, 실무 대규모 서비스에서는 *시스템을 살리는 일*.

## 2. 스냅샷의 본질 — *살아있는 도메인과의 단절*

학습자의 초기 직관: *"Post 가 담고 있는 데이터가 토씨 하나 바뀌지 않고 그대로 ArchivedPost 로 옮겨져야 한다"*.

### 직관 정정

이 직관 자체는 틀린 게 아니다. **"그대로" 의 단위가 무엇인가** 를 정교화하면 된다.

> **"그대로" 의 단위는 *데이터 값* 이지 *관계 객체* 가 아니다.**

- `title`, `content`, `createdAt` 같은 *값* → 그대로 박제 (✅)
- `Member member`, `List<Comment> comments` 같은 *살아있는 객체와의 연결* → 박제가 아니라 *누수*

### 왜 관계 매핑이 누수인가

`Member member` 로 관계를 들고 있으면, *Member 의 username 이 바뀌면 아카이브된 글의 작성자도 따라 바뀐다*. 탈퇴하면 더 큰 문제 (cascade 설정에 따라 깨지거나 사라짐).

**관계는 살아있는 도메인의 일부다.** 그걸 들고 있으면 *스냅샷이 오염된다*.

진짜 스냅샷:

```java
private Long memberId;          // 식별자만 보존
private String authorUsername;  // 시점 박제 (옵션) — Member.username 이 바뀌어도 영향 없음
private String authorEmail;     // 시점 박제 (옵션)
```

→ *현재 Member 와 단절된, 그 시점의 값들*. 이게 스냅샷.

### 한 줄 정의

> **스냅샷 = 시간이 흘러 다른 데이터가 바뀌어도 영향받지 않는, 박제된 값의 집합.**

JPA 관계 매핑은 *반대 방향* — 서로 영향을 주고받기 위해 존재한다. 그래서 *스냅샷에 관계 매핑이 어울리지 않는 것* 은 우연이 아니라 **목적이 정반대** 라서 그렇다.

## 3. 판단 프레임워크 (재사용 가능)

다음 번에 아카이브 / 감사 로그 / 이벤트 소싱 / 이력 테이블 같은 *불변 기록* 을 설계할 때 사용할 기준:

| 분류 | 기본 정책 |
|--|--|
| **기본값** | 날것의 값만 박제 (id, 본문 값, 시간 메타데이터) |
| **추가 박제** | 특정 시나리오에서 필요한 *시점 값* 이 있을 때만 (예: 소송 대응 → `authorUsername` 시점 박제, 분석 목적 → `category_id`, `year`) |
| **금지** | 살아있는 도메인과의 *관계 매핑* (양방향 OneToMany, 객체 참조 ManyToOne) |

**특별 요구사항이 없으면 기본값** — 이게 디폴트.

---

## 4. 이번 학습 프로젝트의 결정

5가지 시나리오를 다 만족시키려 들면 무거워진다. 학습 목적에 맞춰 단순화:

- **주 목적**: 운영 테이블 가볍게 유지 (4번) + 최소한의 복구/감사 가능성 (1, 2번)
- **불필요**: 작성자 username/email 시점 박제, ArchivedComment, 분석용 메타데이터

→ `ArchivedPost` 의 최종 필드:

```java
@Id private Long id;              // 원본 Post.id 보존 (= PK)
private String title;
private String content;
private Long memberId;            // 객체가 아닌 ID 만
private LocalDateTime createdAt;
private LocalDateTime updatedAt;
private LocalDateTime archivedAt; // 메타데이터
```

## 5. 일반화되는 통찰

이 통찰은 *이벤트 소싱*, *감사 로그(audit log)*, *데이터 웨어하우스*, *이력 테이블* 같은 다른 영역에서도 똑같이 적용된다.

> **불변 기록을 만들 땐 살아있는 도메인과 끊는다.**

- 이벤트 소싱: 이벤트 객체에 *현재 도메인 참조 금지*. 모든 정보는 이벤트 자체에 박제.
- 감사 로그: *작성 시점의 사용자명* 을 텍스트로 박제. 사용자 정보 변경에 영향받지 않게.
- 데이터 웨어하우스: ETL 시점에 *비정규화* 하여 시점 스냅샷으로 저장.

같은 원칙이다 — *"기록은 그 시점의 박제, 살아있는 도메인의 변화에 흔들리지 않아야 한다"*.
