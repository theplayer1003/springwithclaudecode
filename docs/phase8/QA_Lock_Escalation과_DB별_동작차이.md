# QA — Lock Escalation 과 DB 별 동작 차이

> Phase 8 정리 문서 B1(Tasklet 으로 1억 건을 처리하면 안 되는 이유) 답변을 작성하던 중,
> *"X 락이 일정 수준 쌓이면 RDBMS 가 Row Lock 을 테이블 Lock 으로 승격시킨다(Lock Escalation)"* 라고 단언했다가
> **그 동작이 모든 RDBMS 의 공통 동작이 아니라는 사실**을 발견하며 정리한 함정.
>
> 핵심 교훈 한 줄: **"Lock 관련 동작은 DB 엔진마다 다르다 — 'RDBMS가 ~한다'는 단언 옆에는 항상 '어느 DB?'를 둬라."**

---

## 질문 배경

1억 건을 하나의 트랜잭션으로 처리하면 "트랜잭션 폭증"으로 DB 가 마비된다 — 여기까진 맞다.
그 메커니즘을 설명하면서 *"X 락이 누적되다 한계에 닿으면 RDBMS 가 행 락을 풀고 테이블 락으로 격상시킨다(Lock Escalation)"* 라고 적었다.

문제는 이 문장이 **특정 DB(SQL Server, DB2)의 동작을 모든 RDBMS 의 공통 동작인 양 일반화**했다는 점이다.
현재 프로젝트가 쓰는 H2 / MySQL(InnoDB) / PostgreSQL 은 **Lock Escalation 을 하지 않는다.**
즉 결론("DB 가 마비된다")은 맞지만 **그 원인으로 든 메커니즘이 우리 스택에서는 일어나지 않는** 상황이었다.

---

## 핵심 답변

### 1. Lock Escalation 이란

**락의 "개수"가 일정 임계치를 넘으면, 잘게 잡고 있던 행(Row) 락들을 풀고 더 큰 단위(페이지/테이블) 락 하나로 바꿔 잡는** DB 의 동작이다.

- **왜 이런 게 있나?** 락 하나하나가 메모리를 차지하기 때문이다. 수백만 개의 행 락을 메모리에 들고 있으면 **락 관리 메모리 자체가 부담**이 된다. 그래서 "차라리 테이블 통째로 잠그고 락 1개만 관리하자"는 트레이드오프를 선택하는 것.
- **대가:** 테이블 락이 걸리는 순간 그 테이블에 손대려는 다른 트랜잭션이 전부 막힌다(동시성 급락).

### 2. DB 별 동작 차이 (이 QA 의 핵심)

| DB 엔진 | Lock Escalation | 행 락을 어디에 저장하나 |
|---|---|---|
| **SQL Server** | **한다** (행/페이지 → 테이블로 승격) | 메모리(Lock Manager) — 그래서 개수 압박에 민감 → 승격으로 해소 |
| **DB2** | **한다** | 메모리 — 임계치 초과 시 승격 |
| **MySQL (InnoDB)** | **하지 않는다** | 페이지당 **비트맵**으로 효율적으로 표현 → 행 락이 많아도 메모리 압박이 작음 → 승격 불필요 |
| **PostgreSQL** | **하지 않는다** | 행 락 정보를 **튜플(행) 자체**에 기록(`xmax` 등) → 락 개수가 메모리를 누르지 않음 → 승격 개념 자체가 없음 |

> 핵심 인과: **"행 락을 메모리에 들고 있느냐"가 갈림길.** 메모리에 들고 있는 DB(SQL Server/DB2)는 개수 압박 때문에 승격이라는 탈출구가 필요하고, 메모리에 부담을 안 주는 방식으로 저장하는 DB(InnoDB의 비트맵, PostgreSQL의 튜플 기록)는 승격할 이유가 없다.

### 3. 그렇다면 InnoDB/PostgreSQL 에서 1억 건 단일 트랜잭션의 "진짜" 문제는?

Lock Escalation 으로 테이블이 잠겨서 마비되는 게 아니다. 진짜 문제는 이것들이다:

1. **장시간 락 점유** — 수백만 개의 X 락(행 락)을 **트랜잭션이 끝날 때까지** 그대로 붙잡는다. 같은 행을 건드리려는 다른 트랜잭션은 **lock wait timeout / 데드락**으로 줄줄이 실패.
2. **Undo Log(History list) 비대화** — 변경 전 원본이 계속 쌓이고, 트랜잭션이 안 끝나니 정리(purge)되지 않는다. 다른 트랜잭션의 MVCC 스냅샷 읽기가 **긴 Undo 체인**을 거슬러야 해서 느려지고, 디스크 I/O 가 폭증.
3. **Undo 테이블스페이스 / 복제 지연** — 거대 트랜잭션 하나가 Undo 공간을 잠식하고, 복제(replication)에서는 커밋될 때까지 적용이 밀려 **복제 지연(lag)** 을 유발.

→ 결론은 같다("DB 가 위태로워진다")**. 다만 메커니즘이 "테이블 락 승격"이 아니라 "장시간 락 점유 + Undo 비대화"** 라는 것.

### 4. 보너스 — MVCC 와 Undo Log 의 이중 역할 (왜 읽기는 안 막히나)

InnoDB/PostgreSQL 은 **MVCC(Multi-Version Concurrency Control)** 를 쓴다. 변경 전 원본을 Undo 에 스냅샷으로 남겨두기 때문에:

- **일반 SELECT(스냅샷 읽기)는 락을 안 잡는다.** 쓰기 트랜잭션이 행 락을 잔뜩 쥐고 있어도, 읽기는 Undo 의 과거 버전을 보고 지나간다. → "읽기와 쓰기가 서로를 막지 않는다."
- 그래서 Undo Log 는 **롤백용 + MVCC 읽기용**의 두 얼굴을 갖는다.
- 단, 위 2번처럼 Undo 체인이 길어지면 그 "막히지 않는 읽기"마저 **느려진다.**

## 재사용 가능한 교훈 (판단 프레임)

> **"RDBMS 가 ~한다"** 라는 단언을 쓰거나 들을 때, 자동으로 **"어느 DB 엔진 기준인가?"** 를 붙여라.
> Lock, 격리 수준의 구현, MVCC 유무, 인덱스 구조 등은 **표준이 아니라 엔진별 구현**이다.
> 특히 *Oracle/SQL Server 문서나 글*에서 배운 동작을 *MySQL/PostgreSQL* 에 그대로 옮기면 틀리기 쉽다 (Lock Escalation 이 대표 사례).
