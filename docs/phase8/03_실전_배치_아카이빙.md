# 실전 배치 — 오래된 게시글 아카이빙

## 1. 시나리오

> **매일 새벽 3시에, 1년 이상 된 게시글을 아카이빙 테이블로 옮긴다.**

배치 작업의 목적은 *비즈니스 로직 엔티티가 오래되어 더 이상 사용될 일이 없을 때 이를 삭제하는 것 대신 다른 테이블로 분리시켜두는 것*. 이를 통해 **비즈니스 로직에 직접적으로 쓰이는 테이블을 경량화** 해 검색 속도를 높이는 효과를 얻는다.

(아카이빙의 본질과 다양한 활용 시나리오에 대해서는 `QA_아카이빙_스냅샷.md` 참조.)

## 2. 전체 그림

```
[Scheduler] cron 만나면
   ↓
[JobLauncher] archivePostJob 실행
   ↓
[archivePostJob] → Step 실행
   ↓
[archivePostStep] chunk-oriented, chunkSize = 1000
   ↓
   청크 트랜잭션 시작
   ↓
   [Reader] cutoff 이전 Post 를 한 건씩 읽음
   ↓
   [Processor] Post → ArchivedPost 변환 (id 보존, archivedAt 설정)
   ↓
   [Writer]
     ├─ archivedPostRepository.saveAll(items)              ← INSERT
     └─ postRepository.deleteAllByIdInBatch(originalIds)   ← DELETE
   ↓
   COMMIT (둘 다 성공 or 둘 다 롤백)
   ↓
   다음 청크 → 반복 → 끝
```

> **참고**: Spring Batch 의 Step 은 *두 종류* 가 있다. *Tasklet 기반 Step* 과 *Chunk-oriented Step*. 이번 아카이빙 시나리오에서는 *대량 데이터를 흘려보내며 처리* 해야 하므로 **Chunk-oriented Step** 을 사용한다. (Tasklet 은 `01_Spring_Batch_입문_Tasklet.md` 참조.)

## 3. 테스트 데이터 — data.sql

배치 실행 검증을 위해 *두 그룹* 의 데이터를 삽입:

```sql
-- 옛 게시글 1500건 (아카이빙 대상)
INSERT INTO POST (title, content, member_id, created_at, updated_at)
SELECT 'post title' || X,
       'post content' || X,
       999,
       DATEADD('YEAR', -2, CURRENT_TIMESTAMP),
       DATEADD('YEAR', -2, CURRENT_TIMESTAMP)
FROM SYSTEM_RANGE(1, 1500);

-- 최신 게시글 1500건 (아카이빙 비대상)
INSERT INTO POST (title, content, member_id, created_at, updated_at)
SELECT 'post title' || X,
       'post content' || X,
       999,
       CURRENT_TIMESTAMP,
       CURRENT_TIMESTAMP
FROM SYSTEM_RANGE(1, 1500);
```

### 핵심 포인트

- **`SYSTEM_RANGE(1, 1500)`** — H2 가 제공하는 *범위 기반 SELECT*. *1부터 1500까지 1500개의 행을 만들어* INSERT 의 소스로 사용.
- **`DATEADD('YEAR', -2, ...)`** — H2 의 시간 함수. *2년 전 시각* 으로 created_at 설정.
- *대상/비대상 두 그룹 분리* — 배치 후 *대상만 옮겨졌는지* 검증 가능.

## 4. ArchivedPost 엔티티 — 스냅샷의 본질

### 설계 결정

```java
@Entity
public class ArchivedPost implements Persistable<Long> {

    @Id
    private Long id;                  // ← 원본 Post 의 id 그대로 보존

    private String title;
    private String content;
    private Long memberId;            // ← 객체 관계가 아닌 ID 만 보존
    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
    private LocalDateTime archivedAt; // ← 아카이빙 메타데이터

    // ...
}
```

### 왜 *별도 엔티티* 가 필요한가

두 가지 이유:

1. **JPA 의 엔티티-테이블 1:1 매핑 원칙**
   - JPA 는 *한 엔티티 = 한 테이블*. *Post* 객체를 사용하려면 *post 테이블* 에 접근해야 함. 테이블 분리가 안 됨.
   - 별도 테이블에 저장하려면 *별도 엔티티* 가 필수.

2. **불필요한 데이터 제거 + 메타데이터 추가**
   - 살아있는 도메인의 *관계 (Comment, Member 객체 참조)* 는 *스냅샷에 불필요* — 오히려 *오염 위험*.
   - *archivedAt* 같은 아카이빙 *전용 메타데이터* 추가 가능.

### Persistable 구현 — *왜 필요한가*

`ArchivedPost` 는 **`Persistable<Long>` 인터페이스를 구현** 한다:

```java
@Override
public boolean isNew() {
    return true;  // 항상 새 엔티티로 취급
}
```

이유: *`@Id` 가 미리 채워진 엔티티* 를 `saveAll` 할 때, JPA 의 기본 `isNew()` 판정은 *id 가 null 아니므로 기존 엔티티* 로 판단 → `merge` 동작 → *DB 에서 SELECT 후 INSERT/UPDATE*. **불필요한 SELECT** 발생.

`Persistable.isNew()` 를 `true` 로 명시하면 *항상 persist (= INSERT)* 만 실행. SELECT 가 사라진다.

(자세한 내용은 `QA_save_isNew_Persistable.md` 참조.)

## 5. Reader — 페이지 시프트 함정과 Keyset 해법

### 첫 시도 — RepositoryItemReader

```java
@Bean
public RepositoryItemReader<Post> archivePostReader() {
    return new RepositoryItemReaderBuilder<Post>()
        .name("archivePostReader")
        .repository(postRepository)
        .methodName("findByCreatedAtBefore")
        .arguments(LocalDateTime.now().minusYears(1))
        .pageSize(500)
        .sorts(Collections.singletonMap("id", Direction.ASC))
        .build();
}
```

`findByCreatedAtBefore(cutoff, Pageable)` 메서드를 호출 → `LIMIT pageSize OFFSET pageSize*page` 방식으로 페이징.

### 함정 — OFFSET 페이지 시프트

학습자 시나리오 (chunk 1000, page 500, 옛 데이터 1500건) 의 실제 진행:

```
[청크 1]
  Reader: page 0 (OFFSET 0) → id 1~500
  Reader: page 1 (OFFSET 500) → id 501~1000
  Processor + Writer 처리 → id 1~1000 아카이빙 + 삭제
  COMMIT
  
  ↑ 이 시점 DB 상태: 옛 데이터는 id 1001~1500 (500건) 만 남음

[청크 2]
  Reader: page 2 (OFFSET 1000)
  → 현재 cutoff 이전 데이터는 500건 (id 1001~1500) 뿐
  → OFFSET 1000 으로는 0건 반환
  Reader 가 null 반환 → Step 종료
  
  ↑ id 1001~1500 (500건) 이 건너뛰어짐!
```

### 함정의 본질

> **OFFSET 은 *위치 기반*. 데이터가 변하면 위치의 의미도 변한다.**

- *Reader 가 자신의 페이지 카운터 (page) 를 들고 진행*
- *현실 DB 의 데이터는 변함* (DELETE 됨)
- → *카운터의 위치* 와 *실제 데이터의 위치* 가 어긋남

이게 *읽으면서 같은 데이터를 변경하는 모든 시나리오* 에서 일어나는 본질적 함정.

(자세한 내용은 `QA_OFFSET_페이지_시프트_Keyset.md` 참조.)

### Keyset Pagination 의 본질

> **OFFSET 의 *위치* 대신, *변하지 않는 값 (id)* 을 기준으로 진행한다.**

```sql
-- OFFSET 방식 (변하는 위치)
WHERE created_at < :cutoff ORDER BY id LIMIT 500 OFFSET 500

-- Keyset 방식 (변하지 않는 값)
WHERE created_at < :cutoff AND id > :lastSeenId ORDER BY id LIMIT 500
```

`id > :lastSeenId` 조건은 *데이터가 사라져도 의미가 변하지 않음*. *식별자의 값* 이라는 *절대 기준*.

### 커스텀 Reader 구현 — NoOffSetItemReader

Spring Batch 의 기본 Reader 중 Keyset 을 지원하는 게 없어서 **`ItemReader` 인터페이스를 직접 구현**:

```java
public class NoOffSetItemReader implements ItemReader<Post> {

    private final PostRepository postRepository;
    private final LocalDateTime cutoff;
    private final int pageSize;

    private Long lastSeenId = 0L;
    private Iterator<Post> currentPageIterator = Collections.emptyIterator();

    @Nullable
    @Override
    public Post read() throws Exception {
        if (!currentPageIterator.hasNext()) {
            fetchNextPage();
        }
        if (!currentPageIterator.hasNext()) {
            return null;
        }
        Post nextPost = currentPageIterator.next();
        this.lastSeenId = nextPost.getId();
        return nextPost;
    }

    private void fetchNextPage() {
        PageRequest pageRequest = PageRequest.of(0, pageSize);  // ← page 0 고정
        List<Post> posts = postRepository.findByCreatedAtBeforeAndIdGreaterThanOrderByIdAsc(
            cutoff, lastSeenId, pageRequest);
        if (!posts.isEmpty()) {
            this.currentPageIterator = posts.iterator();
        } else {
            this.currentPageIterator = Collections.emptyIterator();
        }
    }
}
```

### 상태 관리의 본질

이 Reader 가 **상태를 가집니다**:
- `lastSeenId` — 마지막으로 읽은 아이템의 id
- `currentPageIterator` — 현재 페이지의 버퍼

**Reader 의 버퍼링 패턴**:
- *DB 통신은 페이지 단위로 묶음* (500개를 한 번에)
- *외부 (Spring Batch) 에는 한 개씩 노출* (Iterator 의 `next()`)
- → *DB 왕복 줄이면서 흐름은 한 건씩*

### Iterator 패턴의 선택

*for 문* 으로 한 번에 처리하지 않고 *Iterator* 를 사용하는 이유:

- `ItemReader.read()` 는 *외부 (Spring Batch) 에서 호출* 받는 구조
- *호출 사이에 상태 유지* 필요
- *Iterator 가 List + 위치 추적을 한 객체로 캡슐화* → 자연스러움

자세한 내용은 `QA_OFFSET_페이지_시프트_Keyset.md` 참조.

## 6. PostRepository — 메서드 이름 쿼리

NoOffSetItemReader 가 사용하는 메서드:

```java
List<Post> findByCreatedAtBeforeAndIdGreaterThanOrderByIdAsc(
    LocalDateTime cutoff, Long lastSeenId, Pageable pageable
);
```

### 메서드 이름 쿼리의 파싱

Spring Data JPA 가 메서드 이름을 *토큰 단위로 파싱* 해서 JPQL 생성:

```
findByCreatedAtBeforeAndIdGreaterThanOrderByIdAsc
├── find                ← SELECT
├── By                  ← WHERE 절 시작
├── CreatedAtBefore     ← p.createdAt < ?
├── And                 ← AND
├── IdGreaterThan       ← p.id > ?
└── OrderByIdAsc        ← ORDER BY p.id ASC
```

자동 생성되는 JPQL:
```jpql
SELECT p FROM Post p
WHERE p.createdAt < ?1 AND p.id > ?2
ORDER BY p.id ASC
```

### 왜 `Page<Post>` 가 아닌 `List<Post>` 인가

Keyset 페이징에는 *총 개수가 의미 없음* — *페이지 개념 자체가 사라짐*. `Page<T>` 는 *총 개수 쿼리도 함께 날리는 비용* 이 있어서 *불필요*.

→ `List<Post>` 로 받는 게 적합.

(자세한 내용은 `QA_Spring_Data_JPA_메커니즘.md` 참조.)

## 7. Processor — Post → ArchivedPost 변환

```java
@Bean
public ItemProcessor<Post, ArchivedPost> archivePostProcessor() {
    return post -> new ArchivedPost(
        post.getId(),         // ← 원본 id 보존
        post.getTitle(),
        post.getContent(),
        post.getCreatedAt(),
        post.getUpdatedAt(),
        LocalDateTime.now(),  // ← archivedAt = 변환 시점
        post.getAuthorId()    // ← Member 객체가 아닌 id 만
    );
}
```

### 핵심 포인트

- **원본 id 보존** — Writer 에서 *원본 Post 의 id 추적* 에 사용
- **`LocalDateTime.now()` 의 위치** — *람다 안* 이라 *호출 시점마다 새 값*. Bean 시점 고정 함정을 자연스럽게 회피.
- **`post.getAuthorId()`** — Member 객체가 아닌 *id 만* 보존. 스냅샷 본질.
- **Stateless** — 외부 상태에 의존하지 않는 *순수 함수* 처럼 동작.

### 람다와 함수형 인터페이스

`ItemProcessor<I, O>` 는 *함수형 인터페이스 (SAM)*. 추상 메서드가 *`process(I item)` 하나* 뿐이라 람다로 구현 가능.

```java
// 람다는 다음과 동등
new ItemProcessor<Post, ArchivedPost>() {
    @Override
    public ArchivedPost process(Post post) {
        return new ArchivedPost(...);
    }
}
```

## 8. Writer — INSERT + DELETE 의 함께 처리

```java
@Bean
public ItemWriter<ArchivedPost> archivePostWriter() {
    return chunk -> {
        List<? extends ArchivedPost> items = chunk.getItems();
        
        archivedPostRepository.saveAll(items);  // INSERT (벌크)
        
        List<Long> originalIds = items.stream()
            .map(ArchivedPost::getId)
            .toList();
        postRepository.deleteAllByIdInBatch(originalIds);  // DELETE (벌크)
    };
}
```

### Chunk 트랜잭션 안의 INSERT + DELETE

Writer 의 *두 작업이 한 chunk 트랜잭션* 안에서 일어남:

```
BEGIN TRANSACTION (chunk 시작)
   Reader.read() x 1000번
   Processor.process() x 1000번
   Writer.write()
       saveAll(items)              ← INSERT
       deleteAllByIdInBatch(ids)   ← DELETE
COMMIT (chunk 끝)
```

*INSERT 와 DELETE 가 함께 성공하거나 함께 롤백* — **원자성 보장**. 정합성 어긋남 (한쪽만 일어남) 이 발생하지 않음.

> Writer 의 트랜잭션이 *별도로* 일어나는 게 아니다. *Reader/Processor/Writer 가 모두 같은 chunk 트랜잭션* 안에서 동작.

### `Chunk<? extends T>` 의 의미

`ItemWriter<ArchivedPost>` 의 `write` 시그니처:

```java
void write(Chunk<? extends ArchivedPost> chunk);
```

`? extends ArchivedPost` 는 *공변성을 위한 와일드카드*. *ArchivedPost 또는 그 하위 타입의 chunk* 도 받을 수 있음.

→ `chunk.getItems()` 의 반환 타입이 `List<? extends ArchivedPost>`. 변수 선언 시 *같은 와일드카드* 를 써야 함.

자세한 내용은 `QA_제네릭_와일드카드_PECS.md` 참조.

### 벌크 메서드의 선택

- **`saveAll`**: 단건 `save` 반복보다 효율. `hibernate.jdbc.batch_size` 설정 시 JDBC 배치 INSERT.
- **`deleteAllByIdInBatch`**: *단일 DELETE IN 쿼리*. *영속성 컨텍스트 우회*, SELECT 없이 바로 DELETE.

```sql
-- deleteAllByIdInBatch 가 만드는 쿼리
DELETE FROM post WHERE id IN (1, 2, 3, ..., 1000);
```

→ 1000건 삭제가 *1번의 쿼리*. *영속성 컨텍스트의 추상화* 를 의도적으로 우회해서 *성능* 확보. 트레이드오프.

자세한 내용은 `QA_deleteById_vs_InBatch.md` 참조.

## 9. Step / Job / Scheduler

### Step

```java
@Bean
public Step archivePostStep(JobRepository jobRepository,
                              PlatformTransactionManager transactionManager,
                              ItemReader<Post> archivePostReader,
                              ItemProcessor<Post, ArchivedPost> archivePostProcessor,
                              ItemWriter<ArchivedPost> archivePostWriter) {
    return new StepBuilder("archivePostStep", jobRepository)
            .<Post, ArchivedPost>chunk(1000, transactionManager)
            .reader(archivePostReader)
            .processor(archivePostProcessor)
            .writer(archivePostWriter)
            .build();
}
```

`.<Post, ArchivedPost>chunk(1000, transactionManager)` — Reader 의 출력 타입과 Processor 의 출력 타입을 제네릭으로 명시.

### Job

```java
@Bean
public Job archivePostJob(JobRepository jobRepository, Step archivePostStep) {
    return new JobBuilder("archivePostJob", jobRepository)
            .start(archivePostStep)
            .build();
}
```

### Scheduler

```java
@Scheduled(cron = "0 0 3 * * *")  // 매일 새벽 3시
public void runArchiveBatchJob() {
    try {
        String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
        JobParameters jobParameters = new JobParametersBuilder()
                .addString("requestDate", now)
                .toJobParameters();
        jobLauncher.run(archivePostJob, jobParameters);
    } catch (Exception e) {
        // 실전에서는 로깅 필요
    }
}
```

> **JobParameters 의 `requestDate`** — 매 실행마다 *다른 시각* 이라서 *새 JobInstance* 로 인식됨. Spring Batch 가 *같은 JobInstance 의 중복 실행* 을 막는 메커니즘.

> ⚠️ **Cron 표현식 6필드 함정** — 초 필드를 `*` 로 두면 *매 초마다 실행* 됨. 학습 단계에서 실제로 마주친 함정. 자세한 내용은 `QA_Cron_표현식_함정.md` 참조.

## 10. 검증 방법

### SQL 로그 (`spring.jpa.show-sql=true`)

배치 실행 후 보아야 할 쿼리:

```
1. Reader: SELECT p FROM Post p WHERE p.createdAt < ? AND p.id > ? ORDER BY p.id LIMIT 500
   ← Keyset 페이징, OFFSET 없음

2. Writer (saveAll): INSERT INTO archived_post (id, title, ...) VALUES (?, ?, ...)
   ← Persistable 효과로 SELECT 없이 INSERT 만

3. Writer (deleteAllByIdInBatch): DELETE FROM post p WHERE p.id IN (?, ?, ..., ?)
   ← 단일 IN 쿼리
```

### DB 상태 검증 (H2 콘솔)

```sql
SELECT COUNT(*) FROM POST;           -- 비대상만 남아야 함 (1500건)
SELECT COUNT(*) FROM ARCHIVED_POST;  -- 1500건 (모두 아카이빙됨)

-- 데이터 손실 검증
SELECT (SELECT COUNT(*) FROM POST) + (SELECT COUNT(*) FROM ARCHIVED_POST);
-- 예상: 3000 (원본 데이터 수와 일치)
```

### 디버깅의 함정 — *도구의 표시 제한*

학습 중 마주친 함정: H2 콘솔의 *최대 출력 1000건* 제한. 3000건 데이터가 있는데 콘솔에서 *1000건만 보여* *데이터가 사라진 것처럼* 보이는 현상.

> **메타 통찰**: *"내가 본 것 ≠ 실제 상태"*. 디버깅 도구의 *기본 제한* 을 의식 못하면 *허상의 문제* 를 추적하게 됨. 항상 도구의 표시 제한을 의식.

## 11. 메타 통찰 정리

### 추상화의 한계와 우회

이번 시나리오에서 *JPA 의 추상화* 를 두 번 우회했다:

1. **Persistable** — *@Id 가 미리 채워진 엔티티는 기존 엔티티* 라는 JPA 의 기본 가정을 우회. *항상 새 엔티티* 임을 명시.
2. **deleteAllByIdInBatch** — *영속성 컨텍스트를 통한 DELETE* 의 추상화를 우회. *직접 DELETE 쿼리* 발급.

> **JPA 의 추상화는 *보통의 경우* 에 맞게 설계됨. 비표준 시나리오 (대량 배치, id 직접 관리) 에서는 *의도적 우회* 가 필요.**
> 우회용 사다리 (Persistable, InBatch 메서드) 가 *프레임워크에 의도적으로 마련되어 있음*. 우회 자체가 부정적인 게 아님.

### 위치 기반 vs 값 기반

페이지 시프트 함정의 본질:
- **위치 기반 (OFFSET)**: 데이터가 변하면 의미가 변함
- **값 기반 (Keyset)**: *id 같은 절대 식별자* 위에서 진행. 변화에 면역

> **상태를 어떻게 표현할 것인가가 문제의 본질.** 같은 진행을 *위치 (page)* 로 표현하느냐, *값 (lastSeenId)* 으로 표현하느냐. 표현 방식에 따라 *대응할 수 있는 변화* 가 달라짐.

### 학습 코드 ≠ 실전 코드

이번 시나리오에서 *학습용 단순화* 한 부분:
- `LocalDateTime.now().minusYears(1)` 을 Bean 시점에 박음 (실전: `@StepScope` + JobParameters)
- ItemStream 미구현 → 재시작 불가 (실전: 재시작 가능성 필요)
- 단일 Step (실전: 여러 Step + 조건부 분기)

> *학습 코드는 학습을 위해 단순화*. 실전 시나리오는 *추가적 고려* 가 필요함을 의식하고 가는 것이 *판단력의 시작*.
