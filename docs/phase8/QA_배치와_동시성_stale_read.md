# QA — 배치와 동시성: 락, MVCC, 그리고 stale read

> Phase 8 정리 문서 B4(page size / chunk size)를 다루던 중,
> *"Reader 가 읽을 때 락을 안 건다"* 는 사실에서 출발해
> **"읽은 뒤 다른 트랜잭션이 원본을 바꿔버리면 잘못된 배치 작업이 되는 것 아닌가?"** 라는 의문으로 이어진 흐름을 정리.
>
> 이 의문은 **틀린 게 아니라 동시성 제어의 핵심 분기점에 스스로 도달한 것**이며,
> Phase 10(낙관적/비관적 락)의 직접적인 예고편이다.

---

## 0. 먼저 — 락이란 무엇이고, 왜 읽기엔 원래 필수가 아닌가

락(Lock)을 "모든 DB 작업에 기본으로 걸리는 것"이라고 오해하기 쉽다. **아니다.**
락은 **"변경 충돌을 막기 위한 장치"** 다. 그러므로 충돌이 생길 수 있을 때만 필요하다.

| 동시 작업 조합 | 데이터를 바꾸나? | 락이 본질적으로 필요한가 |
|---|---|---|
| 읽기 ↔ 읽기 | 둘 다 안 바꿈 | **불필요** (충돌 없음) |
| 읽기 ↔ 쓰기 | 한쪽이 바꿈 | 충돌 가능 |
| 쓰기 ↔ 쓰기 | 둘 다 바꿈 | **필요** (충돌 확실) |

→ 락이 본질적으로 꼭 필요한 작업은 **쓰기**다. 읽기는 데이터를 바꾸지 않으므로 **원래 락이 필수가 아니다.**
"모든 작업에 락"이 아니라 **"변경하는 작업에 락"** 이 출발점이다.

### 그런데 왜 옛날엔 읽기에도 락을 걸었나

읽기 자체는 데이터를 안 바꾸지만, **"읽는 도중에 남이 바꾸면 이상한 값(절반만 바뀐 상태 등)을 읽을 수 있다."**
이 일관성 문제를 막으려고, MVCC 이전의 락 기반 DB 는 **읽을 때도 공유 락(S 락)** 을 걸어
*"내가 읽는 동안 아무도 쓰지 마"* 를 강제했다. → 그 결과 **읽기와 쓰기가 서로를 막아** 동시성이 떨어졌다.

---

## 1. MVCC — 읽기 락을 "없앤" 게 아니라 "더 나은 방법으로 대체"

MVCC(Multi-Version Concurrency Control, 다중 버전 동시성 제어)의 해법은
*"읽는 동안 쓰기를 막자"* 가 아니라:

> **쓰기는 새 버전을 만들고, 읽기는 과거의 안정된 버전(스냅샷)을 본다.**

그 결과:
- 읽기는 **락 없이도** 일관된 값을 본다 (변경 중인 최신이 아니라, **확정된 과거 스냅샷**).
- 쓰기는 읽기가 끝나기를 기다릴 필요가 없다.
- → **읽기와 쓰기가 서로를 막지 않는다** (동시성 ↑).

정확한 그림은 *"원래 다 락이 걸렸는데 MVCC 가 읽기 락을 뺐다"* 가 아니라,
**"읽기에 락을 걸던 낡은 방식을, MVCC 가 스냅샷이라는 더 나은 방식으로 대체했다"** 이다.

> 참고: InnoDB / PostgreSQL 의 일반 SELECT 가 락 없이 스냅샷을 읽는 것이 이 동작이다.
> 변경 전 원본은 Undo Log 에 보관되어 이 스냅샷의 재료가 된다 → [[QA_Lock_Escalation과_DB별_동작차이]] 에서 다룬 Undo Log 의 이중 역할.

---

## 2. 그 대가 — stale read (낡은 읽기)

MVCC 가 "최신 대신 안정된 과거 스냅샷"을 보여주는 순간, **부작용이 따라온다.**
읽은 이후에 원본이 바뀌어도, 트랜잭션은 **읽은 시점의 옛 값**으로 계속 작업한다. 이를 **stale read** 라 한다.

### 아카이빙 배치에서의 시나리오 (학습자가 직접 그린 것)

```
시각 →

t1  Reader:    게시글 #500 읽음 (제목 = "A")   ← 락을 걸지 않음 (MVCC 스냅샷)
                    ↓
t2  (그 사이) 다른 트랜잭션: #500 제목을 "B" 로 UPDATE + commit
                    ↓
t3  Processor: 제목 "A" 기준으로 ArchivedPost 가공
                    ↓
t4  Writer:    commit → 아카이브에는 "A" 가 저장됨
```

최신 값은 "B" 였는데, 아카이브에는 낡은 값 "A" 가 들어갔다.
**이것은 버그가 아니라, "읽기에 락을 안 거는" 성능 설계의 정해진 대가다.**
빠른 읽기(앞면)와 stale 가능성(뒷면)은 **하나의 선택에서 동시에 나오는 양면**이다.

---

## 3. 트랜잭션 범위 ≠ 락 범위

stale read 가 "왜 끼어들 틈이 있는가" 는 이 분리로 설명된다.

```
트랜잭션 범위:  [============== chunk 전체 (begin ~ commit) ==============]

락 범위:                                   [Writer 쓰기 ~ commit]
                (read 구간엔 락 없음 ← 여기로 stale read 가 끼어든다)
```

- **트랜잭션** 은 chunk 시작 시 begin, write 후 commit — chunk 전체에 걸린다.
- **락** 은 그 트랜잭션 *안에서* "쓰기가 일어나는 순간부터 commit 까지" 만 걸린다.
- 읽기(SELECT)는 같은 트랜잭션 안에 있어도 MVCC 라 **락이 없다.**
- → 따라서 read 구간(t1~t2)에 다른 트랜잭션의 변경이 끼어들 수 있다.

핵심: **트랜잭션이 열려 있다 ≠ 락이 걸려 있다.** 락 비용을 결정하는 건 **쓰기 양 = chunk size** 다 (page size 와 무관).

---

## 4. 그래서 — 정답/오답이 아니라 "선택"이다

학습자의 원래 의문:
> "Reader 가 읽는 순간부터 락을 걸어야 하는 것 아닌가? 중간에 다른 트랜잭션이 대상을 바꿔버리면 잘못된 배치 아닌가?"

**답: 읽는 순간부터 락을 거는 것도 가능하다.** `SELECT ... FOR UPDATE`(비관적 락)를 쓰면
Reader 가 읽으면서 쓰기 락을 함께 걸어 중간 변경을 **원천 차단**할 수 있다. 다만 그러면 그 행을
다른 트랜잭션이 못 건드려 **동시성이 떨어진다.**

즉 "락을 거느냐 마느냐" 는 **정합성 ↔ 성능의 트레이드오프** 이고, **대상의 성격에 따라 고르는 선택** 이다.

| 배치 대상 | stale read 가 문제인가 | 선택 |
|---|---|---|
| **안 변하는 데이터** (우리 아카이빙 — 오래된 비활성 게시글) | 사실상 무문제 | 락 없이 읽어 성능 확보 |
| **활발히 변하는 데이터** (실시간 주문 → 정산 이관 등) | 치명적 (금액이 틀어짐) | 비관적/낙관적 락으로 보호 |

### 우리 아카이빙 배치의 판단 (학습자 정리)

- 시나리오: *"오래된 게시글을 따로 옮겨놓자."*
- 대상이 **이미 잘 안 쓰이는 오래된 게시글** → 배치가 도는 그 짧은 순간에 수정이 들어올 확률이 **매우 낮다.**
- 그 순간 락을 안 걸어 **동시성을 확보**하는 것이 성능에 유리하다.
- 확률이 "매우 낮다" 는 것이지 **0 은 아니다.** 다만 **비즈니스 정책상 이 리스크를 감수해도 성능 이점이 더 크다** 고 판단한 것.

→ 그래서 학습자의 의문은 **틀린 게 아니라, "그 방법(읽는 순간 락)도 있지만 안 쓰기로 선택한 것"** 이다.

---

## 5. 해결 도구 지도 (Phase 10 예고)

"활발히 변하는 데이터" 를 다뤄 stale read 를 막아야 할 때의 선택지. 지금은 *지도* 로만 잡아둔다.

1. **대상을 안 변하는 것으로 한정** — 우리 아카이빙처럼 "비활성/완료 상태" 만 고른다. **가장 단순하고 강력.**
2. **상태 플래그로 격리** — 배치 시작 시 대상을 `status = ARCHIVING` 으로 먼저 표시해 다른 트랜잭션이 건드리지 못하게 한다 (애플리케이션 레벨 락).
3. **비관적 락 (`SELECT ... FOR UPDATE`)** — 읽을 때 쓰기 락을 함께 걸어 변경을 차단. 정합성↑ 동시성↓.
4. **낙관적 락 (version 컬럼)** — 변경 시점에 "내가 읽은 버전 == 현재 버전?" 을 확인, 다르면 충돌로 처리.

3·4 가 **Phase 10(동시성 제어 — 낙관적/비관적 락)** 의 주제다.

---

## 메타 교훈

> **"락 없는 빠른 읽기" 는 공짜가 아니라 stale read 가능성과 맞바꾼 것이다.**
> 락은 변경 충돌을 막는 장치라 본질적으로 *쓰기* 에 필요하고, 읽기는 원래 락이 필수가 아니다.
> MVCC 는 읽기의 일관성을 *스냅샷* 으로 제공해 읽기 락을 걷어냈고, 그 부작용이 stale read 다.
> 배치에서 이를 감수할지 / 락으로 막을지는 **정답이 아니라, 대상의 성격에 따른 비즈니스 판단** 이다.
