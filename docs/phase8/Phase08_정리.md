학습자가 답변을 작성하고, 클로드가 항목별 통과/수정 필요 판정을 한다.
모든 항목이 통과될 때까지 보강 → 재리뷰를 반복한다.

각 질문 아래에 답변을 작성한다. 모르는 부분은 *"잘 모르겠다"* 라고 솔직히 적어도 OK — 그 부분이 보강의 출발점이 된다.

## 영역 A — Spring Batch 입문 / Tasklet

### A1.
```text
배치 처리 (Batch Processing) 의 *특성* 을 정리하고,
*왜 이런 작업이 일반 API 호출이나 수동 SQL 실행이 아닌 배치 프레임워크로 처리되어야 하는지* 설명하시오.
```
**답변:**

```text
배치 처리는 대용량 데이터를 반복적이고 주기적으로 처리해야할 필요가 있을 때 적용하는 프로세스 입니다.
예를 들어 '오래된 게시글 아카이빙' 같은 작업은 일정 기간 이상의 모든 데이터를 대상으로 하기 때문에 대용량 데이터일 수 있고,
시간이 지날때마다 처리해야할 데이터가 생신되기 때문에 반복적이고 주기적으로 처리해줘야합니다.

이러한 작업을 배치 작업이라고 하는 특별한 프로세스 없이
일반적인 관리자 페이지에서 API 호출로 처리한다거나, 관리자가 하나하나 수동으로 SQL 을 실행한다고 가정해보겠습니다.

만약 관리자 페이지에서 API 호출로 배치작업을 수행하도록 요청한다면 배치 작업의 특성상 HTTP 의 타임아웃이 발생할 수 있습니다.
HTTP 타임아웃은 네트워크 통신 중 작업이 지정된 시간 내에 완료되지 않으면 연결을 강제로 종료하는 설정을 말합니다.
이는 무한정 대기로 서버, 클라이언트의 자원이 고갈되는 것을 방지하는 필수적인 장치입니다.
네트워크 통신에는 여러 인프라가 참여하며 이들 각자가 타임아웃을 가지고 있습니다.
브라우저 -> CDN -> Load Balancer -> API Gateway -> Proxy -> 서버 순으로 요청이 들어간다고 하면,
이 모든 경로에서 타임아웃이 존재하기 때문에 설령 서버에서 배치 처리를 끝마쳤다고 하더라도 앞선 네트워크 단에서 기다리지 못하고
에러를 반환하며 통신이 종료됩니다.
따라서 응답이 오래 걸릴 수 밖에 없는 배치 작업들은 API 호출로 구현하기에 부적합합니다.

다음으로 관리자가 DB 콘솔에 접속해 수동으로 SQL 을 날려 실행한다고 가정해보겠습니다.
이 경우 사람이 직접 수행하기 때문에 새벽 시간대 등 핵심 비즈니스 로직 요청이 한가한 시간대를 이용할 수 없게 됩니다.
배치 작업 등은 보통 핵심 비즈니스 로직이 아닌 운영을 위한 부가적인 처리인데 핵심 비즈니스 요청이 활발한 시간대에 처리하게 되면
자원 낭비가 발생할 수 있습니다.
또 자동화가 아닌 직접 처리기 때문에 휴먼 에러 발생으로 실패 추적이 어렵고 재시도, 트랜잭션 통제 등의 어려움이 있을 수 있습니다.

배치 처리는 이런 단점을 최소화하고 자동화하여 비교적 비즈니스 로직 호출이 한가한 시점에 관리자가 신경 쓰지 않아도 자동으로 필요한 작업을
수행할 수 있도록 합니다.
```

### A2.
```text
Spring Batch 의 *Job > Step > Tasklet* 의 계층 구조를 설명하고,
 *JobLauncher*, *JobRepository*, *JobParameters* 가 각각 어떤 역할을 하는지 정리하시오.
  특히 *JobParameters 가 왜 필요한지* — 매 실행마다 다른 값 (예: requestDate) 을 넣는 이유를 포함할 것.
```
**답변:**

```text
Job 은 하나의 정의된 배치 작업 자체를 뜻합니다.
'오래된 게시글 아카이빙' 같은 배치 작업을 실제로 구현해낸 것이 Job 객체가 됩니다.

Step 은 Job 안의 하나의 작업 단위입니다.
한 Job 안에 여러 Step 이 순차로 들어갈 수 있고, 각 Step 은 Tasklet 형태나 Chunk-oriented 형태로 구현됩니다. 
'오래된 게시글 아카이빙' 을 예시로 들면(Chunk-oriented 형태가 적합),
'아카이빙 대상 게시글을 조회한다(reader)', '조회된 엔티티를 아카이빙하기 적절하게 변환한다(processor)', '변환된 엔티티를 저장한다(writer)' 등이 모여 
Step 을 구성하게 됩니다.

Tasklet 은 Chunk-oriented 구조와 함께 Step 의 구현체 입니다.
앞서 언급한 조회, 변환, 저장 등으로 쪼개지는 작업은 Chunk-oriented 형태로 구현하며,
Tasklet 은 한번에 끝나는 단순한 작업 단위를 구현합니다.
예를 들어 임시 파일을 삭제한다던지, 외부 시스템에 알림을 발송하는 등에 사용합니다.
Step 이 Tasklet 또는 Chunk-oriented 구조를 하나 가지는 1:1 관계가 됩니다.

JobLauncher 는 job 과 jobParameters 를 파라미터로 받아 Job 을 실행시키는 객체입니다.
먼저 JobLauncher 는 넘겨받은 파라미터를 토대로 JobInstance 를 생성합니다. JobInstance 는 수행하려는 작업의 논리적 실행 단위 입니다.
이 객체는 넘겨받은 job 이름과 jobParameters 를 조합해 해시한 값을 담은 객체 입니다. 유니크 제약을 통해 DB에 유일하도록 저장됩니다.

다음으로, job 을 수행하려 할때 JobExecution 을 생성합니다. 이는 JobInstance 를 실행하려는 실제 물리적인 시도를 표현한 객체입니다.
JobExecution 은 JobInstance 를 외래키로 가져 어떤 job 을 실행시키려 했는지 기록하고,
시작 시간, 종료 시간, 진행상황, 종료 결과 등의 메타 데이터를 저장합니다.

JobLauncher 의 수행과정을 하나의 시나리오로 정리해보겠습니다.
먼저 26년 6월 9일에 배치 작업이 시작된다고 가정해보겠습니다.
JobLauncher 가 생성되고 run 메서드가 호출되면서 작업과 날짜 파라미터가 전달됩니다.
JobLauncher 는 JobRepository 를 통해 DB 조회를 합니다. 작업 이름과 날짜 파라미터 조합의 해시 값이 JobInstance 테이블에
존재하는지 체크합니다. 첫번째 수행임을 가정해 값이 없고 새로운 인스턴스를 삽입합니다.
이제 작업을 등록했으니 실제 실행을 시도합니다. 방금 만들어진 JobInstance 의 키를 외래키로 하며 이 작업을 수행하려고 시도한 순간의
메타 데이터를 삽입합니다. 진행상황은 STARTED 값으로 등록됩니다.

진행 과정에서 N번째 데이터에서 에러가 발생해 멈추는 상황을 가정해보겠습니다.
JobLauncher 는 일단 해당 JobExecution 의 진행상황을 FAILED 로 업데이트 하고 종료합니다.
개발자가 버그를 수정하고 다시 배치 작업을 호출합니다.
같은 작업 이름과 날짜로 호출되면 앞서 저장한 JobInstance 를 발견합니다.
해당 JobInstance 의 JobExecution 을 살펴보니 FAILED 상태입니다. Spring 은 지난번 실패 때문에 다시 배치 작업을 호출함을 판단하고
기존 인스턴스를 재사용합니다. 해당 인스턴스에 새로운 JobExecution 이 붙습니다.
지난번 중단된 N번째 기록부터 시작해 성공적으로 끝나면 진행상황을 COMPLETED 로 업데이트하고 종료합니다.
(N번째 중단 부터 재시작하는 기능을 구현하려면 Reader 가 ItemStream 을 구현해 진행 상황을 ExecutionContext 에 기록해야함.)

이후 실수로 누군가 똑같은 job 과 날짜파라미터를 넘겨서 호출했다고 가정해보겠습니다.
JobLauncher 는 해시를 통해 JobInstance 를 조회하고 이미 존재함을 확인합니다. 마지막 실행 기록 JobExecution 의 상태를 조회하여
COMPLETED 임을 확인하면 Spring 은 이 배치는 성공해서 끝났으니 다시 돌릴 필요가 없음을 판단하고
JobInstanceAlreadyCompleteException 예외를 던지고 실행을 차단합니다.
  

앞서 등장한대로 JobRepository 는 배치작업에 필요한 객체들을 저장하는 DB 와 통신하는 객체입니다.
해당 객체는 배치 작업에 필요한 DB 통신을 안전하게 하기 위해 트랜잭션 등을 관리합니다.

JobParameters 는 Job 을 호출할때 들어갈 파라미터를 관리하는 객체입니다.
이 객체 담긴 값을 토대로 JobInstance 를 새로 만들지 결정하는 해시 값이 생성됩니다.
```
```text
추가)
현재 BatchScheduler 코드는 동기 작업으로 작성되어 있습니다.
따라서 배치 작업이 호출되면 해당 스레드는 배치 작업이 끝날 때 까지 멈춰있어야 합니다.
실전 코드에서 동기 작업으로 처리하게 될 경우 매우 오래 걸리는 배치 작업일 시 다음 배치 작업 수행 시점까지 이전 배치 작업이 안 끝날 수 있습니다.
따라서 이를 비동기 실행 + 중복 실행 방지 형식으로 작성할 필요가 있습니다.
```

### A3.
```text
`Tasklet` 인터페이스를 *함수형 인터페이스* 관점에서 설명하고,
가장 단순한 Tasklet 의 구현 (예: Hello Tasklet) 이 *어떻게 동작* 하는지 설명하시오.
`RepeatStatus.FINISHED` 와 `RepeatStatus.CONTINUABLE` 의 차이도 포함.
```
**답변:**

```text
Tasklet 은 함수형 인터페이스입니다. 함수형 인터페이스란 구현할 추상 메서드가 하나만 선언되어 있는 인터페이스를 말합니다.
따라서 Tasklet 을 구현하기 위해 번거롭게 별도 클래스로 implements Tasklet 할 필요 없이 람다식으로 깔끔하게 구현이 가능합니다.

Tasklet 은 Step 의 일종으로 Spring Batch 에서 복잡하지 않은 간단한 작업을 쉽게 정의할때 사용합니다.
프로젝트에서 작성한 Hello Tasklet 의 동작 과정은 다음과 같습니다.
Job 이 시작되고 Step 이 Tasklet 객체를 호출합니다.

Tasklet 을 구현할때 파라미터로 넘긴 StepContribution 과 ChunkContext 는 현재 배치의 진행 상태를 담은 데이터 가방입니다.
이 객체를 호출 직후 생성해 주입해 줍니다.
StepContribution 객체는 이름 그대로 현재 Step 의 실행 상태에 기여하는 객체입니다.
읽은 아이템 수, 쓴 아이템 수, 필터링한 아이템 수 등의 통계 업데이트를 하고 Tasklet 안에서 진행 상태를 갱신합니다.
ChunkContext 객체는 현재 청크의 컨텍스트 정보를 담습니다.
JobParameters 를 가지며 Step 의 ExecutionContext 에 접근 가능합니다. Step 의 메타 데이터도 포함됩니다.

이제 구현한 배치 작업이 호출됩니다. 따로 StepContribution 과 ChunkContext 를 통한 처리가 없기 때문에
그냥 콘솔에 문장을 출력하고 RepeatStatus 를 반환합니다.

RepeatStatus 란 해당 Tasklet 을 지금 끝낼지, 한 번 더 실행할지 결정하는 제어 장치입니다.
FINISHED 는 Tasklet 의 실행 끝남을 의미하며 Step 이 종료됩니다.
CONTINUABLE 은 Tasklet 을 다시 호출하길 요청합니다. 대용량 파일을 100건씩 읽는다고 하면 '더 이상 읽을 데이터가 없을 때 까지'
같은 처리가 필요할 수 있습니다. 이 때 CONTINUABLE 을 반환해 반복하게 하다 더 이상 데이터가 없으면 루프를 탈출해 FINISHED 를 반환하는
구조를 만들 수 있습니다.
```

## 영역 B — Chunk-oriented Step

### B1.
```text
*Tasklet 으로 1억 건의 게시글을 처리하면 안 되는 이유* 를 세 가지로 정리하고,
그래서 *Chunk-oriented Step* 이라는 패턴이 등장한 본질을 설명하시오.
```
**답변:**

```text
Tasklet 으로 대용량 게시글을 처리하기에는 크게 세가지 한계점이 있습니다.

OOM 문제와 트랜잭션 폭증 문제, 부분 실패에 대한 재시작 불가능과 자원 낭비 입니다.

먼저 OOM 문제입니다.
Tasklet 안에 1억 건의 엔티티를 조회해오는 메서드가 있다고 가정하겠습니다.
엔티티가 하나에 평균 2KB 라면 1억 건의 경우 200GB 가 됩니다. 이는 JVM 힙으로는 한번에 메모리에 올릴 수 없는 크기입니다.

두번째는 트랜잭션 폭증 문제입니다.
먼저 살펴볼 부분은 Undo Log 입니다.
Undo Log 는 RDBMS가 트랜잭션에서 update, 수정하려는 데이터를 작업 이전 버전으로 저장해둬 롤백이 가능하도록 하거나 트랜잭션 작업 중에 다른 Update 가 아닌 세션이 이를 읽어
처리되는 트랜잭션 이전 버전의 데이터를 읽어갈 수 있도록 합니다.(이를 MVCC, Multi-Version Concurrency Control 다중 버전 동시성 제어 라고 합니다.)
1억 건의 단일 트랜잭션이 시작되면 RDBMS는 기존 데이터 원본을 수정하기 전에 Undo Log 에 복사해둬야하고 트랜잭션 처리 시간동안 이를 유지해야합니다.
때문에 디스크 I/O 대역폭을 크게 소모시키고 다른 세션이 과거 데이터를 읽기 위해 거대한 Undo Log 체인을 탐색해야하기 때문에 RDBMS 성능 저하를 유발하게 됩니다.
그 다음 요소는 Lock 경합입니다.
Lock 은 DBMS 에서 여러 트랜잭션이 동시에 데이터에 접근해서 작업을 수행할때 데이터 손실이나 불일치를 막기 위한 장치입니다.
Lock 에는 S 락과 X 락이 있으며 현대 RDBMS 에는 MVCC 아키텍처가 도입되어 있습니다.
S 락은 읽기 작업을 위해 얻는 락입니다. X 락은 변경 작업을 위해 얻는 락입니다. MVCC 는 DB 의 동시성을 극대화 하기 위해 트랜잭션이 시작되면
원본 데이터를 읽어 Undo Log 를 만들어 스냅샷을 보관해둡니다. 이는 롤백에도 활용되며 트랜잭션이 진행되는 동안 다른 서비스나 스레드가 해당 데이터를 읽어서 작업을 차리하도록 합니다. 

이제 1억건의 단일 트랜잭션이 들어오는 상황을 가정해보겠습니다.
트랜잭션이 시작되고 업데이트 대상 행 마다 X 락이 걸리기 시작합니다. 이 락들은 RDBMS 메모리에 누적되기 시작합니다.
또 MVCC 로 인해 데이터 변경 전 원본이 Undo Log 에 기록되기 시작합니다.
어느정도 X 락이 쌓이면 RDBMS 가 한계를 감지해 Row Lock 을 해제하고 전체 테이블을 대상으로 Lock 걸게 됩니다.

(Lock Escalation, SQL Server 나 DB2 같은 DB 에서는 락 격상이 일어나지만
현재 프로젝트에서 사용되는 MySQL InnoDB 와 postgreSQL 은 Lock Escalation 을 하지 않습니다. 현재 프로젝트는
테이블 락으로 번지지 않고 수백만 개의 Row Lock 이 잡히는 것으로 문제가 발생합니다.)

이제 해당 테이블에 대한 다른 서비스, 스레드들은 작업을 수행하지 못해 장시간 대기하거나 실패하게 됩니다.
테이블 락이 걸려 있어도(혹은 개별 row lock 이어도 동일) 다른 '읽기 작업'들은 MVCC 의 Undo log 를 통해 수행이 가능합니다.
하지만 1억건이라는 방대한 Undo log 에서 읽기 작업을 수행해야하기 때문에 디스크 I/O 가 폭증하고 읽기 속도가 크게 저하됩니다.

세번째는 부분 실패의 비용입니다.
1억건을 하나의 트랜잭션으로 묶으면 1억건을 대상으로 전부 성공하거나 전부 실패하는 트랜잭션의 특성이 적용됩니다.
어떤 원인이던간에 1억건 중 한 건을 남겨놓고 실패하면 지금까지의 처리가 모두 롤백되어야하기 때문에 장시간 자원을 들여 수행한 작업이
체크포인트 없이 모두 무용지물이 됩니다. 심지어 실패해서 수행하는 이 롤백 과정에서도 방대한 리소스가 소모됩니다.

이러한 한계를 해결하기 위해 단일 구조인 Tasklet 대신 Chunk-oriented step 구조의 reader -> processor -> writer 의 파이프라인으로
OOM 문제를 해결하고 트랜잭션을 분리해 RDBMS 의 오버헤드를 없애고 락 점유 시간을 줄이며, chunk 단위로 커밋될 때마다 처리됨을 기록하고 재시작이 유연해지도록 합니다.
```

```text
Chunk-oriented Step 의 *Reader / Processor / Writer 호출 패턴* 을 의사코드로 또는 시퀀스로 설명하시오.
특히 *Reader 와 Processor 는 한 건씩 호출되는데 Writer 는 청크당 한 번만 호출되는 비대칭* 의 의미를 짚을 것.
```
**답변:**

```text
Job 이 호출되고 Step 이 순차적으로 호출되면 Chunk oriented 구조가 시작됩니다.
먼저 reader 를 통해 처리 대상을 읽어옵니다.
페이징 기법을 사용해 대상 데이터의 일부를 가져올 수 있습니다.
혹은 고정된 id 값을 기반으로 쿼리를 날려 조회해 올 수도 있습니다. 조회해온 데이터들을 iterator 로 감싸 한 건씩 처리할 수 있도록 합니다.
reader 로부터 한건이 읽어지면 해당 엔티티를 Processor 에게 넘겨줍니다.
Processor 는 해당 엔티티로부터 값을 뽑아 배치 처리 결과물을 작성합니다.
가공된 결과물은 writer 가 보관합니다. 이후 reader - processor 과정을 반복해 chunk 크기 만큼 엔티티를 모으면
writer 가 일괄로 저장 쿼리를 내보냅니다.
이렇게 하나의 트랜잭션이 끝나고 처리할 대상이 남아있다면 다시 reader 부터 시작해 반복합니다.

Reader 와 Processor 는 한번에 한 건씩 처리하는 동일한 인터페이스 구조를 가지고 있습니다.

processor 입장을 살펴보면, 필터링이 자연스러워지기 때문입니다.
프로세서가 null 을 반환하면 그 건은 writer 로 넘어가지 않고 제외됩니다. 한 건을 받아서 처리해서 돌려줘야 writer 까지 이 판단이 간단하게 처리됩니다.
만약 프로세서에서 list 를 반환했다면 외부에서 그 리스트를 보고 null 이 있는지 검사하는 과정이 필요해집니다.
또 변환이라는 작업 자체가 한 건 단위로 처리되는게 자연스럽습니다. Post -> ArchivedPost 로 변환되기 때문에 인터페이스 역시 한 건 단위가 자연스럽습니다.
에러 처리에 있어서도 단건 처리 구조가 유리합니다. 만약 list 단위로 움직인다면 '어느 지점에서 실패했는지' 알리기 위한 또다른 자료구조가 필요해집니다.
단건 처리는 에러 발생 지점을 외부에서 곧바로 알아차리고 실패 지점부터 작업할 수 있도록 할 수 있습니다.
종합적으로, 상태가 없는, 순수 함수 구조를 만들어야 테스트, 재시작이 쉽고 병렬화도 쉬워집니다.

reader 입장에서는 한건 씩 노출해야 chunk size를 조절할 수 있게 됩니다.
만약 reader 가 읽어온 데이터를 1000건 씩 리스트로 반환해버린다고 하면 커밋 단위인 chunk size 역시 1000건 단위로 강제됩니다.
한건 씩 넘겨야 외부에서 chunk size 만큼 센 후 트랜잭션을 커밋할 수 있습니다.
또 한건 씩이라는 계약을 해뒀기 때문에 Step 은 데이터 출처를 신경쓰지 않아도 됩니다.
DB의 페이지 단위로 오든, CSV 파일에서 한 줄 씩 읽던, 메세지 큐에서 하나의 메시지 씩 오던 똑같이 동작할 수 있습니다.
어떤 출처에서든지 똑같이 적용할 수 있는 단위가 '한 건 씩 처리하기' 입니다.


반면 Writer 는 Chunk 단위로 처리되어야하는데 이는 Writer 가 외부 DB 와의 통신을 맡고 있기 때문입니다.
외부 통신은 비용이 비싼 작업이기 때문에 배치 대상 만큼 DB 와의 통신을 하게 되면 막대한 자원 손실이 발생합니다. 따라서 일정량을 모아 한번에 처리하는
효율적 구조를 채택했습니다.
```

```text
Page size 와 Chunk size 의 차이를 설명하고, 
둘이 항상 같지 않아도 되는 이유 와 대부분의 경우 일치시키는 것이 권장되는 이유 를 정리하시오.
```

**답변:**

```text
Page size 는 Reader 가 청크 처리 동안 한번에 읽어올 데이터의 양입니다.
Chunk size 는 하나의 트랜잭션 크기로 하나의 Step 을 완료하기 까지 나눠 처리할 데이터 양입니다. 

효율을 위해 생각해볼 포인트는 다음과 같습니다.
페이지 크기에 의한 DB와의 통신 자원, DB 부하와 통신 횟수
청크 크기(트랜잭션 크기)에 의한 락 점유 시간과 롤백 비용
처리하는 동안 필요한 메모리 양입니다.

먼저 페이지 크기와 청크 크기가 같은 경우 입니다.
이 경우 한번의 읽기 작업 후 이를 모두 처리하면 곧바로 청크 크기를 채우기 때문에 바로 커밋이 진행됩니다.
제일 단순한 상태로 파악이 쉬워 디버깅, 재시작 처리 등에 유리합니다. 많은 경우 이 케이스를 디폴트로 사용합니다.

다음은 페이지 크기가 청크 크기보다 작은 경우 입니다. (page < chunk)
페이지 크기가 100, 청크 크기가 1000 이라고 가정해보겠습니다.
먼저 트랜잭션이 시작되고 Reader 가 100건을 읽어옵니다. 이후 Processor 가 한건씩 넘겨 받아 가공하고 Writer 가 보관합니다.
다시 Reader 가 100건을 읽어오며 작업을 반복해 총 10번, 1000건이 Writer에 누적되면 커밋해서 트랜잭션을 종료합니다.
하나의 청크 처리 완료를 위해서는 1000건이 필요한데 이를 100건 씩 나눠 읽어오기 때문에 DBMS 입장에서는 부하가 분산되게 됩니다.
반면 애플리케이션 입장에서는 1000건을 한번에 가져오지 않고 100건 씩 나누기 때문에 10번의 통신이 필요해 상대적으로 통신 부하가 발생합니다.
또, 1000건의 상대적으로 높은 락 점유 시간과 롤백비용이 발생하게 됩니다.
처리하는 동안 필요한 메모리는 Reader 는 100건, 청크 입장에서는 1000건이 됩니다.

이번엔 청크 크기가 페이지 크기보다 작은 경우 입니다. (page > chunk)
페이지 크기를 1000, 청크 크기를 100 이라고 가정하겠습니다.
트랜잭션이 시작되면 Reader 는 1000건을 읽어옵니다. 이후 Processor 가 한건씩 넘겨 받으며 가공하고 Writer 에게 넘겨주는데 100건이 모이면 커밋하며 트랜잭션이 종료됩니다.
다음 청크 작업이 진행되며 Reader 는 이미 읽어온 데이터가 있기 때문에 읽지 않고 이어서 Processor 에게 한 건 씩 넘겨주고 Writer 가 100건이 모이면 커밋 후 트랜잭션이 종료됩니다.
이를 10번 반복하면 같은 양의 처리가 이루어집니다.
이 경우 Reader 는 한번에 모든 데이터를 불러왔기 때문에 RDBMS 의 부하가 상대적으로 커지게 됩니다. 애플리케이션 입장에서는 한번의 통신으로 끝나기 때문에 부하가 줄어듭니다.
매 청크 처리마다 100건의 트랜잭션이 처리되므로 상대적으로 락 점유 시간과 롤백 비용이 줄어듭니다.
처리하는 동안 필요한 메모리는 Reader 1000건, 청크 입장에서는 100건 씩이기 때문에 앞서 다룬 케이스에서 위치만 변경되었을 뿐 같습니다.

각각의 특징에 따른 선택 과정을 정리해보겠습니다.
페이지 크기와 청크 크기가 같은 상황이 디폴트입니다. 특별한 이유가 없다면 단순하며 처리가 쉬운 이 형태가 우선시 됩니다.

만약 DBMS 의 부하가 신경 쓰이는 상황이라면 페이지 크기가 청크 크기보다 작은게 좋습니다.
예를 들어 낮 시간대에 수행할 배치인데 해당 DB 를 실사용자 요청 처리에도 쓴다면 DB 의 성능을 신경써야 합니다.
페이지 크기를 청크 크기보다 줄여 가벼운 조회를 여러번에 걸쳐 수행하는게 유리해집니다.

또, DB가 네트워크 너머 외부 서비스에 있다면 통신 비용이 비싸지기 때문에 페이지 크기가 청크 크기보다 큰게 유리합니다.
예를 들어 해당 처리를 위해 DB 가 물리적으로 먼, 미국에 있는 데이터 센터와 통신할 필요가 있다면 통신 비용을 줄이기 위해 횟수를 줄여야합니다.
따라서 페이지 크기를 키워 가급적 적은 회수로 데이터를 읽어오도록 한 뒤 처리하는게 유리합니다.
```
