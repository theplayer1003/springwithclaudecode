모든 항목이 통과될 때까지 보강 → 재리뷰를 반복한다.

각 질문 아래에 답변을 작성한다. 모르는 부분은 *"잘 모르겠다"* 라고 솔직히 적어도 OK — 그 부분이 보강의 출발점이 된다.

## 영역 A — Spring Batch 입문 / Tasklet

### A1.
```text
배치 처리 (Batch Processing) 의 *특성* 을 정리하고,
*왜 이런 작업이 일반 API 호출이나 수동 SQL 실행이 아닌 배치 프레임워크로 처리되어야 하는지* 설명하시오.
```
**답변:**

```text
배치 처리는 대용량 데이터를 반복적이고 주기적으로 처리해야할 필요가 있을 때 적용하는 프로세스 입니다.
예를 들어 '오래된 게시글 아카이빙' 같은 작업은 일정 기간 이상의 모든 데이터를 대상으로 하기 때문에 대용량 데이터일 수 있고,
시간이 지날때마다 처리해야할 데이터가 생신되기 때문에 반복적이고 주기적으로 처리해줘야합니다.

이러한 작업을 배치 작업이라고 하는 특별한 프로세스 없이
일반적인 관리자 페이지에서 API 호출로 처리한다거나, 관리자가 하나하나 수동으로 SQL 을 실행한다고 가정해보겠습니다.

만약 관리자 페이지에서 API 호출로 배치작업을 수행하도록 요청한다면 배치 작업의 특성상 HTTP 의 타임아웃이 발생할 수 있습니다.
HTTP 타임아웃은 네트워크 통신 중 작업이 지정된 시간 내에 완료되지 않으면 연결을 강제로 종료하는 설정을 말합니다.
이는 무한정 대기로 서버, 클라이언트의 자원이 고갈되는 것을 방지하는 필수적인 장치입니다.
네트워크 통신에는 여러 인프라가 참여하며 이들 각자가 타임아웃을 가지고 있습니다.
브라우저 -> CDN -> Load Balancer -> API Gateway -> Proxy -> 서버 순으로 요청이 들어간다고 하면,
이 모든 경로에서 타임아웃이 존재하기 때문에 설령 서버에서 배치 처리를 끝마쳤다고 하더라도 앞선 네트워크 단에서 기다리지 못하고
에러를 반환하며 통신이 종료됩니다.
따라서 응답이 오래 걸릴 수 밖에 없는 배치 작업들은 API 호출로 구현하기에 부적합합니다.

다음으로 관리자가 DB 콘솔에 접속해 수동으로 SQL 을 날려 실행한다고 가정해보겠습니다.
이 경우 사람이 직접 수행하기 때문에 새벽 시간대 등 핵심 비즈니스 로직 요청이 한가한 시간대를 이용할 수 없게 됩니다.
배치 작업 등은 보통 핵심 비즈니스 로직이 아닌 운영을 위한 부가적인 처리인데 핵심 비즈니스 요청이 활발한 시간대에 처리하게 되면
자원 낭비가 발생할 수 있습니다.
또 자동화가 아닌 직접 처리기 때문에 휴먼 에러 발생으로 실패 추적이 어렵고 재시도, 트랜잭션 통제 등의 어려움이 있을 수 있습니다.

배치 처리는 이런 단점을 최소화하고 자동화하여 비교적 비즈니스 로직 호출이 한가한 시점에 관리자가 신경 쓰지 않아도 자동으로 필요한 작업을
수행할 수 있도록 합니다.
```

### A2.
```text
Spring Batch 의 *Job > Step > Tasklet* 의 계층 구조를 설명하고,
 *JobLauncher*, *JobRepository*, *JobParameters* 가 각각 어떤 역할을 하는지 정리하시오.
  특히 *JobParameters 가 왜 필요한지* — 매 실행마다 다른 값 (예: requestDate) 을 넣는 이유를 포함할 것.
```
**답변:**

```text
Job 은 하나의 정의된 배치 작업 자체를 뜻합니다.
'오래된 게시글 아카이빙' 같은 배치 작업을 실제로 구현해낸 것이 Job 객체가 됩니다.

Step 은 Job 안의 하나의 작업 단위입니다.
한 Job 안에 여러 Step 이 순차로 들어갈 수 있고, 각 Step 은 Tasklet 형태나 Chunk-oriented 형태로 구현됩니다. 
'오래된 게시글 아카이빙' 을 예시로 들면(Chunk-oriented 형태가 적합),
'아카이빙 대상 게시글을 조회한다(reader)', '조회된 엔티티를 아카이빙하기 적절하게 변환한다(processor)', '변환된 엔티티를 저장한다(writer)' 등이 모여 
Step 을 구성하게 됩니다.

Tasklet 은 Chunk-oriented 구조와 함께 Step 의 구현체 입니다.
앞서 언급한 조회, 변환, 저장 등으로 쪼개지는 작업은 Chunk-oriented 형태로 구현하며,
Tasklet 은 한번에 끝나는 단순한 작업 단위를 구현합니다.
예를 들어 임시 파일을 삭제한다던지, 외부 시스템에 알림을 발송하는 등에 사용합니다.
Step 이 Tasklet 또는 Chunk-oriented 구조를 하나 가지는 1:1 관계가 됩니다.

JobLauncher 는 job 과 jobParameters 를 파라미터로 받아 Job 을 실행시키는 객체입니다.
먼저 JobLauncher 는 넘겨받은 파라미터를 토대로 JobInstance 를 생성합니다. JobInstance 는 수행하려는 작업의 논리적 실행 단위 입니다.
이 객체는 넘겨받은 job 이름과 jobParameters 를 조합해 해시한 값을 담은 객체 입니다. 유니크 제약을 통해 DB에 유일하도록 저장됩니다.

다음으로, job 을 수행하려 할때 JobExecution 을 생성합니다. 이는 JobInstance 를 실행하려는 실제 물리적인 시도를 표현한 객체입니다.
JobExecution 은 JobInstance 를 외래키로 가져 어떤 job 을 실행시키려 했는지 기록하고,
시작 시간, 종료 시간, 진행상황, 종료 결과 등의 메타 데이터를 저장합니다.

JobLauncher 의 수행과정을 하나의 시나리오로 정리해보겠습니다.
먼저 26년 6월 9일에 배치 작업이 시작된다고 가정해보겠습니다.
JobLauncher 가 생성되고 run 메서드가 호출되면서 작업과 날짜 파라미터가 전달됩니다.
JobLauncher 는 JobRepository 를 통해 DB 조회를 합니다. 작업 이름과 날짜 파라미터 조합의 해시 값이 JobInstance 테이블에
존재하는지 체크합니다. 첫번째 수행임을 가정해 값이 없고 새로운 인스턴스를 삽입합니다.
이제 작업을 등록했으니 실제 실행을 시도합니다. 방금 만들어진 JobInstance 의 키를 외래키로 하며 이 작업을 수행하려고 시도한 순간의
메타 데이터를 삽입합니다. 진행상황은 STARTED 값으로 등록됩니다.

진행 과정에서 N번째 데이터에서 에러가 발생해 멈추는 상황을 가정해보겠습니다.
JobLauncher 는 일단 해당 JobExecution 의 진행상황을 FAILED 로 업데이트 하고 종료합니다.
개발자가 버그를 수정하고 다시 배치 작업을 호출합니다.
같은 작업 이름과 날짜로 호출되면 앞서 저장한 JobInstance 를 발견합니다.
해당 JobInstance 의 JobExecution 을 살펴보니 FAILED 상태입니다. Spring 은 지난번 실패 때문에 다시 배치 작업을 호출함을 판단하고
기존 인스턴스를 재사용합니다. 해당 인스턴스에 새로운 JobExecution 이 붙습니다.
지난번 중단된 N번째 기록부터 시작해 성공적으로 끝나면 진행상황을 COMPLETED 로 업데이트하고 종료합니다.
(N번째 중단 부터 재시작하는 기능을 구현하려면 Reader 가 ItemStream 을 구현해 진행 상황을 ExecutionContext 에 기록해야함.)

이후 실수로 누군가 똑같은 job 과 날짜파라미터를 넘겨서 호출했다고 가정해보겠습니다.
JobLauncher 는 해시를 통해 JobInstance 를 조회하고 이미 존재함을 확인합니다. 마지막 실행 기록 JobExecution 의 상태를 조회하여
COMPLETED 임을 확인하면 Spring 은 이 배치는 성공해서 끝났으니 다시 돌릴 필요가 없음을 판단하고
JobInstanceAlreadyCompleteException 예외를 던지고 실행을 차단합니다.
  

앞서 등장한대로 JobRepository 는 배치작업에 필요한 객체들을 저장하는 DB 와 통신하는 객체입니다.
해당 객체는 배치 작업에 필요한 DB 통신을 안전하게 하기 위해 트랜잭션 등을 관리합니다.

JobParameters 는 Job 을 호출할때 들어갈 파라미터를 관리하는 객체입니다.
이 객체 담긴 값을 토대로 JobInstance 를 새로 만들지 결정하는 해시 값이 생성됩니다.
```
```text
추가)
현재 BatchScheduler 코드는 동기 작업으로 작성되어 있습니다.
따라서 배치 작업이 호출되면 해당 스레드는 배치 작업이 끝날 때 까지 멈춰있어야 합니다.
실전 코드에서 동기 작업으로 처리하게 될 경우 매우 오래 걸리는 배치 작업일 시 다음 배치 작업 수행 시점까지 이전 배치 작업이 안 끝날 수 있습니다.
따라서 이를 비동기 실행 + 중복 실행 방지 형식으로 작성할 필요가 있습니다.
```

### A3.
```text
`Tasklet` 인터페이스를 *함수형 인터페이스* 관점에서 설명하고,
가장 단순한 Tasklet 의 구현 (예: Hello Tasklet) 이 *어떻게 동작* 하는지 설명하시오.
`RepeatStatus.FINISHED` 와 `RepeatStatus.CONTINUABLE` 의 차이도 포함.
```
**답변:**

```text
Tasklet 은 함수형 인터페이스입니다. 함수형 인터페이스란 구현할 추상 메서드가 하나만 선언되어 있는 인터페이스를 말합니다.
따라서 Tasklet 을 구현하기 위해 번거롭게 별도 클래스로 implements Tasklet 할 필요 없이 람다식으로 깔끔하게 구현이 가능합니다.

Tasklet 은 Step 의 일종으로 Spring Batch 에서 복잡하지 않은 간단한 작업을 쉽게 정의할때 사용합니다.
프로젝트에서 작성한 Hello Tasklet 의 동작 과정은 다음과 같습니다.
Job 이 시작되고 Step 이 Tasklet 객체를 호출합니다.

Tasklet 을 구현할때 파라미터로 넘긴 StepContribution 과 ChunkContext 는 현재 배치의 진행 상태를 담은 데이터 가방입니다.
이 객체를 호출 직후 생성해 주입해 줍니다.
StepContribution 객체는 이름 그대로 현재 Step 의 실행 상태에 기여하는 객체입니다.
읽은 아이템 수, 쓴 아이템 수, 필터링한 아이템 수 등의 통계 업데이트를 하고 Tasklet 안에서 진행 상태를 갱신합니다.
ChunkContext 객체는 현재 청크의 컨텍스트 정보를 담습니다.
JobParameters 를 가지며 Step 의 ExecutionContext 에 접근 가능합니다. Step 의 메타 데이터도 포함됩니다.

이제 구현한 배치 작업이 호출됩니다. 따로 StepContribution 과 ChunkContext 를 통한 처리가 없기 때문에
그냥 콘솔에 문장을 출력하고 RepeatStatus 를 반환합니다.

RepeatStatus 란 해당 Tasklet 을 지금 끝낼지, 한 번 더 실행할지 결정하는 제어 장치입니다.
FINISHED 는 Tasklet 의 실행 끝남을 의미하며 Step 이 종료됩니다.
CONTINUABLE 은 Tasklet 을 다시 호출하길 요청합니다. 대용량 파일을 100건씩 읽는다고 하면 '더 이상 읽을 데이터가 없을 때 까지'
같은 처리가 필요할 수 있습니다. 이 때 CONTINUABLE 을 반환해 반복하게 하다 더 이상 데이터가 없으면 루프를 탈출해 FINISHED 를 반환하는
구조를 만들 수 있습니다.
```

## 영역 B — Chunk-oriented Step

### B1.
```text
*Tasklet 으로 1억 건의 게시글을 처리하면 안 되는 이유* 를 세 가지로 정리하고,
그래서 *Chunk-oriented Step* 이라는 패턴이 등장한 본질을 설명하시오.
```
**답변:**

```text
Tasklet 으로 대용량 게시글을 처리하기에는 크게 세가지 한계점이 있습니다.

OOM 문제와 트랜잭션 폭증 문제, 부분 실패에 대한 재시작 불가능과 자원 낭비 입니다.

먼저 OOM 문제입니다.
Tasklet 안에 1억 건의 엔티티를 조회해오는 메서드가 있다고 가정하겠습니다.
엔티티가 하나에 평균 2KB 라면 1억 건의 경우 200GB 가 됩니다. 이는 JVM 힙으로는 한번에 메모리에 올릴 수 없는 크기입니다.

두번째는 트랜잭션 폭증 문제입니다.
먼저 살펴볼 부분은 Undo Log 입니다.
Undo Log 는 RDBMS가 트랜잭션에서 update, 수정하려는 데이터를 작업 이전 버전으로 저장해둬 롤백이 가능하도록 하거나 트랜잭션 작업 중에 다른 Update 가 아닌 세션이 이를 읽어
처리되는 트랜잭션 이전 버전의 데이터를 읽어갈 수 있도록 합니다.(이를 MVCC, Multi-Version Concurrency Control 다중 버전 동시성 제어 라고 합니다.)
1억 건의 단일 트랜잭션이 시작되면 RDBMS는 기존 데이터 원본을 수정하기 전에 Undo Log 에 복사해둬야하고 트랜잭션 처리 시간동안 이를 유지해야합니다.
때문에 디스크 I/O 대역폭을 크게 소모시키고 다른 세션이 과거 데이터를 읽기 위해 거대한 Undo Log 체인을 탐색해야하기 때문에 RDBMS 성능 저하를 유발하게 됩니다.
그 다음 요소는 Lock 경합입니다.
Lock 은 DBMS 에서 여러 트랜잭션이 동시에 데이터에 접근해서 작업을 수행할때 데이터 손실이나 불일치를 막기 위한 장치입니다.
Lock 에는 S 락과 X 락이 있으며 현대 RDBMS 에는 MVCC 아키텍처가 도입되어 있습니다.
S 락은 읽기 작업을 위해 얻는 락입니다. X 락은 변경 작업을 위해 얻는 락입니다. MVCC 는 DB 의 동시성을 극대화 하기 위해 트랜잭션이 시작되면
원본 데이터를 읽어 Undo Log 를 만들어 스냅샷을 보관해둡니다. 이는 롤백에도 활용되며 트랜잭션이 진행되는 동안 다른 서비스나 스레드가 해당 데이터를 읽어서 작업을 차리하도록 합니다. 

이제 1억건의 단일 트랜잭션이 들어오는 상황을 가정해보겠습니다.
트랜잭션이 시작되고 업데이트 대상 행 마다 X 락이 걸리기 시작합니다. 이 락들은 RDBMS 메모리에 누적되기 시작합니다.
또 MVCC 로 인해 데이터 변경 전 원본이 Undo Log 에 기록되기 시작합니다.
어느정도 X 락이 쌓이면 RDBMS 가 한계를 감지해 Row Lock 을 해제하고 전체 테이블을 대상으로 Lock 걸게 됩니다.

(Lock Escalation, SQL Server 나 DB2 같은 DB 에서는 락 격상이 일어나지만
현재 프로젝트에서 사용되는 MySQL InnoDB 와 postgreSQL 은 Lock Escalation 을 하지 않습니다. 현재 프로젝트는
테이블 락으로 번지지 않고 수백만 개의 Row Lock 이 잡히는 것으로 문제가 발생합니다.)

이제 해당 테이블에 대한 다른 서비스, 스레드들은 작업을 수행하지 못해 장시간 대기하거나 실패하게 됩니다.
테이블 락이 걸려 있어도(혹은 개별 row lock 이어도 동일) 다른 '읽기 작업'들은 MVCC 의 Undo log 를 통해 수행이 가능합니다.
하지만 1억건이라는 방대한 Undo log 에서 읽기 작업을 수행해야하기 때문에 디스크 I/O 가 폭증하고 읽기 속도가 크게 저하됩니다.

세번째는 부분 실패의 비용입니다.
1억건을 하나의 트랜잭션으로 묶으면 1억건을 대상으로 전부 성공하거나 전부 실패하는 트랜잭션의 특성이 적용됩니다.
어떤 원인이던간에 1억건 중 한 건을 남겨놓고 실패하면 지금까지의 처리가 모두 롤백되어야하기 때문에 장시간 자원을 들여 수행한 작업이
체크포인트 없이 모두 무용지물이 됩니다. 심지어 실패해서 수행하는 이 롤백 과정에서도 방대한 리소스가 소모됩니다.

이러한 한계를 해결하기 위해 단일 구조인 Tasklet 대신 Chunk-oriented step 구조의 reader -> processor -> writer 의 파이프라인으로
OOM 문제를 해결하고 트랜잭션을 분리해 RDBMS 의 오버헤드를 없애고 락 점유 시간을 줄이며, chunk 단위로 커밋될 때마다 처리됨을 기록하고 재시작이 유연해지도록 합니다.
```

**답변:**

```text
Job 이 호출되고 Step 이 순차적으로 호출되면 Chunk oriented 구조가 시작됩니다.
먼저 reader 를 통해 처리 대상을 읽어옵니다.
페이징 기법을 사용해 대상 데이터의 일부를 가져올 수 있습니다.
혹은 고정된 id 값을 기반으로 쿼리를 날려 조회해 올 수도 있습니다. 조회해온 데이터들을 iterator 로 감싸 한 건씩 처리할 수 있도록 합니다.
reader 로부터 한건이 읽어지면 해당 엔티티를 Processor 에게 넘겨줍니다.
Processor 는 해당 엔티티로부터 값을 뽑아 배치 처리 결과물을 작성합니다.
가공된 결과물은 writer 가 보관합니다. 이후 reader - processor 과정을 반복해 chunk 크기 만큼 엔티티를 모으면
writer 가 일괄로 저장 쿼리를 내보냅니다.
이렇게 하나의 트랜잭션이 끝나고 처리할 대상이 남아있다면 다시 reader 부터 시작해 반복합니다.

Reader 와 Processor 는 한번에 한 건씩 처리하는 동일한 인터페이스 구조를 가지고 있습니다.

processor 입장을 살펴보면, 필터링이 자연스러워지기 때문입니다.
프로세서가 null 을 반환하면 그 건은 writer 로 넘어가지 않고 제외됩니다. 한 건을 받아서 처리해서 돌려줘야 writer 까지 이 판단이 간단하게 처리됩니다.
만약 프로세서에서 list 를 반환했다면 외부에서 그 리스트를 보고 null 이 있는지 검사하는 과정이 필요해집니다.
또 변환이라는 작업 자체가 한 건 단위로 처리되는게 자연스럽습니다. Post -> ArchivedPost 로 변환되기 때문에 인터페이스 역시 한 건 단위가 자연스럽습니다.
에러 처리에 있어서도 단건 처리 구조가 유리합니다. 만약 list 단위로 움직인다면 '어느 지점에서 실패했는지' 알리기 위한 또다른 자료구조가 필요해집니다.
단건 처리는 에러 발생 지점을 외부에서 곧바로 알아차리고 실패 지점부터 작업할 수 있도록 할 수 있습니다.
종합적으로, 상태가 없는, 순수 함수 구조를 만들어야 테스트, 재시작이 쉽고 병렬화도 쉬워집니다.

reader 입장에서는 한건 씩 노출해야 chunk size를 조절할 수 있게 됩니다.
만약 reader 가 읽어온 데이터를 1000건 씩 리스트로 반환해버린다고 하면 커밋 단위인 chunk size 역시 1000건 단위로 강제됩니다.
한건 씩 넘겨야 외부에서 chunk size 만큼 센 후 트랜잭션을 커밋할 수 있습니다.
또 한건 씩이라는 계약을 해뒀기 때문에 Step 은 데이터 출처를 신경쓰지 않아도 됩니다.
DB의 페이지 단위로 오든, CSV 파일에서 한 줄 씩 읽던, 메세지 큐에서 하나의 메시지 씩 오던 똑같이 동작할 수 있습니다.
어떤 출처에서든지 똑같이 적용할 수 있는 단위가 '한 건 씩 처리하기' 입니다.


반면 Writer 는 Chunk 단위로 처리되어야하는데 이는 Writer 가 외부 DB 와의 통신을 맡고 있기 때문입니다.
외부 통신은 비용이 비싼 작업이기 때문에 배치 대상 만큼 DB 와의 통신을 하게 되면 막대한 자원 손실이 발생합니다. 따라서 일정량을 모아 한번에 처리하는
효율적 구조를 채택했습니다.
```

```text
Page size 와 Chunk size 의 차이를 설명하고, 
둘이 항상 같지 않아도 되는 이유 와 대부분의 경우 일치시키는 것이 권장되는 이유 를 정리하시오.
```

**답변:**

```text
Page size 는 Reader 가 청크 처리 동안 한번에 읽어올 데이터의 양입니다.
Chunk size 는 하나의 트랜잭션 크기로 하나의 Step 을 완료하기 까지 나눠 처리할 데이터 양입니다. 

효율을 위해 생각해볼 포인트는 다음과 같습니다.
페이지 크기에 의한 DB와의 통신 자원, DB 부하와 통신 횟수
청크 크기(트랜잭션 크기)에 의한 락 점유 시간과 롤백 비용
처리하는 동안 필요한 메모리 양입니다.

먼저 페이지 크기와 청크 크기가 같은 경우 입니다.
이 경우 한번의 읽기 작업 후 이를 모두 처리하면 곧바로 청크 크기를 채우기 때문에 바로 커밋이 진행됩니다.
제일 단순한 상태로 파악이 쉬워 디버깅, 재시작 처리 등에 유리합니다. 많은 경우 이 케이스를 디폴트로 사용합니다.

다음은 페이지 크기가 청크 크기보다 작은 경우 입니다. (page < chunk)
페이지 크기가 100, 청크 크기가 1000 이라고 가정해보겠습니다.
먼저 트랜잭션이 시작되고 Reader 가 100건을 읽어옵니다. 이후 Processor 가 한건씩 넘겨 받아 가공하고 Writer 가 보관합니다.
다시 Reader 가 100건을 읽어오며 작업을 반복해 총 10번, 1000건이 Writer에 누적되면 커밋해서 트랜잭션을 종료합니다.
하나의 청크 처리 완료를 위해서는 1000건이 필요한데 이를 100건 씩 나눠 읽어오기 때문에 DBMS 입장에서는 부하가 분산되게 됩니다.
반면 애플리케이션 입장에서는 1000건을 한번에 가져오지 않고 100건 씩 나누기 때문에 10번의 통신이 필요해 상대적으로 통신 부하가 발생합니다.
또, 1000건의 상대적으로 높은 락 점유 시간과 롤백비용이 발생하게 됩니다.
처리하는 동안 필요한 메모리는 Reader 는 100건, 청크 입장에서는 1000건이 됩니다.

이번엔 청크 크기가 페이지 크기보다 작은 경우 입니다. (page > chunk)
페이지 크기를 1000, 청크 크기를 100 이라고 가정하겠습니다.
트랜잭션이 시작되면 Reader 는 1000건을 읽어옵니다. 이후 Processor 가 한건씩 넘겨 받으며 가공하고 Writer 에게 넘겨주는데 100건이 모이면 커밋하며 트랜잭션이 종료됩니다.
다음 청크 작업이 진행되며 Reader 는 이미 읽어온 데이터가 있기 때문에 읽지 않고 이어서 Processor 에게 한 건 씩 넘겨주고 Writer 가 100건이 모이면 커밋 후 트랜잭션이 종료됩니다.
이를 10번 반복하면 같은 양의 처리가 이루어집니다.
이 경우 Reader 는 한번에 모든 데이터를 불러왔기 때문에 RDBMS 의 부하가 상대적으로 커지게 됩니다. 애플리케이션 입장에서는 한번의 통신으로 끝나기 때문에 부하가 줄어듭니다.
매 청크 처리마다 100건의 트랜잭션이 처리되므로 상대적으로 락 점유 시간과 롤백 비용이 줄어듭니다.
처리하는 동안 필요한 메모리는 Reader 1000건, 청크 입장에서는 100건 씩이기 때문에 앞서 다룬 케이스에서 위치만 변경되었을 뿐 같습니다.

각각의 특징에 따른 선택 과정을 정리해보겠습니다.
페이지 크기와 청크 크기가 같은 상황이 디폴트입니다. 특별한 이유가 없다면 단순하며 처리가 쉬운 이 형태가 우선시 됩니다.

만약 DBMS 의 부하가 신경 쓰이는 상황이라면 페이지 크기가 청크 크기보다 작은게 좋습니다.
예를 들어 낮 시간대에 수행할 배치인데 해당 DB 를 실사용자 요청 처리에도 쓴다면 DB 의 성능을 신경써야 합니다.
페이지 크기를 청크 크기보다 줄여 가벼운 조회를 여러번에 걸쳐 수행하는게 유리해집니다.

또, DB가 네트워크 너머 외부 서비스에 있다면 통신 비용이 비싸지기 때문에 페이지 크기가 청크 크기보다 큰게 유리합니다.
예를 들어 해당 처리를 위해 DB 가 물리적으로 먼, 미국에 있는 데이터 센터와 통신할 필요가 있다면 통신 비용을 줄이기 위해 횟수를 줄여야합니다.
따라서 페이지 크기를 키워 가급적 적은 회수로 데이터를 읽어오도록 한 뒤 처리하는게 유리합니다.
```

### C3.
```text
`saveAll` 호출 시 *예상하지 못한 SELECT 쿼리* 가 archived_post 테이블에 발생한 이유를 설명하시오.

```

**답변:**

```text
학습하면서 배치를 실제로 돌려보는 중에 의도하지 않은 쿼리가 생성되는걸 발견했습니다.
의도대로였다면 Post 엔티티를 읽고 ArchivedPost 를 생성한 후 새 테이블에 저장하는건데
로그를 살펴보니 쿼리 중에 ArchivedPost 테이블을 조회하는 쿼리가 발견되었습니다.

원인은 배치작업의 writer 에 선언된 saveAll() 메서드에 있었습니다.
이 메서드는 JPA 가 제공하는 기본 메서드 입니다.(SimpleJpaRepository)
해당 증상을 알기 위해 먼저 JPA 의 기본 동작 부터 파악해야합니다.
JPA 는 객체를 DB의 테이블 컬럼과 일대일 매칭해서 사용할 수 있도록 해줍니다.

JPA 는 새로운 엔티티를 저장할 때 해당 엔티티가 이미 저장되어 있는지(튜플) 아니면 새로운 엔티티인지 확인합니다. (isNew 메서드)
만약 새로운 엔티티라면 persist() 메서드를 호출해 엔티티를 저장합니다. (INSERT)
이미 존재하는 튜플이라면 merge() 메서드를 호출합니다. 이름 대로 기존 튜플과 병합하겠다는 메서드입니다.
이 메서드는 우선 기존 튜플을 조회(SELECT) 하고, 조회에 성공하면 UPDATE, 만약 조회 되지 않으면 생성한 엔티티를 저장합니다.(INSERT)

이 지점이 SELECT 쿼리 발생의 원인입니다.
JPA는 새로 생성한 ArchivedPost 엔티티를 새로 생성했다고 판단하지 않고 기존에 저장된 튜플로 파악해서 SELECT 쿼리가 나가게 되었습니다.

이제 isNew 의 판단 기준을 살펴봐야합니다.
JPA 의 isNew 판단 기준은 id 값 여부에 있습니다. (AbstractEntityInformation 의 isNew())
isNew 메서드는 엔티티에서 id 값을 추출한 뒤
id 가 래퍼 타입일 경우 null 인지 검사해서 null 이면 id 값이 없다고 판단해 새로운 엔티티로 취급하며,
원시 타입일 경우 0 인지 검사해서 0 이면 id 값이 없다고 판단해서 새로운 엔티티로 취급합니다.

JPA 를 활용할 시 새로운 엔티티를 생성한다면 @GeneratedValue 로 자동 생성 전략을 취하는게 기본 옵션입니다.
현 프로젝트에서는 아키이빙을 위해 기존에 존재하는 Post 의 정보를 읽어와 ArchivedPost 를 생성하는데 이 과정에서 이미 id 값이 할당되어 있습니다.
이 부분이 불필요한 메서드 호출을 유발하게 되었습니다.

JPA 는 id 값이 이미 존재하는 ArchivedPost 를 이미 있는 튜플로 파악하며 merge() 메서드를 호출합니다.
merge() 를 위해 SELECT 쿼리를 수행하면 먼저 영속성 컨텍스트를 확인하게 되며 ArchivedPost 는 방금 생성되었기 때문에 조회되지 않습니다.
이어서 DB 를 조회하지만 마찬가지로 조회되지 않습니다.
SELECT 된 튜플이 없기 때문에 JPA 는 해당 엔티티를 INSERT 하게 됩니다.

결론적으로, 배치 작업이 커지면 커질 수록 무의미한 SELECT 쿼리가 계속 나가게 됩니다. 불필요한 쿼리 수행 방지를 위해 해결해야 할 문제입니다.

해결책은 ArchivedPost 에 Persistable 인터페이스를 구현하는 것 입니다.
Persistable 인터페이스는 JPA 에서 엔티티가 데이터베이스에 새로 저장될 객체인지 여부를 명시적으로 지정하게 해주는 인터페이스 입니다.
isNew() 메서드는 기본적으로 @GeneratedValue 를 전제로 동작하지만,
해당 인터페이스를 구현하면 isNew() 메서드를 직접 정의할 수 있고 여기에 항상 참을 반환하도록 지정해 문제를 해결했습니다.
ArchivedPost 는 기존 Post 값을 스냅샷 한다는 목적 상 항상 새로운 엔티티가 생성될 수 밖에 없기 때문에
항상 참을 반환하도록 해도 문제가 없다고 판단됩니다.
```

### C4.
```text
`deleteById` 가 *이름과 다르게 SELECT 부터 수행하는 이유* 를 네 가지로 정리하시오. 
그리고 `deleteAllByIdInBatch` 가 *왜 이 추상화를 의도적으로 우회하는지*, *대량 처리에서 어떤 트레이드오프가 일어나는지* 설명하시오.
```

**답변:**

```text
deleteById 의 실제 동작 과정을 먼저 살펴보겠습니다.
메서드 이름 대로 Id 를 기반으로 튜플을 삭제하는 메서드입니다.
단순히 생각하면 DELETE FROM ... WHERE id ... 쿼리가 나가서 튜플을 삭제하면 될 것처럼 보입니다.
하지만 JPA 는 영속성 컨텍스트로 인해 조금 다르게 수행됩니다.

JPA 는 먼저 Id 를 이용해 해당 엔티티를 조회합니다. 영속성 컨텍스트를 확인하고, 없으면 DB 에 쿼리를 날려 조회 후 영속성 컨텍스트에 적재합니다.
그 다음 delete 메서드를 호출해 영속성 컨텍스트의 엔티티를 삭제 표시 하고, 최종적으로 트랜잭션 커밋 시점에 DELETE 쿼리를 DB 에 날려 실제 튜플이 삭제됩니다.

JPA 는 DB 테이블의 튜플들을 객체로 다루기 위한 기술스택입니다. 이를 위해 영속성 컨텍스트를 이용하는게 기본 옵션입니다.
영속성 컨텍스트란 DB의 튜플들을 객체로 다루기 위해 조회해온 뒤 메모리에 객체 형태로 보관하면서 생성, 수정, 삭제 등의 변화를 추적하고
관리할 수 있게 하는 1차 캐시 공간 입니다.
JPA 는 영속성 컨텍스트를 이용해 다양한 작업을 지원합니다.

첫번째는 라이프사이클 콜백 지원 입니다.
@PreRemove, @PostRemove 등을 통해 삭제 전 후 시점에 특정 처리를 할 수 있게 합니다.
이를 구현하기 위해서는 삭제 요청에 즉각 DB 에 쿼리를 날리지 않고 우선 영속성 컨텍스트에 대상 엔티티를 가져올 필요가 있습니다.(SELECT)
삭제 요청 시 대상을 영속성 컨텍스트에 올리고, 콜백을 처리한 다음 영속성 컨텍스트에 엔티티 삭제 처리 표시를 합니다.
이후 트랜잭션이 커밋되면 그때 진짜로 DB 에 DELETE 쿼리를 날려 실제 튜플을 삭제합니다.

두번째는 Cascade 연관관계 처리 입니다.
예를 들어 cascadeType.REMOVE 는 대상 엔티티 삭제 시 연관 엔티티도 지우는 옵션입니다.
연관 관계 처리를 위해 JPA 는 영속성 컨텍스트에 엔티티를 올리고 연관 객체 그래프를 탐색해 대상을 지운 뒤 요청이 들어온 엔티티도 삭제합니다.

세번째는 dirty checking 입니다.
해당 기능은 영속 상태인 엔티티의 변경을 감지해 자동으로 UPDATE 쿼리를 생성해주는 기능입니다.

이렇게 JPA 는 DB의 튜플들을 객체로 다루기 위해 기본적으로 영속성 컨텍스트를 활용해 작업을 처리하도록 설계되어 있습니다.
JPA 는 반드시 엔티티가 영속성 컨텍스트 일때(managed 상태일때)만 DELETE 쿼리를 생성하도록 되어 있습니다.(SELECT 가 반드시 필요)

문제는 대용량 배치 작업 처리 시 영속성 컨텍스트를 이용하며 발생합니다.
앞서 언급한대로 JPA 는 DELETE 요청을 SELECT + DELETE 로 수행합니다. 이런 점이 대용량 배치 작업과 만나 N+N 쿼리가 발생하게 됩니다.
Chunk 크기 만큼 배치 작업이 수행될때마다 2N 의 쿼리가 수행되어 엄청난 DB 왕복 비용이 소모됩니다.

이를 해결하기 위해 JPA 는 deleteAllByIdInBatch 메서드를 제공합니다.
해당 메서드는 배치 처리용 삭제 메서드로 앞서 설명한 영속성 컨텍스트를 의도적으로 따르지 않고 SELECT 없이 곧바로 JPQL DELETE 쿼리를 날립니다.
DELETE FROM Post p WHERE p.id IN :ids 쿼리 처럼 처리 대상 리스트를 한번에 보내서 삭제하도록 하기 때문에
DB 네트워크 통신 비용도 한번에 그칩니다.

하지만 이러한 메서드는 트레이드오프가 있습니다.
기본적인 영속성 컨텍스트 처리를 따르지 않기 때문에 앞서 말한 영속성 컨텍스트를 통한 기능들을 건너뛰게 되며
dirty checking 부분에서 유령 객체를 만들 여지가 있습니다.

유령 객체 문제는 영속성 컨텍스트를 거치지 않고 곧바로 DELETE 를 하게 되는 경우 발생합니다.
예를 들어 어떤 트랜잭션에서 영속성 컨텍스트에 어떤 게시글이 저장되어 있는데, deleteAllByInBatchId 가 호출되고 이를 곧바로 DB 에서 DELETE 시키면,
영속성 컨텍스트에만 해당 엔티티가 살아남게 됩니다. 이후 해당 트랜잭션에서 set 등의 변경 작업이 일어나면 dirty checking 이 작동되어
UPDATE 를 시도하게 되고, 실제 DB 에는 이미 튜플이 사라져 있는 문제가 발생합니다.

현재 프로젝트에서는 아카이빙 작업 특성 상 원본 엔티티를 제거한 이후에 조작할 일이 없기 때문에 문제를 유발하지 않습니다.
```

### D1.
```text

*왜 1000건만 처리되고 500건이 누락되었는지*, 그리고 *그 함정의 본질이 무엇인지* — 한 문장으로 압축해 정리할 것.

```

**답변:**

```text
프로젝트를 수행하면서 만난 현상은 다음과 같습니다.
Chunk 크기가 1000 이며, Page 크기는 500, 배치 대상 데이터는 1500건인 시나리오입니다.
의도대로였다면 두 개의 chunk 에 걸쳐 500건 씩 세번 작업해 1500건을 대상으로 배치 처리가 되어야 했지만, 실제 결과는 1000건 까지만 처리되었습니다.

원인을 알기 위해 페이징 기법의 작동 구조와 Reader 의 생명 주기, 우리가 작성한 배치 작업의 특성에 대해 알아야합니다.

먼저 페이징 기법은 결과 집합에서의 순서상 위치를 기반으로 작동됩니다.
500건의 페이지 크기가 설정되면 쿼리는 ORDER BY (조건) LIMIT 500 OFFSET 0 으로 시작됩니다.
이후 읽어온 500건이 다 소진되고 나면 OFFSET 값은 500이 오른채로 상태가 유지됩니다.
다음 Reader 의 읽기 호출이 들어오면 쿼리는 ORDER BY (조건) LIMIT 500 OFFSET 500 으로 나가게 되며,
이는 '결과 집합 중에 500건을 건너뛰고 500개를 불러와라' 라는 의미가 됩니다.
페이징은 이런식으로 작동합니다.

Reader 의 생명주기는 'Step 단위'로 유지됩니다.
배치 작업이 시작되고 하나의 Step 이 생성되며 그 안에 Reader 가 생성됩니다. 청크 마다 Reader 는 필요한만큼 read() 하며
하나의 청크 트랜잭션이 커밋되어도 Reader 는 상태를 유지하고 Step 이 끝나야 비로소 닫히게 됩니다.

이 두가지 특징이 작성한 배치 작업과 모여 의도치 않은 동작이 일어났습니다.
작업이 시작되면 Reader 가 페이지 크기 500 만큼 데이터를 읽어옵니다. 데이터를 처리 후 Writer 에 500건이 누적됩니다.
Chunk 크기만큼 모이지 않았기 때문에 Reader 는 다시 500건을 읽어오는데 이때 쿼리 결과에 대해 앞의 500건을 건너뛰고 가져오게됩니다.
Writer 에 1000건이 모였기 때문에 ArchivedPost 엔티티를 저장하고, 원본 Post 1000건을 삭제하여 커밋, 트랜잭션을 끝냅니다.

다음 청크 작업이 시작되며 Reader 는 생명 주기가 끝나지 않았기 때문에 아까 그 상태를 유지합니다.
Reader 는 페이지 크기를 500 만큼 읽어오려고 하는데 앞선 Chunk 에서 1000건을 읽었기 때문에 현재 OFFSET 값은 1000 입니다.
쿼리를 날려 조회해보니 1000건을 지웠기 때문에 500건 밖에 조회가 되지 않습니다. 1000건을 건너 뛰고 500 건을 읽어오려고 하면 결과는 없습니다.
배치 작업은 작업 대상이 이제 없다고 판단해 배치가 종료됩니다.

결론적으로, 결과 집합에서 위치 기반으로 데이터를 읽어왔는데 두번째 청크부터 삭제 작업으로 인해 결과 집합이 바뀌어버렸기 때문에
페이징 처리가 의도대로 수행되지 않았습니다.

위와 같은 증상을 해결하기 위해 결과 집합의 순서상 위치 기반이 아닌 고정된 값,
예를 들어 튜플들의 id 값을 통해 처리를 해야 해당 현상을 해결할 수 있습니다.
```

```text

```

### E1.
```text
자바 제네릭의 불공변성 (invariance) 이 무엇이고 왜 그렇게 설계되었는지 설명하시오.
'List<Dog> 가 List<Animal> 이 아니다' 라는 사실이 타입 안전성을 위해 왜 필요한지 시뮬레이션으로 보일 것.
```

**답변:**

```text
먼저 자바의 제네릭에 대해 알아보겠습니다.

자바 1.4 이전 버전은 컬렉션으로 Object 타입만 다룰 수 있었습니다. 때문에 코드 작성 시 의도치 않은 타입 섞임이 일어나고 런타임 에러가 발생했습니다.
해당 버전의 자바에서는,
ArrayList list = new ArrayList();
list.add("Hello");
list.add(123);
같은 코드를 작성해도 모든 타입은 Object 의 하위 타입이기 때문에 컴파일 에러 없이 넘어갔습니다.
따라서 데이터를 꺼내면 수동 형 변환을 해야 했습니다.
(String) list.get(0);
자칫 123 을 꺼내면서 문자열로 캐스팅한다면 ClassCastException, 런타임 에러가 발생하게 됩니다.

이를 해결하고자 자바 5+ 에서 제네릭이 도입되었습니다.
제네릭을 통해 컬렉션들은 자기가 다루는 타입이 어떤 타입인지를 표시하게 되었습니다. List<Integer> : Integer 타입을 다루는 List 컬렉션
이제 컴파일 타임에 안정성을 보장하고 수동 캐스팅과 그 실패 위험을 제거할 수 있게 됩니다.

없던 제네릭이 생겼기 때문에 자바는 하위 호환성을 신경써줘야 합니다. 이를 위해 타입 소거를 도입하게 됩니다.
코드를 사용할때는 제네릭을 작성하고 이를 컴파일러가 읽고 잘못된 코드에 대해 경고를 해줍니다. 작성이 끝난 후 컴파일 되면 실제 컴파일 코드에서는
제네릭을 전부 지우고 제네릭이 없던 시절의 Object 기반 코드로 돌아가 알맞은 캐스팅을 붙여주게 됩니다.
List<String> list = new ArrayList<>(); -> List list = new ArrayList();
list.add("apple");
String fruit = list.get(0); -> String fruit = (String) list.get(0); 

이제 불공변에 대해 알아보겠습니다.

먼저 공변이란 'A 타입의 하위 타입인 B 가 있다면, A 컬렉션으로 선언된 저장 공간에 B 타입 컬렉션을 할당할 수 있다' 는 뜻 입니다.
대표적인 공변 컬렉션은 배열입니다. 배열을 예로 들면,
Integer 타입은 Number 타입의 하위 타입이기 때문에, Number[] 배열에는 Integer[] 값이 할당 될 수 있다는 뜻 입니다.
Number[] numberAry = new Integer[10];
공변성은 제네릭 등장 이전부터 적용되던 개념이었고
이는 다형성(Integer 타입을 Number 타입 변수에 대입 가능)을 컬렉션으로 확장하기 위해 적용된 개념입니다.
(사실 배열이 공변인건 자바의 유명한 결함입니다. 제네릭이 불공변이 되게 된 계기입니다.
배열이 공변이기 때문에
Object[] arr = new Integer[3];
arr[0] = "hello";
같은 코드가 가능해지고 이는 컴파일 타임에는 넘어가지만 런타임에 ArrayStoreException 을 발생시킵니다.
이런 점을 막기 위해 제네릭이 불공변이 되었습니다.)

불공변은 'A 타입의 하위 타입인 B 가 있다면, A 컬렉션으로 선언된 저장 공간에 B 타입 컬렉션을 할당할 수 없다' 는 뜻 입니다.
제네릭을 도입하면서 자바 컬렉션에 불공변 개념이 적용되었습니다.
List<Number> numberList = new ArrayList<Integer>(); 가 불가능하단 뜻 입니다.

만약 제네릭이 도입되었는데 공변하다고 가정하면 다음과 같은 문제가 발생합니다.
List<Number> list = new ArrayList<Integer>(); 가 가능해진다고 하면,
알맹이가 List<Integer> 여도 겉으로는 list 는 여전히 Number 타입의 컬렉션이기 때문에 Double 등의 Number 의
하위 타입을 담을 수 있게 되고 이는 타입 안정성을 보장하지 못하게 됩니다.
따라서 제네릭 컬렉션은 불공변을 적용하게 되었습니다.

다만, 이렇게 불공변으로만 취급해버리면 완벽히 안전하겠지만 코드의 유연성이 많이 떨어지게 됩니다.
이를 해결하기 위해 도입된게 '?' 라는 와일드 카드 입니다.
자바는 이제 와일드카드를 통해 공변, 반공변을 부여할 수 있게 됩니다. 

먼저 공변 부여 입니다.
<? extends Number>
제네릭을 이렇게 선언하면 이는 '제네릭에 들어올 타입을 Number 타입과 그 하위 타입으로만 제한하고 읽기 전용으로만 쓸 수 있다' 는 뜻이 됩니다.

다음으로 반공변 부여 입니다.
<? super Integer>
이 제네릭은 '제네릭에 들어올 타입을 Integer 타입과 그 상위 타입으로만 제한하고 쓰기 전용으로만 쓸 수 있다' 는 뜻이 됩니다.

예시를 통해 이렇게 정해진 이유를 정리해보겠습니다.
스마트폰 과 아이폰, 갤럭시 란 클래스가 있으며, 아이폰과 갤럭시는 스마트폰의 하위 타입이라고 가정하겠습니다.

<? extends 스마트폰> 이라는 변수 공간이 만들어지면 여기에는 스마트폰의 하위 타입까지 들어올 수 있습니다.
따라서 이 변수 공간에는 List<아이폰> 이 들어올 수 있습니다.
<? extends 스마트폰> list = 아이폰리스트;
만약 쓰기를 허용하게 되면 아이폰 리스트로 쓸 변수 공간에 list.add(갤럭시) 같은 코드를 작성해도 컴파일러가 이를 감지해 미리 막지 못합니다.
이를 막기 위해 컴파일러는 원천 차단을 위해 해당 변수를 읽기 전용으로 취급하게 됩니다.

마찬가지로
<? super 아이폰> 이라는 변수 공간이 만들어지면 여기에는 아이폰의 상위 타입이 들어올 수 있습니다.
즉, 최소 아이폰 임을 보장하는 변수 공간이 됩니다.
<? super 아이폰> list = new ~;
list 변수 공간에는 아이폰리스트가 들어올 수도 있고, 스마트폰리스트가 들어올 수도 있고, 오브젝트리스트가 들어올 수 도 있습니다.

따라서, 컴파일러는 아이폰의 하위타입에 대한 쓰기를 지원할 수 있습니다.
어떤 리스트가 공간에 할당되던간에, 아이폰 객체와 아이폰X,아이폰8플러스 등의 아이폰 하위타입은 쓰기가 가능합니다.
list.add(아이폰8플러스);
list.add(아이폰X);

문제는 읽기 작업입니다.
컴파일러는 list 가 어떤 타입인지 알지 못 합니다. 그렇기 때문에 안전하게 받으려면 무조건 Object 타입임을 가정해야합니다.
그래야 실제로 어떤 타입 리스트가 할당되던지 처리가 가능합니다. 하지만 이러면 객체를 Object 로 꺼내봐야 실질적 비즈니스 로직을 수행할 수 없습니다.
따라서 반공변은 읽기 작업에 대해서는 부적합(쓸모없음) 쓰기 전용으로 사용하게 됩니다.
```

### E2.

```text
? extends T 와 ? super T 의 차이를 설명하고, **PECS 원칙** (Producer Extends, Consumer Super) 의 의미를 정리하시오. 
? extends T 가 읽기 전용인 이유 와 ? super T 가 쓰기 전용인 이유 를 컴파일러가 무엇을 모르는가 의 관점에서 설명할 것.
```

**답변:**

```text
? extends T
T 가 상한선인 제네릭, 해당 저장공간에는 T 와 그 하위타입의 컬렉션들을 할당할 수 있으며 읽기 작업 전용으로 쓰인다.
? super T
T 가 하한선인 제네릭, 해당 저장공간에는 T 와 그 상위타입의 컬렉션들을 할당할 수 있으며 쓰기 작업 전용으로 쓰인다.

컴파일러는 작성된 <? extends T>, <? super T> 에 대해 실제로 어떤 타입이 올지 알 수 없습니다.
대상은 런타임에 정해지기 때문에
extends 의 경우 하위 타입 컬렉션이 할당되었을 때 상위 타입의 객체를 쓰려고하면 컴파일 에러가 발생합니다.
super 의 경우 어디까지의 상위 타입 컬렉션이 할당 될지 알 수 없기 때문에 항상 성공하려면 오브젝트 타입으로만 객체를 다뤄야하며
이는 실질적으로 활용할 여지가 없습니다.
따라서 각각 읽기 전용, 쓰기 전용으로 활용처를 제한해 컴파일 타임에 오류를 붙잡을 수 있게 됩니다.

오브젝트 - 스마트폰 - 아이폰 - 아이폰X, 아이폰8
               ㄴ 갤럭시 - 갤럭시10
PECS, Producer Extends, Consumer Super
해당 원칙은 extends 제네릭은 읽기 전용, 공급자로만 쓰이며, super 제네릭은 쓰기 전용, 소비자로만 쓰인다는 뜻입니다.
얼핏 단어만 보면 읽기 전용이 소비자이고 쓰기 전용이 공급자로 보이지만
실제 활용처에서는 해당 제네릭 컬렉션이 다른 컬렉션을 대상으로 공급, 소비 하기 때문에 붙여진 이름입니다.

주로 이 와일드카드 제네릭이 선언된 컬렉션들은 다른 컬렉션들을 받아서 처리하기 위한 공간으로 쓰입니다.
? extends T 부터 살펴보겠습니다.
파라미터에 <? extends 스마트폰> 제네릭 타입의 저장 공간을 선언하면, 해당 메서드는 스마트폰과 그 하위의 어떤 타입이던지 넘겨받을 수 있게 됩니다.
따라서 해당 저장공간은 최소 스마트폰 타입 저장공간으로 넘겨받은 컬렉션을 다룰 수 있기 때문에 스마트폰 타입으로 읽기에 안전합니다.
이를 이용해 그것이 아이폰이던, 갤럭시던 상관하지 않으니 대상 컬렉션들의 가격을 읽어 총 가격을 구하는 등의 메서드를 작성할 수 있습니다.
따라서 해당 메서드에서 제네릭 컬렉션은 외부에서 넘겨받은 스마트폰 혹은 그 하위 타입의 어떤 컬렉션을 메서드에 공급하게 됩니다. 

? super T 도 비슷한 형태로 쓰입니다.
파라미터에 <? super 아이폰> 제네릭 타입의 저장 공간을 선언하면, 해당 메서드는 아이폰, 스마트폰, 오브젝트 타입의 컬렉션을 넘겨받을 수 있게 됩니다.
따라서 해당 저장공간은 아이폰과 그 하위 타입은 항상 들어갈 수 있기 때문에 쓰기에 안전합니다.
이를 이용해 파라미터로 아이폰 컬렉션과 ? super 아이폰 컬렉션을 함께 받도록 선언하면
아이폰 컬렉션의 객체들을 상위 타입의 바구니에 옮겨 담을 수 있습니다.
외부에서 함께 넘겨받은 컬렉션을 소비해 같은, 혹은 상위 타입의 컬렉션을 대상으로 작업하기 때문에 소비하는 모양이 됩니다.
```

```text
먼저 cron 의 표현 방식은 다음과 같습니다.
(* * * * * *) 에서 각 자리는 앞에서부터
초, 분, 시간, 일, 월, 요일 을 의미합니다.

예를 들어 초는 0부터 59초까지 존재합니다.

만약 초 자리에 30 이라고 적으면 이는 30초 를 지정한다는 의미가 됩니다.

* 기호는 컴퓨터에서 Asterisk, 아스테리스크 라고 부르며 '전체' 라는 의미를 가집니다.

초 자리 */30 이 오면 이는 간격을 의미합니다.
'*' : 모든 초를 대상으로 하는데
'/30' : 30초 간격으로 수행한다 는 의미가 됩니다. 이 값은 이제 0초, 30초, 0초, ... 를 의미하게 됩니다.

초 자리에 1,2,3 이 오면 이는 1초, 2초, 3초 를 지정한다는 의미입니다.

'-' 기호는 범위입니다. 1-3 은 1부터 3까지, 1초, 2초, 3초 의 범위를 지정하게 됩니다.


이제 질문의 cron 을 읽어보면, (* */1 * * * *)
모든 요일, 매월, 매일, 매시간(0시~23시), 1분 간격 마다, 매초
라는 의미가 됩니다.
*/1 이라고 쓰면 모든 분을 대상으로 1분 간격을 지정하기 때문에 결국 모든 분을 대상으로 한다는 것과 같은 뜻이 됩니다.
결국 해당 cron 은 (* * * * * *) 으로 모든 시간 매초마다 배치를 실행하도록 하게 합니다.

본래 목적은 1분 마다 실행이었기 떄문에 (0 * * * * *) 으로 매분을 대상으로 0초마다 수행, 따라서 1분마다 수행되도록 작성할 수 있습니다.
혹은 (0 */1 * * * *) 도 모든 분이 1분 간격이기 때문에 똑같은 의미가 됩니다.

원래 유닉스에서 온 표준 cron 표기법은 초를 빼고 5가지 필드를 가지지만 Spring 은 더 정밀한 표현을 지원하기 위해 6가지 필드를 가집니다.
```
