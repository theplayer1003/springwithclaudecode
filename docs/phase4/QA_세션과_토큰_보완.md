# 세션과 토큰의 단점 보완

## Q. 세션과 토큰 각각의 단점을 현대 애플리케이션에서는 어떻게 보완하는가?

### 세션의 단점 보완 — 외부 세션 저장소 (Redis)

세션의 핵심 단점은 서버 메모리에 세션을 저장하므로 서버 확장이 어렵다는 것이었다. Redis 같은 외부 저장소를 사용하면 모든 서버가 공유 저장소를 바라보므로 어떤 서버로 요청이 가든 세션을 조회할 수 있다. 확장성 문제가 해결되고 강제 로그아웃(세션 삭제)도 가능하다.

### JWT의 단점 보완 — Refresh Token

JWT의 핵심 단점은 토큰이 탈취되면 만료 전까지 막을 수 없다는 것이었다. 토큰을 두 개로 분리하여 완화한다.

- **Access Token** — 수명 짧음 (15분~1시간). API 요청에 사용
- **Refresh Token** — 수명 김 (7일~30일). Access Token 재발급에만 사용

Access Token이 탈취되어도 짧은 시간 후 만료된다. Refresh Token은 서버 DB에 저장하므로 강제 폐기도 가능하다.

### 하이브리드 방식

실무에서는 순수한 세션만, 순수한 JWT만 쓰는 경우는 드물다. 세션과 토큰을 동시에 사용하는 것이 아니라, **JWT의 무상태 장점은 유지하면서 제어가 필요한 부분만 서버 측 저장소를 활용**하는 방식이다.

- Access Token(JWT) + Refresh Token(서버 DB/Redis 저장) — 가장 일반적
- JWT + Redis 블랙리스트 — 로그아웃한 토큰을 Redis에 등록하여 차단
