# 패키지 구조 결정

## Q. 새 클래스를 만들 때 어떤 패키지에 넣어야 하는지 어떻게 판단하나?

**A:** 핵심 질문은 하나다: **"이 클래스가 하는 일의 성격은 무엇인가?"**

### 판단 기준 (자문할 것)

```
1. 이 클래스가 하는 일의 성격은? (설정? 로직? 데이터 접근? 초기화?)
2. 기존 패키지 중 같은 성격의 것이 있는가?
3. 없다면 — 앞으로 같은 성격의 클래스가 더 생길 수 있는가?
   → 있으면 새 패키지 생성
   → 없으면 가장 가까운 기존 패키지에 배치
```

### 예시: 기존 패키지의 역할

```
controller/   → HTTP 요청을 받아서 응답하는 클래스
service/      → 비즈니스 로직
repository/   → DB 접근
entity/       → DB 테이블과 매핑되는 객체
dto/          → 요청/응답 데이터 전달 객체
exception/    → 예외 정의
security/     → 인증/인가 관련 (Filter, JWT, SecurityConfig)
```

각 패키지는 "이 안의 클래스들은 이런 역할을 한다"라는 공통점을 가진다.

## Q. `config`와 `init` 중 어디에 넣어야 하나? (AdminInitializer 사례)

**A:** 둘 다 틀린 답은 아니다. 성격의 차이를 이해하고 선택하면 된다.

| 패키지      | 의미                    | 들어가는 클래스                              |
|----------|-----------------------|---------------------------------------|
| `config` | 애플리케이션 구성, 빈 등록, 설정   | SecurityConfig, CORS 설정, Swagger 설정 등 |
| `init`   | 초기 데이터 투입, 기동 시 1회 실행 | 초기 ADMIN 생성, 테스트 데이터 삽입 등             |

`config`는 "애플리케이션이 어떻게 동작할지 정의"하는 것이고, `init`은 "애플리케이션이 시작될 때 데이터를 준비"하는 것이다.

초기 ADMIN 생성은 "설정"이 아니라 "데이터 준비"이므로 `init`이 더 명확하다. 다만 `config`에 넣는 프로젝트도 많으며, 이 경우 "CommandLineRunner도 Spring Bean이고 기동
시 구성의 일부"라는 근거를 가진다.

## Q. 패키지 구조에 절대적인 정답이 있는가?

**A:** 없다. 중요한 건 두 가지다:

1. **프로젝트 내 일관성** — 같은 성격의 클래스가 여기저기 흩어져 있지 않은가
2. **직관성** — "이 코드 어디 있지?"라고 생각했을 때 자연스럽게 찾을 수 있는가

하나의 패키지에 성격이 다른 것들이 섞이기 시작하면, 분리할 수 있는지 고민해보는 것이 좋다. 예를 들어 `config`에 SecurityConfig, CorsConfig, AdminInitializer가 모두
들어가면 "설정"과 "초기화"가 섞이므로, `init`으로 분리하는 편이 유지보수에 유리하다.
