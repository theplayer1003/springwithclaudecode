# HTTP 상태 코드

## Q. HTTP 상태 코드란?

**A:** 서버가 클라이언트의 요청을 처리한 결과를 숫자 3자리로 알려주는 약속이다. 첫 번째 자리가 카테고리를 나타낸다.

| 범위 | 카테고리 | 의미 |
|------|---------|------|
| 2xx | 성공 | 요청이 정상 처리됨 |
| 3xx | 리다이렉션 | 다른 곳으로 이동 필요 |
| 4xx | 클라이언트 오류 | 요청 쪽에 문제 있음 |
| 5xx | 서버 오류 | 서버 쪽에 문제 있음 |

---

## Q. Spring에서 상태 코드는 어떻게 사용하나?

**A:** `HttpStatus` enum으로 선언되어 있고, `ResponseEntity`와 함께 사용한다.

```java
return ResponseEntity.status(HttpStatus.CREATED).body(response);  // 201
return ResponseEntity.status(HttpStatus.CONFLICT).body(error);    // 409
```

---

## Q. 자주 사용하는 상태 코드는?

### 2xx — 성공

| 코드 | Spring 선언 이름 | 의미 | 사용 상황 |
|------|-----------------|------|----------|
| 200 | `OK` | 요청 성공 | GET 조회, PUT 수정 후 결과 반환 |
| 201 | `CREATED` | 리소스 생성 완료 | POST로 새 리소스 생성 시 |
| 204 | `NO_CONTENT` | 성공, 반환 본문 없음 | DELETE 성공 시 |

### 3xx — 리다이렉션

| 코드 | Spring 선언 이름 | 의미 | 사용 상황 |
|------|-----------------|------|----------|
| 301 | `MOVED_PERMANENTLY` | 영구 이동 | URL이 영구적으로 변경됨 |
| 302 | `FOUND` | 임시 이동 | 임시 리다이렉트 |
| 304 | `NOT_MODIFIED` | 변경 없음 | 캐시된 리소스가 여전히 유효 |

### 4xx — 클라이언트 오류

| 코드 | Spring 선언 이름 | 의미 | 사용 상황 |
|------|-----------------|------|----------|
| 400 | `BAD_REQUEST` | 잘못된 요청 | 입력값 검증 실패, 요청 형식 오류 |
| 401 | `UNAUTHORIZED` | 인증 필요 | 로그인하지 않은 사용자의 접근 |
| 403 | `FORBIDDEN` | 권한 없음 | 로그인은 했지만 권한 부족 |
| 404 | `NOT_FOUND` | 리소스 없음 | 존재하지 않는 게시글 조회 |
| 405 | `METHOD_NOT_ALLOWED` | 허용되지 않는 메서드 | GET만 허용된 URL에 POST 요청 |
| 409 | `CONFLICT` | 충돌 | 중복 아이디, 동시 수정 충돌 |
| 422 | `UNPROCESSABLE_ENTITY` | 처리 불가 | 형식은 맞지만 비즈니스 규칙 위반 |
| 429 | `TOO_MANY_REQUESTS` | 요청 과다 | API 호출 횟수 제한 초과 |

### 5xx — 서버 오류

| 코드 | Spring 선언 이름 | 의미 | 사용 상황 |
|------|-----------------|------|----------|
| 500 | `INTERNAL_SERVER_ERROR` | 서버 내부 오류 | 예상치 못한 예외 발생 |
| 502 | `BAD_GATEWAY` | 게이트웨이 오류 | 프록시/로드밸런서가 백엔드 응답 실패 |
| 503 | `SERVICE_UNAVAILABLE` | 서비스 불가 | 서버 점검, 과부하 |
| 504 | `GATEWAY_TIMEOUT` | 게이트웨이 타임아웃 | 백엔드 응답 시간 초과 |

## Q. REST API에서 자주 쓰는 조합은?

```
GET    성공 → 200    / 없으면 → 404
POST   성공 → 201    / 중복 → 409    / 입력 오류 → 400
PUT    성공 → 200    / 없으면 → 404
DELETE 성공 → 204    / 없으면 → 404
인증 실패 → 401      / 권한 부족 → 403
```

---

## Q. 401과 403의 차이는?

**A:** 인증(Authentication)과 인가(Authorization)의 차이다.

- **401 UNAUTHORIZED**: "너 누구야?" — 인증 자체가 안 된 상태. 토큰이 없거나 만료됨.
- **403 FORBIDDEN**: "너 누군지는 알지만, 이건 못 해" — 인증은 됐지만 해당 리소스에 대한 권한이 없음.

```
일반 사용자가 관리자 API 호출:
  로그인 안 한 상태  → 401
  로그인은 했지만 USER 권한  → 403
```

---

## Q. 400과 409는 언제 구분하나?

**A:**

- **400 BAD_REQUEST**: 요청 자체가 형식적으로 잘못됨. 필수 필드 누락, 타입 불일치, 검증 실패.
- **409 CONFLICT**: 요청 형식은 정상이지만, 현재 서버 상태와 충돌. 이미 존재하는 아이디로 가입 시도 등.

```
비밀번호 빈 값으로 회원가입 → 400 (입력값 검증 실패)
이미 있는 아이디로 회원가입 → 409 (서버 데이터와 충돌)
```
