# 비밀번호 암호화와 솔트

## Q1. 비밀번호를 왜 암호화해야 하는가?

**A:** DB가 해킹당하면 평문 비밀번호가 그대로 노출된다. 사람들은 여러 사이트에서 같은 비밀번호를 쓰는 경우가 많아서 하나가 뚫리면 다른 서비스까지 연쇄 피해를 준다.

## Q2. 해시만으로 충분하지 않은 이유는?

**A:** 해시는 단방향 변환이라 역산은 불가능하지만 두 가지 문제가 있다.

1. **같은 비밀번호는 같은 해시** — DB를 보면 같은 비밀번호를 쓰는 사용자를 바로 식별 가능
2. **레인보우 테이블 공격** — 흔한 비밀번호 수백만 개를 미리 해시로 만들어둔 표와 대조하면 원본을 찾을 수 있음. 해시를 역산한 게 아니라 미리 만들어둔 답안지와 대조하는 것.

## Q3. 솔트(Salt)가 이 문제를 어떻게 해결하는가?

**A:** 각 사용자마다 랜덤한 값(솔트)을 비밀번호에 섞어서 해시한다. 같은 비밀번호 "1234"라도 솔트가 다르면 완전히 다른 해시가 나온다. 레인보우 테이블에는 "1234"의 해시는 있지만 "1234 +
랜덤솔트"의 해시는 없으므로 기존 테이블이 무력화된다.

## Q4. 솔트가 해시에 포함되어 드러나 있는데 의미가 있는가?

**A:** 솔트의 목적은 비밀 유지가 아니라 **같은 비밀번호라도 다른 해시가 나오게 만드는 것**이다. 공격자가 솔트를 알아도 사용자별로 레인보우 테이블을 따로 만들어야 하므로 현실적으로 불가능하다. 결국 한
명씩 무차별 대입을 해야 하는데, BCrypt는 의도적으로 해시 연산을 느리게 만들어서 이것마저 비현실적으로 만든다.

## Q5. BCrypt의 matches()는 어떻게 비교하는가?

**A:** BCrypt 해시 문자열 안에 알고리즘, 비용, 솔트, 해시값이 모두 포함되어 있다.

```
$2a$10$N9qo8uLOickgx2ZMRZoMye IjZAgcfl7p92ldGxad68LJZdL17lhWy
 알고리즘 비용  솔트(22자)              해시값
```

`matches("평문", "저장된해시")` 호출 시:

1. 저장된 해시에서 솔트 추출
2. 입력된 평문 + 추출한 솔트로 해시 생성
3. 생성한 해시와 저장된 해시 비교 → 일치하면 true
