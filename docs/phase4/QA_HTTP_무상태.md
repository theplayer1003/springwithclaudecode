# HTTP 무상태(Stateless)의 이점

## Q. 무상태 프로토콜을 사용해서 얻는 이점은?

**A:**

### 1. 서버 확장이 쉬움 (가장 큰 이점)

상태가 있으면 클라이언트 A의 요청은 항상 "A의 상태를 기억하는 서버"로 가야 한다.

```
[상태 있음]
클라이언트 A → 서버 1 (A의 세션이 여기 있음)
클라이언트 A → 서버 2 (A의 세션이 없음 → 인증 실패!)
```

무상태면 아무 서버나 요청을 처리할 수 있다. 서버를 10대, 100대로 늘려도 문제없다.

```
[무상태]
클라이언트 A → 서버 1 (토큰 검증 → OK)
클라이언트 A → 서버 2 (토큰 검증 → OK)
클라이언트 A → 서버 3 (토큰 검증 → OK)
```

대규모 서비스에서 트래픽이 몰리면 서버를 추가하는 것으로 대응한다. 무상태여야 이것이 가능하다.

### 2. 서버 장애에 강함

서버 1이 죽어도 클라이언트는 서버 2로 요청하면 된다. 상태가 서버에 없으니 잃어버릴 것도 없다.

상태가 있으면 서버 1이 죽으면 그 서버에 저장된 모든 세션(사용자 로그인 상태)이 사라진다.

### 3. 서버 리소스 절약

서버가 사용자별 상태를 메모리에 보관하지 않아도 된다. 동시 접속자가 10만 명이면 10만 개의 세션을 메모리에 유지해야 하는데, 무상태면 그럴 필요가 없다.
