# 인증과 인가 개념

---

## 1. 인증(Authentication)과 인가(Authorization)

이 두 단어는 비슷해 보이지만 완전히 다른 개념입니다.

- **인증(Authentication)** — "너는 누구인가?" → 신원 확인
- **인가(Authorization)** — "너는 이걸 할 수 있는가?" → 권한 확인

현실 비유:

- 회사 건물에 들어갈 때 **사원증을 보여주는 것** → 인증 (이 사람이 직원이 맞는지 확인)
- 서버실에 들어가려 할 때 **출입 권한이 있는지 확인** → 인가 (직원이긴 하지만, 서버실 출입 권한이 있는가?)

인증이 먼저, 인가가 그 다음입니다. 누구인지 모르면 권한을 확인할 수도 없습니다.

---

## 2. 현재 우리 게시판의 문제

지금 게시판은 **누구나 모든 것을 할 수 있습니다.**

```
아무나 → POST /posts → 게시글 생성 (누가 쓴 건지 알 수 없음)
아무나 → DELETE /posts/1 → 남의 게시글 삭제 가능
아무나 → PUT /posts/1 → 남의 게시글 수정 가능
```

해결해야 할 것:

1. **회원가입/로그인** — 사용자를 식별할 수 있어야 함
2. **게시글에 작성자 정보** — 누가 쓴 글인지 기록
3. **권한 제어** — 본인 글만 수정/삭제 가능, 관리자는 모든 글 관리 가능

## 3. 웹에서 인증이 어려운 이유 — HTTP의 무상태(Stateless) 특성

HTTP는 **무상태 프로토콜**입니다. 각 요청은 독립적이고, 서버는 이전 요청을 기억하지 않습니다.

```
요청 1: POST /login (아이디: user1, 비밀번호: 1234) → 로그인 성공!
요청 2: GET /posts → 서버: "너 누구야?" (요청 1을 기억하지 못함)
```

그래서 로그인 후 "나 이미 인증된 사람이야"를 매 요청마다 증명해야 합니다. 이를 위한 대표적인 방법이 두 가지 있습니다.

### 세션(Session) 기반

```
1. 클라이언트: POST /login (아이디, 비밀번호)
2. 서버: 인증 성공 → 서버 메모리에 세션 저장 → 세션 ID를 쿠키로 전달
3. 클라이언트: 이후 모든 요청에 쿠키(세션 ID) 포함
4. 서버: 세션 ID로 서버 메모리를 조회 → "아, user1이구나"
```

- 장점: 구현이 간단, 서버가 세션을 관리하므로 강제 로그아웃 가능
- 단점: **서버가 상태를 저장해야 함**. 서버가 여러 대일 때 세션 공유 문제 발생

### 토큰(Token) 기반 — JWT

```
1. 클라이언트: POST /login (아이디, 비밀번호)
2. 서버: 인증 성공 → JWT 토큰 생성 → 클라이언트에 전달
3. 클라이언트: 이후 모든 요청의 헤더에 토큰 포함
4. 서버: 토큰을 검증 → "아, user1이구나" (서버 메모리 조회 필요 없음)
```

- 장점: **서버가 상태를 저장하지 않음**. 서버가 여러 대여도 문제없음 (확장성)
- 단점: 토큰이 탈취되면 만료 전까지 막을 수 없음 (강제 로그아웃 어려움)

우리 프로젝트에서는 **JWT 기반 인증**을 사용합니다. 대규모 서비스에 관심이 있으므로 확장성이 좋은 방식을 선택합니다.

## 4. JWT (JSON Web Token) 구조

JWT는 세 부분으로 이루어진 문자열입니다. 점(`.`)으로 구분합니다.

```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyMSIsInJvbGUiOiJVU0VSIiwiZXhwIjoxNzE3MjAwMDAwfQ.abc123signature
```

```
Header.Payload.Signature
```

### Header — 토큰 메타정보

```json
{
  "alg": "HS256",
  // 서명에 사용할 알고리즘
  "typ": "JWT"
  // 토큰 타입
}
```

### Payload — 실제 데이터 (Claims)

```json
{
  "sub": "user1",
  // subject: 사용자 식별자
  "role": "USER",
  // 사용자 역할
  "exp": 1717200000
  // 만료 시간 (Unix timestamp)
}
```

Payload에는 비밀번호 같은 민감 정보를 넣으면 안 됩니다. JWT는 **암호화가 아니라 인코딩**이므로 누구나 Payload를 읽을 수 있습니다.

### Signature — 위변조 방지

```
HMACSHA256(
    base64(header) + "." + base64(payload),
    서버만 아는 비밀키
)
```

서버만 아는 비밀키로 서명합니다. 누군가 Payload를 조작하면(예: role을 ADMIN으로 변경) Signature가 맞지 않게 되어 서버가 위변조를 감지합니다.

## 5. 역할(Role) 기반 인가

인증된 사용자라도 모든 것을 할 수 있는 건 아닙니다. **역할(Role)**에 따라 접근 가능한 범위가 다릅니다.

| 역할            | 할 수 있는 것                         |
|---------------|----------------------------------|
| 비로그인          | 게시글/댓글 조회만 가능                    |
| USER (일반 사용자) | 조회 + 게시글/댓글 작성 + **본인 글만** 수정/삭제 |
| ADMIN (관리자)   | 모든 기능 + **모든 글** 수정/삭제           |

## 6. 전체 인증/인가 흐름 (우리가 구현할 것)

```
[회원가입]
클라이언트 → POST /auth/signup (username, password, email)
서버 → 비밀번호 암호화 → DB에 저장 → 성공 응답

[로그인]
클라이언트 → POST /auth/login (username, password)
서버 → DB에서 사용자 조회 → 비밀번호 검증 → JWT 토큰 생성 → 토큰 반환

[인증이 필요한 요청]
클라이언트 → POST /posts (헤더: Authorization: Bearer eyJhbG...)
서버 → 토큰 검증 → 사용자 정보 추출 → 요청 처리

[인가가 필요한 요청]
클라이언트 → DELETE /posts/1 (헤더: Authorization: Bearer eyJhbG...)
서버 → 토큰 검증 → 사용자 정보 추출 → "이 게시글의 작성자인가?" 확인 → 삭제 실행 또는 거부
```

## 7. 핵심 정리

| 개념                     | 설명                                                            |
|------------------------|---------------------------------------------------------------|
| **인증(Authentication)** | "너는 누구인가?" — 신원 확인                                            |
| **인가(Authorization)**  | "너는 이걸 할 수 있는가?" — 권한 확인                                      |
| **무상태(Stateless)**     | HTTP는 이전 요청을 기억하지 않으므로 매 요청마다 인증 정보를 보내야 함                    |
| **세션 기반**              | 서버가 상태를 저장. 간단하지만 확장성 문제                                      |
| **JWT 기반**             | 서버가 상태를 저장하지 않음. 확장성 좋음                                       |
| **JWT 구조**             | Header + Payload + Signature. Payload는 누구나 읽을 수 있으므로 민감 정보 금지 |
| **역할(Role)**           | 사용자별 접근 가능한 범위를 구분 (USER, ADMIN 등)                            |
