## 1. IoC와 DI, Bean

### 의존성이란

A 클래스가 필드로 B 클래스를 가지고 있을 경우 이를 의존성이 있다고 말한다.
A 클래스는 B 클래스가 있어야만 동작할 수 있는 상황이다.

이를 해결하는 방법에는 직접 생성하는 방법과 외부에서 주입해주는 방법이 있다.

### 직접 생성의 문제점

직접 생성하는 방법의 경우 두 클래스가 강하게 결합하여 문제점이 발생한다.

- **변경에 취약하다** — B 클래스가 변경되면 자연스럽게 이를 소유하고 있는 A 클래스에도 변경의 여파가 일어난다.
- **테스트가 어렵다** — B 클래스가 외부에 통신을 할 경우 테스트 코드 작성 시 필요에 따라 가짜 객체를 넣어줘야 할 수 있는데, 이미 명시적으로 어떤 객체가 필요한지 선언되어 있어 추후에 이를 교체해줄 수가 없다.

### DI (의존성 주입)

이런 문제를 해결하기 위해 외부에서 주입해주는 의존성 주입, DI의 개념이 필요하다.
A 클래스의 필드를 곧바로 초기화하지 않고 객체 생성 시 외부에서 해당 필드에 초기화시키고 싶은 객체를 파라미터로 넘겨주는 방식으로 동작한다.

이렇게 하면 인터페이스 등의 다형성을 사용해 B 클래스의 변경에도 인터페이스만 잘 유지해주면 되기 때문에 여파를 적게 받으며, 테스트 시 필요에 따라 개량된 객체를 주입해 처리할 수 있다.

DI에는 세 가지 방식이 있다:

- **생성자 주입 (권장)** — 생성자의 매개변수로 의존성을 받는 방식. `private final`로 선언해 한번 주입되면 변경 불가하여 안전하다. 실무에서 가장 권장되는 방식이다.
- **필드 주입** — `@Autowired`를 필드에 직접 선언. 코드가 짧지만 `final`을 쓸 수 없고 테스트가 어려워 프로덕션 코드에서는 지양한다.
- **세터 주입** — setter 메서드를 통해 주입. 선택적 의존성에 사용할 수 있지만 대부분의 경우 생성자 주입이 더 좋다.

### IoC (제어의 역전)

일반적인 프로그래밍에서는 개발자가 직접 객체를 만들고 연결한다. 스프링에서는 이 객체 생성과 연결의 제어권이 프레임워크로 넘어간다. 이를 제어의 역전이라 한다. DI는 이 IoC를 구현하는 구체적인 방법이다.

### Bean (빈)

Bean은 스프링이 관리하는 객체를 말한다. `@Component`, `@Controller`, `@Service`, `@Repository` 등의 어노테이션을 선언해서 지정할 수 있고, 애플리케이션이 실행되면 스프링 프레임워크가 자동으로 빈 컨테이너에 등록해서 관리하게 된다.

빈은 애플리케이션 실행 후 객체가 한 번만 생성되어 해당 하나의 객체를 돌려 사용하게 된다. 이를 싱글톤이라고 한다. 이를 통해 메모리를 절약하며 해당 클래스의 일관성을 유지할 수 있다.

### @SpringBootApplication

애플리케이션의 진입점 클래스에 선언하는 어노테이션으로, 세 가지 역할을 한 번에 수행한다:

- `@SpringBootConfiguration` — 스프링 부트 설정 클래스임을 표시
- `@EnableAutoConfiguration` — 추가된 라이브러리를 보고 자동으로 설정 (예: web starter가 있으면 Tomcat 자동 설정)
- `@ComponentScan` — 해당 클래스가 있는 패키지와 하위 패키지에서 빈을 자동 탐색

### 스프링의 동작 큰 흐름

1. 스프링 부트 시작
2. 컴포넌트 스캔 (빈 대상 클래스 탐색)
3. Bean 생성
4. 의존성 주입
5. 웹 서버 시작
6. HTTP 요청 수신
7. 컨트롤러가 매핑된 메서드 수행
8. 응답 반환

## 2. REST API

### REST란

REST(Representational State Transfer)는 URI로 자원을 식별하고, HTTP 메서드로 행위를 표현하는 아키텍처 스타일이다. URL에는 명사를 사용하고, 행위(조회, 생성, 수정, 삭제)는 HTTP 메서드(GET, POST, PUT, DELETE)로 표현한다.

### API란

API(Application Programming Interface)는 프로그램끼리 소통하는 규칙이다.
콘솔 프로그램에서는 대개의 경우 컨트롤러가 사용자에게 프롬프트를 출력하고 응답을 받아 프로그램 로직을 수행시킨다.
반면 웹 환경에서는 사용자가 선제적으로 서버에게 요청을 보내는 형태이다. 사용자는 URI에 원하는 자원과 파라미터 등을 실어 보내고, 서버가 이를 해석해서 적절한 메서드를 실행시키는 구조이다.

### 스프링에서 REST API 만들기

`@RestController`를 선언해 REST API 컨트롤러를 만든다. 해당 어노테이션은 `@Controller`와 `@ResponseBody`가 결합된 어노테이션이다.

- `@Controller` — 클래스를 스프링 MVC의 컨트롤러로 지정하는 어노테이션. HTTP 요청을 받아 처리하고 뷰를 반환하는 전통 웹 페이지 구성에 사용됨을 선언한다.
- `@ResponseBody` — 컨트롤러 메서드의 반환 값을 View 대신 HTTP 응답 본문에 직접 작성해 클라이언트에 전달하는 역할을 한다.

둘을 합쳐서 HTTP 요청을 받아 처리하고, 반환 값을 View 대신 HTTP 응답 본문(JSON)에 직접 작성해 클라이언트에 전달하게 된다.

### URL 매핑

`@RequestMapping`은 컨트롤러가 처리할 URL을 지정한다. 클래스에 루트 URL을 지정하고, 각 메서드별로 루트 경로 뒤를 이어 URL을 확장해 나갈 수 있다.

URL에 포함되어 오는 값을 파라미터로 받으려면 `@PathVariable` 어노테이션을 사용한다. 파라미터에 어노테이션을 선언하고 해당 값을 변환할 타입과 변수명을 선언하면 메서드 내에서 사용 가능하다.

### @RequestBody

`@RequestBody`는 클라이언트가 보낸 JSON을 객체로 변환시키는 어노테이션이다. 파라미터에 어노테이션 선언과 함께 변환하고자 하는 클래스 타입을 선언하고 변수명을 지정해주면, JSON의 형식에 맞춰 자동으로 객체를 생성해준다. 이 때 JSON의 키 이름과 만들고자 하는 객체의 필드명이 동일해야 하며, 해당 필드를 대상으로 getter, setter가 선언되어 있어야 스프링이 자동으로 매핑해서 생성해 줄 수 있다.

### 요청-응답 흐름

`GET /posts/1`을 요청했다고 가정한 전체 흐름:

1. 클라이언트가 HTTP 요청 송신
2. Tomcat 서버가 요청을 수신
3. DispatcherServlet이 URL과 매칭되는 컨트롤러 메서드를 탐색
4. 어노테이션으로 지정된 컨트롤러 클래스를 스프링이 찾아 반환
5. 해당 컨트롤러에 매핑된 GET과 `{id}=1` 파라미터를 처리할 수 있는 메서드 실행
6. 메서드가 동작 후 결과를 반환
7. HttpMessageConverter가 객체를 JSON으로 직렬화
8. 클라이언트가 JSON을 수신

## 3. JSON 요청과 응답 처리

### JSON이란

JSON(JavaScript Object Notation)은 본래 자바스크립트에서 만들어진 텍스트 기반 경량 데이터 교환 형식이다. 사람과 기계 모두에게 읽고 쓰기가 쉽고, XML보다 가볍고 단순해 통신에 적합하다 판단되어 사실상 웹 통신의 표준으로 사용되고 있다.

구조는 `key:value` 쌍의 나열이며, 문자열, 숫자, 불리언, 널, 객체, 배열 타입이 사용 가능하다.

### 직렬화와 역직렬화

스프링에 자동으로 포함된 Jackson 라이브러리가 변환 작업을 담당한다.

- **역직렬화** (JSON → 자바 객체) — 클라이언트의 요청이 JSON 형태로 들어오면 이를 자바 객체로 변환시키는 작업. Jackson이 기본 생성자로 객체를 생성한 뒤 setter를 통해 값을 할당한다.
- **직렬화** (자바 객체 → JSON) — 자바 객체를 클라이언트에 반환하기 위해 JSON 형태로 변환하는 작업. Jackson이 getter를 호출해 필드 값을 조회하여 JSON을 구성한다.

두 경우 모두 대상 클래스의 필드명과 JSON의 키 이름이 정확히 일치해야 한다.

### 요청 객체와 응답 객체의 분리

요청과 응답에 사용하는 객체는 서로 분리되어야 한다. 생성 시에는 id가 필요 없지만 조회 시에는 id가 포함되어야 하는 것처럼 각각 필요한 정보가 다르며, 이를 통합해서 사용할 경우 클라이언트가 수정하면 안 되는 필드까지 조작 가능해지는 보안상의 문제가 발생할 수 있다.

### HTTP 상태 코드와 ResponseEntity

HTTP 상태 코드는 요청에 따른 작업이 어떻게 진행되었는지를 알려주는 숫자 코드이다.

| 코드 | 의미 |
|------|------|
| 200 | OK — 성공 |
| 201 | Created — 생성 성공 |
| 204 | No Content — 성공, 반환 데이터 없음 |
| 400 | Bad Request — 잘못된 요청, 필수 값 누락 등 |
| 404 | Not Found — 요청한 자원 없음 |
| 500 | Internal Server Error — 서버 내부 오류 |

`ResponseEntity`는 HTTP 상태 코드 + HTTP 헤더 + 응답 본문(body)을 함께 반환할 수 있는 객체이다. 두 가지 패턴으로 사용한다:

- **body가 있을 때**: `ResponseEntity.ok(post)` 또는 `ResponseEntity.status(HttpStatus.CREATED).body(post)`
- **body가 없을 때**: `ResponseEntity.noContent().build()` 또는 `ResponseEntity.notFound().build()`

`.body()`와 `.build()`는 양자택일이다. body를 담으면 `.body()`로 마무리하고, body 없이 상태 코드만 반환할 때 `.build()`로 마무리한다.
