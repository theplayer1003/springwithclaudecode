# 01. IoC와 DI 그리고 빈 — Q&A

## Q: 필드 주입과 세터 주입은 실전에서 아무런 장점이 없나요? 오직 생성자 주입만 알면 될까요?

결론부터 말하면, **생성자 주입을 기본으로 쓰되, 나머지 방식이 쓰이는 경우도 알아두면 좋다.**

### 필드 주입 — 실전에서의 위치

```java
@Autowired
private PostService postService;
```

**장점:**
- 코드가 가장 짧고 간결함

**실전에서 쓰이는 곳:**
- **테스트 코드** — 테스트 클래스에서는 간결함이 더 중요하고, 테스트 객체를 직접 생성할 일이 없어서 필드 주입을 쓰는 경우가 많음
- **레거시 코드** — 과거에 작성된 스프링 프로젝트에서 흔히 보임. 읽을 줄은 알아야 함

**왜 프로덕션 코드에서 권장하지 않는가:**
- `final`을 붙일 수 없어서, 실행 중에 의존성이 바뀔 수 있음
- 스프링 없이는 객체를 생성할 수 없음 (순수 자바 테스트 불가)
- 의존성이 눈에 잘 안 보여서, 클래스가 너무 많은 의존성을 가져도 알아채기 어려움

### 세터 주입 — 실전에서의 위치

```java
@Autowired
public void setPostService(PostService postService) {
    this.postService = postService;
}
```

**실전에서 쓰이는 곳:**
- **선택적 의존성** — 있어도 되고 없어도 되는 의존성일 때. 예를 들어 캐시가 있으면 사용하고 없으면 무시하는 경우
- **순환 참조 해결** — A가 B를 필요로 하고, B도 A를 필요로 할 때 생성자 주입으로는 해결이 안 되는 경우 (하지만 이 상황 자체가 설계 문제인 경우가 많음)

### 정리

| 방식 | 프로덕션 코드 | 테스트 코드 | 알아야 하는 이유 |
|------|:---:|:---:|------|
| **생성자 주입** | 기본으로 사용 | 사용 가능 | 직접 작성할 방식 |
| **필드 주입** | 지양 | 종종 사용 | 레거시 코드 읽을 때 |
| **세터 주입** | 드물게 사용 | 드물게 사용 | 선택적 의존성 개념 이해 |

**지금 단계에서는 생성자 주입만 사용하면 된다.** 나머지 두 방식은 "이런 코드를 만나면 이런 의미구나" 정도로 알아두면 충분. Phase 5(테스트)에서 필드 주입을 실제로 보게 될 것.
