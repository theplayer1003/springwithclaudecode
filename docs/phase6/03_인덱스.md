# 인덱스

## 인덱스란

DB에서 특정 컬럼의 값을 **정렬된 상태로 별도 저장**해두는 자료구조. 검색과 정렬 성능을 크게 향상시킨다.

### 인덱스가 없을 때 vs 있을 때

```
인덱스 없음 (Full Table Scan):
→ 100만 건을 처음부터 끝까지 하나씩 비교

인덱스 있음:
→ 정렬된 목차에서 이진 탐색처럼 빠르게 위치를 찾아 이동
→ 100만 건에서도 약 20번의 비교로 찾을 수 있다 (log 기반)
```

## 인덱스가 도움이 되는 경우

### 1. WHERE 검색

```sql
-- 인덱스가 title에 있으면 빠르게 찾음
SELECT *
FROM post
WHERE title = 'Spring 입문';
```

### 2. ORDER BY 정렬

인덱스는 이미 정렬된 상태로 저장되어 있으므로:

```
인덱스 없이 ORDER BY created_at DESC:
→ 전체 데이터를 읽고 → 메모리에서 정렬 → 결과 반환 (비용 큼)

인덱스 있을 때:
→ 이미 정렬되어 있으니 순서대로 꺼내기만 하면 됨 (정렬 비용 0)
```

페이징과 결합되면 "정렬된 인덱스에서 20개만 꺼내기"가 되어 매우 효율적이다.

## 인덱스의 비용 (트레이드오프)

인덱스는 "정렬된 목차를 별도로 유지"하는 것이므로, 데이터가 추가/수정/삭제될 때마다 **인덱스도 갱신**해야 한다.

### 인덱스를 걸면 좋은 경우

- `WHERE` 조건에 자주 쓰이는 컬럼
- `ORDER BY`에 자주 쓰이는 컬럼
- 조회가 잦고, 삽입/수정이 상대적으로 적은 컬럼

### 인덱스를 걸면 손해인 경우

- 거의 검색하지 않는 컬럼
- 삽입/수정/삭제가 매우 빈번한 컬럼
- 데이터가 매우 적은 테이블 (Full Scan이 오히려 빠름)

## 자동으로 인덱스가 생성되는 경우

- `@Id` (PK) — 기본키는 항상 자동 인덱스
- 외래키 (FK) — JPA가 외래키 컬럼에 자동으로 인덱스 생성

## JPA에서 인덱스 설정

```java

@Entity
@Table(indexes = @Index(name = "idx_post_created_at", columnList = "created_at"))
public class Post { ...
}
```

- `@Table`은 DB 테이블에 대한 설정이므로, `columnList`에는 **자바 필드명이 아닌 DB 컬럼명**을 사용한다
- JPA 기본 네이밍 전략: camelCase → snake_case (`createdAt` → `created_at`)

## LIKE 검색과 인덱스의 한계

```sql
LIKE 'Spring%'    -- 인덱스 활용 가능 (앞에서부터 매칭)
LIKE '%Spring%'   -- 인덱스 활용 불가 (Full Scan)
LIKE '%Spring'    -- 인덱스 활용 불가
```

앞에 `%`가 붙으면 "중간 글자로 목차를 찾는 것"과 같아서 인덱스를 활용할 수 없다. 키워드 검색이 중요한 서비스는 Elasticsearch 같은 전문 검색 엔진을 사용한다.
