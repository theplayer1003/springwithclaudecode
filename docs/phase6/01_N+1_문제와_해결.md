# N+1 문제와 해결

## N+1 문제란

JPA에서 연관 엔티티를 가진 목록을 조회할 때, **1번의 목록 조회 + N번의 연관 엔티티 조회**가 발생하는 성능 문제.

### 발생 원인

`findAll()` 같은 목록 조회는 단순히 `SELECT * FROM post`만 실행한다. JPA는 결과로 Post 객체를 만든 뒤, 각 Post의 연관 엔티티(Member 등)를 **별도의 쿼리로 하나씩**
조회한다.

```
1. SELECT * FROM post;                    -- 게시글 전체 조회 (1번)
2. SELECT * FROM member WHERE id = 1;     -- 1번 게시글의 작성자
3. SELECT * FROM member WHERE id = 2;     -- 2번 게시글의 작성자
...
→ 게시글 100개면 총 101번의 쿼리 실행
```

### Fetch 전략과 N+1의 관계

| 전략    | 동작                             | 기본 적용        |
|-------|--------------------------------|--------------|
| EAGER | 부모 조회 시 즉시 연관 엔티티도 조회          | `@ManyToOne` |
| LAZY  | 실제로 접근할 때(`get~()` 호출 시) 쿼리 실행 | `@OneToMany` |

기본값이 다른 이유:

- `@ManyToOne` — 연관 객체가 1개. 부담이 적으므로 EAGER
- `@OneToMany` — 연관 객체가 N개. 수백 개일 수 있으므로 LAZY

**LAZY는 N+1의 근본적 해결이 아니다.** 발생 시점을 늦출 뿐, 루프에서 연관 엔티티에 접근하면 동일하게 N+1이 발생한다.

## 해결: JOIN FETCH

JPQL의 `JOIN FETCH` 문법으로 연관 엔티티를 **한 번의 쿼리**로 함께 가져온다.

```java
// Repository
@Query("SELECT p FROM Post p JOIN FETCH p.member ORDER BY p.createdAt DESC")
List<Post> findAllWithMember();
```

실행되는 SQL:

```sql
SELECT p.*, m.*
FROM post p
         JOIN member m ON p.member_id = m.id
ORDER BY p.created_at DESC
```

| 방식            | 쿼리 수 | 네트워크 왕복 |
|---------------|------|---------|
| N+1 (findAll) | 101번 | 101번    |
| JOIN FETCH    | 1번   | 1번      |

JOIN 자체에 비용이 있지만, DB와 애플리케이션 사이의 네트워크 왕복 101번보다 DB 내부에서 JOIN 1번 하는 것이 압도적으로 효율적이다. DB는 JOIN 최적화를 잘 한다(인덱스 활용 등).

## JPQL이란

**Java Persistence Query Language** — SQL과 비슷하지만 테이블이 아닌 **엔티티 객체**를 대상으로 쿼리를 작성하는 언어. JPA 표준 스펙에 포함.

```java
// 쿼리 메서드 — Spring Data JPA가 메서드 이름을 파싱해서 SQL 자동 생성
List<Post> findByTitle(String title);

// JPQL — @Query로 직접 쿼리 작성 (엔티티명, 필드명 사용)
@Query("SELECT p FROM Post p WHERE p.title = :title")
List<Post> findByTitle(@Param("title") String title);
```

쿼리 메서드는 단순 조건에 편리하지만, JOIN FETCH 같은 복잡한 쿼리는 JPQL로 직접 작성해야 한다.

### JPQL 기본 문법

SQL과 거의 같지만 **테이블/컬럼 대신 엔티티/필드를 쓴다:**

```
SQL:    SELECT * FROM post WHERE title = 'hello'
JPQL:   SELECT p FROM Post p WHERE p.title = 'hello'
```

핵심 차이:

- `Post` — 테이블명이 아니라 **엔티티 클래스명** (대문자)
- `p` — 별칭(alias). JPQL에서는 **필수**
- `p.title` — 컬럼명이 아니라 **자바 필드명**
- `SELECT p` — 엔티티 객체 자체를 반환 (`SELECT *`와 비슷)

### 파라미터 바인딩

**1. 이름 기반 — `:파라미터명` + `@Param` (권장)**

```java

@Query("SELECT c FROM Comment c WHERE c.post.id = :postId")
List<Comment> findByPostId(@Param("postId") Long postId);
```

JPQL 안의 `:postId`와 `@Param("postId")`가 연결된다.

**2. 순서 기반 — `?1`, `?2`**

```java

@Query("SELECT p FROM Post p WHERE p.title = ?1")
List<Post> findByTitle(String title);
```

`@Param` 없이 동작하지만 파라미터가 여러 개일 때 순서를 헷갈릴 수 있어 이름 기반을 주로 사용한다.

### 자주 쓰는 패턴

```sql
-- 조건
SELECT p
FROM Post p
WHERE p.title = :title

-- 정렬
SELECT p
FROM Post p
ORDER BY p.createdAt DESC

-- 연관 엔티티 조건 (객체 그래프 탐색)
SELECT c
FROM Comment c
WHERE c.post.id = :postId

-- JOIN FETCH (N+1 해결)
SELECT p
FROM Post p
         JOIN FETCH p.member

-- JOIN FETCH + 조건 조합
SELECT c
FROM Comment c
         JOIN FETCH c.member
WHERE c.post.id = :postId
```

## Spring Data JPA vs JPA vs Hibernate

```
JPA            — 자바 표준 스펙 (인터페이스/규칙 정의). JPQL 포함
Hibernate      — JPA 구현체 (실제 동작하는 코드)
Spring Data JPA — Spring이 만든 편의 계층 (Repository 자동 구현, 쿼리 메서드, @Query 등)
```
