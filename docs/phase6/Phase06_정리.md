## 1. N+1 문제란 무엇이며, 왜 발생하는가?

**A:**
N+1 문제란 JPA 에서 쿼리를 보내 자료를 가져오고 객체를 조립할 때 발생하는 문제입니다.
쿼리를 하나 작성해서 보냈을 때 연관 데이터를 가져오기 위해 N 번의 쿼리가 추가 발생해 N+1문제라는 이름이 붙었습니다.

예를 들어 게시글 전체를 불러오는 쿼리가 실행되었을 때, post 객체가 조립되려면 member 객체가 필요합니다.
그냥 게시글만 불러오는 쿼리였다면 `SELECT * FROM POST` 로 해결될 일이지만, 이를 객체로 조립하기 위해서는 member 객체가 필요하고
그를 위해 `SELECT * FROM MEMBER WHERE id = ...;` 쿼리가 나가게 됩니다.
모든 게시글에 대해 이 작업이 이루어지기 때문에 처음 1번의 전체 게시글 조회 쿼리가 전체 게시글 수(N개) 만큼 늘어나게 되어 성능이 저하 됩니다.


## 2. Fetch 전략(EAGER, LAZY)의 차이는? LAZY로 바꾸면 N+1이 해결되는가?

**A:**
EAGER 는 "즉시", LAZY 는 "나중에" 입니다.
EAGER 전략은 엔티티가 로딩되면 그 즉시 관련된 엔티티도 같이 로딩하는 전략이고 LAZY 전략은 관련된 엔티티의 필드에 실제로 접근하면 그때 로딩하는 전략입니다.
예를 들어 `post.getMember().getUsername()` 작업이 있다면 EAGER 일 경우 post 가 조회된 순간에 이미 member 도 조회되어 있기 때문에
해당 메서드 호출 이전에 이미 엔티티 조회가 끝나있습니다.
LAZY 라면 post가 조회되도 member 는 아직 조회되지 않은 상태입니다.
이때는 post 엔티티에 프록시 member 가 들어가 있습니다. 프록시는 진짜 member 객체가 아니라 hibernate 가 만들어둔 가짜 껍데기 객체입니다.
`post.getMember()` 를 호출해도 아직 DB 쿼리는 나가지 않은 상태고 프록시 객체에는 ID 값만 가지고 있습니다.
이후 .getUsername() 까지 호출된 순간, 필드에 접근한 순간에 쿼리를 날려 데이터를 가져오고 프록시 객체 내부를 채우게 됩니다.
LAZY 전략이 N+1 문제를 해결하지는 못 합니다. 아직 쿼리가 실행이 되지 않았을 뿐이지 호출되면 쿼리가 수행되어 조회가 일어나기 때문입니다.


## 3. JOIN FETCH는 무엇이며, 일반 JOIN과 어떤 차이가 있는가?

**A:**
JPA 의 JOIN 은 그냥 새로운 테이블을 참조하기만 합니다. 데이터 자체는 원래 조회하려고 했던 엔티티의 데이터만 SELECT 해옵니다.
JPQL 의 JOIN 은 SQL JOIN 처럼 두 테이블을 같이 참조하지만 SELECT 절에는 원래 엔티티만 포함됩니다.
`SELECT p FROM Post p JOIN p.member m WHERE m.username = 'kim'` 의 쿼리가 수행된다면 member 테이블을 통해
이름이 kim 인 사람이 걸러질테지만 데이터 자체는 post, 게시글 데이터만 가져오게 됩니다.
반면 JOIN FETCH 는 JOIN 되는 테이블의 엔티티도 함께 가져옵니다.
JOIN FETCH 로 위의 쿼리를 수행한다면 게시글과 함께 멤버의 데이터도 넘어와 엔티티로 만들어집니다.


## 4. JPQL과 SQL의 차이는? JPQL에서 파라미터를 바인딩하는 방법 두 가지를 설명하라.

**A:**
JPQL 과 SQL 의 핵심 차이점은
SQL 의 쿼리는 테이블과 컬럼을 대상으로 작성되고
JPQL 의 쿼리는 엔티티 클래스, 필드를 대상으로 작성됩니다.

`SELECT * FROM post WHERE title = 'hello'`
`SELECT p FROM Post p WHERE p.title = 'hello'`
post 라는 테이블명 대신 Post 라는 클래스 명이 작성됩니다. p 는 엔티티 객체 입니다.
SQL 의 title 은 컬럼명인데 JPQL 의 title 은 p.title 로 p 객체의 필드 명을 의미합니다.

바인딩 방법은 먼저 이름 기반 방법이 있습니다.
`@Query("SELECT c FROM Comment c JOIN FETCH c.member WHERE c.post.id = :postId")`
`Page<Comment> findByPostIdWithMember(@Param("postId") Long postId, Pageable pageable);`

`@Param` 어노테이션으로 쿼리 문자열을 지정하고 뒤에 메서드 호출 시 넘겨받을 자료형을 선언합니다.
선언된 쿼리 문자열은 쿼리를 작성할때 `:...` 콜론 뒤에 배치되어 사용됩니다.

두 번째 방법은 순서 기반 방법입니다.
`@Query("SELECT p FROM Post p WHERE p.title = ?1")`
`List<Post> findByTitle(String title);`

`?1` 에서 ? 뒤가 순서이고 첫 번째 파라미터에 매핑됩니다. 이때는 `@Param` 어노테이션 없이 순서대로 매핑을 할 수 있습니다.
이 방법은 파라미터가 여러 개일 때 순서를 헷갈리 수 있기 때문에 이름 기반이 주로 쓰입니다.


## 5. 쿼리 메서드와 @Query의 차이는? 각각 언제 사용하는가?

**A:**
쿼리 메서드는 JpaRepository 를 상속한 인터페이스에서 선언되는 메서드들을 말합니다.
이 메서드 이름들은 그냥 의미에 맞게 지어지는게 아니라 규칙에 따라 작성되어 쿼리로 변환됩니다.
단순 조건 조회가 필요하면 쿼리 메서드를 사용해 구현합니다.

`@Query` 어노테이션은 쿼리 메서드를 사용하지 않고 해당 메서드에 개발자가 따로 쿼리를 지정하는 어노테이션입니다.
쿼리 메서드로는 복잡한 쿼리를 작성할 수 없습니다.
`findAllWithMemberJoinFetchOrderByCreatedAtDesc()` 같은 건 쿼리 메서드로 표현이 어렵습니다.
여러 테이블에 걸친 조건이나 서브쿼리가 필요한 경우 사용되는게 `@Query` 어노테이션 입니다.
JPQL 문법을 써서 쿼리를 작성하고 넘겨받은 파라미터를 쿼리에 맵핑 할 수 있습니다.


## 6. 페이징이 필요한 이유는? Page 객체에는 어떤 정보가 담겨 있는가?

**A:**
먼저 페이징이란 어떤 데이터를 조회할 때 지정된 양 만큼 잘라서 하나의 묶음으로 취급하는 기법을 말합니다.
예를 들어 게시글이 10,000 개가 있다고 할 때 전체 게시글 조회가 이루어진다면 방대한 데이터를 다 가져오지만 실질적으로 다 사용되지 않을 가능성이 높습니다.
따라서 페이징 기법을 도입해 첫번째 페이지, 게시글 N 개. 이런 식으로 데이터를 나눠 가져오게 할 수 있습니다.
Page 객체는 페이징 처리를 도와주는 객체입니다. 서버 내부에서는 객체로 다뤄지고 클라이언트에 나갈 때 JSON 으로 변환됩니다.
JSON 형태로 객체를 살펴보면 다음과 같습니다.
```json
{   
  "content": [...],
  "totalPages": 50,
  "totalElements": 1000,
  "size": 20,
  "number": 0
}
```
먼저 실제 데이터들이 담기는 content 가 있습니다.
totalPages 는 조회하려는 데이터의 전체 페이지 수 입니다. 이 값으로 페이지 네비게이션을 만듭니다.(1, 2, 3, ... N 개의 페이지)
totalElements 는 전체 데이터 수이며
size 는 한 페이지 당 크기 입니다.
number 는 현재 페이지 번호 입니다.


## 7. Page와 Slice의 차이는? 각각 어떤 UI에 적합한가?

**A:**
둘 다 데이터 페이징을 구현한 인터페이스 입니다.
먼저 Page 는 앞서 말한대로 전체 페이지 수를 알 수 있어 번호 기반 페이징에 적합합니다.
Slice 는 Page 와 다르게 전체 카운트 등을 하지 않고 다음 페이지의 존재 여부만 확인합니다.
이는 요청된 페이지 사이즈보다 하나를 더 조회해서 존재하는지 확인합니다.

Page 는 전통적인 게시판 하단에 1, 2, 3, 4, ... N 페이지로 나눠 표현할 때 유용하며
Slice 는 무한 스크롤, 더보기 기능 등에 유리합니다.

Slice 는 Page 의 상위 인터페이스입니다. Page 가 Slice 를 상속합니다.
Slice 는 content(데이터 목록), size(페이지 크기), number(현재 페이지 번호), `hasNext()`(다음 페이지 존재 여부) 를 가집니다.
즉 Page 에서 Count 관련 값들이 빠집니다. Slice 는 COUNT(*) 쿼리를 실행하지 않기 때문에
만약 데이터가 수백만 건이라 COUNT 쿼리가 무거울 경우 페이징이 필요하지 않다면 Slice 를 사용하는게 더 효율적일 수 있습니다.


## 8. 컨트롤러에서 Pageable 파라미터는 어떻게 생성되는가? 클라이언트는 어떻게 페이지를 요청하는가?

**A:**
컨트롤러의 메서드에서 받는 파라미터에 Pageable 이 선언되어 있으면 Spring 이 자동으로 설정값을 읽어 생성해 넣어줍니다.
`application.properties` 에 페이지 사이즈 등을 설정할 수 있고 따로 설정하지 않으면 default 값이 적용됩니다.

클라이언트는 쿼리 파라미터로 원하는 페이지를 지정하게 됩니다.
`GET /posts?page=0&size=10`: 10개 크기로 1페이지 요청
이런 식으로 요청이 들어오면 Spring 이 값을 읽고 자동으로 Pageable 객체로 변환해 줍니다.
만약 클라이언트가 아무런 값을 안 보내면 `application.properties` 기본 값이 적용 됩니다.


## 9. 인덱스란 무엇이며, 검색(WHERE)과 정렬(ORDER BY) 모두에 도움이 되는 이유는?

**A:**
인덱스는 DBMS 가 테이블을 대상으로 만들어두는 색인, 목차 같은 겁니다.
B-Tree 형식으로 자료가 정렬됩니다.
WHERE 절의 경우 특정 값을 찾을 때 처음부터 끝까지 Full Scan 하지 않고 이진 탐색으로 더 빠르게 찾아낼 수 있습니다.
ORDER BY 절의 경우 인덱스가 이미 정렬된 상태를 보장하기 때문에 정렬 비용이 0 이 됩니다.
만약 인덱스가 없다면 데이터를 모두 불러와서 메모리에 올려서 정렬 과정을 거쳐야 합니다.


## 10. 인덱스를 모든 컬럼에 걸면 안 되는 이유는? 인덱스가 적합한 경우와 부적합한 경우를 설명하라.

**A:**
먼저 인덱스 자체도 하나의 데이터로 공간을 차지합니다. 인덱스를 걸게되면 해당 테이블에 대해 별도의 자료구조(B-Tree)로 디스크에 저장되게 됩니다.
따라서 모든 컬럼을 대상으로 인덱스를 걸면 낭비가 발생하게 됩니다.
또한 인덱스를 걸어뒀을 때 데이터의 수정, 삭제 등의 업데이트가 발생되면 인덱싱을 다시 수행해야합니다.
한 테이블의 모든 컬럼에 인덱스가 걸려 있다면 이로 인해 많은 리소스 낭비를 유발하게 됩니다.
마지막으로 데이터가 얼마 없는 경우에는 인덱스보다 full scan 이 더 빠를 수 있습니다.
인덱스를 통해 조회하는 과정은
B-Tree 탐색 -> 실제 데이터 위치로 이동 -> 데이터 읽기
인데,
데이터가 10건이면 Full scan 으로 10건을 순서대로 읽으면 끝날 일을 인덱스 탐색 -> 해당 위치 이동 -> 해당 지점에서 데이터 읽기 시작
이라는 간접 비용이 발생합니다. 이런 경우 인덱스 탐색보다 full scan 이 더 효율적일 수 있습니다.

인덱싱이 효율적일지 Full scan 이 효율적일지 판단하는 기준이 몇 가지 있습니다.
먼저 선택도(Selectivity) 입니다. 해당 조건으로 조회 되는 데이터가 전체 데이터의 몇 %인지 보는 방법입니다.
일반적으로 전체의 10~15% 이하를 조회하면 인덱스가 유리하며 그 이상일 경우 Full scan 이 더 나을 수 있습니다.
앞서 말한 인덱스의 간접비용 때문인데 조회 대상이 결과적으로 적으면 간접비용이 적게 발생합니다.
대량 조회의 경우 이 간접비용이 늘어나기 때문에 차라리 디스크 I/O 가 효율적인 순차 읽기, full scan 이 더 빠르게 됩니다.

정리하면 소량 조회는 인덱스를 통해 필요한 것만 딱딱 찍어 가져오는게 효율적이지만
많은 수의 데이터를 대량 조회해야한다면 어차피 대부분의 데이터를 읽는 과정이기 때문에 처음부터 쭉 읽는게 효율적이게 됩니다.
많은 데이터를 조회할 때 인덱스가 유리한 경우는 테이블 전체 건수가 많을 때 해당하는 말이고, 조회 결과가 전체의 많은 비율을 차지하면
오히려 full scan 이 유리하게 됩니다.

데이터 건수로도 판단할 수 있습니다. 수백 건 이하면 그냥 Full scan 이 빠릅니다.

쓰기 빈도로도 판단합니다. `INSERT/UPDATE/DELETE` 작업이 매우 빈번하면 앞서 설명한 것 처럼 인덱스 유지 비용이 커집니다.

이러한 판단은 DB 옵티마이저가 자동으로 하기도 합니다. EXPLAIN 명령어를 써서 옵티마이저가 자동으로 어떤 판단을 했는지 알 수 있습니다.


## 11. LIKE '%keyword%' 검색에서 인덱스가 활용되지 못하는 이유는?

**A:**
`%keyword%` 패턴은 keyword 앞 뒤로 뭐가 와도 좋다는 뜻 입니다.
인덱스는 제일 앞글자부터 정렬이 되는 구조 입니다.
`keyword%` 라고 하면 로 시작하는 구간을 찾아 가서 탐색을 시작합니다.
하지만 `%keyword%` 는 중간 어딘가에서부터 keyword 가 포함된 자료를 찾아야하기 때문에 인덱스 목차에서 찾아갈 시작점이 없습니다.
따라서 인덱싱이 불가능한 패턴입니다.

이런 인덱싱이 불가능한 검색이 필요할 경우 `Elasticsearch` 같은 전문 검색 엔진을 동원할 수 있습니다.
엘라스틱 서치는 역인덱스를 통해 이런 검색을 구현합니다.
일반 인덱스가 문서 -> 단어 의 구조라면, 역인덱스는 단어 -> 해당 단어가 포함된 문서 목록 을 찾습니다.
예를 들어 일반 인덱스가 created_at 컬럼에 걸려있다면 해당 값 기준으로 정렬해서 "이 값이 어느 행에 있는지" 를 알려줍니다.
특정 컬럼 값의 인덱스(문서) -> 해당 행의 위치(단어) 를 빠르게 찾게 됩니다.
`Spring Boot 입문 가이드` 라는 데이터를 예로 들면
S 에 해당하는 인덱스 문서를 보고 해당 행의 위치로 점프해 데이터를 읽어나가 해당 문서를 찾게 됩니다.

역인덱스는
`Spring` -> `[게시글1, 게시글5, 게시글42]`
`Boot` -> `[게시글1, 게시글5]`
`입문` -> `[게시글1, 게시글 30]`
`가이드` -> `[게시글1, 게시글 22]`
같은 형태로 저장됩니다.
이후 Spring 이라는 키워드로 조회 요청이 들어오면 곧바로 해당 게시글 리스트를 불러와 빠르게 조회를 처리할 수 있게 됩니다.
엘라스틱 서치는 해당 문서가 저장될 때 텍스트를 토큰, 단어 단위로 분리해 모든 단어에 대해 역인덱스로 자동 분류합니다.
설명만 보면 역인덱스의 도입 비용이 커보입니다.
역인덱스 도입에 대한 비용을 따져보면,
저장 공간이 원본 데이터보다 커질 수 있고 문서의 추가, 삭제 시 역인덱스 갱신 비용도 발생하지만 이건 DB 인덱스의 비용과 같은 트레이드 오프입니다.
또, '%Spring%' 으로 100만 건을 full scan 하는 것 보다 역인덱스의 'Spring' 키를 찾아 목록을 꺼내는게 압도적으로 빠르기 때문에
쓰기 비용을 감수하고 읽기 성능을 얻는 구조로 역시 DB의 인덱스와 동일한 트레이드오프입니다.


## 12. @Cacheable과 @CacheEvict의 역할과 동작 흐름을 설명하라.

**A:**
`@Cacheable` 어노테이션은 Spring 에게(Spring Cache) 해당 데이터를 캐싱하라는 의미입니다.
현재 프로젝트에서는 `spring.cache.type=redis` 로 설정되어 있어 redis 에 저장되게 됩니다.

파라미터로 저장소 이름을 지정할 수 있고 그 뒤엔 키 값을 지정할 수 있습니다.(key-value 형태)
만약 메서드 파라미터가 하나고 어노테이션 파라미터의 키값을 생략하면 자동으로 메서드 파라미터 값이 키 값으로 지정됩니다.

`@CacheEvict` 어노테이션은 해당 메서드 수행 후 해당 엔티티가 캐싱된 데이터를 삭제하라는 의미입니다.
만약 캐싱된 데이터가 이후 업데이트 혹은 삭제되었는데 별다른 조치가 없다면 클라이언트의 조회 요청에 캐싱된 데이터가 나갈 수 있습니다.
따라서 캐싱을 도입했을 경우 데이터의 변경에 대한 관리가 필요하고 이를 수행하는게 해당 어노테이션 입니다.
`@Cacheable` 가 마찬가지로 저장소 이름과 키값을 지정할 수 있고 키 값을 생략해 하나의 파라미터를 키값으로 쓸 수 있습니다.


## 13. Spring 기본 캐시(메모리) 대신 Redis를 사용하는 이유는?

**A:**
Spring 에 탑재된 기본 캐시는 메모리 기반 캐시입니다. 만약 서버가 다시 시작될 일이 있다면 캐싱 데이터는 모두 사라지게 됩니다.
얼핏 '다시 캐싱 데이터가 쌓이면 되는거 아닌가?' 생각이 들지만 규모가 큰 애플리케이션의 경우 서버 시작 직후 캐싱된 데이터가 없어
수많은 요청이 DB 조회로 몰릴 수 있습니다. 이를 캐시 스탬피드 라고 합니다.

또 서버 규모가 커질 경우 스케일 아웃을 고려하게 되어 여러 대의 서버가 생길 수 있는데 이 경우 각 서버간의 캐싱 데이터 공유 문제가 발생합니다.
예를 들어 1번 서버에서 조회가 일어나 해당 엔티티가 캐싱되었다고 가정하겠습니다.
이후 2번 서버에서 해당 엔티티의 업데이트가 일어날 수 있습니다. 만약 Spring 캐싱이 적용되면 해당 업데이트에 대한 캐싱 정보는 2번 서버에만 존재합니다.
다시 1번 서버에서 해당 엔티티의 조회가 일어나면 2번 서버에서 일어난 변경을 반영하지 못하고 그냥 옛날 캐싱 데이터가 나가게 됩니다.
Redis 같은 외부 캐싱 저장소를 사용할 경우 모든 서버가 Redis 저장소를 바라보기 때문에 이런 캐싱 데이터 공유 문제를 해결할 수 있습니다.


## 14. Redis에 Java 객체를 저장할 때 직렬화 설정이 필요한 이유는? @class 타입 정보가 없으면 어떤 문제가 발생하는가?

**A:**
Redis 는 외부 저장소이므로 Java 객체를 그대로 저장할 수가 없고 JSON 문자열로 변환해야 합니다.(직렬화)
프로젝트를 진행하면서 겪은 문제점은
1. LocalDateTime 직렬화 에서 Jackson 이 LocalDateTime 을 모르기 때문에 JavaTimeModule 을 등록해야 했습니다.
좀 더 구체적으로,
Jackson 은 코어 모듈이 Java 8 이전에 만들어졌고 LocalDateTime은 Java 8에서 추가된 클래스입니다.
때문에 Jackson 은 JavaTimeModule 이라는 확장 모듈, 플러그인을 만들어 대응하도록 했습니다.
`objectMapper.registerModule(new JavaTimeModule());`
Jackson의 설계 철학이 코어를 가볍게 유지하는 것이기 때문에 LocalDateTime 을 사용하기 위해 따로 모듈을 등록해줘야합니다.
2. 날짜가 `[2026,5,7,13,44,9]` 같은 배열 형태로 저장되어 WRITE_DATES_AS_TIMESTAMPS 를 비활성화하고
`"2026-05-07T13:44:09"` 처럼 ISO 문자열로 저장되도록 했습니다.
배열은 사람이 읽기에 어려워 디버깅 등에 불리합니다. 또 다른 프론트 엔드 등의 다른 시스템에서도 데이터를 읽을 때 대부분
ISO 문자열을 표준으로 채택하기 때문에 배열 형태를 비활성화하는게 필수는 아니지만 사실상 항상 권장됩니다.
3. `@Class` 타입 정보를 알려주지 않아 Jackson 이 Redis 에서 JSON 을 꺼내 어떤 클래스로 변경해야할지 몰라 기본 값인
`LinkedHashMap` 으로 역직렬화 했습니다. `activateDefaultTyping` 을 설정해 JSON에 `@Class` 필드를 추가하면 역직렬화 시
해당 클래스로(`PostResponse`) 변환 되도록 했습니다.
   `objectMapper.activateDefaultTyping(
   BasicPolymorphicTypeValidator.builder()
   .allowIfBaseType(Object.class)
   .build(),
   DefaultTyping.EVERYTHING,
   As.PROPERTY
   );`
activateDefaultTyping 으로 설정해주면 직렬화 시 JSON 에 "@class": "com.study.board.dto.PostResponse" 필드가 자동으로 삽입됩니다.
이제 역직렬화 시 jackson 이 @class 값을 읽어 변환해야할 클래스가 뭔지 알게되어 자동으로 처리됩니다.


## 15. @Transactional 롤백은 Redis 캐시에 영향을 주는가? 테스트에서 이 문제를 어떻게 해결했는가?

**A:**
`@Transactional` 어노테이션은 프로덕트 코드에서는 이름대로 메서드를 트랜잭션 취급하도록 합니다.
하지만 테스트코드에서는 추가로 모든 코드를 수행하고 결과를 롤백 하는 기능이 붙습니다. 이는 테스트 간의 간섭을 없애고 일관된 테스트 환경을 보장하기 위함입니다.

하지만 Redis 의 캐싱은 Spring 트랜잭션 범위와 별개의 이야기입니다.
Spring 자체를 로드하지 않는 단위 테스트나 컨트롤러 계층만 로드하고 서비스는 Mock 으로 처리하는 슬라이스 테스트는 괜찮지만
통합 테스트는 모든 애플리케이션 구성요소를 다 로드해서 테스트하기 때문에 캐싱 기능이 작동합니다.
통합 테스트 코드에서 `@Cacheable` 어노테이션이 달린 메서드를 호출하게 되면 해당 엔티티는 Redis에 캐싱되게 되고
이를 정리하지 않으면 다음 테스트 코드에 영향을 받게 됩니다.
따라서 캐싱이 일어나는 경우 이를 `@AfterEach` 메서드나 아니면 개별 메서드에서 CacheManager 를 생성해 직접 관리해줘야 정상적인 테스트 수행이 가능합니다.


## 16. Docker를 사용하는 이유는? 직접 설치하는 것과 비교했을 때의 장점은?

**A:**
Docker 는 애플리케이션을 컨테이너로 패키징하여 어디서든 동일하게 실행할 수 있게 하는 플랫폼 입니다.
Docker 를 사용해 Redis 를 설치하게 되면 서버 컴퓨터에 직접 Redis 를 설치하지 않아도 됩니다.
애플리케이션을 개발하다보면 여러 사람이 하나의 프로젝트를 작업하게 되는데 이때 Docker 를 사용하지 않고 Redis 를 각자 설치해서 쓰게 되면
버전 관리 등의 환경이 통일되지 않아 개발에 어려움이 있을 수 있습니다.

간편한 설치 제거도 장점입니다. docker가 대신 관리해주기 때문에 `docker run`, `docker rm` 등으로 설정 파일, 환경 변수 등까지
한번에 제어가 가능합니다. 직접 설치해서 수동으로 관리하면 자잘한 설정들을 따로 관리해줘야 합니다.

애플리케이션이 여러 서비스와 연결될 경우 이들을 각각 컨테이너로 관리해 충돌 없이 독립적으로 실행, 중지가 가능해집니다.
예를 들자면 로컬에 Redis 5 가 설치되어 있었는데 새 프로젝트에서는 Redis 7 이 필요하다고 가정합니다.
같은 포트 6379를 쓰려면 기존 걸 멈춰야 하고, 설정 파일이 섞일 수도 있습니다.
docker 의 경우 컨테이너로 각 redis 가 격리되기 때문에 포트만 다르게 설정해 독립적으로 실행이 가능하고
서로의 설정이나 데이터에 영향이 가지 않습니다.


## 17. 댓글이 추가/삭제될 때 게시글 캐시를 제거해야 하는 이유는? deleteCommentForced에서 @CacheEvict 대신 CacheManager를 직접 사용한 이유는?

**A:**
어떤 댓글이 조회를 통해 이미 캐싱되어 있었다고 가정하겠습니다.
이후에 또 다른 요청으로 해당 댓글이 추가되거나 삭제 되는 등 변경이 일어나 DB의 데이터가 새로워졌다면 이를 캐싱에도 반영해야합니다.
그렇지 않은 채 다음 조회 요청을 받게되면 애플리케이션은 비싼 DB 커넥트 대신 빠른 캐싱을 선택하게 되어 변경이 일어나기 전 데이터를 반환하게 됩니다.
따라서 캐싱을 사용할 경우 캐싱 된 데이터의 추가적인 관리가 필요해집니다.

댓글의 경우 게시글에 종속된 엔티티 입니다. 따라서 댓글의 변경은 게시글의 변경을 의미하기도 합니다.
기존 댓글 서비스의 강제삭제의 경우 파라미터로 commentId 만을 받았습니다.
`@CacheEvict` 어노테이션은 키 값을 통해 캐싱 데이터에 접근하기 때문에 캐싱된 게시글에 접근 할 수가 없었습니다.
따라서 commentId 를 통해 직접 comment 엔티티를 꺼내고 여기서 속한 게시글의 id 를 얻어 캐시 매니저로 직접 정리해줘야했습니다.
