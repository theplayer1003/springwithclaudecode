# 01. JPA와 데이터베이스 기본 — Q&A

## Q: JPA의 한계점은 무엇인가요? 현업에서 그런 한계점에도 불구하고 JPA가 표준처럼 사용되나요?

### JPA의 한계점

**1. 복잡한 쿼리에 약하다**

JPA는 단순한 CRUD에 강하지만, 복잡한 통계 쿼리(GROUP BY, HAVING, 서브쿼리 등)는 JPA 쿼리 메서드로 표현하기 어렵다. 이를 위해 JPQL(객체 기반 쿼리 언어), 네이티브 쿼리(직접 SQL 작성), QueryDSL 같은 별도 라이브러리를 함께 사용한다. JPA를 쓰더라도 SQL 지식은 여전히 필요하다.

**2. 성능 제어가 어렵다**

JPA가 SQL을 자동 생성하기 때문에 어떤 쿼리가 실행되는지 개발자가 바로 알기 어렵다. 대표적인 문제가 N+1 문제로, 게시글 10개를 조회했을 때 각 게시글의 댓글을 가져오기 위해 추가로 10번의 쿼리가 발생하는 현상이다. JPA를 잘 쓰려면 내부 동작(영속성 컨텍스트, 지연 로딩 등)을 깊이 이해해야 한다.

**3. 학습 곡선이 높다**

단순 사용은 쉽지만, 제대로 쓰려면 영속성 컨텍스트, 지연/즉시 로딩, 프록시, 변경 감지, 벌크 연산 등 이해해야 할 개념이 많다. 내부 동작을 모르고 쓰면 직접 SQL을 쓰는 것보다 성능이 나빠질 수 있다.

### 현업에서의 JPA 위치

한국 기준 JPA(Spring Data JPA)가 사실상 표준이다. 다만 단독으로 쓰기보다 복합적으로 사용한다:

| 상황 | 사용 기술 |
|------|----------|
| 단순 CRUD | Spring Data JPA (쿼리 메서드) |
| 복잡한 동적 쿼리 | QueryDSL + JPA |
| 대량 데이터 조회, 통계 | 네이티브 SQL 또는 MyBatis 병행 |
| 극한 성능이 필요한 구간 | 직접 SQL (JDBC) |

실무 코드의 70~80%는 단순 CRUD이므로, 이 대부분을 극도로 간결하게 처리해주는 생산성 이점이 한계점을 상쇄한다. 복잡한 20~30%만 다른 도구로 보완하는 방식이다.
