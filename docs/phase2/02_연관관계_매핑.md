# 연관관계 매핑 (게시글 ↔ 댓글)

Phase 1에서 게시글(Post)과 댓글(Comment)을 독립적으로 만들었습니다.
하지만 실제로는 "이 댓글은 어떤 게시글에 달린 것인가?"라는 관계가 필요합니다.
JPA에서는 이 관계를 **연관관계 매핑**으로 표현합니다.

---

## 1. 데이터베이스에서의 관계

### 외래키 (Foreign Key)

데이터베이스에서 테이블 간의 관계는 **외래키**로 표현합니다.

```
posts 테이블                    comments 테이블
+----+--------+               +----+---------+---------+
| id | title  |               | id | content | post_id |  ← 외래키
+----+--------+               +----+---------+---------+
| 1  | 첫 글  |               | 1  | 좋아요  | 1       |  ← 1번 게시글의 댓글
| 2  | 두번째 |               | 2  | 감사!   | 1       |  ← 1번 게시글의 댓글
+----+--------+               | 3  | 안녕!   | 2       |  ← 2번 게시글의 댓글
                              +----+---------+---------+
```

`comments` 테이블의 `post_id`가 외래키입니다. "이 댓글이 어떤 게시글에 속하는지"를 나타냅니다.

### 관계의 종류

| 관계 | 설명 | 예시 |
|------|------|------|
| **1:N (일대다)** | 하나가 여러 개를 가짐 | 게시글 1개 : 댓글 N개 |
| **N:1 (다대일)** | 여러 개가 하나에 속함 | 댓글 N개 : 게시글 1개 |
| **1:1 (일대일)** | 하나가 하나를 가짐 | 사용자 1명 : 프로필 1개 |
| **N:M (다대다)** | 여러 개가 여러 개와 연결 | 게시글 N개 : 태그 M개 |

우리 게시판에서는 **게시글(1) : 댓글(N)** 관계를 구현합니다.

---

## 2. 데이터베이스 vs 자바 객체의 관계 표현

여기서 패러다임 불일치가 다시 등장합니다.

### 데이터베이스 방식 — 외래키로 연결

```
comments 테이블에 post_id 컬럼을 두어 관계를 표현
→ 양방향 탐색 가능: JOIN으로 어느 쪽에서든 조회 가능
```

### 자바 객체 방식 — 참조로 연결

```java
// 댓글 → 게시글 (참조를 가지고 있으면 접근 가능)
comment.getPost()     // ✅ 가능

// 게시글 → 댓글 (참조가 없으면 접근 불가)
post.getComments()    // ❌ Post에 List<Comment>가 없으면 불가능
```

자바 객체는 **참조를 가진 쪽에서만 상대방에 접근**할 수 있습니다.
데이터베이스는 외래키 하나로 양쪽 모두 JOIN이 가능합니다.

이 차이 때문에 JPA에서는 **단방향/양방향** 개념이 등장합니다.

---

## 3. 단방향 vs 양방향

### 단방향 관계

한쪽에서만 상대방을 참조합니다.

```java
// Comment → Post 방향만 존재
@Entity
public class Comment {

    @ManyToOne                         // 댓글 여러 개가 게시글 하나에 속함
    @JoinColumn(name = "post_id")      // DB에서 외래키 컬럼 이름
    private Post post;                 // 이 댓글이 속한 게시글
}
```

- `comment.getPost()` → 가능
- `post.getComments()` → 불가능 (Post에 댓글 목록이 없으므로)

### 양방향 관계

양쪽 모두 상대방을 참조합니다.

```java
// Comment 쪽 (관계의 주인)
@Entity
public class Comment {

    @ManyToOne
    @JoinColumn(name = "post_id")
    private Post post;
}

// Post 쪽 (읽기 전용 — mappedBy)
@Entity
public class Post {

    @OneToMany(mappedBy = "post")      // Comment의 "post" 필드에 의해 매핑됨
    private List<Comment> comments = new ArrayList<>();
}
```

- `comment.getPost()` → 가능
- `post.getComments()` → 가능

---

## 4. 관계의 주인 (mappedBy)

양방향 관계에서 가장 중요한 개념입니다.

### 왜 주인이 필요한가?

데이터베이스에서는 외래키가 **하나**입니다 (`comments.post_id`).
그런데 자바에서는 양쪽 모두 참조를 가지고 있습니다.

```java
comment.setPost(post);         // 댓글에서 게시글 설정
post.getComments().add(comment); // 게시글에서 댓글 추가
```

둘 다 관계를 변경할 수 있으면, JPA는 어느 쪽을 기준으로 DB를 업데이트해야 할까요?
이 혼란을 방지하기 위해 **한쪽을 관계의 주인**으로 지정합니다.

### 규칙

> **외래키가 있는 쪽이 관계의 주인이다.**

```
comments 테이블에 post_id(외래키)가 있다
→ Comment가 관계의 주인
→ Comment.post 필드가 DB의 외래키를 관리
```

- **주인 쪽 (Comment)**: `@ManyToOne` + `@JoinColumn` → 외래키를 직접 관리. 이쪽에서 값을 설정해야 DB에 반영됨
- **반대쪽 (Post)**: `@OneToMany(mappedBy = "post")` → 읽기 전용. 여기에만 값을 넣으면 DB에 반영되지 않음

### 중요: 양쪽 다 설정하기

DB 반영은 주인 쪽만 하면 되지만, 자바 객체의 일관성을 위해 **양쪽 모두 설정**하는 것이 좋습니다.

```java
// 올바른 방법: 양쪽 모두 설정
comment.setPost(post);              // 주인 쪽 설정 → DB 반영됨
post.getComments().add(comment);    // 반대쪽도 설정 → 객체 일관성 유지

// 잘못된 방법: 반대쪽만 설정
post.getComments().add(comment);    // DB에 반영되지 않음!
```

## 5. @ManyToOne, @OneToMany 정리

### @ManyToOne — "나는 여러 개 중 하나, 상대는 하나"

```java
@Entity
public class Comment {

    @ManyToOne
    @JoinColumn(name = "post_id")
    private Post post;
}
```

- 댓글 **여러 개(Many)**가 게시글 **하나(One)**에 속함
- 외래키를 가지는 쪽 → **관계의 주인**
- 가장 많이 쓰는 연관관계

### @OneToMany — "나는 하나, 상대는 여러 개"

```java
@Entity
public class Post {

    @OneToMany(mappedBy = "post")
    private List<Comment> comments = new ArrayList<>();
}
```

- 게시글 **하나(One)**가 댓글 **여러 개(Many)**를 가짐
- `mappedBy`로 주인이 아님을 표시
- 반드시 `new ArrayList<>()`로 초기화 (NullPointerException 방지)

## 6. 지연 로딩 (Lazy Loading) vs 즉시 로딩 (Eager Loading)

게시글을 조회할 때, 댓글도 함께 가져와야 할까요?

### 즉시 로딩 (Eager Loading)

```java
@OneToMany(mappedBy = "post", fetch = FetchType.EAGER)
private List<Comment> comments;
```

게시글을 조회하면 **댓글도 무조건 함께** 가져옵니다.
- 게시글만 보고 싶어도 댓글까지 불필요하게 조회
- 데이터가 많으면 성능 문제 발생

### 지연 로딩 (Lazy Loading)

```java
@OneToMany(mappedBy = "post", fetch = FetchType.LAZY)
private List<Comment> comments;
```

게시글을 조회할 때 **댓글은 가져오지 않습니다**. `post.getComments()`를 실제로 호출하는 시점에 DB 쿼리가 실행됩니다.

- 필요할 때만 조회하므로 효율적
- **JPA 기본 권장 전략**

### 기본값

| 어노테이션 | 기본 로딩 전략 |
|-----------|--------------|
| `@ManyToOne` | **EAGER** (즉시 로딩) |
| `@OneToMany` | **LAZY** (지연 로딩) |

`@ManyToOne`의 기본값이 EAGER인 것은 주의가 필요합니다. 실무에서는 보통 `@ManyToOne(fetch = FetchType.LAZY)`로 명시적으로 지연 로딩을 설정합니다.

## 7. cascade — 연쇄 작업

게시글을 삭제하면 달린 댓글도 함께 삭제되어야 하지 않을까요?

```java
@OneToMany(mappedBy = "post", cascade = CascadeType.ALL)
private List<Comment> comments = new ArrayList<>();
```

`cascade = CascadeType.ALL`을 설정하면:
- 게시글을 저장하면 → 댓글도 함께 저장
- 게시글을 삭제하면 → 댓글도 함께 삭제

### 주요 CascadeType

| 타입 | 동작 |
|------|------|
| `PERSIST` | 저장 시 연쇄 저장 |
| `REMOVE` | 삭제 시 연쇄 삭제 |
| `MERGE` | 수정 시 연쇄 수정 |
| `ALL` | 위의 모든 것 포함 |

cascade는 **부모가 자식의 생명주기를 완전히 관리**할 때만 사용합니다.
게시글이 삭제되면 댓글도 의미가 없으므로 적합한 경우입니다.
반면, 회원이 탈퇴한다고 게시글까지 삭제해야 할지는 비즈니스 요구사항에 따라 다릅니다.

---

## 8. orphanRemoval — 고아 객체 제거

```java
@OneToMany(mappedBy = "post", cascade = CascadeType.ALL, orphanRemoval = true)
private List<Comment> comments = new ArrayList<>();
```

`orphanRemoval = true`를 설정하면, 부모와의 관계가 끊어진 자식 엔티티를 자동으로 삭제합니다.

```java
post.getComments().remove(comment);  // 리스트에서 제거
// → orphanRemoval이 true이면 comment가 DB에서도 삭제됨
```

cascade와 함께 사용하면 부모 엔티티가 자식의 생명주기를 완전히 관리하게 됩니다.

## 9. 핵심 정리

| 개념 | 설명 |
|------|------|
| **외래키** | 다른 테이블의 기본키를 참조하는 컬럼. 테이블 간 관계를 표현 |
| **1:N 관계** | 하나가 여러 개를 가지는 관계. 게시글(1) : 댓글(N) |
| **@ManyToOne** | N쪽에 선언. 외래키를 관리하는 관계의 주인 |
| **@OneToMany** | 1쪽에 선언. mappedBy로 주인이 아님을 표시 |
| **관계의 주인** | 외래키가 있는 쪽. 이쪽에서 설정해야 DB에 반영됨 |
| **양방향 관계** | 양쪽 모두 참조를 가짐. 양쪽 다 값을 설정해야 함 |
| **지연 로딩 (LAZY)** | 연관 엔티티를 실제 사용할 때 조회. 권장 전략 |
| **즉시 로딩 (EAGER)** | 연관 엔티티를 즉시 함께 조회. 성능 주의 |
| **cascade** | 부모 엔티티의 작업을 자식에게 연쇄 적용 |
| **orphanRemoval** | 부모와 관계 끊어진 자식을 자동 삭제 |
