# Phase 2: 데이터베이스 연동 — 학습 정리

## 1. JPA와 데이터베이스 기본

Phase 1이나 콘솔 프로그램에서는 메모리에 데이터를 저장했기 때문에 애플리케이션이 꺼지면 동시에 저장한 값들이 모두 지워진다.
이를 방지하기 위해 실제 서비스에서는 외부에 DB, 데이터를 영구적으로 기억할 장치를 마련한다.

데이터베이스에는 RDBMS, NoSQL이 존재하며 우리는 RDBMS를 학습한다.

### SQL을 직접 쓰면 생기는 문제

RDBMS(Relational Database Management System)는 관계형 데이터베이스로 데이터를 테이블(표) 형태로 저장하며 테이블 간의 관계를 정의한다.
SQL은 DBMS에게 질의하는 언어이다. 자바 코드에서 SQL문을 String으로 작성해서 DB에 관련된 처리를 할 수 있지만 몇 가지 문제점이 있다.

1. **반복적인 코드** — 쿼리를 문자열로 작성하고 이를 PreparedStatement에 탑재해 실행시킨다. 쿼리 결과를 얻고 나면 결과를 순회하면서 하나하나 파라미터를 모두 꺼내서 조작해줘야 한다. 상당히 긴 코드가 작성되며 그럴수록 추후 유지보수 비용 또한 상승한다.
2. **패러다임 불일치** — 객체 지향과 관계형 DB는 사고방식이 다르다. 객체는 객체에서 바로 다른 객체에 접근이 가능하지만, RDBMS는 테이블이 분리되어 있어 JOIN이라는 별도 연산을 통해 접근이 가능해진다.
3. **DB에 종속** — ANSI SQL로 공용 쿼리가 있지만 각 RDBMS마다 자체적으로 쓰는 문법들이 있다. 즉 어떤 DB를 사용하면 종속성이 발생해 추후 다른 DB로 전환이 필요할 때 유지보수 비용이 증가한다.

### ORM과 JPA

이러한 불편한 점을 해결하고자 ORM(Object-Relational Mapping)이라는 기술이 도입된다. 객체를 관계로 매핑해주는 기술이다. 이를 통해 SQL을 제거하고 프로그램을 작성하던 Java 언어로 쿼리(DB를 조작)를 작성할 수 있게 되며, DB 교체의 필요성이 생겨도 Java로 작성된 쿼리들은 유지가 된다.

Java 진영의 ORM을 JPA(Java Persistence API)라고 부른다. 자바에서 ORM을 어떻게 구현할 것인지에 대한 약속, 인터페이스를 의미한다. 그것을 직접 구현한 구현체는 따로 있으며, 제일 표준으로 쓰이는 게 Hibernate이다.

Spring Data JPA는 한 단계 더 나아가 JPA를 더 추상화한 계층으로, Hibernate를 직접 다루는 것보다 더 쉽고 자동화되어 ORM 기술을 사용할 수 있게 해준다.

### Entity

Entity는 DB 테이블과 1:1로 매핑되는 자바 클래스이다. 클래스를 선언하고 앞에 `@Entity` 어노테이션만 붙여주면 해당 클래스는 그대로 DB와 매핑되어 저장이 가능해진다. `@Id` 어노테이션으로 기본키를 지정하고, `@GeneratedValue` 어노테이션으로 키 생성 전략을 지정할 수 있다. 클래스의 각 필드는 컬럼이 되어 저장된다.

JPA를 사용하기 위해선 protected 접근 제어자로 된 비어있는 기본 생성자와 getter/setter가 필수로 요구된다. `@Column` 어노테이션을 통해 컬럼 이름, 길이, NOT NULL 같은 제약사항 등을 지정할 수 있으며, 생략하면 스프링 부트의 네이밍 전략에 따라 자동으로 지정된다.

### Repository

DB를 도입했으니 DB 관련 처리를 맡아줄 코드가 필요하다. 본래라면 자바 코드로 쿼리를 작성하고 해당 쿼리를 처리할 메서드 등을 작성해야 했겠지만, JPA에서는 이미 필요한 기능을 만들어놨다. interface를 선언해 `JpaRepository`를 상속받는 것으로 끝나며, 구현체는 Spring이 자동으로 생성해서 주입해준다.

미리 구현되어 있는 보편적인 DB 처리 메서드 외에 필요한 메서드가 있다면, Spring Data JPA가 지정한 메서드 명명 규칙에 맞게 메서드를 선언해주면 자동으로 해당 의미에 따라 쿼리가 작성되고 처리된다.

### 영속성 컨텍스트 (Persistence Context)

영속성 컨텍스트란 Entity를 관리하는 JPA의 내부 저장소로, 캐시를 의미한다. 매번 자원이 많이 드는 DB 조회 대신 자주 쓰일만한 자료들을 미리 불러다 놓는 개념이다.

조회 작업이 일어나면 우선적으로 영속성 컨텍스트에서 해당 데이터가 있는지 찾아 더 빠르게 값을 반환한다. 만약 영속성 컨텍스트에 없다면 그때 DB에 쿼리를 날려 필요한 데이터를 찾아와 처리하고 동시에 영속성 컨텍스트에 저장해둔다.

### 변경 감지 (Dirty Checking)

DB에는 DB대로 값이 있는 상태고, 똑같은 값이 영속성 컨텍스트에도 존재하는 상태이다. 같은 값이지만 두 곳에 존재하기 때문에 하나가 변경되면 다른 하나도 변경되어야만 한다.

영속성 컨텍스트에 있는 엔티티 데이터를 변경하면 JPA는 별도의 save 호출이 없어도 자동으로 변경사항을 DB에 UPDATE 쿼리를 날려 저장해준다. 이를 위해 영속성 컨텍스트가 Entity의 초기 상태를 스냅샷으로 저장해두고, 트랜잭션이 끝났을 때 현재 상태와 비교해서 변경된 부분만 UPDATE 해준다.

### 트랜잭션 (Transaction)

트랜잭션이란 DB 처리 관련된 작업을 묶어서 하나의 작업으로 취급하는 단위이다.

예를 들어 'A 계좌에서 B 계좌로 10만원을 송금한다'라는 작업은:
1. A계좌에서 10만원을 차감
2. B계좌에 10만원을 추가

두 가지 작업으로 나뉜다. 이 두 작업은 반드시 둘 다 성공해야 한다. A계좌에서 10만원 차감만 성공하고 B 계좌에 10만원을 추가하는 걸 실패하면 A계좌에서 돈만 증발하는 치명적인 오류가 발생한다.

그렇기 때문에 이를 '트랜잭션'이라는 단위로 묶고 모든 작업이 무사히 성공함을 보장하도록 처리해야 한다. 만약 2번 작업이 실패한다면 1번 작업을 성공했어도 다시 상태를 되돌려 놔야 하며 이를 **롤백**이라고 한다.

영속성 컨텍스트의 변경 감지는 트랜잭션 안에서만 동작한다. 스프링에서는 `@Transactional` 어노테이션을 메서드에 붙이면 해당 메서드가 하나의 트랜잭션으로 처리된다.

## 2. 연관관계 매핑 (게시글 ↔ 댓글)

### 외래키

앞서 RDBMS는 테이블 간의 관계를 정의한다고 했다. Phase 1에서는 게시글은 게시글대로, 댓글은 댓글대로 따로 구현했고 둘의 관계는 따로 표현되지 않았다. 하지만 실제로 댓글은 게시글에 종속되어야 한다. 이것이 관계이며 JPA에서는 이 관계를 '연관관계 매핑'으로 표현한다.

RDB에서 테이블에는 그 값을 독자적으로 식별 가능한 키(기본키)가 존재한다. 이 키 값은 중복되지 않은 유일한 값으로 해당 행을 대표하는 값이 된다. 관계를 표현하기 위해 테이블에서는 **외래키**를 사용한다. 다른 테이블의 기본키를 자신 테이블에 입력해둬 해당 테이블과 자신 테이블의 행을 연관 지을 수 있게 된다.

### 관계의 종류

관계는 4가지 케이스가 있다:
- **1:N (일대다)** — 게시글 1개에 N개의 댓글을 달 수 있음
- **N:1 (다대일)** — 댓글 N개는 1개의 게시글에 속함
- **1:1 (일대일)** — 사용자 1명은 프로필을 1개만 가질 수 있음
- **N:M (다대다)** — 게시글 N개에는 태그가 M개 달릴 수 있음

### 단방향과 양방향

자바의 객체에서는 참조를 통해 두 객체가 연결되며, 이는 단방향/양방향 개념으로 나뉜다. 객체는 참조를 가진 경우에만 상대방에게 접근이 가능하다.

- `comment.getPost()` — 댓글 객체가 게시글 필드를 가지고 있으면 접근 가능
- `post.getComments()` — 게시글 객체가 댓글 리스트 필드를 가지고 있어야 접근 가능

하지만 데이터베이스는 어느 한 테이블에 외래키가 존재하면 JOIN을 통해 두 테이블이 서로를 인식할 수 있다. RDB는 기본적으로 양방향의 모양을 가지게 된다.

이 차이가 DB의 테이블을 객체로 그대로 치환하기 어려움을 유발하며, JPA가 이를 해결해준다.

### @ManyToOne과 @OneToMany

```java
@Entity
public class Comment {
    @ManyToOne
    @JoinColumn(name = "post_id")
    private Post post;
}
```

`@ManyToOne`은 "댓글(Many)이 게시글(One)에 속한다"는 관계를 선언한다. `@JoinColumn`은 DB에 생성될 외래키 컬럼 이름을 지정한다. Post 타입을 필드로 가짐으로써 댓글에서 게시글을 참조할 수 있게 된다.

```java
@Entity
public class Post {
    @OneToMany(mappedBy = "post")
    private List<Comment> comments = new ArrayList<>();
}
```

`@OneToMany`는 "게시글(One)이 댓글(Many) 여러 개를 가진다"는 관계를 선언한다. `mappedBy = "post"`는 Comment 클래스의 `post` 필드가 관계의 주인임을 나타낸다.

### 관계의 주인

양방향 관계에서 댓글을 게시글에 연결하는 방법이 두 가지 있다:

```java
comment.setPost(post);              // Comment 쪽에서 설정
post.getComments().add(comment);    // Post 쪽에서 설정
```

그런데 DB에는 외래키가 하나(`comments.post_id`)뿐이다. JPA가 양쪽의 변경을 모두 DB에 반영하면, 두 쪽이 서로 다른 값을 설정했을 때 어느 쪽을 따라야 할지 알 수 없다. 그래서 JPA는 **한쪽만 DB에 반영되도록** 관계의 주인을 정한다.

**규칙: 외래키가 있는 쪽(N쪽)이 관계의 주인이다.**

- **주인(Comment)**: `@JoinColumn`으로 외래키를 관리. 이쪽에서 `setPost()`를 해야 DB에 반영됨
- **반대쪽(Post)**: `mappedBy`로 주인이 아님을 표시. 읽기 전용

실무에서는 양쪽 다 설정하는 것이 안전하다:
```java
comment.setPost(post);              // 주인 쪽 → DB 반영
post.getComments().add(comment);    // 반대쪽 → 자바 객체 일관성 유지
```

### 지연 로딩과 즉시 로딩

연관된 엔티티를 **언제** 가져올지 결정하는 전략이다.

- **즉시 로딩 (EAGER)** — 엔티티를 조회할 때 연관 엔티티도 무조건 함께 가져온다. 필요 없을 때도 가져오므로 성능 문제가 발생할 수 있다.
- **지연 로딩 (LAZY)** — 연관 엔티티를 실제로 사용할 때(`getComments()` 호출 시) 비로소 DB에서 조회한다. 필요할 때만 가져오므로 효율적이다.

`@ManyToOne`의 기본값이 EAGER이므로, 실무에서는 `@ManyToOne(fetch = FetchType.LAZY)`로 명시적으로 지연 로딩을 설정한다. 모든 연관관계를 LAZY로 설정하고, 함께 필요한 경우에만 fetch join 등으로 명시적으로 가져오는 것이 권장된다.

### cascade — 연쇄 작업

부모 엔티티의 작업을 자식 엔티티에 연쇄 적용하는 기능이다.

```java
@OneToMany(mappedBy = "post", cascade = CascadeType.ALL)
private List<Comment> comments = new ArrayList<>();
```

`cascade = CascadeType.ALL`을 설정하면:
- 게시글을 저장하면 → 댓글도 함께 저장
- 게시글을 삭제하면 → 댓글도 함께 삭제

부모가 자식의 생명주기를 완전히 관리할 때 사용한다. 게시글이 삭제되면 댓글도 의미가 없으므로 적합한 경우이다.

### orphanRemoval — 고아 객체 제거

```java
@OneToMany(mappedBy = "post", cascade = CascadeType.ALL, orphanRemoval = true)
private List<Comment> comments = new ArrayList<>();
```

부모와의 관계가 끊어진 자식 엔티티를 자동으로 삭제한다. cascade와의 차이:

| 상황 | cascade = REMOVE | orphanRemoval = true |
|------|:---:|:---:|
| 부모 삭제 | 자식도 삭제 | 자식도 삭제 |
| 리스트에서 자식 제거 | 자식 DB에 남아있음 | **자식 DB에서 삭제** |

```java
post.getComments().remove(comment);  // 리스트에서 제거
// orphanRemoval이 true이면 → comment가 DB에서도 삭제됨
```

cascade와 orphanRemoval을 함께 사용하면 부모 엔티티가 자식의 생명주기를 완전히 관리하게 된다.
