# 02. 연관관계 매핑 — Q&A

## Q1: 패러다임 불일치가 주목받는 이유가 DB와 자바 객체를 양방향으로 변환하기 위함인가요?

맞다. DB는 테이블/행/외래키로, 자바는 객체/필드/참조로 데이터를 표현한다. 이 둘을 양방향으로 변환하기 위해 연관관계 매핑이 필요하다:

- 자바 객체 → DB 저장: 객체의 참조 관계를 외래키로 변환
- DB 조회 → 자바 객체: 외래키 관계를 객체의 참조로 변환

JPA의 연관관계 어노테이션(`@ManyToOne`, `@OneToMany` 등)은 이 변환 규칙을 정의하는 것이다.

## Q2: @ManyToOne의 기본값이 EAGER인 것이 왜 주의가 필요한가요?

EAGER가 기본이면, 댓글을 조회할 때마다 게시글도 무조건 함께 조회된다. 댓글 100개를 조회하면 각 댓글마다 게시글을 가져오는 쿼리가 추가 실행될 수 있다. 댓글 내용만 필요한 상황에서도 불필요한 조회가
발생한다.

LAZY로 설정하면 필요할 때만 가져온다. 함께 조회가 필요하면 fetch join 등 명시적인 방법이 있으므로, 기본을 LAZY로 두는 것이 안전하다.

**실무 관례: 모든 연관관계를 LAZY로 설정하고, 필요한 경우에만 명시적으로 함께 조회한다.**

## Q3: cascade와 orphanRemoval의 차이가 무엇인가요?

둘은 트리거 조건이 다르다.

- **cascade = REMOVE**: 부모 엔티티가 삭제될 때 자식도 함께 삭제
- **orphanRemoval = true**: 부모와의 관계가 끊어질 때 자식을 삭제. 부모가 삭제되지 않아도 동작

```java
// 게시글은 살아있고, 댓글 목록에서 하나를 빼는 경우
post.getComments().

remove(comment);

// cascade만 있으면: comment는 DB에 남아있음 (부모가 삭제된 게 아니므로)
// orphanRemoval이 있으면: comment가 DB에서 삭제됨 (부모와 관계가 끊어졌으므로)
```

| 상황          | cascade = REMOVE | orphanRemoval = true |
|-------------|:----------------:|:--------------------:|
| 부모 삭제       |      자식도 삭제      |        자식도 삭제        |
| 리스트에서 자식 제거 |   자식 DB에 남아있음    |      자식 DB에서 삭제      |

cascade의 MERGE(수정)는 자식의 내용을 함께 수정하는 것이지 삭제가 아니다. "부모의 자식 목록에서 빠졌다"는 것은 수정이 아니라 관계의 해제이며, 이것을 감지하는 것이 orphanRemoval이다.
