# 분산 시스템의 이름 계약 — 멱등 선언, 합의, 조용한 실패

## 질문 배경

Phase 7 정리 C3 답변 중 RabbitMQ 의 Binding 정의 위치를 다루다가 다음 의문이 등장했다.

- *"Producer 측에서 Binding 을 정의할 수도 있고 Consumer 측에서 정의할 수도 있고, 양쪽 모두 정의할 수도 있다고 했다. 그런데 양쪽 모두 정의 가능이라는 게 *각자 다른 내용으로* 정의해버리면 통신이 안 되지 않나? 그러면 양쪽이 어떤 Exchange, Queue, Binding 을 쓸지 서로 따로 상의해서 결정하는 건가?"*

이 의문은 *분산 시스템에서 두 시스템이 어떻게 합의에 도달하는가* 라는 본질적 질문을 짚는다. 메시지 큐의 통신은 결국 *이름 기반 계약* 에 의존하며, 이 계약을 어떻게 관리하느냐가 분산 시스템 설계의 핵심 영역이다.

---

## 1. 멱등 선언의 의미

RabbitMQ 의 핵심 설계 원칙 중 하나는 **멱등성 (idempotent) 선언** 이다.

> 같은 이름의 Exchange, Queue, Binding 을 *여러 번 선언해도 충돌이 없다*. 이미 있으면 무시되고, 없으면 생성된다.

이 멱등성 덕분에 *누가 어디서 선언하든 결과는 같다*. 다음과 같은 패턴들이 모두 가능하다.

**Producer 측 선언**. A 애플리케이션 config 에 Exchange, Queue, Binding 을 모두 정의. Producer 가 인프라까지 관리하는 모델. A 가 먼저 떠야 인프라가 구성된다.

**Consumer 측 선언**. B 애플리케이션 config 에 정의. 본 학습 프로젝트가 선택한 방식. *Consumer 가 자기가 받을 큐를 책임지는* 시각의 자연스러운 표현.

**양쪽 모두 선언**. 멱등성으로 충돌 없이 양쪽의 선언이 받아들여진다. 어느 쪽이 먼저 떠도 OK 라 가용성이 높아진다.

**외부 도구 선언**. Terraform, RabbitMQ 관리 UI, 스크립트 등으로 별도 관리. 인프라 코드와 비즈니스 코드의 분리.

핵심은 *Exchange, Binding, Queue 자체가 RabbitMQ 브로커 내부의 객체* 라는 점이다. 누가 등록 명령을 내리든 브로커에 등록되면 그만이다. 어디서 선언했는지는 *논리적 배치의 선호* 일 뿐 본질이 아니다.

---

## 2. 멱등성의 함정 — 같은 *이름* 일 때만 멱등

학습자가 짚은 의문의 핵심이 여기에 있다. *"양쪽이 자유롭게 다른 내용으로 선언해버리면?"*

멱등성이 보장하는 것은 *같은 이름과 같은 구성으로 선언될 때* 다. 이름이나 구성이 다르면 멱등성이 깨진다.

A 가 `exchange.comment` 를 Fanout 으로 선언하고 B 가 같은 `exchange.comment` 를 Direct 로 선언하면? **충돌, 에러 발생** 한다. 같은 이름인데 타입이 다르기 때문에 멱등이 아니다.

A 가 `exchange.foo` 에 발행하고 B 가 `exchange.bar` 에서 받으려 하면? **충돌은 없지만 별개의 Exchange 두 개가 등록** 된다. 양쪽 다 정상 동작하는 것처럼 보이지만 *메시지가 서로에게 도달하지 않는다*.

즉 *"자유롭게 다른 내용으로 선언해도 된다"* 가 아니라 *"같은 내용이면 양쪽이 동시에 선언해도 된다"* 가 멱등성의 정확한 의미다.

---

## 3. 분산 시스템의 *계약 (Contract)*

학습자의 직관 *"서로 상의해서 결정하는 건가?"* — 이게 분산 시스템에서 **계약 (Contract)** 이라고 부르는 개념이다.

HTTP API 통신에 비유하면 명확해진다. A 가 `/api/comment` 에 POST 한다고 약속하고 B 가 `/api/comment` 에서 받는다고 약속해야 통신이 성립한다. 만약 B 가 `/api/comments` (s 추가) 에서 받으려 하면 A 가 보내도 B 에 안 도달한다. *양쪽이 같은 엔드포인트를 쓰기로 합의* 해야 한다.

메시지 큐도 동일하다. A 가 `exchange.comment` 에 발행한다고 약속하고 B 가 `exchange.comment` 의 `queue.email` 에서 받는다고 약속해야 통신이 성립한다. **메시지 큐 통신은 결국 *이름 기반 계약*** 이다.

이 계약은 *코드 외부에 존재하는 약속* 이다. 멱등 선언은 그 약속을 *안전하게 등록* 해주는 도구일 뿐, 약속 자체를 만들어주지는 않는다.

---

## 4. 실전에서 *이름 합의* 를 보장하는 방법

분산된 두 시스템이 *같은 이름을 쓰도록 보장* 하는 실무 방법들.

**명세 문서 (Documentation)**. API 명세처럼 메시지 명세를 작성한다. AsyncAPI 스펙, 사내 위키, ADR (Architecture Decision Record) 등을 통해 *어떤 Exchange/Queue/Binding 이 있는지* 를 문서화한다. 가장 가벼운 방법이지만 *드리프트 (drift)* 의 위험이 있다 (문서와 실제 구현이 어긋남).

**공유 라이브러리 (Shared Library)**. 이름 상수를 *공통 모듈* 에 두고 양쪽이 import 한다. 예를 들어 `messaging-spec` 같은 별도 jar/패키지를 만들어 거기에 `EXCHANGE_NAME = "comment.event.exchange"` 같은 상수를 두고 A 와 B 가 모두 참조한다. 본 학습 프로젝트는 *같은 JVM* 이라 `RabbitMQProducerConfig.EXCHANGE_NAME` 을 직접 import 해서 썼지만, 실제 분리 시에는 공유 라이브러리로 빼는 게 일반적이다.

**인프라 코드 (Infrastructure as Code)**. Terraform 같은 IaC 도구가 *단일 진실 공급원* 이 된다. RabbitMQ 의 Exchange, Queue, Binding 을 IaC 로 정의하고 A 와 B 는 그저 *이름을 참조* 한다. 대규모 시스템에서 일반적인 방식.

**환경 변수 / 설정 중앙화**. Spring Cloud Config 같은 설정 서버에서 이름을 중앙 관리한다. A 와 B 가 같은 설정 서버를 참조해서 이름을 가져온다.

어떤 방식을 쓰든 본질은 *"양쪽이 같은 이름을 쓰도록 강제하는 메커니즘"* 이다.

---

## 5. *조용한 실패* — 이름 어긋남의 가장 큰 위험

이름이 어긋났을 때의 가장 큰 함정은 *에러가 안 난다는 점* 이다.

A 가 `exchange.foo` 에 발행하고 B 가 `queue.bar` 에서 대기 중이라면, A 의 발행은 성공적으로 완료되고 (A 입장에서는 OK), 메시지는 `exchange.foo` 에 들어가지만 *아무 Binding 도 없어서 폐기되거나 미배달* 된다. B 는 `queue.bar` 에서 대기 중인데 *아무 메시지도 오지 않는다* (B 입장에서는 그냥 조용하다).

양쪽 다 *에러 없이 정상 동작* 하는 것처럼 보이지만 *통신이 안 된다*. 코드가 에러를 잡아주지 않으니 *사전 합의와 명세가 없으면 디버깅이 거의 불가능* 하다. *분산 시스템 디버깅에서 가장 짜증나는 유형* 의 문제다.

이런 조용한 실패를 방지하기 위해 *합의 메커니즘* 이 필수다. 공유 라이브러리나 IaC 같은 방식은 *컴파일 시점이나 배포 시점에 이름 불일치를 감지* 할 수 있게 해준다.

---

## 6. 학습자 질문에 대한 직답

> *"양쪽이 자유롭게 선언하면 통신 못 함?"*

정확함. 자유롭게 *다른 내용* 으로 선언하면 통신 불가. *조용히 실패* 까지 동반.

> *"그러면 양쪽이 상의해야 하는 것 아닌가?"*

정확함. 이게 *분산 시스템의 계약*. 멱등성은 약속을 등록하는 도구일 뿐, 약속을 만드는 책임은 시스템 설계자에게 있다.

> *"양쪽 모두 선언 가능이란 게 무슨 뜻?"*

*같은 내용일 때* 멱등으로 받아들여진다는 뜻. 내용을 자유롭게 정한다는 뜻이 아님. 멱등성이 보장하는 건 *같은 선언이 충돌 없이 여러 번 받아들여짐* 이고, 보장하지 않는 건 *양쪽이 다른 내용을 선언했을 때의 일치* 다.

---

## 한 줄 요약

분산 시스템의 통신은 결국 *이름 기반 계약* 에 의존한다. 멱등 선언은 *같은 이름의 선언을 충돌 없이 받아들이는* 도구이지 *양쪽의 합의를 자동으로 만드는* 도구가 아니다. 이름이 어긋나면 *조용히 실패* 하므로, *명세 문서, 공유 라이브러리, 인프라 코드* 같은 *합의 메커니즘* 이 분산 시스템 설계의 필수 요소다.
