# HTTP 직접 호출 vs 메시지 큐

## 질문 배경

Phase 7 정리 C2 답변 중 HTTP 직접 호출 대비 메시지 큐의 장점 다섯 가지를 정리하다가 다음 의문들이 등장했다.

- HTTP 직접 호출의 본질적 특성이 무엇이고 메시지 큐와 정확히 어떻게 대비되는가
- 메시지 큐도 결국 *A → 메시지 큐 → B* 의 통신이라면 *A → 메시지 큐* 단계에서도 실패가 발생할 수 있는데, 어떻게 "더 안전한" 시스템이라고 할 수 있는가
- HTTP 직접 호출에서 *장애 전파* 가 구체적으로 어떤 메커니즘으로 일어나는가

이 세 가지를 정리해서 *왜 메시지 큐가 HTTP 직접 호출보다 신뢰성이 높다고 평가되는가* 의 본질을 분명히 한다.

---

## 1. HTTP 직접 호출의 본질적 특성

HTTP 는 *두 시스템이 직접 통신하는 가장 단순한 방식* 이다. 그 단순함의 대가로 다음과 같은 특성이 자연스럽게 따라온다.

**요청-응답 사이클**. 클라이언트가 요청을 보내고 응답을 받기 전까지 기다린다. 동기적 통신이다. 응답을 받는 그 시간 동안 *호출자의 스레드는 점유* 된다.

**1:1 통신**. 한 번의 호출은 *하나의 서버* 에게만 도달한다. 여러 시스템에 같은 메시지를 보내려면 여러 번 호출해야 한다.

**상태 없음 (Stateless)**. 매 요청이 독립적이고 서버는 이전 요청을 기억하지 않는다. 이건 HTTP 의 강점이기도 하다.

**즉시 처리 가정**. 서버는 요청을 받으면 *즉시 처리하고 응답해야* 한다. 처리에 시간이 걸리면 클라이언트가 타임아웃을 겪는다.

**양방향 가용성 필요**. 호출 시점에 *클라이언트와 서버 모두 살아있어야* 한다. 한쪽이 죽어있으면 통신 자체가 불가능하다.

**즉시 결과 확인**. 응답 코드 (2xx, 4xx, 5xx) 로 성공/실패를 *그 자리에서* 알 수 있다.

---

## 2. 메시지 큐와의 본질적 대비

같은 *두 시스템 간 통신* 문제를 메시지 큐는 정반대 방향으로 해결한다.

HTTP 직접 호출에서는 호출자가 피호출자에게 직접 요청을 보내고 응답을 기다린다. 메시지 큐에서는 호출자가 *메시지 큐에 발행만 하고* 끝낸다. 피호출자는 *큐에서 자기 페이스로 가져가* 처리한다.

이 차이가 만드는 결과:
- **동기 vs 비동기** — HTTP 는 응답까지 대기, 메시지 큐는 발행 즉시 리턴
- **즉시성 vs 시간 분리** — HTTP 는 즉시 처리 필요, 메시지 큐는 컨슈머가 자기 속도로 처리
- **직접성 vs 간접성** — HTTP 는 피호출자 주소를 알아야 함, 메시지 큐는 큐 이름만 알면 됨
- **일회성 vs 영속성** — HTTP 는 호출 실패 시 호출자가 재시도 책임, 메시지 큐는 큐가 메시지 보관

HTTP 는 *즉시 응답이 필요한 동기적 요청* 에 적합하다. 사용자가 요청한 데이터를 즉시 돌려줘야 하는 경우. 메시지 큐는 *이벤트성 알림, 작업 위임* 에 적합하다. 일이 발생했음을 알리고 처리는 나중에 되어도 OK 인 경우.

## 3. HTTP 에서 장애가 전파되는 메커니즘

*"한쪽의 장애가 다른 쪽으로 전파된다"* 는 표현이 추상적으로 들릴 수 있지만, 실제 메커니즘은 매우 구체적이다. **Little's Law** 로 정확히 설명된다.

A 가 B 에 HTTP 호출을 한다고 가정하자. 평상시 B 가 100ms 에 응답한다면, A 는 호출 한 건당 *자기 스레드를 100ms 동안 점유* 한다. 초당 50 건의 호출이 들어온다면 동시 점유 스레드는 *L = λ × W = 50 × 0.1 = 5 개*. A 의 스레드 풀이 100 개라면 충분히 여유롭다.

이제 B 가 다운되거나 느려진다고 하자. B 가 응답을 안 보내면 A 는 *타임아웃 시간까지 대기* 한다. 보통 30 초. 같은 초당 50 건이 들어오면 동시 점유 스레드가 *50 × 30 = 1500 개* 가 된다. A 의 스레드 풀 100 개를 훨씬 초과한다.

이때 A 는 *B 와 무관한 다른 요청들 (댓글 조회, 로그인, 회원가입 등) 도 처리하지 못한다*. 스레드가 모두 *B 의 응답을 기다리느라 점유* 되어 있기 때문이다. **A 가 외부에서 보기에 다운된 것처럼 보이게 된다**.

이게 **연쇄 실패 (Cascade Failure)** 의 본질이다. B 의 장애가 A 의 장애로 *전파* 되는 것은 *A 의 스레드가 B 의 응답을 기다리느라 묶이기 때문* 이다. 호출 방식이 동기적이라는 점이 이 전파의 근본 원인이다.

메시지 큐는 이 메커니즘을 끊는다. A 는 메시지 큐에 발행한 즉시 스레드를 풀어준다. B 가 죽어있든 느리든 *A 의 스레드 점유 시간은 발행에 걸린 수 ms* 뿐이다. Little's Law 의 동시 점유 수가 폭증하지 않으므로 A 의 풀이 고갈되지 않는다. *호출자의 자원 점유를 피호출자의 처리 시간과 분리* 하는 것이 메시지 큐의 진짜 효과다.

## 4. 메시지 큐도 결국 통신인데 더 안전한 이유

학습자가 짚은 본질적 의문 — *"A → B 의 통신을 A → 메시지 큐로 바꾼 것뿐인데, 메시지 큐가 100% 가용성을 전제하는 건가? A → 메시지 큐 발행도 실패할 수 있지 않나?"*

이 직관은 정확하다. 메시지 큐도 *완벽한 신뢰성* 을 제공하지는 않는다. 다만 *영속성 책임을 전문 인프라에 위임* 한다는 점에서 일반 애플리케이션 간 직접 통신보다 신뢰성이 *질적으로 다른 수준* 으로 올라간다.

메시지 큐는 *메시지를 안전하게 보관* 한다는 단 하나의 임무에 특화된 시스템이다. 다음과 같은 메커니즘들이 신뢰성을 끌어올린다:

**Publisher Confirms** — 메시지가 브로커에 도착했는지 발행자에게 확인 응답을 보낸다. 발행 실패를 즉시 알 수 있다.

**Persistent Messages** — 메시지를 메모리뿐 아니라 *디스크에도 저장* 한다. 브로커가 재시작되어도 메시지가 보존된다.

**Mirrored Queues / Clustering** — 큐를 여러 노드에 복제한다. 한 노드가 다운되어도 다른 노드가 인계한다.

**Disk Replication** — 디스크 자체를 여러 곳에 복제한다.

A 와 B 가 직접 통신하면 두 일반 비즈니스 애플리케이션 사이의 신뢰성에 의존한다. 메시지 큐를 두면 *영속성을 위해 설계된 전문 시스템* 이 중간에 들어온다. *신뢰성의 수준이 다른 영역* 으로 올라가는 셈이다.

## 5. 그래도 남는 문제 — 이중 쓰기 문제

학습자의 직관이 더 깊이 짚는 또 하나의 문제가 있다. *이중 쓰기 문제 (Dual Write Problem)*.

본 학습 시나리오에서 다음 흐름을 보자:

1. A 가 DB 에 댓글을 저장하고 트랜잭션을 커밋한다.
2. A 가 메시지 큐에 발행한다.

만약 2 단계 직전에 A 가 죽는다면? DB 에는 댓글이 저장되었지만 알림 메시지는 발행되지 않았다. *DB 와 메시지 큐는 서로 다른 시스템* 이라 *하나의 원자적 트랜잭션* 으로 묶을 수 없다. 이게 이중 쓰기 문제다. 분산 시스템의 고전적 난제.

해결 패턴 — **Outbox Pattern**. DB 트랜잭션 안에서 *댓글 저장* 과 *발행할 메시지를 outbox 테이블에 저장* 을 함께 처리한다. 이 둘은 *같은 DB 트랜잭션* 이라 원자적이다. 별도 프로세스나 스레드가 outbox 테이블을 폴링해서 미발행 메시지를 메시지 큐로 발행하고, 발행 성공하면 outbox 에서 표시한다.

```
1. DB 트랜잭션 시작
2. comments 테이블에 댓글 저장
3. outbox 테이블에 발행할 메시지 저장
4. 트랜잭션 커밋 (원자적)
─────────────────────
별도 발행기:
5. outbox 테이블 폴링
6. 미발행 메시지를 메시지 큐로 발행
7. 발행 성공 시 outbox 에서 표시
```

이 패턴은 *DB 와 메시지 큐를 하나의 트랜잭션으로 묶기* 의 우회법이다. 댓글 저장과 메시지 보관을 같은 DB 트랜잭션에 묶고, 별도 발행기가 이를 메시지 큐로 옮기는 구조. Phase 10 (아키텍처 심화) 에서 본격적으로 다룰 수 있는 주제다.

---

## 6. 한 줄 요약

메시지 큐는 *완벽한 신뢰성* 을 제공하지는 않지만, *영속성을 위해 설계된 전문 인프라* 가 통신의 중간에 들어옴으로써 일반 애플리케이션끼리의 직접 통신보다 *훨씬 더 신뢰성 있는 영역* 으로 옮겨준다. 그리고 *호출자의 자원 점유를 피호출자의 처리 시간과 분리* 함으로써 *연쇄 실패의 본질적 메커니즘을 끊어준다*. 다만 *이중 쓰기 문제* 같은 분산 시스템의 본질적 난제는 여전히 남아있으며, 이는 Outbox 패턴 같은 별도 설계로 보완된다.
