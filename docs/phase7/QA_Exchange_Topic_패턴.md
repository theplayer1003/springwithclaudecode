# Exchange Topic 패턴 — 마이크로서비스 이벤트 라우팅

## 질문 배경

Phase 7 정리 C4 답변 중 Exchange 의 네 가지 타입 (Direct, Fanout, Topic, Headers) 을 비교하다가 다음 의문이 등장했다.

- *"Topic 을 좀 실사례 시나리오로 학습하고 싶다. 어떤 경우에 유용한가?"*

Fanout 과 Topic 은 *"여러 큐로 메시지를 보낸다"* 라는 점에서 표면적으로 비슷해 보이지만, 사용 의도가 매우 다르다. Topic 의 진짜 가치는 *마이크로서비스 환경의 이벤트 라우팅* 같은 영역에서 드러난다.

---

## 1. Topic Exchange 의 기본 메커니즘

Topic Exchange 는 *Routing Key 의 패턴 매칭* 으로 메시지를 라우팅한다. Producer 가 메시지를 발행할 때 Routing Key 를 지정하고, Consumer 측의 Binding 이 *패턴* 을 정의하면 패턴에 매칭되는 메시지가 해당 큐로 라우팅된다.

패턴 문법은 두 가지 와일드카드를 사용한다. `*` 는 *정확히 한 단어* 와 매칭되고 (`order.*` 는 `order.created` 와 매칭되지만 `order.line.added` 와는 매칭 안 됨), `#` 는 *0개 이상의 단어* 와 매칭된다 (`order.#` 는 `order.created`, `order.line.added`, `order` 모두와 매칭).

이 패턴 기반 라우팅이 Topic 의 본질이다.

---

## 2. Fanout vs Topic 의 본질적 차이

표면적으로 둘 다 *여러 큐에 메시지를 보낼 수 있다*. 하지만 사용 의도가 다르다.

**Fanout 의 의도** — *"하나의 이벤트에 여러 컨슈머가 관심을 가진다"*. 같은 메시지를 *모든 구독자에게 동일하게 전달* 한다. 본 학습 프로젝트의 알림 시나리오가 이 패턴이다. 댓글 작성이라는 *하나의 이벤트* 에 *이메일 알림과 SMS 알림* 두 구독자가 모두 같은 메시지를 받는다.

**Topic 의 의도** — *"여러 종류의 이벤트 중 각 컨슈머가 일부만 관심을 가진다"*. 한 Exchange 에 *다양한 종류의 메시지* 가 들어오고 각 구독자는 *패턴으로 자기 관심사만 필터링* 한다.

→ *Fanout 은 "이벤트의 종류는 하나, 구독자 여럿"*, *Topic 은 "이벤트의 종류가 다양, 각 구독자가 일부 선택"* 의 구조다.

이 차이는 *처음에는 같은 일을 두 방식으로 구현할 수 있는 것처럼 보이지만* 시스템이 커지면서 결정적인 차이를 만든다.

---

## 3. 실사례 시나리오 — 마이크로서비스 이벤트 라우팅

Topic 의 진가가 발휘되는 대표 영역은 *마이크로서비스 환경의 이벤트 버스* 다.

전자상거래 시스템을 가정해보자. 다음과 같은 이벤트들이 발행된다:

```
order.created
order.cancelled
order.paid
payment.failed
payment.succeeded
delivery.started
delivery.completed
user.signup
user.deleted
```

이 모든 이벤트가 하나의 Topic Exchange (`events.exchange`) 로 들어온다. 각 팀/서비스의 큐는 *자기에게 필요한 이벤트만 패턴으로 구독* 한다:

- **재고팀 큐** — `order.*` 패턴으로 Binding → 모든 주문 이벤트만 받음. 재고 차감/복구 처리.
- **결제팀 큐** — `payment.*` 패턴 → 결제 관련 이벤트만. 결제 상태 추적.
- **알림팀 큐** — `*.completed` 패턴 → 완료성 이벤트만. 고객에게 완료 알림.
- **감사 로그 큐** — `#` 패턴 → 모든 이벤트. 영구 보관.
- **환영 메일 큐** — `user.signup` 패턴 (정확 매칭) → 회원가입만.
- **실패 알림 큐** — `*.failed` 패턴 → 모든 실패 이벤트. 운영팀에 경고.

새로운 이벤트가 추가될 때 (예: `order.refunded`), 기존 구독자들은 *자기 패턴에 맞으면 자동으로 받기 시작* 한다. Producer 코드는 변경 없이 그대로다. Consumer 측도 *자기 패턴을 수정하지 않는 한 영향 없음*. 매우 유연한 확장.

---

## 4. 왜 Fanout 으로 못 하는가

위 시나리오를 Fanout 으로 구현하려면 *이벤트 종류만큼 Exchange 를 만들어야* 한다. `order.created.exchange`, `order.cancelled.exchange`, `payment.failed.exchange` 등등. 각 Exchange 에 *관심 있는 큐들이 모두 Binding* 되어야 한다.

이 방식의 문제:
- Exchange 수가 폭증한다
- 새 이벤트 종류 추가 시 *새 Exchange 생성 + 모든 관심 있는 큐의 Binding 추가* 가 필요하다
- Producer 가 *어떤 Exchange 에 발행할지* 결정해야 한다 (이벤트 종류에 따라)
- *"모든 완료성 이벤트를 받는다"* 같은 *횡단 관심사* 구독이 어렵다

Topic 은 이 모든 문제를 *Exchange 하나 + 패턴 매칭* 으로 우아하게 해결한다. *이벤트의 종류는 늘어나도 인프라는 그대로* 다.

---

## 5. Topic 이 적합한 다른 시나리오들

마이크로서비스 외에도 Topic 이 자연스러운 영역들이 있다.

**다국가/다지역 시스템**. `event.us.order.created`, `event.eu.order.created` 같은 패턴으로 *지역별 처리* 가 가능. 미국 처리 시스템은 `event.us.*` 만 구독, EU 처리 시스템은 `event.eu.*` 만 구독.

**로깅/모니터링 시스템**. `log.error.payment`, `log.warn.order` 같은 식으로 *심각도와 도메인의 조합* 으로 라우팅. 에러 알림 시스템은 `log.error.*` 구독, 결제 로그 분석은 `log.*.payment` 구독.

**알림 시스템의 채널별 분기**. `notification.email.user_signup`, `notification.sms.user_signup` 같은 패턴으로 *채널별 분기*. 이메일 발송 시스템은 `notification.email.*` 만, SMS 발송은 `notification.sms.*` 만.

이 모든 시나리오의 공통점은 *이벤트의 종류가 다양하고, 구독자별로 관심사가 다르고, 확장이 자주 일어난다* 는 점이다.

---

## 6. 본 학습 프로젝트에서 Fanout 을 선택한 이유

본 학습 프로젝트는 *댓글 작성 알림* 시나리오로, *하나의 이벤트 (CommentCreatedEvent) 에 여러 구독자 (이메일, SMS) 가 모두 관심* 인 단순한 패턴이다. 모든 구독자가 같은 메시지를 받으면 되므로 Fanout 이 적합하다.

만약 향후 *다양한 이벤트 종류가 추가* 된다면 (예: 게시글 작성, 좋아요, 활동 점수 변경 등) 그때 Topic 으로 전환하는 게 자연스러운 발전이다. 지금 시점에서는 *이벤트 종류가 한 가지* 라 Fanout 으로 충분하고, 미리 Topic 을 도입하는 건 *오버 엔지니어링* 이 될 수 있다.

이런 결정이 *YAGNI (You Aren't Gonna Need It)* 원칙의 적용 사례다. *현재 필요한 만큼만 도입하고 필요해질 때 발전시킨다*.

---

## 7. Topic 의 장단점 정리

**장점**
- 이벤트 종류가 다양한 환경에서 *Exchange 인프라가 단순*
- *횡단 관심사* (예: 모든 에러, 모든 완료) 구독이 쉬움
- 새 이벤트 추가 시 *기존 구독자 영향 최소*
- *패턴 매칭의 유연성*

**단점**
- Routing Key 의 *계층 구조 설계* 가 필요 (네이밍 규칙 미리 합의)
- 패턴이 복잡해지면 *디버깅이 어려움*
- Fanout 대비 *오버헤드가 약간 더 큼* (패턴 매칭 비용)
- *이벤트가 한 종류* 인 단순 시나리오에서는 *과한 도구*

---

## 한 줄 요약

Fanout 은 *"하나의 이벤트, 여러 구독자"* 의 단순 브로드캐스트, Topic 은 *"여러 종류의 이벤트, 각 구독자가 패턴으로 선택"* 의 라우팅이다. Topic 의 진가는 *마이크로서비스의 이벤트 버스* 처럼 *다양한 이벤트 종류와 구독자별 다른 관심사* 가 있는 환경에서 발휘된다. 단순한 시나리오에서는 Fanout 이 충분하므로 *필요해질 때 발전시키는* 것이 자연스럽다.
