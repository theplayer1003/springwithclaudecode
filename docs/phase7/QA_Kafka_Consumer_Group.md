# Kafka Consumer Group — 컨슈머 협력 단위의 이해

## 질문 배경

Phase 7 정리 C6 답변 중 Kafka 를 정리하다가 다음 의문이 등장했다.

- *"컨슈머의 협력 단위라는 표현이 잘 이해되지 않는다. 어떤 시나리오에서 컨슈머들이 무슨 협력을 해서 처리한다는 것인가?"*

이 질문은 Kafka 의 핵심 메커니즘인 *Consumer Group* 의 의미를 묻는다. *왜 Kafka 에는 Consumer Group 이라는 개념이 별도로 있고*, *RabbitMQ 와는 무엇이 다른가* 를 정리할 필요가 있다.

---

## 1. Consumer Group 의 정의

Consumer Group 은 *Kafka 의 같은 토픽을 함께 읽기 위해 여러 컨슈머를 하나의 팀으로 묶은 단위* 다. 같은 토픽을 여러 컨슈머가 분담해서 처리하고 싶을 때 그룹으로 묶는다.

핵심 원리는 *파티션 단위로 컨슈머에게 할당* 된다는 점이다. 그룹 안의 각 컨슈머는 토픽의 *일부 파티션* 을 담당하고, 같은 파티션의 메시지는 *항상 같은 컨슈머* 가 처리한다. 그룹 내에서는 *메시지가 중복 처리되지 않고 부하만 분산* 된다.

다른 그룹은 *서로 무관하게 같은 토픽을 독립적으로* 구독할 수 있다. 그룹 A 가 처리한 메시지를 그룹 B 가 자기 페이스로 또 처리한다.

---

## 2. 같은 그룹 안에서의 협력

같은 Consumer Group 에 속한 컨슈머들은 *완전히 같은 역할의 인스턴스들* 이다. 같은 코드를 가진 같은 종류의 컨슈머. 보통은 *같은 Spring Boot 애플리케이션을 여러 서버에 띄운 것* 또는 *같은 애플리케이션 안의 여러 스레드* 다.

토픽이 4 개의 파티션을 가지고 그룹에 컨슈머가 2 개 있다면 Kafka 가 자동으로 할당한다:
- Consumer A → 파티션 P0, P1 담당
- Consumer B → 파티션 P2, P3 담당

이게 *"협력"* 의 의미다. 둘이서 4 개 파티션을 *나눠 처리* 한다. 어느 컨슈머든 처리할 일은 *이메일 발송* 같은 *같은 작업* 이다. 다만 *어느 메시지를 받느냐* 가 다를 뿐이다.

컨슈머가 추가되면 *전체 재분배 (rebalancing)* 가 발생한다. 3 개로 늘어나면 P0/P1, P2, P3 식으로 다시 할당. 한 컨슈머가 죽으면 그가 담당하던 파티션이 *남은 컨슈머에게 재할당* 된다. 이렇게 *수평 확장* 과 *장애 복원* 이 자연스럽게 이뤄진다.

---

## 3. 다른 그룹 사이에서의 독립성

서로 다른 Consumer Group 은 *완전히 무관하게 같은 토픽을 구독* 한다.

```
[Consumer Group "주문처리팀"]
  Consumer A → P0, P1
  Consumer B → P2, P3
  → 주문 처리 로직 수행

[Consumer Group "분석팀"]
  Consumer C → P0, P1, P2, P3 (혼자라 다 받음)
  → 분석을 위한 다른 로직 수행
```

두 그룹은 각자 *독립적으로 같은 메시지를 처리* 한다. 주문처리팀이 메시지를 읽었다고 분석팀이 못 읽는 게 아니다. 메시지는 토픽에 *계속 남아있고* 각 그룹이 *자기 offset 으로* 읽어간다.

이게 Kafka 의 강력함이다. *"같은 토픽을 여러 시스템이 다른 목적으로 동시 구독, 각 시스템 안에서는 부하 분산"* 의 구조가 자연스럽다.

---

## 4. *"내 메시지를 어떻게 찾아가는가"* 의 함정

학습자가 *"컨슈머가 자기 메시지를 어떻게 찾아가는가"* 같은 표현으로 의문을 던졌는데, 이건 RabbitMQ 사고에서 비롯된 자연스러운 오해다.

Kafka 에서는 *"내가 처리해야 할 메시지" 라는 개념 자체가 없다*. 같은 Consumer Group 안의 컨슈머들은:
- 완전히 같은 역할의 인스턴스들
- 어느 메시지를 받아도 똑같이 처리할 수 있음
- 누가 어느 메시지를 처리하든 결과는 동일

옆 파티션에 *자신이 처리해야 할 메시지* 가 있어서 가져와야 하는 일은 없다. 옆 파티션은 *동료 컨슈머의 영역* 이고 동료가 똑같이 처리한다. *"내 것/네 것"* 의 구분이 애초에 없다.

Kafka 가 *파티션을 컨슈머에 자동 할당* 하고 컨슈머는 *할당된 파티션의 모든 메시지를 그냥 처리* 하면 끝이다. *"누가 어디서 일을 가져갈지"* 만 분담될 뿐 *"누가 어떤 일을 하는지"* 는 모두 같다.

---

## 5. 왜 파티션이라는 구조가 있는가

Consumer Group 을 이해하려면 *파티션의 존재 이유* 도 같이 이해해야 한다.

**병렬 처리** — 한 토픽의 모든 메시지를 한 컨슈머가 순차 처리하면 처리량이 제한된다. 파티션으로 나누면 *여러 컨슈머가 동시에 다른 파티션을 처리* 가능해서 처리량이 늘어난다. 파티션 4 개면 최대 4 명의 컨슈머가 동시에 처리 가능.

**순서 보장** — 같은 파티션 안에서는 메시지 순서가 보장된다. Producer 가 *키 (key) 를 지정해서* 발행하면 Kafka 가 키를 해싱해서 *항상 같은 파티션* 에 배치한다. 같은 키의 메시지는 *같은 파티션, 같은 컨슈머* 가 순서대로 처리하게 된다. 예를 들어 *"사용자 A 의 주문 → 결제 → 배송"* 을 *키를 사용자 A 로* 발행하면 *같은 컨슈머* 가 *순서대로* 처리하므로 결제 전에 배송이 처리되는 사고가 없다.

**수평 확장 단위** — 파티션 수가 *최대 병렬도의 상한* 이다. 파티션 4 개면 한 그룹에 최대 4 명의 컨슈머까지만 의미 있다. 5 명을 띄우면 1 명은 *할당받을 파티션이 없어서 놀게 된다*. 처리량을 늘리려면 *파티션 수를 미리 충분히 확보* 해야 한다.

→ **파티션 = 병렬 처리의 단위 + 순서 보장의 단위 + 수평 확장의 단위**.

## 6. RabbitMQ 의 부하 분산과의 비교

RabbitMQ 에서도 부하 분산은 가능하다. 같은 큐에 여러 컨슈머가 붙으면 큐가 *round-robin 으로 분배* 한다. 누가 더 빠르면 그 컨슈머가 더 많이 받는다.

차이점:
- RabbitMQ — *메시지 단위 분배*. 큐가 그 순간 손이 빈 컨슈머에게 다음 메시지를 준다. 동적이다.
- Kafka — *파티션 단위 할당*. 한 컨슈머가 자기 파티션의 메시지를 *연속해서* 처리한다. 정적이다 (재할당 전까지).

비유로:
- RabbitMQ — *공장의 컨베이어 벨트*. 작업자 누구든 다음 부품을 집어간다.
- Kafka — *도서관의 책장*. 각 작업자에게 *책장 구역* 이 할당되어 그 구역의 책만 읽는다.

이 차이가 왜 중요한가? *순서 보장* 때문이다. Kafka 는 *같은 키의 메시지를 같은 컨슈머가 순서대로* 처리하도록 보장할 수 있지만, RabbitMQ 는 round-robin 분배라 *같은 큐의 메시지라도 컨슈머가 여럿이면 처리 순서 보장이 깨진다*.

따라서:
- *순서가 중요하지 않은 작업 큐* — RabbitMQ 가 자연스럽다 (동적 부하 분산, 빠른 흡수)
- *키별 순서 보장이 중요한 스트리밍* — Kafka 가 자연스럽다 (파티션 단위 정적 할당)

## 7. 실제 시나리오 예시

전자상거래 시스템에서 *주문 이벤트* 가 발생하면:

```
"order.created" 토픽 (파티션 4 개)

[Group "재고팀"]      → 재고 차감
  Consumer R1 → P0, P1
  Consumer R2 → P2, P3

[Group "결제팀"]      → 결제 처리 추적
  Consumer P1 → P0, P1, P2, P3 (혼자라 다 받음)

[Group "알림팀"]      → 고객에게 알림
  Consumer N1 → P0, P1
  Consumer N2 → P2, P3

[Group "분석팀"]      → BI 데이터 적재
  Consumer A1 → P0, P1, P2, P3

[Group "감사팀"]      → 감사 로그
  Consumer L1 → P0, P1, P2, P3
```

다섯 그룹이 *같은 주문 이벤트* 를 받지만 *각자 다른 처리* 를 한다. 각 그룹 안에서는 *여러 컨슈머가 부하를 분담* 한다.

본 학습 프로젝트의 Fanout Exchange + 이메일/SMS 두 큐 구조를 Kafka 로 옮긴다면:
- 토픽: `comment.created` (파티션 N 개)
- Consumer Group "이메일 알림팀" — 이메일 발송 처리
- Consumer Group "SMS 알림팀" — SMS 발송 처리
- 향후 추가될 수도 있는 Consumer Group "분석팀" — 댓글 통계 처리

이런 구조다. RabbitMQ 의 *각 큐가 독립된 메시지를 가짐* 과 Kafka 의 *같은 메시지를 여러 그룹이 독립 offset 으로 읽음* 의 차이가 분명히 드러난다.

---

## 한 줄 요약

Consumer Group 은 *같은 토픽을 함께 읽는 컨슈머들의 협력 단위* 다. 같은 그룹 안에서는 *파티션 단위로 부하가 분산* 되고 (중복 처리 없음), 다른 그룹끼리는 *완전히 독립적으로 같은 토픽을 구독* 한다 (각자 자기 페이스). 컨슈머들은 *완전히 같은 역할의 인스턴스들* 이며 *"내 메시지"* 라는 개념은 없고 *할당된 파티션의 모든 메시지를 그저 처리* 할 뿐이다.
