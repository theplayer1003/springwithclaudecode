## 영역 A — 비동기 처리 (`@Async`)

### A1.

다음 시나리오의 *세 가지 문제* 를 각각 설명하고, 각 문제가 서로 어떻게 다른지 정리하시오.

> *"댓글 작성 시 게시글 작성자에게 이메일 알림을 보낸다. 이메일 발송은 평균 1.5초 걸리는 외부 시스템 호출이다. 이 코드를 `CommentService.createComment()` 안에 직접 호출로
작성했다."*

**답변:**

첫번째 문제는 트랜잭션 문제입니다. createComment 메서드는 하나의 트랜잭션 단위로 묶이는데 이 안에 '이메일 알림을 보낸다' 코드가 들어가면 필연적으로 return 문 전에 작성됩니다.
따라서 createComment() 호출 -> 이메일 알림 보내기 -> 트랜잭션 커밋 의 순서가 되는데 만약 이메일 알림이 나가고 어떤 이유로 트랜잭션이 롤백될 경우 이미 나간 이메일 알림을 돌릴 수가 없는 문제가
생깁니다.

두번째 문제는 이메일이 외부 시스템 호출인 점으로 createComment 메서드의 성능이 외부환경에 종속되는 점입니다. 앞으로 createComment() 메서드는 항상 평균 1.5초의 지연을 겪어야하며
이메일 알림 기능이 늦어질 경우 댓글 작성 기능도 같이 느려지는 문제점이 발생합니다.

세번째 문제는 장애 전파 문제입니다. createComment 의 핵심 동작은 댓글을 작성하는 기능인데 이를 알리는 기능에 장애가 생겨 끝까지 수행되지 못할 수 있습니다.
알림이 어떤 이유로 안 갈지언정 댓글은 작성되는게 애플리케이션의 핵심 기능이기 때문에 이를 분리해야 합니다.

세 문제들의 본질적 차이는 다음과 같습니다.
트랜잭션 문제는 DB 와 외부 시스템 상태가 어긋나는 정합성의 문제입니다.
응답 지연 문제는 사용자 경험 문제 입니다.
장애 전파는 시스템 안정성에 관한 문제입니다.

두번째 문제와 세번째 문제는 비동기, `@Async` 의 도입으로 해결할 수 있지만 첫번째 문제는 비동기만으로 해결되지 않습니다.
별도 스레드에 일을 시킨다고 해도 커밋 이전에 이메일이 발송될 수 있기 때문입니다. 이를 위해 `@TransactionalEventListener` 가 필요합니다.

세번째 장애 전파 문제에 대해 좀 더 알아보겠습니다.
구체적인 시나리오는 다음과 같습니다.
SMTP 서버 응답이 평상 시 평균 1.5초가 걸린다고 가정합니다.
댓글 요청이 초당 10개가 들어온다고 가정하면 10개의 알림 스레드가 1.5초를 대기합니다. 10 스레드 ~ 20 스레드를 왔다 갔다 하기 때문에
동시 점유 스레드는 약 15개라고 할 수 있습니다. 이런 상황에서는 톰캣 풀 스레드의 디폴트 값이 200개 이기 때문에 여유롭습니다.

하지만 STMP 서버에 장애가 발생해서 응답이 30초로 늘어났다고 가정하면
20초 째에 요청 누적이 200개가 됩니다. 이제 새 요청이 들어와도 처리할 스레드가 없기 때문에 전략에 따라 대기하거나 요청이 거부되어야 합니다.
비동기가 없다면 이 시점에 댓글 작성과 알림 서비스 때문에 이와 무관한 게시글 목록 조회, 로그인, 회원가입 등의 모든 서비스가 멈추게 됩니다.
이런 상황이 장애 전파 입니다.

### A2.

`@Async` 어노테이션의 동작 메커니즘을 *프록시* 와 *스레드 풀* 관점에서 설명하시오.
*"어노테이션 하나 붙였을 뿐인데 어떻게 별도 스레드에서 동작하는가?"* 에 대한 답을 포함할 것.

**답변:**

비동기 구현 시나리오는 다음과 같습니다.
먼저 비동기로 수행되길 바라는 메서드(b) 와 그 메서드를 호출할 메서드(a) 가 있습니다. 둘은 서로 다른 클래스에 존재합니다.(B 와 A)
b 메서드에 `@Async` 어노테이션이 붙으면 스프링이 켜지고 빈을 등록할때 해당 객체를 프록시로 교체하게 됩니다.

프록시란 b 메서드가 구현된 객체 B를 참조하고 있는 Spring 이 만들어준 가짜 객체 B\` 입니다.
겉으로는 B 와 똑같이 b 메서드를 가진거처럼 보이지만 실제로는 b\` 인거죠.

개발자는 B 를 참조하며 b 메서드를 호출한다고 생각하고 코드를 작성하지만 Spring 이 실행되고 나면 이것들이 모두 프록시로 대체 되는겁니다.

이제 코드가 실행되고 a 메서드가 호출됩니다.
a 메서드가 실행되면서 b 메서드 호출 코드를 만납니다. Spring 에 의해 런타임에서는 b\` 이 됩니다.
프록시가 메서드를 호출 받았기 때문에 진짜 b 메서드를 호출하기 전에 Spring 이 준비한 코드를 삽입할 수 있게 됩니다.
여기서 비동기를 위해 TaskExecutor 에게 b 메서드를 넘기게 되고 프록시는 즉시 리턴을 합니다.
A 객체는 해당 일을 비동기로 처리하고 이어서 자신의 작업을 이어나갈 수 있습니다.

`@Async` 메서드는 반환 타입에 제한이 있습니다.
먼저 `void` 반환이 가능합니다. 이는 fire-and-forget 전략으로 호출 측에서 일을 시키고 더 이상 신경 쓰지 않는 형태입니다.
다음은 `Future<T>`, `CompletableFuture<T>` 가 있습니다. 이것들은 호출 뒤 빈 객체를 만들어두고 이후 스레드가 일을 마친 뒤 반환하면
그때 비어있는 객체에 작업 결과를 입력하는 형태입니다.

### A3.

`@Async` 의 함정 — *자기 호출(self-invocation)* 문제가 무엇인지, 왜 발생하는지, 어떻게 해결하는지 설명하시오.

**답변:**

자기 호출 문제는 비동기 메서드가 해당 메서드를 호출하는 같은 클래스에 선언되었을 때 발생합니다.
앞서 A2 에서 프록시의 작동 시나리오를 정리했습니다.
해당 시나리오에 맞춰 생각해보면, 자기 호출은 프록시 안의 진짜 객체(참조)에 호출하는 메서드와 호출 당하는 메서드가 같이 있게 됩니다.
요청 -> 프록시가 요청을 받음 -> a\` 메서드 수행(부가 기능 없이 a 메서드 호출) -> a 메서드 수행 중 this.b 메서드 호출을 만남
-> 프록시의 메서드 호출 없이 곧바로 b 메서드가 실행

이렇게 진행되어 프록시가 가로챌 틈이 없어집니다.

해결책으로 제일 권장되는건 비동기 메서드를 별도의 클래스로 분리하는 것 입니다.
분리하게 되면 분리된 클래스에만 `@Async` 어노테이션이 붙고 이를 프록시가 감싸서 호출이 들어왔을 때 무사히 가로채서 비동기 로직을 수행할 수 있게 됩니다.
권장 되지 않는 방법으로는 자기 자신을 주입 받는 방법도 있습니다. 객체 안에 자기 자신을 한번 더 주입 받으면 주입될 객체가 프록시로 감싸져 있기 때문에
비동기 로직이 작동될 수 있게 됩니다.

### A4.

**스레드 풀** 과 **커넥션 풀** 의 차이를 본질적 측면에서 정리하시오 (재사용 자원의 종류, 풀이 가득 찼을 때의 기본 동작 차이).

**답변:**

스레드 풀은 OS 레벨의 연산자원을 관리하는 풀이고 커넥션 풀은 외부 시스템과의 연결자원을 관리하는 풀 입니다.
스레드 풀의 스레드는 CPU 명령을 실행하고 연산을 수행하는 주체, 스레드가 대기하고 있으며
커넥션 풀은 TCP/IP 연결 설정과 인증 과정을 거쳐 완성된 외부와의 연결선, 커넥션이 대기하고 있습니다.
이것들은 만드는데 자원이 많이 들기 때문에 애플리케이션이 시작 하고 미리 지정된 수를 만들어 재사용하는 전략이 풀(pool) 전략입니다.

둘 다 대기 로직이 있지만 다른 점은 제한 방식입니다.
스레드 풀은 코어 풀 사이즈 만큼 스레드를 미리 만들어두고 작업을 받은 뒤, 모든 코어 스레드가 작업 중일때 다음 작업을 대기 큐에 적재합니다.
대기 큐도 가득 차면 맥스 풀 사이즈까지 추가 스레드를 생성해서 처리합니다.
만약 맥스 풀 사이즈 스레드까지 가득 차고 큐도 가득 차면 거부 정책이 발동됩니다.

커넥션 풀은 가용 커넥션이 있으면 전부 가동합니다.
모두 사용 중일 경우 다음 요청은 일정 시간 대기 합니다.(connection-timeout)
대기 시간 내에 가용이 나오면 빌려주고 시간을 초과하면 예외가 발생합니다.

스레드 풀은 큐의 크기, 공간으로 제한하고 커넥션 풀은 대기 시간으로 제한합니다.
스레드 풀은 거부 정책 선택지가 다양하게 있으며 커넥션 풀은 시간 초과 예외가 일반적입니다.

## 영역 B — 이벤트 발행/구독

### B1.

비동기 처리(`@Async`) 만으로 부족한 두 가지 문제 — *트랜잭션 커밋 시점* 과 *책임 비대화* — 를 각각 설명하시오.

**답변:**

A 섹션에서 다룬 것처럼 비동기 처리는 어떤 작업을 다른 스레드에 위임하고 현재 스레드는 그 다음 일을 이어나가는 방식을 말합니다.
단순히 비동기만 도입한다면 트랜잭션 커밋 시점과 책임 비대화 문제를 유발할 수 있습니다.

먼저 트랜잭션 커밋 시점은 비동기 메서드를 호출하는 메서드가 트랜잭션으로 처리될 때 발생합니다.
트랜잭션이란 해당 메서드를 하나의 작업 뭉치로 보며 모두 성공하거나 모두 실패해야하는 작업으로 취급하는데 여기에 비동기 메서드가 끼어들면
트랜잭션 메서드가 수행되던 스레드와 컨텍스트가 분리된 새로운 스레드에서 수행되기 때문에 비동기 메서드는 트랜잭션과 무관하게 작업이 완료되어 버립니다.
비동기 메서드가 수행되었는데 트랜잭션이 장애 등의 이유로 실패일 경우 결론적으로 비동기 메서드는 잘못된 작업을 수행하는 꼴이 됩니다.
따라서 비동기 메서드는 트랜잭션 메서드에서 호출될 경우 트랜잭션의 성공 여부를 기다릴 필요가 있습니다.
Spring 에서 제공하는 어노테이션을 통해 트랜잭션이 무사히 COMMIT 됨을 기다리고 그 다음 비동기 메서드를 실행할 수 있습니다.

책임 비대화 문제는 비동기 메서드를 호출하는 쪽 클래스의 책임에 관한 문제 입니다.
비동기 메서드는 '어떤 작업이 다른 스레드에 어떤 작업을 시킨다.' 의 형태 입니다.
관심사가 분리되어 나가는 형태이기 때문에 호출하는 클래스 쪽에서는 비동기 처리가 필요해질 수록 주입 받아야하는 외부 객체가 늘어나게 됩니다.
이는 코드의 가독성을 헤치고 호출 클래스이 책임을 불명확하게 만듭니다.
이 문제를 해결하기 위해 이벤트 발행/구독 로직을 도입할 수 있습니다.
호출 클래스는 비동기 처리를 위한 여러 클래스들에게 일일히 메서드를 호출하지 않고 '어떤 사건이 일어났으니 이를 처리해 달라' 는 이벤트를 발행하기만 합니다.
비동기를 위해 호출되던 클래스들은 이 이벤트를 구독하고 있다가 이벤트가 생성되면 이를 받아서 각자의 작업을 처리하는 방식입니다.

### B2.

Spring 이벤트 시스템의 **핵심 객체 4가지** (이벤트, 발행자, 멀티캐스터, 어댑터)의 역할을 설명하시오.
그리고 *학습자가 직접 다루는 객체* 와 *Spring이 내부에서 처리하는 객체* 를 구분하시오.

**답변:**

이벤트는 어떤 사건이 발생했을 때 만들어지는 객체 입니다. 해당 객체에는 사건에 관한 내용들이 담깁니다.
개발자가 이를 직접 작성하여 발행하게 되고 소비자는 이 객체를 통해 필요한 정보를 꺼내 자신의 작업을 수행합니다.

발행자는 이름 그대로 만들어진 이벤트를 이벤트 발행/구독 시스템에 넘기는 역할입니다.
개발자가 이벤트 발행이 필요한 메서드에 publishEvent 하게 되면 Spring 이 이를 받아 리스너에게 이벤트를 넘겨줍니다.

멀티캐스터는 Spring 이 처음 실행될 때 Listener를 읽어와 관리하는 라우터 입니다.
Listener 란 발행된 이벤트를 받아 처리하는 객체의 보편적 명칭입니다.
발행자가 이벤트를 생성해 Spring 시스템에 알리면 스프링 내부에서 미리 관리하던 목록을 돌며 해당 이벤트를 해결할 수 있는 리스너를 찾습니다.
`@EventListener` 를 달고 있는 클래스가 리스너로 등록되며 현재 프로젝트에서는 이 어노테이션을 포함한 `@TransactionalEventListener` 가 이를 포함하고 있습니다.
이제 멀티캐스터가 발견된 리스너에게 이 이벤트를 넘겨줍니다. 리스너는 이제 이벤트를 받아 필요한 로직을 수행합니다.
현재 프로젝트에서는 글을 게시한 사람과 댓글을 작성한 사람이 같다면 이벤트를 발행할 필요가 없기 때문에 이를 확인하고 최종적으로 RabbitMQ 에 이벤트를 발행하게 됩니다.

어댑터는 이벤트 리스너를 감싸는 프록시 입니다.
앞서 멀티캐스터가 Listener 를 관리한다고 했습니다. 좀 더 정확히 멀티 캐스터는 자바 인터페이스 규격인 `ApplicationListener` 만 관리할 줄 압니다.
본래 어댑터가 없던 때에는 개발자들은 `ApplicationListener` 인터페이스 규격에 맞춰서 코드를 작성해야했습니다.
Spring 종속적이고 불편한 코드를 개선하기 위해 어댑터가 등장했고
이제 개발자는 작성하고 싶은 코드를 작성한 뒤 어노테이션을 붙여주면(`@EventListener` 등) Spring 에서 이를 `ApplicationListener` 프록시로 감싸
멀티캐스터가 이를 받아 처리할 수 있게 됩니다.

최종적으로 개발자가 직접 다루는 객체는 이벤트와 발행자 뿐입니다. 나머지 요소는 Spring 에서 자동으로 처리해주게 됩니다.

### B3.

`@EventListener` 가 다른 어노테이션(`@Async`, `@Transactional`, `@Cacheable`) 과 다른 점은 무엇인가?

**답변:**

뒤의 세 어노테이션은 모두 프록시 이지만 `@EventListener` 는 어댑터 패턴이며 그 동작 목적에 약간의 차이가 있습니다.

뒤의 세 어노테이션들은 모두 어떤 메서드 앞뒤에 임의의 작업을 추가하고 싶을 때 사용하는 어노테이션 입니다.

반면 `@EventListener` 는 어떤 메서드를 외부 시스템과 연결하고 싶을 때 사용하는 어노테이션 입니다.
개발자가 직접 이 메서드를 호출하지 않고 스프링이 `@EventListener`, 어댑터로 감싸진 객체를 멀티캐스터에 등록해뒀다가
이벤트가 발행되면 멀티캐스터가 해당 어댑터를 호출하는 방식으로 동작합니다.

뒤의 세 어노테이션은 여전히 호출자 클래스가 피호출자 클래스, 프록시 객체를 알아야합니다.

호출자와 피호출자 모두 서로를 모른채 Spring 표준 인터페이스만 알면 문제를 해결할 수 있게 됩니다.

### B4.

`@TransactionalEventListener(phase = AFTER_COMMIT)` 이 어떤 메커니즘으로 *트랜잭션 커밋 후* 시점에 호출되는지 설명하시오. (
`TransactionSynchronization` 의 역할 포함)

**답변:**

`TransactionSynchronization` 이 바로 `@TransactionalEventListener`, 리스너가 동작을 트랜잭션 커밋 후로 미루는 매커니즘입니다.
`TransactionSynchronization` 은 트랜잭션 관리자가 트랜잭션 생명주기 시점에 따라 호출할 콜백들을 관리하는 동기화 슬롯에 등록되는 콜백 객체입니다.

```java
public interface TransactionSynchronization {
    void beforeCommit(boolean readOnly);

    void afterCommit();

    void beforeCompletion();

    void afterCompletion(int status);
}
```

각각 커밋 직전, 커밋 성공 이후, 커밋이나 롤백 완료 직전, 커밋이나 롤백 완료 직후에 해당하는 메서드로,
트랜잭션 관리자가 자기 생애주기 중에 적절한 타이밍에 외부 코드를 끼워넣을 수 있는 hook 를 제공하게 됩니다.

실제 동작 수행은 다음과 같습니다.
트랜잭션 메서드가 호출되고, publishEvent() 메서드로 이벤트를 발행하게 됩니다.
멀티캐스터가 이를 받아 알맞은 리스너를 찾습니다.
이때 `@TransactionalEventListener` 로 등록된 리스너가 발견됩니다.
만약 `@EventListenr` 였다면 작업을 즉시 수행했겠지만 이 경우엔 네가지 콜백 시점 중 하나에 등록되고 해당 시점이 되면(우리 코드에서는 afterCommit())
콜백을 실행하게 됩니다.
이제 리스너 메서드가 호출되어 이벤트가 메시지 큐로 발행됩니다.

`@TransactionalEventListener` 에 `fallbackExecution = true` 옵션을 줄 수 있습니다. 이 경우 활성 트랜잭션이 없을 때 즉시 메서드를 실행하게 됩니다.
디폴트 값이 false 기 때문에 활성 트랜잭션이 없으면 리스너 메서드도 실행이 되지 않습니다.
이 부분은 디버깅할 때 리스너가 호출 안되는 원인으로 주의할 필요가 있습니다.

### B5.

`@Async`(시간 분리) 와 이벤트 발행(결합 분리)이 *서로 다른 축* 의 도구라는 의미를 설명하시오.
두 도구를 함께 쓰는 본 학습의 흐름을 예로 들 것.

**답변:**

`@Async` 비동기의 핵심 관점은 시간의 분리 입니다.
메인 스레드가 일을 진행하다가 비동기 메서드를 호출하게 되면 해당 작업을 본인이 수행하는게 아닌 새로운 스레드에게 위임하고
자신은 하던 일을 마저하게 됩니다. 이것이 시간 분리의 효과 입니다.

이벤트 발행의 핵심 관점은 결합의 분리 입니다.
비동기의 경우 메인 스레드의 어떤 객체는 반드시 비동기 메서드를 가진 다른 객체를 알고 있어야 합니다. 이를 의존성이 있다고 합니다.
이 구조를 이벤트 발행으로 처리하게 되면 어떤 객체는 다른 객체를 알지 못 해도 일을 처리할 수 있게 됩니다.
두 객체 간의 의존성이 제거되고 Spring 이 대신 이벤트를 중개해줘 작업을 진행할 수 있게 합니다. 이를 통해 확장을 쉽게 만듭니다.

비동기와 이벤트 발행 구조의 관점에서 현재 작성된 애플리케이션 구조를 정리하면 다음과 같습니다.
먼저 하나의 애플리케이션에 코드가 작성되어 있지만 논리적으로는 A, B 애플리케이션이 있다고 가정합니다.
패키지 consumer 에 작성된 코드들은 B 애플리케이션 코드라고 가정하고 그외 코드들은 A 애플리케이션에서 수행된다고 가정합니다.
A 애플리케이션에서 createComment() 요청을 처리하기 시작합니다.
해당 메서드는 댓글을 생성하고 DB 에 저장을 요청하며 이벤트를 생성해 Spring 에 전달합니다.
Spring 의 이벤트멀티캐스터는 알맞은 Listener(CommentNotificationListener) 를 호출해 일을 처리하도록 합니다.
이 지점이 이벤트 발행의 결합의 분리 입니다.
댓글을 생성하는 객체는 이벤트를 발행하기만 할뿐 이 이벤트를 처리하는 Listener 의 존재를 전혀 모릅니다.
거꾸로 Listener 또한 이 이벤트를 만들어낸 CommentService 의 존재를 모릅니다.
이것이 결합의 분리, 의존성의 제거 입니다. 두 객체는 모두 Spring 에만 의존하며 Spring 이 이벤트를 중개해줘 작업을 이어나가게 합니다.

이제 Listener 가 이벤트를 받아 RabbitMQ, 네트워크를 통해 외부 시스템 메시지 큐에 이벤트를 등록합니다.
B 애플리케이션은 이를 구독하고 있습니다. @RabbitListener 가 선언된 Consumer 클래스들이 그 객체입니다.
B 애플리케이션은 등록된 이벤트를 가져와 처리하기 시작하고 이벤트를 Service 객체에 넘겨 로직을 수행합니다. 해당 로직은 @Async 로 작성되어 있습니다.
B 애플리케이션의 Consumer 를 실행하던 스레드는 해당 작업을 새로운 스레드에게 위임하고 자신은 다음 코드를 수행하러 넘어갑니다.
이 지점이 비동기의 시간 분리 지점입니다.
B 애플리케이션 안에서 Consumer 와 Service 객체는 각자 동시에 자신이 할 일을 처리해나가게 됩니다.

추가로, 이벤트 구조를 도입하면서 '메시지 큐' 도 같이 도입했습니다. (둘은 반드시 세트가 아닙니다.)
메시지 큐를 도입해서 얻은 효과는 해당 이벤트를 처리할 또 하나의 애플리케이션 도입입니다.
메시지 큐는 프로세스의 분리를 의미합니다. 이제 두 애플리케이션이 협력해 이벤트 관련 작업을 처리하기 때문에 하나의 JVM 위에서 같이 작업하지 않게 되어
장애 격리와 독립 배포가 가능해집니다.

학습용 코드를 작성하면서 편의상 실제 애플리케이션 분리 없이 논리적으로 두 개의 애플리케이션이 있다고 가정했고,
Phase07의 여러 주제 중 비동기와 메시지 큐 주제가 겹쳐 마치 메시지 큐 구조를 위해 비동기가 필요한 것처럼 보일 수 있습니다.
하지만 B 애플리케이션에서 이벤트의 처리를 비동기로 한 것은 A 애플리케이션과는 관계가 없습니다.
B 애플리케이션의 내부적으로 어떤 효율적인 처리를 위해 비동기를 도입했다고 봐야합니다.

## 영역 C — 메시지 큐 (RabbitMQ)

### C1.

이벤트 발행/구독(소주제 2) 까지 만든 흐름의 한계 — *같은 JVM 안의 한계* — 가 무엇인지, 메시지 큐가 그것을 어떻게 해결하는지 설명하시오.

**답변:**

이벤트 발행/구독 시스템은 하나의 애플리케이션에서 이루어지는 작업입니다.
CommentService 가 다른 객체에 의존하지 않고 Spring 에게 이벤트를 발행하기만 하면
Spring 은 해당 이벤트를 처리할 수 있는 Listener 를 찾아 이벤트를 넘겨주고 처리하도록 합니다. Listener 역시 CommentService 에 대한 의존이 없습니다.
이를 통해 확장성의 이점을 얻을 수 있습니다.

하지만 아직 하나의 애플리케이션 위에서, 즉 같은 JVM 위에서 처리되는 작업이라는 한계가 있습니다.

같은 JVM 위에 두 가지 작업을 처리할 경우 발생하는 한계는 다음과 같습니다.
첫번째로 장애 격리가 불가능한 점 입니다. 해당 서버가 죽으면 모든 기능이 정지된다는걸 의미합니다.

두번째로 자원의 격리가 불가능한 점 입니다. 하나의 서버, JVM 위에서 작동하기 때문에 CPU, 메모리, GC, 스레드 풀 등을 모두 공유합니다.
만약 알림 서비스가 무거운 작업이라면 자원 점유율이 높아지고 이는 댓글 서비스에도 영향을 주게 됩니다.

세번째로 독립 배포가 불가능한 점 입니다. 만약 추후 알림 서비스를 수정해야한다면 수정 없이 그대로인 댓글 서비스도 같이 재배포되어야 합니다.
일부분의 수정 재배포 때문에 서비스 전체의 가용성이 줄어들게 됩니다.

네번째로 독립 스케일링 불가 입니다. 서비스가 지속되면서 알림 서비스에 대한 수요가(처리량) 늘어도 알림에 대한 자원 증설이 불가능합니다.
하나의 애플리케이션이기 때문에 전체 자원을 늘려야만 합니다.

다섯번째로 기술 통일이 강제됩니다. 하나의 애플리케이션에 묶여 있기 때문에 Java 로 작성되었으면 계속 Java 로 작성되어야 합니다.
알림 등의 기능을 Python 으로 구현하고 싶어도 분리가 불가능해집니다.

이를 개선하고자 도입하는 구조가 메시지 큐 입니다.
메시지 큐는 발행된 이벤트를 네트워크를 통해 받아 보관해뒀다 구독하는 다른 애플리케이션에게 전달하는 역할을 합니다.
따라서, 메시지 큐를 통해 CommentService 와 Listener 의 작업을 별개의 애플리케이션으로 분리해서 진행할 수 있게 됩니다.
하나의 JVM 위에 같이 돌아가는 애플리케이션이 아닌 완전히 격리된 두 개의 JVM 위 애플리케이션으로 나뉘게 되면
기존의 이벤트 발행/구독 시스템이 가진 장애 문제를 격리된 환경으로 원천 차단할 수 있게 됩니다.

### C2.

HTTP 직접 호출 대비 메시지 큐의 장점 5가지를 정리하시오
(응답 대기, 가용성 결합, 처리 능력 결합, 메시지 영속성, 시스템 격리).

**답변:**

먼저 HTTP 호출에 대해 정리해보겠습니다.
HTTP 직접 호출은 두 애플리케이션 A, B 간의 제일 단순한 연결 방법으로 서로 직접 통신하는 전략 입니다.
응답을 대기해야 하며, 양쪽 시스템 모두 정상 작동해야 동작하고, 호출된 즉시 처리를 해야하며, 호출 실패 시 메시지는 유실되고, 한쪽의 장애가 다른 쪽으로 전파 됩니다.

이런 한계로 HTTP 를 이용한 직접 통신은 장애를 유발할 수 있습니다. 시나리오는 다음과 같습니다.
A 애플리케이션과 B 애플리케이션이 직접 통신한다고 하면
B가 다운된 경우 A는 B와의 통신을 시도하고 타임아웃까지 대기해야합니다. 그 시간 동안 A 스레드는 점유되어 자원이 낭비됩니다.
B가 다운은 아니어도 평소보다 느려진다고 하면 A의 스레드는 그 느려진 시간 동안 점유되어 성능 저하가 일어납니다.

하지만 메시지 큐를 도입하여 이 문제를 해결할 수 있습니다.
A, B 애플리케이션 사이에 메시지 큐가 끼어들어
A 애플리케이션이 B 애플리케이션의 응답을 대기할 필요 없이 메시지 큐에 이벤트를 넘기고 즉시 리턴 받습니다.
A 애플리케이션은 B 애플리케이션이 다운되도 메시지 큐에 이벤트를 넘기만 하면 됩니다.
B 애플리케이션은 호출 즉시 처리할 필요 없이 자신의 수용 용량 만큼 메시지 큐에서 이벤트를 꺼내 처리하면 됩니다.
A 애플리케이션이 메시지 큐에 이벤트를 발행하기만 하면 호출 실패 없이 이벤트는 메시지 큐가 관리해줍니다.
A 애플리케이션과 B 애플리케이션 사이에 메시지 큐가 존재하기 때문에 서로의 장애 전파가 일어나지 않습니다.

즉, 메시지 큐는 영속성 책임을 전문으로하는 인프라 입니다. 얼핏 A 와 B 간의 통신을 메시지 큐 시스템이 대신할 뿐 아닌가 싶지만
애플리케이션 끼리 직접 통신하는 것 보다 전문 인프라에 일을 맡기는 것이 직접 통신하는 것보다 훨씬 신뢰성을 높일 수 있습니다.
메시지 큐 시스템은 신뢰성을 높이기 위해 클러스터링, 복제 등으로 가용성을 더 끌어올립니다.

하지만 메시지 큐가 모든 부분에서 만능은 아닙니다. 만약 DB 커밋 후 이벤트 발행을 호출하려고 하는 순간에 A 애플리케이션이 죽으면
해당 이벤트는 DB 상으로는 존재하지만 알림은 안 나간 엔티티가 됩니다.
즉, DB 작업과 메시지 발행을 트랜잭션으로, 하나의 원자적인 작업 단위로 묶을 수가 없습니다.
이를 Dual Write Problem, 이중 쓰기 문제라고 합니다.
해결 패턴으로는 Outbox pattern 이 있습니다.
간단하게, DB 트랜잭션이 시작되고 댓글을 저장한 후 DB 의 outbox 테이블에 발행할 메시지와 함께 저장한 뒤
트랜잭션을 커밋하는 겁니다. 이후 별도 스레드에서 outbox 테이블을 폴링해서 미발행 메시지를 메시지 큐로 발행하고 성공하면 outbox 에 표시하는 식 입니다.

### C3.

RabbitMQ의 **핵심 구성요소** 5가지 (Producer, Exchange, Binding, Queue, Consumer)의 역할을 설명하시오.
*Producer가 Queue에 직접 보내지 않는다는 점* 포함.

**답변:**

먼저 시스템 구성요소를 A 애플리케이션, B 애플리케이션, 메시지 큐 시스템으로 가정하겠습니다.
A 애플리케이션 <--- 네트워크 ---> 메시지 큐 시스템 <--- 네트워크 ---> B 애플리케이션 의 구조 입니다.

여기서 Exchange, Binding, Queue 는 모두 메시지 큐 시스템 내부의 객체를 뜻합니다.
A 애플리케이션이 Producer 가 되고 B 애플리케이션이 Consumer 인 시나리오 입니다.

Producer 는 이벤트를 발행하는 주체입니다.
A 애플리케이션은 요청을 처리하면서 이벤트를 생성하고, 발행 해줄 클래스(Listener)를 호출하게 됩니다.
생성된 이벤트는 Spring 을 거쳐 Listener 가 받아 외부로 발행하게 됩니다.
A 애플리케이션의 Config 에 보통 이벤트를 수신하게 할 Exchange 가 정의되어 있습니다.

이벤트는 메시지 큐 시스템에 들어와 Exchange 객체에 먼저 저장됩니다.
Exchange 는 라우팅 객체로 받은 메시지를 Binding 규칙에 따라 적절한 Queue 에 배치하는 역할을 합니다.
덕분에 Producer 는 몇 개의 큐가 있는지, 이름은 무엇이고 어느 큐에 이벤트를 발행할지 신경 쓰지 않습니다. 그저 이벤트를 발행하기만 하면
분류는 메시지 큐 시스템의 Exchange 가 알아서 해줍니다.

Binding 은 Exchange 와 Queue 를 매핑하는 연결 규칙 입니다. 이 Exchange의 메시지가 어느 Queue 에 배치될지 결정합니다.

Queue 는 Binding 규칙을 통해 메시지를 전달 전 보관하는 대기열 객체입니다. FIFO 구조로 처리되며 여기서 Consumer 의 호출에 따라
순차적으로 전달되어 처리됩니다.

Consumer는 B 애플리케이션으로, Queue 에 연결되어 메시지를 꺼내와 필요한 작업을 수행합니다.

현재 작성된 프로젝트를 기반으로 몇 가지 관점을 살펴보겠습니다.

먼저 Binding 이 정의 되는 위치 입니다. 작성된 프로젝트에서 Binding 규칙은 Consumer 쪽에 정의되어 있습니다.
Producer 측에서는 Exchange 정보만 가지고 있고 그곳에 메시지를 배치해달라고 합니다.
Consumer 측은 Config 파일에 Exchange 와 Queue 를 지정해 메시지를 배치해달라고 합니다.
이렇게 작성된 이유는 이벤트를 가져와야 할 Consumer 가 자기가 받을 큐를 책임지는게 작업 흐름상 자연스럽기 때문입니다.
이 방식에서 Producer 는 Consumer 가 늘어나도 변경될 점이 없습니다. Consumer 측만이 늘어날 때마다 자기 큐를 가져오도록 설정하면 됩니다.

하지만 이 방식만 가능한건 아닙니다.
Producer 측에서 Binding 까지 지정할 수 있습니다. Config에 Exchange/Queue/Binding 을 모두 정의할 수 있으며
이는 Producer 가 인프라까지 관리하는 성격을 띄게 됩니다. 이 경우 A 가 먼저 떠야 인프라가 구성됩니다.
Consumer 는 Queue 만 지정해 처리하고자 하는 메시지를 받아올 수 있습니다.

Producer 와 Consumer 양쪽에 모두 선언할 수도 있습니다. 멱등성을 구현했기 때문에 충돌 없이 메시지를 받을 수 있으며
이 경우 양쪽이 같은 이름을 쓰도록 보장하기 위해 명세 문서, 공유 라이브러리, 인프라 코드(Terraform 등), 환경 변수, 설정 중앙화 등의 방법을
활용합니다.(만약 모종의 이유로 이름이 어긋나면 에러가 나지 않고 그냥 메시지가 도달하지 않게 됩니다. 분산 시스템 디버깅 시 유의해야합니다.)

규칙을 어느 측도 아닌 외부 도구로 선언할 수도 있습니다. Terraform, RabbitMQ 관리 UI, 스크립트 등으로 별도 관리도 가능합니다.
인프라 코드와 비즈니스 코드가 분리되는 효과를 얻을 수 있습니다.

다음 관점은 메시지를 누구 주도로 전달하는가 입니다.
메시지 큐 시스템이 Consumer 에게 보내줄 수 도 있고, Consumer 가 메시지 큐 시스템에서 직접 빼올 수도 있습니다.
전자를 push 방식, 후자를 pull(poll) 방식이라고 합니다.

push 방식은 메시지를 보유하고 있는 메시지 큐 시스템이 주도적으로 보내기 때문에 지연이 없습니다.
pull 방식을 채택한다면 n초 마다, 폴링 주기만큼 가져오기를 시도해야하기 때문에 실제 메시지가 적재되고 나가기까지 지연이 발생합니다.

push 방식이 자원 효율이 더 좋습니다. 메시지를 보유한 메시지 큐가 메시지가 있을 때만 내보내주기 때문입니다.
pull 의 경우 메시지가 없을 수도 있지만 지속적으로 체크해야합니다.

push 방식은 필요한 트래픽만 발생시킵니다. pull 방식은 의미 없는 요청이 다수 발생할 수 있습니다.

이런 특성 때문에 Push 가 실시간성과 자원 효율 모두 우수한 방식입니다.

RabbitMQ 에서도 둘 다 구현 가능하지만 Push 방식이 디폴트 값입니다. Consumer 가 시작되고 RabbitMQ 에 구독을 등록하면,
즉 queue 가 지정되면 RabbitMQ 는 Consumer 와의 TCP 연결을 유지합니다.
이후 큐에 메시지가 들어오면 RabbitMQ 가 알아서 Consumer 에게 메시지를 전송해줍니다. Consumer 는 이를 받아 즉시 처리하고 ack 응답을 합니다.
basic.get 명령으로 pull 을 구현할 수 있지만 거의 사용 하지 않습니다. 일회성 디버깅 정도나 테스트 시나리오 정도로 활용됩니다.

Kafka 는 정반대로 pull 방식이 기본입니다. Kafka 는 디스크에 데이터를 순차적으로 기록해두고 보관합니다.
이후 B 애플리케이션이 스스로 처리 가능한 속도 만큼 가져가도록 하며 알아서 가져가는 형태기 때문에 "한번에 100 개 씩" 등의
배치 처리도 스스로 결정할 수 있습니다. 이런 경우 네트워크 비용을 줄이는데 효과적입니다.

### C4.

Exchange의 **4가지 타입** (Direct, Fanout, Topic, Headers)을 비교하시오.
**Fanout vs Topic** 의 본질적 차이를 *메시지 종류* 관점에서 설명할 것.

**답변:**

Exchange, 메시지를 Producer 에게 받아 Queue 에 Binding 할때는 4가지 전략이 있습니다.
첫번째는 Direct 입니다.
이름 그대로 Exchange 와 Queue 의 1대1 매칭입니다.
이 매칭을 위해 Routing Key 를 사용합니다. Producer 와 Consumer 가 지정한 이 Routing Key 가 정확히 일치하면 라우팅되는 방식입니다.

두번째는 Fanout 입니다.
Fanout 은 Exchange 에서 바인딩 된 모든 큐로 브로드캐스트 하는 전략입니다.
Exchange 를 구독하는 Queue 를 선언하는 식으로 정의되어 등록된 모든 큐에 메시지를 적재하는 방식입니다.
알림 이벤트가 발생했다고 하면 이메일 큐, SMS 큐, 로그 큐 등이 이를 바인딩해 메시지를 받아옵니다.
이런 형태의 하나의 이벤트에 여러 처리가 필요할때 사용됩니다.

세번째는 Topic 입니다.
Topic 은 Routing Key 에 패턴을 짜서 매칭시켜 라우팅 합니다.
`user.*.created` 같은 방식으로 바인딩을 하면 패턴에 맞는 모든 큐가 Exchange 에서 메시지를 가져옵니다.
얼핏 Fanout 과 비슷해보이고 실제로 똑같은 일을 두 방식으로 구현할 수 있지만 둘은 사용 의도가 다릅니다.
Fanout 은 하나의 이벤트를 여러 컨슈머가 관심을 가질 때 사용하며,
Topic 은 여러 이벤트 중 각 컨슈머 일부만이 관심을 가질 때 사용합니다.

주문 처리 시스템의 마이크로서비스 이벤트 라우팅을 예시로 알아보겠습니다.
Exchange "events.exchange" 가 있다고 가정해보겠습니다.
이 exchange 는 다양한 이벤트를 받아들입니다.
order.created
order.cancelled
order.paid
payment.failed
payment.succeeded
delivery.started
delivery.completed
user.signup
user.deleted
...

이제 Topic 방식으로 처리하기 위해 Routing key 패턴을 작성합니다.
[재고팀 큐] -> "order.*"        모든 주문 이벤트만 수신
[결제팀 큐] -> "payment.*"      결제 관련만 수신
[알림팀 큐] -> "*.completed"    완료성 이벤트만 수신
[감사 로그 큐] -> "#"            모든 이벤트 수신
[환영메일 큐] -> "user.signup"   회원가입 이벤트만 수신
[실패 알림 큐] -> "*.failed"     모든 실패 이벤트 수신

이런 식으로 하나의 Exchange 에 들어오는 여러 종류의 이벤트를 분류할 필요가 있을 때 Topic 구조가 적합합니다.
큐들은 자신이 관심있는 패턴의 메시지만 필터링해서 받을 수 있게 됩니다.


마지막은 Headers 입니다.
메세지 헤더 값 기반으로 아주 드물게 사용합니다.

### C5.

메시지 큐에서 **JSON 직렬화를 권장하는 이유** 두 가지를 정리하시오 (자바 ObjectInputStream의 한계 포함).

**답변:**

AMQP는 Advanced message Queuing Protocol 기반의 메시징 솔루션을 쉽게 사용할 수 있도록 지원하는 오픈소스 프레임워크 입니다.
자바 개발자가 비동기 메시지 기반의 시스템을 구축할 때 필요한 템플릿과 추상화 레이어를 제공하여 복잡한 인프라 연동 코드를 최소화 시켜줍니다.
AMQP 를 구현한 Spring AMQP 의 기본 MessageConverter 인 SimpleMessageConverter 가 Serializable 객체에 대해 기본적으로 사용하는게
Java 직렬화 객체인 ObjectInputStream, ObjectOutputStream 입니다.

Java 에서 기본적으로 지원하는 직렬화 방식이지만 사실 보안상 취약점이 있습니다.
해당 모듈은 역직렬화 시 데이터가 어떤 클래스인지 검증하지 않고 곧바로 코드를 실행시킵니다.
이러한 특징 때문에 원격 코드 실행에 취약합니다. 악의적인 코드가 섞인 스트림을 보내 원격에서 코드를 실행시킬 수 있습니다.
따라서 메시지 큐 구조를 도입할때 데이터의 전송은 JSON 직렬화를 사용합니다.
ObjectInputStream 은 객체의 데이터 뿐만 아니라 클래스의 구조 정보와 역직렬화 과정에서 실행되어야할 내부 메서드 호출 경로까지
바이트 배열이 담깁니다. 공격자는 이 것들을 조작해 원격 코드를 실행합니다.
하지만 JSON 은 이런 것들, 코드가 없는 순수 데이터 덩어리입니다. 오직 key, value 쌍만 텍스트로 존재하며
문자열 자체로는 서버에서 실행될 명령어나 로직을 가질 수 없습니다.
또 JSON 의 수신 측은 서버에 이미 안전하게 정의되어 있는 특정 DTO, 클래스 틀을 정의해두며 JSON 파서로 데이터를 읽어
이 클래스에 값을 채워 넣기만 합니다. 수신 측이 정의해둔 클래스기 때문에 악성 코드가 포함될 수 없습니다.
마지막으로 JSON 은 텍스트 형태라 검증이 매우 쉽습니다. 반면 ObjectInputStream, 자바 바이트 배열은 그 안에 어떤 코드가 숨겨져있는지
중간에 파악하기가 불가능에 가깝습니다.

이러한 이유 외에도 ObjectInputStream 은 Java 에 종속된다는 한계점이 있습니다. JSON 은 현 시대에서 정보 운반의 표준으로
다른 기술 스택에도 호환되는 장점을 가집니다.

### C6.

**RabbitMQ vs Kafka** 의 본질적 차이를 한 줄로 표현하시오.
그리고 본 학습 시나리오에 RabbitMQ를 선택한 이유를 설명하시오.

**답변:**

두 시스템은 메시지 큐 구조를 구현하는데 있어서 공통점이 있지만 기본적으로 추구하는 바가 다릅니다.
RabbitMQ 는 메시지를 전파하는 브로커가 본질이며 Kafka 는 분산 로그의 성질을 가지고 있습니다.
RabbitMQ 에서는 생산자가 메시지를 보내면 메시지 큐에 메시지가 담겨져 있고 메시지 큐에서 소비자에게 메시지를 전달하고 큐에서 삭제됩니다.

Kafka 는 생산자가 메시지를 보내면 카프카의 토픽이란 곳에 저장되어 보관됩니다.
메시지는 Topic, 주제 별로 구분되며 알맞은 Topic 속에 나뉘어 있는 여러 파티션 중 하나에 저장됩니다.
RabbitMQ 와 다르게 이 메시지는 카프카에 계속 남아 있습니다.

이제 컨슈머는 카프카라는 장부가 가지고 있는 메시지를 읽어서 처리해야합니다.
여기에 컨슈머 그룹이란 개념이 있습니다. 컨슈머 그룹이란 카프카에서 데이터를 나눠 처리하기 위해 여러 개의 컨슈머를 하나의 팀으로 묶은 단위입니다.
컨슈머 그룹을 통해 카프카에 쌓여있는 대량의 데이터를 빠르게 병렬 처리하고 컨슈머 하나가 멈춰도 다른 컨슈머가 해당 토픽을 계속 처리할 수 있어 가용성을 확보하게 됩니다. 
컨슈머 그룹 배치는 기본적으로 하나의 파티션에 하나의 컨슈머 배치 입니다.
제일 이상적인 형태로 3개의 파티션이 있다면 3개의 컨슈머가 각각 배치되어 작업을 처리합니다.
만약 파티션 수에 비해 컨슈머(스레드)가 부족하면 일부 스레드가 여러 파티션을 담당할 수 도 있습니다. 이때 성능저하가 발생합니다.
반대로 파티션보다 스레드가 많다면 남는 스레드는 대기 상태가 됩니다. 카프카의 규칙으로 한 파티션에 두개의 스레드가 작업을 할 수 없습니다.

따라서 현재 프로젝트의 요구사항인 '댓글이 작성되면 이메일 등으로 알림을 보낸다' 는 한번 수행되고 폐기되도 좋은 성격의 메시지기 때문에
Kafka 보단 RabbitMQ 시스템이 더 적합합니다.

Kafka 시스템은 영속성이 핵심입니다.
메시지를 Kafka 가 계속 보관하기 때문에 컨슈머는 필요에 따라 offset 을 과거로 돌려 메시지를 다시 읽고 재처리가 가능합니다.
또 여러 시스템이 같은 이벤트를 다른 목적으로 사용할 때 유리합니다. RabbitMQ 는 이런 요구사항에 대응하기 위해서는 메시지를 복제해
여러 컨슈머한테 전달하는 구조지만 Kafka 는 계속 메시지를 보관하고 있기 때문에 컨슈머가 와서 같은 메시지를 읽어갈 수 있습니다.
마지막으로 영속성을 이용해 배치 처리 작업이 가능해집니다.

## 영역 D — 학습 중 만난 함정과 메타 통찰

### D1.

`@SpyBean` (또는 `@MockitoSpyBean`) + `@Async` 조합의 충돌 사례를 설명하시오.
*근본 원인* 과 *해결책* (또는 우회법) 을 포함할 것.

**답변:**

비동기 메서드의 테스트를 위해 스파이 빈을 도입하려다 충돌이 난 사례입니다.
먼저 비동기 메서드의 작동 시나리오 입니다.
A 스레드의 메서드 수행 -> 비동기 메서드 호출 -> B 스레드가 비동기 메서드를 위임 받아 수행, A 스레드는 다음 코드로 넘어감
문제는 테스트 수행을 위해 비동기 메서드로 수행된 결과를 봐야하는데 이 메서드는 다른 스레드로 넘어가 컨텍스트가 분리되어 관측이 어렵다는 문제가 있습니다.

당시 충돌의 진짜 원인은 *프록시가 두 번 쌓이면서 서로의 메커니즘을 무력화시킨 것* 이었습니다.
CommentNotificationService 에는 `@Async` 가 붙어있어, 스프링이 이를 처리하기 위해 AOP 프록시를 만들어 빈을 감쌌습니다.
이 프록시가 호출을 가로채서 별도 스레드로 위임하는 게 `@Async` 의 동작 방식이었습니다.
그런데 테스트에서 `@SpyBean` 을 추가로 붙이면 Mockito 가 그 위에 CGLIB 기반의 Spy 프록시를 한 겹 더 씌웁니다.
이렇게 프록시가 두 겹이 되면 외부 프록시 (Spy) 가 호출을 먼저 가로채는데,
이 시점에서 Spy 가 `doAnswer` 로 메서드를 동기적으로 실행해버리면 안쪽의 `@Async` 는 동작할 기회 자체를 잃어버립니다.
결과적으로 비동기로 실행되어야 할 메서드가 같은 스레드에서 그대로 실행되어, 비동기 분리가 일어나지 않은 상태로 테스트가 진행된 것입니다.

학습 중 시도한 해결책 흐름을 복기하면,
첫 시도는 CommentNotificationService 자체를 `@SpyBean` 으로 감싸고 `doAnswer` 로 호출을 가로채는 방식이었습니다.
이 방식에서는 `@Async` 가 무력화되어 비동기 동작 자체가 검증 대상이 되지 못했습니다. 
두 번째 시도는 Executor 자체를 `@SpyBean` 으로 감싸서 `execute` 호출을 가로채는 방식이었습니다.
어느 정도 동작했지만 결과가 비결정적이라 가끔은 통과하고 가끔은 실패했습니다.

최종 해결은 `@TestBean` 으로 Executor 빈을 *교체* 하는 방식이었습니다.
Spy 로 감싸지 않고 테스트용 ThreadPoolTaskExecutor 를 새로 만들어 기존 빈을 대체한 겁니다.
그리고 이 테스트용 Executor 에 TaskDecorator 를 등록해서,
비동기 작업이 실행되는 그 순간 안쪽에서 스레드 이름을 캡처하고 CountDownLatch 를 down 시키도록 만들었습니다.
메인 테스트 스레드는 `latch.await` 로 비동기 완료를 대기했고, 작업이 끝나면 캡처된 스레드 이름을 검증할 수 있었습니다.

이 해결책의 핵심은 *프록시를 추가로 쌓지 않고 빈 자체를 교체했다* 는 점입니다. 프록시가 두 겹이 되는 상황을 원천적으로 회피한 셈입니다.
TaskDecorator 가 비동기 작업 내부에서 동작하기 때문에 별도 스레드의 컨텍스트 안에서 관측이 가능했고,
CountDownLatch 가 시간적 동기화를 제공해서 테스트 스레드가 비동기 완료 전에 끝나버리는 문제도 함께 해결되었습니다.

근본 원인을 한 줄로 정리하면 *이미 프록시로 감싸진 빈을 또 다른 프록시로 감싸려 할 때 두 프록시의 인터셉트 메커니즘이 서로를 무력화시킨다* 는 것입니다.
이 충돌을 우회하는 패턴은 빈 자체를 교체하거나 (`@TestBean`), 인터페이스 분리로 프록시 타입을 통일하거나,
비동기 메서드를 별도 클래스로 분리해 단위 테스트로 검증하는 방향이 있습니다.

상세한 메커니즘은 별도로 정리한 QA 문서
(`QA_Mockito_Spy와_Async_충돌.md`, `QA_프록시_JDK_vs_CGLIB.md`, `QA_CountDownLatch.md`) 에 남아 있습니다.

### D2.

Spring 컨텍스트 캐싱이 통합 테스트 격리에 어떻게 영향을 미치는지 설명하시오.
*"테스트 격리는 절대적이지 않다"* 는 통찰의 의미.

**답변:**

먼저 통합 테스트는 단위 테스트와 다르게 모든 스프링 컨텍스트를 실제로 불러와서 코드가 의도대로 동작하는지 점검하는 테스트입니다.
스프링이 불러오는 컨텍스트에는 캐싱에 관련된 컨텍스트도 있기 때문에 통합 테스트가 여러 개 묶여서 수행될 경우 문제가 발생할 수 있습니다.
A 테스트에서 수행했던 작업들이 스프링 컨텍스트에 남아 있어 B 테스트를 하는 도중에 이전 테스트의 자료들이 섞여 기대하던 결과와 다르게 나오는 문제가 있습니다.
이를 해결하는 방법 중 하나로
`@DirtiesContext(classMode = ClassMode.BEFORE_CLASS)`
어노테이션을 통해 이전 테스트와 무관하게 이 테스트를 수행할 때 새롭게 스프링을 부트하도록 할 수 있습니다.

보강하자면, 이 문제의 본질은 *스프링 테스트 프레임워크가 ApplicationContext 자체를 캐싱한다* 는 사실에서 출발합니다.
`@Cacheable` 같은 애플리케이션의 캐시 기능과는 별개의 메커니즘입니다.
통합 테스트에서 ApplicationContext 를 로딩하는 작업은 수십 초가 걸릴 만큼 비용이 큰 작업이라,
매 테스트마다 새로 만드는 건 현실적이지 않습니다. 그래서 스프링 테스트는 같은 설정의 컨텍스트를 캐싱해서 여러 테스트 클래스가 공유하도록 합니다.
`@SpringBootTest`, `@MockitoBean`, `@TestPropertySource` 같은 설정이 같으면 같은 컨텍스트 인스턴스가 재사용되는 식입니다.

여기서 문제가 생기는 이유는 컨텍스트가 공유되면 그 안의 빈 상태도 공유되기 때문입니다.
한 테스트에서 캐시 매니저에 값을 넣으면 다음 테스트에 그대로 남고, ThreadPoolTaskExecutor 의 작업 큐도 그대로 남으며,
static 필드에 넣어둔 CountDownLatch 같은 정적 상태도 다음 테스트로 흘러갑니다.
거기에 더해 데이터베이스나 메시지 큐 같은 외부 시스템의 상태까지 더해지면 테스트 간 격리가 자연스럽게 깨지게 됩니다.

학습 중 직접 겪은 사례가 정확히 이것이었습니다. CommentAsyncIntegrationTest 에서 `static CountDownLatch` 와 `static AtomicReference` 를 사용했고, 다른 테스트와 함께 돌리면 컨텍스트 캐싱과 정적 변수와 RabbitMQ 큐 상태가 얽혀서 비결정적으로 실패했습니다. 단독으로 돌리면 통과했지만 전체 빌드에서는 실패하는 상태였습니다.

`@DirtiesContext` 의 의미는 *"이 테스트가 컨텍스트를 더럽혔으니 폐기하라"* 는 명시적 신호입니다. `classMode = BEFORE_CLASS` 옵션은 이 테스트 클래스가 시작되기 *전* 에 새 컨텍스트를 만들라는 뜻이고, `AFTER_CLASS` 는 *끝난 후* 폐기하라는 뜻입니다. `methodMode` 옵션도 있어 메서드 단위로 더 세밀하게 제어할 수 있지만 그만큼 느려집니다. 학습 중 `BEFORE_CLASS` 를 선택한 이유는 *"이전 테스트들이 남긴 흔적을 일단 다 지우고 시작하자"* 는 의도였습니다. 다만 이것만으로는 RabbitMQ 큐의 메시지까지 다 정리되진 않아서, 결국 테스트를 `@Disabled` 처리한 것이 마지막 결정이었습니다.

이 사례가 가리키는 통찰이 *"테스트 격리는 절대적이지 않다"* 입니다. 단위 테스트는 격리가 비교적 쉽지만, 통합 테스트로 가면 컨텍스트 캐싱과 외부 시스템 상태, 정적 변수 등 격리되지 않는 요소가 빠르게 늘어납니다. *"한 번에 모든 테스트가 통과한다"* 와 *"한 테스트만 따로 돌렸을 때 통과한다"* 가 다르다면, 이는 격리가 깨졌다는 분명한 신호입니다. 이 통찰은 D4 의 *"비결정적 테스트는 무가치하다"* 와 직접 연결됩니다. 격리되지 않은 테스트는 결국 비결정적 동작으로 귀결되기 때문입니다.

### D3.

**메시지 큐의 영속성** 이 왜 *강점이자 함정* 인지 본 학습 중 학습자가 직접 체험한 사례로 설명하시오.

**답변:**

메세지 큐란 애플리케이션에서 발행된 이벤트를 보관해뒀다 적절한 애플리케이션에게 처리를 위임하는 구조 입니다.
이 구조를 통해 얻을 수 있는 점은 애플리케이이션 간의 실행 환경의 분리 입니다. 메시지 큐 없이는 같은 JVM 위에서 수행되는 한계점이 있어
장애 등이 발생하면 같이 영향을 받을 수 밖에 없지만 메시지 큐 구조를 통해 두 애플리케이션을 분리해 원천적으로 차단해 가용성을 올릴 수 있습니다.

단점으로는 영속성 자체가 함정이 될 수 있다는 점이 있습니다.
메시지가 사라지지 않는다는 것은 *잘못된 메시지도 사라지지 않는다* 는 의미이기도 합니다.

학습 중 직접 체험한 사례가 이 함정을 잘 보여줍니다.
처음에 Spring AMQP 의 기본 컨버터 (SimpleMessageConverter) 를 그대로 사용했을 때 자바 직렬화 방식으로 메시지가 큐에 발행되었습니다.
보안상의 이유로 SecurityException 이 발생했고, 이를 해결하기 위해 Jackson2JsonMessageConverter 로 교체했습니다.
컨버터를 바꾸자 새로 발행되는 메시지는 JSON 형식이 되었고 컨슈머도 JSON 으로 역직렬화하도록 설정되었습니다. 
그런데 *그 사이 이미 자바 직렬화로 발행되어 큐에 남아있던 옛 메시지들이 그대로 남아있었습니다*. 
새 JSON 컨슈머가 이 옛 메시지들을 받자 역직렬화에 실패했고, 
RabbitMQ 의 기본 재시도 메커니즘 때문에 같은 메시지를 무한히 재시도하면서 에러가 반복되었습니다. 
결국 RabbitMQ 관리 UI 에서 큐를 수동으로 Purge 해서 옛 메시지를 비워야만 해결되었습니다.

이 경험이 보여주는 것은 *영속성의 양면성* 입니다. 큐가 메시지를 보관해주는 덕분에 컨슈머가 다운되어도 메시지가 안전하지만,
동시에 *처리할 수 없는 형식의 메시지가 들어왔을 때 그 메시지가 무한히 큐에 남아 컨슈머를 막아버리는 위험* 도 함께 생깁니다.
영속성이 좋은 일만 하는 게 아니라 *책임을 지는 만큼 짐도 진다* 는 사실을 직접 본 사례였습니다.

이 함정은 학습 중 만난 특정 사건만의 문제가 아니라 일반화된 패턴이기도 합니다.
이벤트 클래스의 스키마가 진화하면 옛 메시지를 새 코드로 읽지 못하는 상황이 생기고,
버그가 있는 코드로 잘못 발행된 메시지가 큐에 영구히 남는 일도 있으며, 컨슈머가 느리거나 멈추면 큐가 무한히 비대해질 수 있습니다.
실무에서는 이런 문제들에 대응하기 위해 Dead Letter Queue (처리 실패한 메시지를 별도 큐로 격리), 메시지 버저닝 (포맷 변경에 대비한 버전 필드 부여), 
스키마 레지스트리 (메시지 포맷을 중앙에서 관리) 같은 패턴들이 사용됩니다.

학습 중 *"메시지 큐 도입 = 무조건 더 안전"* 이라는 단순한 인식에서 출발해,
실제 사건을 거치며 *"영속성은 책임이고 책임은 짐이기도 하다"* 는 이중성을 인식하게 된 것이 이 항목의 핵심 통찰입니다.

### D4.

다음 학습자 통찰들을 본인 언어로 다시 정리하시오:

- *"결합도 0은 환상, 결합의 위치 옮기기"*
- *"코드 변화 ≠ 본질 변화"*
- *"인터페이스는 미래 확장만이 목적이 아니다"*
- *"비결정적 테스트는 무가치하다"*

**답변:**

**"결합도 0은 환상, 결합의 위치 옮기기"**

비동기, 이벤트 발행, 메시지 큐 같은 도구들을 도입하면 결합이 사라지는 것처럼 보이지만,
실제로 일어나는 일은 *결합의 위치가 옮겨지는 것* 에 가깝습니다.
CommentService 가 EmailService 를 직접 호출하던 강결합은 이벤트 발행 구조로 바꾸면 사라지는 것 같지만,
사실은 양쪽이 *CommentCreatedEvent 라는 새로운 매개체* 에 결합됩니다.
메시지 큐로 가도 마찬가지로 양쪽이 *Exchange/Queue 이름이라는 계약* 에 결합됩니다.
어느 단계에서도 결합 자체가 사라지진 않고 결합의 매개체와 위치가 바뀌는 것뿐입니다.

이 통찰의 가치는 *"분리했으니까 끝"* 이라는 안일한 사고를 막아준다는 점에 있습니다.
도구를 도입할 때 진짜 물어야 할 질문은 *"결합을 어디로 옮길 것인가"*, *"옮긴 결합이 운영과 확장에 어떤 영향을 주는가"* 입니다.
단순히 *"결합을 없앴다"* 가 아니라.

**"코드 변화 ≠ 본질 변화"**

코드 구조가 바뀌어도 본질적 문제는 그대로 남을 수 있습니다. 학습 중 본 가장 명확한 예시가 컨슈머 측 `@Async` 안티 패턴이었습니다.
코드만 보면 *"메시지 큐를 쓰는 안전한 구조"* 처럼 보이지만, ack 타이밍이 어긋나면서 메시지 큐의 안전 약속이 깨지는 본질적 위험이 숨어 있었습니다.
또 학습 코드에 추가된 컨슈머 측 `@Async` 도 코드 흐름상으로는 *"비동기 분리가 추가되어 더 좋아 보이는"* 모습이지만,
실전 의사결정 측면에서는 *"학습용 시연일 뿐 호출자에게 실질적 이득은 없는"* 것이었습니다.

이 통찰은 *"코드를 이렇게 짰으니 이렇게 동작할 것이다"* 는 가정의 위험을 가리킵니다. 코드의 형태와 시스템의 본질을 구분하는 사고가 필요합니다.
*"어떻게 짰는가"* 를 넘어 *"실제로 어떤 보장을 제공하는가"*, *"무엇이 깨질 수 있는가"* 를 묻는 시각이 그것입니다.

**"인터페이스는 미래 확장만이 목적이 아니다"**

인터페이스를 도입할 때 흔히 드는 이유는 *"미래에 다른 구현이 생길 때를 위해"* 입니다.
그러나 학습 중 인터페이스가 현재의 기술적 필요만으로도 충분히 정당화되는 사례를 직접 만났습니다.
CommentNotificationService 를 인터페이스로 분리한 결정의 진짜 이유는 *Mockito Spy 와 @Async 의 프록시 충돌 회피* 였습니다.
JDK Dynamic Proxy 가 동작하려면 인터페이스가 필요한데, 이 요건을 만족시켜야 두 프록시가 같은 방식으로 동작해 충돌이 줄어들기 때문이었습니다.

이 외에도 인터페이스는 테스트 가능성에 직접 기여합니다. 인터페이스가 있으면 mock 객체를 깔끔하게 주입할 수 있고 의존성 격리가 쉬워집니다.
또 *지금 코드를 읽는 사람에게 "이 객체가 무엇을 할 책임이 있는가" 를 명시* 하는 계약의 역할도 합니다.
다른 구현이 영원히 생기지 않더라도, 인터페이스는 이미 현재의 코드에 가치를 더하는 것입니다.

이 통찰은 *"미래 가정으로만 인터페이스를 정당화하는 건 오버 엔지니어링이지만,
현재의 기술적 필요나 가독성 측면에서 인터페이스가 필요하다면 그것만으로 충분히 정당화된다"* 는 균형 잡힌 시각을 줍니다.

**"비결정적 테스트는 무가치하다"**

가끔 통과하고 가끔 실패하는 테스트는 통과해도 안전을 보장하지 않습니다.
학습 중 CommentAsyncIntegrationTest 가 정확히 이 상태가 되었습니다.
다른 테스트와 함께 돌리면 실패, 단독으로 돌리면 성공이라는 비결정적 동작을 보였고,
*"몇 번 돌려서 통과하니까 괜찮다"* 는 유혹이 자연스럽게 따라왔습니다.
그러나 그건 우연한 통과일 뿐, 다음에 실패하지 않는다는 보장은 어디에도 없었습니다.

비결정적 테스트의 진짜 위험은 *진짜 버그가 비결정적 실패에 묻혀서 안 보이게 된다* 는 점입니다.
CI 에서 간헐적 실패가 누적되면 사람들은 *"또 그 테스트 가끔 실패하지"* 라며 무시하기 시작하고,
그러다 진짜 실패도 같이 무시되어 결국 의미 없는 안전망이 됩니다.

학습 중 결국 선택한 것은 `@Disabled` 처리였습니다.
비결정적 테스트를 유지하느니 차라리 비활성화하고 추후 재작성으로 미루는 게 더 정직한 결정이었습니다.
*"테스트는 결정적이어야 의미가 있다"*, *"작은 결정적 테스트가 큰 비결정적 테스트보다 가치 있다"* 는 통찰이 이 결정에서 나왔습니다.

## 영역 E — 통합 사고

### E1.

본 학습의 최종 알림 흐름
(CommentService → Spring 이벤트 → 메시지 큐 → Consumer → @Async → 메일/SMS 발송) 을
단계별로 설명하시오.
각 단계에서 *어떤 분리* 가 일어나는지(시간 분리/결합 분리/프로세스 분리) 포함할 것.

**답변:**

먼저 Producer 역할을 하는 A 애플리케이션의 범위는 CommentService -> Spring 이벤트 까지 입니다.
A 애플리케이션의 CommentService, createComment() 메서드를 통해 시나리오가 시작됩니다.
이 메서드는 특정 게시글에 달릴 댓글을 생성하고 댓글이 달리면 글 작성자에게 알림을 보낼 책임을 가집니다.
댓글을 생성해서 DB 에 반영하는 작업은 트랜잭션으로 묶입니다.
만약 댓글을 생성하고 곧바로 이를 알리는 메서드를 호출하게 되면 아직 완전하게 커밋하지 못 했는데 알림이 나갈 위험이 있습니다.
(@TransactionalEventListener 어노테이션으로 트랜잭션 관리자가 제공하는 네 가지 콜백 시점에 리스너가 작동하도록 해 커밋 이전에 이벤트가
나가는걸 방지할 수 있습니다.)
따라서 이를 해결하기 위해 이벤트 발행 구조를 도입합니다.

이벤트 발행 구조는 CommentService 는 댓글이 생성되었다는 이벤트만 알리고 이벤트의 처리를 다른 객체에게 맡기는 구조 입니다.
Spring 은 이 패턴을 손쉽게 구현할 수 있도록 지원합니다. CommentService 는 이벤트를 작성한 뒤 Spring 에게 publish 하기만 하면 됩니다.
Spring 은 이를 받아 애플리케이션 구동 시 미리 탐색해서 등록해놓은 Listener 들 중 처리 가능한 객체에게 대신 전달해줍니다.
이제 CommentService 와 Listener 는 서로를 의존하지 않지만 함께 이벤트를 처리할 수 있게 되었습니다. 이 부분의 결합의 분리 입니다.

최종적으로 메시지 큐 도입을 통해 시간의 분리도 일어납니다. 이제 A 애플리케이션은 이벤트를 발행하기만 하고 다른 프로그램에 작업 처리를 넘긴 뒤
자신의 작업을 이어서 수행할 수 있게 됩니다.

이벤트 메시지 큐 구조를 위한 외부 시스템으로 RabbitMQ 를 채택했습니다.
비슷한 시스템으로 Kafka 가 있지만 Kafka 는 분산 로그로서의 의의가 더 강하기 때문에 현재 요구사항에 맞는 메세지 브로커, RabbitMQ 를 사용합니다.
메시지 큐 시스템인 RabbitMQ 는 Producer 로 부터 이벤트 메시지를 받습니다. 메시지는 메시지 큐에 적재되며 Consumer 가 등록한 Binding 규칙에 따라
큐에 배치되고 이를 RabbitMQ 가 적절한 Consumer 에게 전송해줍니다.

Consumer -> @Async -> 메일/SMS 발송 부분은 Consumer 역할인 B 애플리케이션에서 일어납니다.
B 애플리케이션이 로드되고 Binding 규칙을 RabbitMQ 에 등록합니다. 이제 규칙에 해당하는 메시지가 발행되어 큐에 적재되면 이를 Consumer 에게 보내줍니다.
현재 프로젝트에서 면밀히 작성되진 않았지만 B 애플리케이션은 다양한 작업을 가지고 있고 메시지 처리를 비동기로 해야할 필요가 있다고 가정합니다.
B 애플리케이션의 Notification 객체가 메시지를 수신하면 이를 처리할 수 있는 Service 를 호출합니다.
호출한 메서드가 동기 메서드였다면 Notification 객체를 실행시키던 스레드는 Service 객체를 불러와 작업을 이어나가야 하지만,
비동기 메서드의 경우 해당 스레드는 스레드 풀에 대기 중이던 새로운 스레드를 불러와 이벤트를 넘기고 작업을 지시한 뒤 자신은 Notification 객체 컨텍스트로 복귀해
작업을 이어나갑니다. 새로 호출된 스레드는 알림을 전송한 뒤 일을 끝마치면 스레드 풀로 돌아가 대기 상태가 됩니다.  
이 부분이 B 애플리케이션의 시간 분리입니다.


이렇게 알림 서비스 시나리오의 흐름이 완성됩니다.
마지막으로 A 애플리케이션과 외부 메시지 큐 시스템 그리고 B 애플리케이션, 이 전체 구조가 프로세스의 분리 입니다.
이 작업을 하나의 애플리케이션에서 처리할 경우 가용성, 확장성 등에 저하가 생기기 때문에 프로세스 자체를 분리해서 외부 시스템이 이어서 처리할 수 있도록 합니다.

### E2.

다음 가상 시나리오에 어떻게 대응할지 본인 판단을 적으시오:

> *"댓글 작성 시 알림 외에 추가로 (a) 게시글 활동 점수 갱신 (b) 사용자 활동 로그 기록 (c) 도배 감지 시스템 트리거 가 필요해졌다. 운영 코드의 어디를 어떻게 변경할 것인가?"*

**답변:**

(a)
게시글 활동 점수 갱신을 위해서는 먼저 엔티티에 점수 값이 포함되어야 합니다.
요구사항을 '게시글이나 댓글 작성 시 활동 점수 부여' 라고 가정해보겠습니다.
먼저 post, comment 에 각각 점수 필드가 생겨야하고 member 엔티티에 자신이 활동한 내역의 합산 점수를 기록하는 필드가 있어야 합니다.
member 엔티티의 합점수 필드는 LAZY 로딩을 하는게 좋겠습니다. 매번 게시글, 댓글 등을 순회하면서 점수를 합하면 자원이 낭비됩니다.
이제 post 나 comment 의 create 시 활동에 대한 점수를 계산하고 필드에 추가합니다.

다음으로 살펴볼 것은 트랜잭션 입니다.

도구 선택 관점에서 점수 갱신의 네 가지 판단 기준을 검토하면, 트랜잭션 정합성 면에서 점수가 잠깐 안 맞아도 큰 문제는 없지만 
*댓글이 작성됐는데 점수가 영원히 갱신되지 않는 상태* 는 피해야 합니다.
강한 정합성이 절대적으로 필요한 건 아니지만 갱신 누락은 막아야 하는 정도의 요구입니다.
처리 시간 면에서는 점수 컬럼 UPDATE 정도라 빠르므로 호출자의 응답 시간을 늦출 요소가 아닙니다.
처리 종류 면에서는 *"댓글 작성 → 점수 갱신"* 한 가지 처리뿐이라 후속 처리가 자주 늘어날 일은 없습니다.
다른 시스템 책임 면에서는 같은 게시판 도메인 안의 처리라 별도 시스템으로 분리할 필요가 없습니다.

따라서 결론은 **CommentService 안에서 PostScoreService.increment() 를 직접 호출하되, 같은 트랜잭션 안에서 처리** 하는 것입니다.
`@Transactional` 메서드 안에서 댓글 저장과 점수 증가를 같이 호출하면 DB 트랜잭션이 두 작업을 원자적으로 묶어주므로 
*"댓글은 저장됐는데 점수는 안 갱신된 상태"* 가 발생할 수 없습니다. 강한 정합성을 자동으로 보장받으면서 추가 인프라 비용도 없습니다.

이벤트 발행 구조는 후속 처리의 종류가 다양해질 때 가치가 있는데 점수 갱신은 한 가지 처리뿐이라 이벤트 매개체만 추가될 뿐 이득이 적습니다.
메시지 큐는 프로세스 분리가 필요할 때 가치가 있는데 점수 갱신은 같은 도메인 내부 처리라 분리할 이유가 없습니다.
메시지 큐 도입은 오히려 정합성을 깨고 운영 복잡도를 키웁니다.
학습한 도구를 다 적용하는 함정에 빠지지 않고 가장 단순한 해결책으로 시작하는 것이 YAGNI 원칙에 맞습니다.

미래에 점수 갱신이 무거운 통계 계산으로 발전하거나 외부 시스템 호출이 필요해지면 그때 `@Async` 또는 이벤트 발행 도입을 검토할 수 있습니다.
지금은 직접 호출로 시작하고 필요해질 때 발전시킵니다.

(b)
사용자 활동 로그 기록의 네 가지 판단 기준을 검토하면, 트랜잭션 정합성 면에서 로그는 부가 데이터라 댓글이 작성됐는데
로그가 잠깐 안 기록되어도 비즈니스가 망가지지 않습니다.
다만 *영구히 기록되지 않으면* 분석이나 감사 목적이 무력화되므로 결과적 정합성은 보장되어야 합니다.
처리 시간 면에서는 단순 INSERT 라면 빠르지만 *대량의 로그가 동시에 발생할 수 있어* 호출자 응답 시간을 늦출 가능성이 있습니다.
비동기로 분리하는 게 유리합니다. 처리 종류 면에서는 사용자 활동 종류가 매우 다양합니다.
가입, 로그인, 댓글 작성, 좋아요, 게시글 작성 등 하나의 로그 시스템이 다양한 이벤트를 받아야 하고 
새로운 활동 유형이 추가될 때마다 자연스럽게 따라가야 합니다. 이벤트 발행 구조의 강점이 발휘되는 영역입니다.
다른 시스템 책임 면에서는 로그가 보통 분석팀이나 감사 시스템이 별도로 관리하는 경우가 많습니다. 
영구 보존, 대량 처리, 분석 도구 연동이 필요할 수 있습니다.

따라서 결론은 **이벤트 발행 + 메시지 큐 도입** 입니다. 다양한 사용자 활동 이벤트를 공통 인터페이스로 발행하고 
별도 로그 시스템이 메시지 큐를 통해 비동기로 처리하는 구조입니다.

```java
@Transactional
public void createComment(...) {
    Comment comment = commentRepository.save(...);
    publisher.publishEvent(new UserActivityEvent(userId, "COMMENT_CREATED", ...));
}

// 별도 로그 시스템의 컨슈머
@RabbitListener(queues = "user.activity.log.queue")
public void onUserActivity(UserActivityEvent event) {
    activityLogRepository.save(...);
}
```

이 선택의 근거는 세 가지입니다. 
첫째, 이벤트 발행 구조가 처리 종류의 확장성을 받쳐줍니다. 새로운 활동 이벤트가 추가될 때 호출자 코드를 수정하지 않아도 됩니다. 
둘째, 메시지 큐가 로그 시스템과의 결합을 끊어줍니다. 로그 시스템이 다운되어도 댓글 작성은 영향받지 않고, 
로그 시스템은 자기 페이스로 메시지를 처리합니다. 
셋째, 영속성이 결과적 정합성을 보장합니다. 메시지 큐가 메시지를 영구 보관하므로 로그 시스템이 일시적으로 죽어도 메시지가 사라지지 않습니다.

만약 별도 분석/감사 시스템 없이 같은 애플리케이션 안에서 로그를 처리할 거라면 이벤트 발행 + 비동기 리스너만으로도 충분합니다.
메시지 큐까지 도입할지는 로그 시스템이 별도 시스템이냐에 달려 있습니다. 
일반적으로 로그/감사는 별도 인프라로 분리하는 게 자연스러우므로 메시지 큐 쪽으로 기우는 게 보통입니다. 
더 큰 규모에서는 Kafka 같은 분산 로그가 자연스러운 선택이 될 수도 있는데, 
영속성을 활용한 재처리와 여러 분석 시스템이 같은 이벤트를 다른 목적으로 사용하는 패턴이 잘 맞기 때문입니다.

(c)
도배 감지 시스템 트리거의 네 가지 판단 기준을 검토하면,
트랜잭션 정합성 면에서 도배 감지는 후처리적 분석이라 댓글 작성과 정합성으로 묶일 필요가 없습니다.
댓글이 일단 작성되고 별도로 분석되는 게 자연스럽습니다. 다만 *댓글이 누락 없이 모두 도배 감지에 도달* 해야 하므로 영속성 보장이 중요합니다.
처리 시간 면에서는 도배 감지가 패턴 분석, 통계 계산, 외부 ML 시스템 호출 등을 동반할 수 있어 시간이 오래 걸리는 처리입니다.
호출자의 응답에 영향 없이 비동기로 분리되어야 합니다. 
처리 종류 면에서는 도배 감지 외에도 스팸 감지, 욕설 감지, 광고 감지 등 유사 시스템이 늘어날 수 있습니다. 
댓글 작성 이벤트에 여러 분석 시스템이 붙는 패턴이라 이벤트 발행 구조가 잘 맞습니다. 
다른 시스템 책임 면에서는 도배 감지가 보통 보안팀이나 ML 팀의 별도 시스템일 가능성이 큽니다. 
독립 배포, 독립 스케일링, 장애 격리가 필요합니다. 도배 감지 시스템이 죽어도 댓글 작성은 멈추면 안 됩니다.

따라서 결론은 **이벤트 발행 + 메시지 큐 도입** 입니다. 본 학습 프로젝트의 알림 흐름과 가장 유사한 구조입니다. 
댓글 작성 이벤트를 발행하고 별도 도배 감지 시스템이 메시지 큐를 통해 비동기로 처리합니다.

```java
@Transactional
public void createComment(...) {
    Comment comment = commentRepository.save(...);
    publisher.publishEvent(new CommentCreatedEvent(...));
}

// 별도 도배 감지 시스템의 컨슈머
@RabbitListener(queues = "spam.detection.queue")
public void onCommentCreated(CommentCreatedEvent event) {
    spamDetectionService.analyze(event);
}
```

이 선택의 근거는 네 가지입니다. 

첫째, 프로세스 분리로 장애 격리가 보장됩니다. 도배 감지 시스템이 다운되거나 느려져도 댓글 작성에는 영향이 없습니다. 
둘째, 시간 분리로 호출자 응답성이 유지됩니다. 도배 감지의 느린 처리가 댓글 작성의 응답 시간을 늦추지 않습니다. 
셋째, 영속성이 보장됩니다. 도배 감지 시스템이 일시적으로 죽어도 메시지가 큐에 남아있어 나중에 처리됩니다. 누락 없는 분석이 보장됩니다. 
넷째, 처리 종류의 확장성이 자연스럽게 챙겨집니다. 
같은 댓글 작성 이벤트에 스팸 감지, 욕설 감지 등 다른 분석 시스템이 자유롭게 추가될 수 있습니다. 
호출자 코드 수정 없이 새 컨슈머만 등록하면 됩니다.

이 케이스는 네 가지 판단 기준이 모두 도구 도입을 가리키는 시나리오입니다. 
(a) 가 *도구를 안 쓰는 것이 정답* 인 반대 극단이라면 (c) 는 *학습한 도구를 모두 활용하는 것이 정답* 인 극단입니다.

세 케이스의 종합. 세 케이스를 비교하면 도구 선택의 상황 의존성이 분명히 드러납니다. 
(a) 는 간단한 직접 호출이 가장 적절하고, 
(b) 는 로그 시스템 분리 여부에 따라 이벤트 발행 + 메시지 큐가 적절하며, 
(c) 는 이벤트 발행 + 메시지 큐가 명확히 필요합니다. 같은 도구가 모든 시나리오에 맞지 않는다는 것을 직접 확인할 수 있습니다. 
이 분석이 가르치는 메타 통찰은 *"학습한 도구를 적용하는 것"* 이 아니라 *"상황에 맞는 도구를 선택할 줄 아는 것"* 이 진짜 학습이라는 점입니다. 도구의 사용법을 배우는 것을 넘어 언제 안 쓸지를 판단하는 단계가 Phase 7 학습의 진짜 마무리입니다.
