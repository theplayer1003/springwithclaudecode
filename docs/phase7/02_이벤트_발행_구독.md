# 이벤트 발행/구독 패턴 (Spring Events)

## 1. 문제 — 직접 호출의 한계

소주제 1에서 만든 `CommentService.createComment` 흐름은 다음과 같다.

```java
@Transactional
public CommentResponse createComment(...) {
    Comment saved = commentRepository.save(comment);              // (1)
    
    if (!post.isAuthor(member.getId())) {
        commentEmailNotificationService.notifyNewComment(...);          // (2) 직접 호출
    }
    
    return CommentResponse.from(saved);                            // (3) 메서드 종료 → 커밋
}
```

이 코드에는 두 가지 구조적 문제가 남아 있다.

### 1.1. 트랜잭션 커밋 시점 vs 부수 작업 시점

`(2)` 와 `(3)` 사이의 미묘한 순서:
- `(2)` 에서 비동기 알림 메서드가 호출되면 별도 스레드에서 즉시 이메일 발송 시작
- 그러나 댓글의 DB 커밋은 `(3)` 메서드 종료 시점에 일어남
- 만약 `(3)` 직전에 어떤 예외가 발생해 트랜잭션이 롤백되면 — **댓글은 DB에서 사라졌는데 이메일은 이미 발송됐다.** 정합성 깨짐.

> 본질: *"DB 커밋이 정말 성공한 후에만 외부 시스템에 부수 효과를 일으키고 싶다"* 는 요구.

### 1.2. CommentService의 책임 비대화

```java
public class CommentService {
    private final PostRepository postRepository;
    private final CommentRepository commentRepository;
    private final MemberRepository memberRepository;
    private final CacheManager cacheManager;
    private final CommentNotificationService commentEmailNotificationService;
    // 부수 작업이 늘어나면 여기에 추가 의존성이 계속 쌓임
}
```

지금은 알림 하나뿐이지만, 댓글 생성 시 일어날 수 있는 부수 작업은 더 많다.
- 게시글 활동 점수 갱신
- 사용자 활동 로그 기록
- 멘션된 사용자에게 별도 알림
- 도배 감지 시스템 트리거

이걸 모두 CommentService에 직접 박으면 — *"댓글 작성"이라는 본질 외에 그 후속 작업의 협력자를 모두 알아야 하는* 신 객체가 된다.

> 본질: *"댓글이 작성되었다는 사실만 알리고, 그에 관심 있는 컴포넌트들이 알아서 반응하도록"* 하고 싶다.

→ 두 문제를 푸는 표준 도구가 **이벤트 발행/구독 패턴**이다.

## 2. 배경 — 옵저버 패턴

위 두 번째 문제는 디자인 패턴에서 잘 알려진 **옵저버 패턴 (Observer Pattern)** 의 적용 대상이다.

### 직접 구현

```java
// Subject (발행자)
public class CommentService {
    private final List<CommentCreatedListener> listeners = new ArrayList<>();
    
    public void register(CommentCreatedListener listener) {
        listeners.add(listener);
    }
    
    public CommentResponse createComment(...) {
        Comment saved = commentRepository.save(comment);
        
        for (CommentCreatedListener listener : listeners) {
            listener.onCommentCreated(saved);
        }
        
        return ...;
    }
}

// Observer 인터페이스
public interface CommentCreatedListener {
    void onCommentCreated(Comment comment);
}
```

핵심 아이디어:
- Subject(`CommentService`)는 Observer(listener)를 인터페이스로만 안다 — 구체 구현은 모름
- 새 Observer 추가 시 Subject 코드 변경 없음 — 호출자와 부수 작업이 분리

### 직접 구현의 한계

1. **listener를 누가 등록하나** — `register(...)` 호출 책임이 모호
2. **CommentService가 List 관리 책임을 가짐** — 인프라성 코드가 비즈니스 로직에 섞임
3. **트랜잭션 시점 제어가 어렵다** — listener 호출 위치를 트랜잭션 안에 두느냐 밖에 두느냐를 직접 제어
4. **동기/비동기 전환, 에러 처리** 등 모두 직접 구현

→ Spring이 이 패턴을 **프레임워크 수준에서 자동화**한다. 학습자는 비즈니스 본질만 작성.

## 3. Spring 이벤트 시스템 — 핵심 객체 4개

Spring 이벤트 시스템에는 여러 내부 객체가 등장하지만, 학습 단계에서는 **4개로 압축**해 이해하면 충분하다.

| 객체 | 정체 | 누가 만드나 | 역할 |
|---|---|---|---|
| **이벤트 객체** (예: `CommentCreatedEvent`) | 학습자가 만드는 평범한 클래스 | 학습자가 `new`로 생성 | "어떤 일이 일어났는지"를 담은 데이터 보따리 |
| **발행자** (`ApplicationEventPublisher`) | Spring 인터페이스 | Spring이 자동 제공 (주입받아 사용) | 이벤트를 Spring에게 전달하는 도구 |
| **멀티캐스터** (`ApplicationEventMulticaster`) | Spring 내부 객체 | Spring이 자동 제공 | 리스너 목록을 들고 있다가, 이벤트가 오면 알맞은 리스너들에게 전파 |
| **리스너 어댑터** (`ApplicationListenerMethodAdapter`) | Spring 내부 객체 | Spring이 자동 생성 | 학습자가 작성한 `@EventListener` 메서드를 감싸는 변환기 |

학습자가 **직접 다루는 것은 2개뿐**:
- 이벤트 객체 (직접 `new`)
- 발행자 (주입받아 `publishEvent` 호출)

나머지 둘은 Spring 내부에서 알아서 다룬다. 존재만 인지하면 된다.

## 4. 동작 순서

### Phase A — 애플리케이션 시작 시 (한 번)

```
1. Spring이 모든 빈을 생성
        ↓
2. EventListenerMethodProcessor (Spring 내부 후처리기)가
   모든 빈의 메서드를 스캔: "@EventListener 어노테이션이 붙은 메서드가 있나?"
        ↓
3. 발견된 메서드마다 → ApplicationListenerMethodAdapter (어댑터) 객체 생성
   어댑터는 (빈 인스턴스 + 메서드 정보 + 이벤트 타입) 을 보유
        ↓
4. 어댑터를 ApplicationEventMulticaster의 리스너 목록에 등록
```

Phase A가 끝나면 Spring은 "어떤 이벤트가 발행되면 어떤 메서드를 호출해야 하는지" 를 모두 알고 있는 상태다.

### Phase B — 이벤트 발행 시 (발행할 때마다)

```
1. [CommentService]
   eventPublisher.publishEvent(event)
        ↓
2. [ApplicationContext = ApplicationEventPublisher]
   내부적으로 multicaster.multicastEvent(event) 호출
        ↓
3. [ApplicationEventMulticaster]
   리스너 목록을 순회하며 이 이벤트 타입에 매칭되는 어댑터들 찾음
        ↓
4. [매칭된 Adapter들]
   각 어댑터의 onApplicationEvent(event) 호출
        ↓
5. [Adapter]
   내부에서 reflection으로 원본 빈의 진짜 메서드 호출:
   targetBean.handle(event)
        ↓
6. [학습자가 작성한 @EventListener 메서드 실행]
   실제 비즈니스 로직 (메일 발송 등)
```

## 5. 어댑터와 멀티캐스터의 의미

이름이 직관적이지 않아 헷갈리는 두 객체.

### 멀티캐스터 (Multicaster)

**Multicasting** = 하나의 메시지를 여러 수신자에게 동시에 전달.

Spring의 `ApplicationEventMulticaster`도 동일한 발상:
- 입력: 이벤트 1개
- 출력: 그 이벤트에 관심 있는 모든 리스너에게 동시 전파

```
이벤트 1개 ──→ [멀티캐스터] ──┬──→ 리스너 A
                              ├──→ 리스너 B
                              └──→ 리스너 C
```

### 어댑터 (Adapter)

어댑터의 본질은 **"모양이 다른 둘을 끼워 맞추는 변환기"** 다.

**현실 비유**: 일본 콘센트에 한국 노트북 플러그를 꽂아야 할 때, 모양이 안 맞으므로 사이에 어댑터를 끼운다. 어댑터는 *"한국 플러그를 받는 입구"* 와 *"일본 콘센트에 꽂히는 출구"* 를 양쪽에 가진 변환기.

**Spring 이벤트의 어댑터**가 변환하는 두 모양:

```
[멀티캐스터가 기대하는 모양]              [학습자가 작성한 메서드의 모양]
표준 인터페이스 ApplicationListener        임의의 시그니처
onApplicationEvent(event)                handle(event), notify(event), ...

           ↓ 모양 안 맞음 ↓
                
              [어댑터]
   ┌────────────────────────────────┐
   │  한쪽 입구: onApplicationEvent     │ ← 멀티캐스터가 여기를 호출 (표준 모양)
   │  내부:                             │
   │    reflection으로                  │
   │    targetBean.handle(event) 호출  │
   │  반대쪽 출구: handle(event)         │ → 학습자 메서드가 여기로 호출됨 (임의 모양)
   └────────────────────────────────┘
```

어댑터의 의사 코드:

```java
// Spring 내부 — 학습자가 작성하지 않음
class ApplicationListenerMethodAdapter implements ApplicationListener<Object> {
    private Object targetBean;     // 학습자가 만든 빈
    private Method targetMethod;   // 학습자가 만든 메서드 정보
    
    @Override
    public void onApplicationEvent(Object event) {
        // 멀티캐스터가 호출하는 표준 메서드
        // 내부에서 reflection으로 학습자 메서드를 호출
        targetMethod.invoke(targetBean, event);
    }
}
```

### 왜 이렇게 복잡한가

Spring 4.2 이전에는 학습자가 `ApplicationListener` 인터페이스를 직접 구현해야 했다:

```java
// 옛날 방식
public class CommentNotificationListener implements ApplicationListener<CommentCreatedEvent> {
    @Override
    public void onApplicationEvent(CommentCreatedEvent event) { ... }
}
```

문제:
- 메서드 이름이 `onApplicationEvent` 로 고정
- 한 클래스에서 여러 이벤트를 받기 어려움
- Spring 인터페이스에 강하게 결합

→ Spring 4.2의 `@EventListener` 는 이 자유로움을 가능하게 했다. 임의 메서드 이름, 한 클래스에 여러 listener 메서드. 그 자유로움의 대가가 어댑터의 변환 작업.

## 6. `@TransactionalEventListener` — 트랜잭션 시점 제어

### 일반 `@EventListener` 의 호출 시점

- 발행과 동시에 같은 스레드에서 즉시 호출 (동기)
- 트랜잭션 커밋 시점과 무관

→ 소주제 1에서 마주한 *"커밋 전 발송"* 문제를 그대로 가짐

### `@TransactionalEventListener` 의 호출 시점 제어

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handle(CommentCreatedEvent event) { ... }
```

| Phase | 시점 |
|---|---|
| `BEFORE_COMMIT` | 트랜잭션 커밋 직전 |
| **`AFTER_COMMIT`** (기본값) | 트랜잭션 정상 커밋 직후 |
| `AFTER_ROLLBACK` | 트랜잭션 롤백 후 |
| `AFTER_COMPLETION` | 커밋/롤백 무관하게 완료 후 |

`AFTER_COMMIT` 선택 시:
- 트랜잭션이 정상 커밋되어야만 리스너 메서드 호출
- 롤백 발생 시 리스너는 호출되지 않음 → *"댓글 없는데 이메일 발송"* 사고 원천 차단

### 내부 동작 — `TransactionSynchronization`

`@TransactionalEventListener` 는 별도 어댑터 (`TransactionalApplicationListenerMethodAdapter`) 를 사용한다. 이 어댑터는 호출되어도 즉시 메서드를 실행하지 않고:

```
1. 현재 활성 트랜잭션에 TransactionSynchronization 콜백 등록
   ("커밋 후 이 메서드를 호출하라" 라는 콜백)
        ↓
2. publishEvent 호출자에게는 즉시 리턴
        ↓
   ... 트랜잭션 진행 ...
        ↓
3. 트랜잭션 매니저가 커밋 완료 → 등록된 콜백들의 afterCommit() 호출
        ↓
4. afterCommit 콜백 안에서 → 학습자 listener 메서드 실제 호출
```

`TransactionSynchronization` 은 Spring 트랜잭션 매니저가 제공하는 표준 라이프사이클 콜백:

```java
public interface TransactionSynchronization {
    void beforeCommit(boolean readOnly);
    void beforeCompletion();
    void afterCommit();
    void afterCompletion(int status);
}
```

`phase` 옵션은 이 콜백들 중 어디에 매핑할지를 정한다.

> ⚠️ `@TransactionalEventListener` 는 **활성 트랜잭션이 있을 때 발행된 이벤트에만** 반응한다. 트랜잭션 없이 publishEvent하면 리스너가 호출되지 않거나 경고가 뜬다.

## 7. 다른 어노테이션과의 비교 — 프록시 vs 어댑터

지금까지 학습한 Spring 어노테이션을 메커니즘으로 분류하면:

| 어노테이션 | 메커니즘 | 빈 자체 변형? | 호출 시점 |
|---|---|---|---|
| `@Transactional` | 프록시 | ✅ 프록시로 감쌈 | 외부 호출 시 프록시가 가로챔 |
| `@Cacheable` | 프록시 | ✅ 프록시로 감쌈 | 외부 호출 시 프록시가 가로챔 |
| `@PreAuthorize` | 프록시 | ✅ 프록시로 감쌈 | 외부 호출 시 프록시가 가로챔 |
| `@Async` | 프록시 | ✅ 프록시로 감쌈 | 외부 호출 시 프록시가 가로챔 |
| **`@EventListener`** | **어댑터 + 멀티캐스터** | ❌ 빈은 그대로 | 이벤트 발행 시 멀티캐스터가 어댑터를 통해 호출 |
| **`@TransactionalEventListener`** | **어댑터 + 멀티캐스터 + TransactionSynchronization** | ❌ 빈은 그대로 | 트랜잭션 라이프사이클 콜백 시점에 호출 |

> **"어노테이션 = 프록시"** 라는 단일 모델은 부정확하다. Spring은 어노테이션의 용도에 따라 가장 적합한 메커니즘을 동원한다. `@EventListener` 가 그 첫 번째 예외 사례.

### 부수 효과

`@EventListener` 가 프록시 기반이 아니라는 사실에서 파생되는 특성:

1. **자기 호출 함정 없음** — `@Transactional`/`@Async` 의 *"`this.method()` 호출 시 무시됨"* 문제 없음. 어차피 누구도 listener 메서드를 직접 호출하지 않음
2. **동기가 기본** — 멀티캐스터가 publish와 같은 스레드에서 즉시 호출. 비동기로 만들려면 listener 메서드에 `@Async` 도 같이 부착
3. **같은 클래스에 listener 둬도 됨** — 프록시 자기 호출 문제 없음. 다만 관심사 분리 측면에서는 별 클래스 권장

## 8. 학습자가 작성하는 것 vs Spring이 처리하는 것

**학습자가 직접 작성**:
- 이벤트 클래스 (`CommentCreatedEvent` 등)
- `@EventListener` 또는 `@TransactionalEventListener` 메서드를 가진 리스너 빈
- 발행자 (`ApplicationEventPublisher`) 주입 + `publishEvent` 호출

**Spring이 자동 처리**:
- `EventListenerMethodProcessor` 가 메서드 스캔
- 어댑터 객체 생성
- 멀티캐스터에 리스너 등록
- `publishEvent` 호출 시 알맞은 어댑터 찾기
- 어댑터의 reflection 호출
- `@TransactionalEventListener` 의 트랜잭션 콜백 등록 / 발동

## 9. 정리

- 이벤트 패턴은 *"본질과 부수 작업의 분리"* 와 *"트랜잭션 시점 제어"* 두 가지 문제를 한 번에 해결
- Spring 이벤트 시스템의 핵심 객체는 4개 — 이벤트, 발행자, 멀티캐스터, 어댑터
- 학습자가 직접 다루는 건 이벤트와 발행자 둘뿐. 나머지는 Spring 내부 처리
- **멀티캐스터** — 1:N 분배 센터
- **어댑터** — 멀티캐스터의 표준 호출 형태와 학습자 임의 메서드 사이의 모양 변환기
- `@EventListener` 는 **프록시가 아닌** 어댑터 + 멀티캐스터 메커니즘 — 다른 어노테이션과 본질적으로 다름
- `@TransactionalEventListener(AFTER_COMMIT)` 은 `TransactionSynchronization` 콜백을 활용해 *"커밋 후"* 시점에 호출
