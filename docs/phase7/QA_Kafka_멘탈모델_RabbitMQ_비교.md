# Kafka 의 멘탈 모델 — RabbitMQ 와의 비교

## 질문 배경

Kafka 의 *Consumer Group* 과 *파티션* 개념을 학습하던 중, *"각 메시지가 종류별로 다른 파티션에 있다"* 같은 오해가 발생했다. 추적해보니 RabbitMQ 의 멘탈 모델로 Kafka 를 이해하려다 혼동이 온 케이스. *"같은 메시지 큐 카테고리지만 두 시스템의 멘탈 모델이 정반대 방향"* 이라는 점을 RabbitMQ 와의 명시적 대비를 통해 정리할 필요가 있었다.

---

**정확합니다.** 그게 가장 흔한 함정이에요. *"메시지 큐"* 라는 같은 카테고리에 묶이지만 두 시스템의 *멘탈 모델* 은 거의 정반대 방향이라, RabbitMQ 의 사고로 Kafka 를 이해하면 반드시 막힙니다.

RabbitMQ 와 비교하면서 다시 설명드리겠습니다.

---

## 1. 본질적 멘탈 모델의 차이

| | **RabbitMQ** | **Kafka** |
|---|---|---|
| **본질** | 메시지 *전달자 (broker)* | 분산 *로그 (log)* |
| **메시지의 운명** | 처리되면 *사라짐* | 처리해도 *남아있음* |
| **저장 위치** | 큐 (대기열) | 파티션 (불변 로그) |
| **메시지가 큐/파티션에 들어가는 이유** | *전달되기 전 대기* | *영구히 기록* |

가장 큰 차이: **메시지가 사라지느냐 vs 남느냐**.

RabbitMQ 는 *"메시지를 안전하게 받는 사람에게 전달하고 끝"* 의 모델.
Kafka 는 *"모든 메시지를 영구히 기록해두고 누구든 와서 읽어가게 함"* 의 모델.

이 차이가 모든 다른 차이의 근원입니다.

---

## 2. 학습자가 혼동한 지점 — 콕 집어 정정

학습자가 *"각 메시지가 종류별로 나뉘어 파티션에 있다"* 라고 생각한 부분 — 이게 RabbitMQ 사고의 흔적입니다.

### RabbitMQ 에서는 *큐가 의미별로 분리됨*

```
[RabbitMQ]
  comment.email.queue ← 이메일 처리할 메시지
  comment.sms.queue   ← SMS 처리할 메시지
  comment.log.queue   ← 로그 적재할 메시지

→ 큐가 다르다 = 메시지의 *처리 의미가 다르다*
→ 이메일 큐의 메시지를 SMS 컨슈머가 받으면 잘못된 일
```

### Kafka 에서는 *파티션은 같은 의미의 분할*

```
[Kafka]
  Topic "comment.created" 
    파티션 P0: 댓글 메시지 일부
    파티션 P1: 댓글 메시지 일부
    파티션 P2: 댓글 메시지 일부
    파티션 P3: 댓글 메시지 일부

→ 파티션이 다르다 = *같은 의미의 메시지를 부하 분산용으로 나눠 담음*
→ P0 의 메시지와 P1 의 메시지는 *의미가 똑같음*
→ 어떤 컨슈머가 처리해도 결과는 동일
```

→ **파티션은 *메시지의 종류* 가 아니라 *같은 종류 메시지의 부하 분산 그릇*** 입니다.

## 3. 개념 매칭표 — 대응되는 것, 안 되는 것

### 대응되는 개념

| RabbitMQ | Kafka | 차이 |
|---|---|---|
| Producer | Producer | 거의 동일 |
| Consumer | Consumer | 개념 같지만 동작 다름 (아래 설명) |
| 메시지 | 메시지 (Record) | 동일 |

### 대응이 *애매한* 개념 — 가장 헷갈리는 부분

| RabbitMQ | Kafka | 본질 |
|---|---|---|
| Queue | Topic 의 파티션 | 둘 다 *메시지가 보관되는 곳* 이지만 동작이 매우 다름 |
| Queue (논리적 구분 단위) | Topic | 둘 다 *메시지 분류의 단위* |
| Queue 에 붙은 여러 컨슈머 | Consumer Group | 둘 다 *부하 분산 그룹* 이지만 분배 방식이 다름 |

가장 흔한 매핑: **RabbitMQ 의 Queue ↔ Kafka 의 Topic + 그 안의 파티션 묶음** 으로 대응.

### Kafka 에만 있는 개념

| 개념 | RabbitMQ 에 없는 이유 |
|---|---|
| **파티션** | RabbitMQ 의 큐는 *분할되지 않음* (한 덩어리) |
| **Offset** | RabbitMQ 는 메시지가 사라지니까 *위치* 개념 불필요 |
| **메시지 보관 정책** (시간/크기) | RabbitMQ 는 처리 즉시 사라지니까 *보관* 개념 없음 |
| **Replay (재생)** | RabbitMQ 는 메시지가 사라져서 불가능 |

### RabbitMQ 에만 있는 개념

| 개념 | Kafka 에 없는 이유 |
|---|---|
| **Exchange** | Kafka 는 *Producer 가 직접 토픽에 발행*, 별도 라우팅 컴포넌트 없음 |
| **Binding** | Exchange 가 없으니 Binding 도 없음 |
| **Direct/Fanout/Topic/Headers Exchange** | 모든 라우팅 결정이 *Producer 의 토픽/키 선택* 으로 단순화 |

## 4. 부하 분산의 차이 — 가장 큰 동작 차이

학습자 의문 *"컨슈머가 어떻게 메시지를 가져가는가"* 의 본질.

### RabbitMQ — 큐가 *능동적으로* 컨슈머에게 나눠줌

```
[Queue "comment.notification"]
  [msg1] [msg2] [msg3] [msg4]
        ↓ Queue 가 round-robin으로 분배 (Push)
  Consumer A: msg1 받음 → 처리 → ack → 사라짐
  Consumer B: msg2 받음 → 처리 → ack → 사라짐
  Consumer A: msg3 받음 → 처리 → ack → 사라짐
  Consumer B: msg4 받음 → 처리 → ack → 사라짐
```

특징:
- **누가 더 빠르면 그 사람이 더 많이 받음** (먼저 ack 보내는 컨슈머가 다음 메시지 빨리 받음)
- 메시지가 *어느 컨슈머에 갈지* 는 *런타임에 결정*
- 컨슈머가 추가/제거되어도 *즉시 자연스럽게 적응* (큐는 그저 다음 메시지를 줄 뿐)
- **메시지 단위로 분배**

### Kafka — *파티션 단위로 컨슈머에 정적 할당*

```
[Topic "comment.created" — 파티션 4개]
  P0: [msg1, msg5, msg9...] ← 항상 같은 컨슈머가 받음
  P1: [msg2, msg6, msg10...]
  P2: [msg3, msg7, msg11...]
  P3: [msg4, msg8, msg12...]
        ↓ Kafka 가 파티션 단위로 컨슈머에 할당
  Consumer A → P0, P1 담당 (계속)
  Consumer B → P2, P3 담당 (계속)
```

특징:
- **한 파티션의 메시지는 *항상 같은 컨슈머* 가 처리** (재할당 전까지)
- 컨슈머가 추가/제거되면 *전체 재분배 (rebalancing)* 발생
- **파티션 단위로 분배** (메시지 단위 아님)
- 같은 키의 메시지는 *항상 같은 파티션* → 같은 컨슈머 → *순서 보장*

### 비유로

| | RabbitMQ | Kafka |
|---|---|---|
| 비유 | *공장의 컨베이어 벨트* — 작업자 누구든 다음 부품을 집어감 | *도서관의 책장* — 각 작업자에게 *책장 구역* 이 할당되어 그 구역의 책만 읽음 |
| 누가 어떤 메시지? | *그 순간 손이 빈 사람* | *그 파티션을 할당받은 사람* |
| 동적 vs 정적 | 매우 동적 | 비교적 정적 (재할당 전까지) |

## 5. *"한 메시지를 여러 곳에 보내기"* 의 차이

### RabbitMQ — Fanout Exchange + 여러 Queue

본 학습 프로젝트가 정확히 이 패턴:

```
Producer → [Fanout Exchange "comment.event.exchange"]
              ↓        ↓
       [email.queue] [sms.queue]
              ↓        ↓
       Consumer1    Consumer2
       
→ Exchange 가 두 큐에 *복제* 해서 넣음
→ 각 큐는 독립된 메시지를 가짐
```

### Kafka — 같은 Topic 을 여러 Consumer Group 이 구독

```
Producer → [Topic "comment.created"]
                ↓ (메시지는 단 하나, 사라지지 않음)
   ┌────────────┼────────────┐
   ↓            ↓            ↓
[Group "이메일"] [Group "SMS"] [Group "분석"]
   ↓            ↓            ↓
   각자 다른 오프셋으로 같은 메시지 읽음
```

→ Kafka 는 *복제하지 않음*. 메시지는 토픽에 *하나만 있고*, 여러 그룹이 *각자의 위치 (offset) 로* 읽어감. 모두가 같은 메시지를 본인 페이스로 읽음.

### 본질적 차이

| | RabbitMQ | Kafka |
|---|---|---|
| 메시지 복제 | Exchange 가 *복제* 해서 여러 큐에 넣음 | 복제 없음, *원본 하나* 를 여러 그룹이 읽음 |
| 새 구독자 추가 | 새 큐 + 새 Binding 추가 (Producer 영향 없음) | 새 Consumer Group 추가만 하면 됨 (인프라 변경 없음) |
| 과거 메시지 읽기 | 불가능 (이미 사라짐) | 가능 (offset 을 과거로 돌리면 됨) |

## 6. 그래서 *"동작 차이"* 가 어떻게 나타나는가

### 시나리오: 본 학습 프로젝트를 두 시스템으로 각각 구현

**RabbitMQ 버전 (현재 코드)**:
```
[댓글 생성]
   ↓
Producer 가 Fanout Exchange 에 발행
   ↓
Exchange 가 email.queue, sms.queue 에 *복제* 해서 넣음
   ↓
EmailConsumer 가 email.queue 에서 가져옴 → 처리 → 사라짐
SmsConsumer 가 sms.queue 에서 가져옴 → 처리 → 사라짐
```

**Kafka 버전 (만약)**:
```
[댓글 생성]
   ↓
Producer 가 "comment.created" 토픽에 발행 (파티션 결정은 키 해시)
   ↓
토픽 안의 한 파티션에 *원본 그대로* 보관 (사라지지 않음)
   ↓
Email Consumer Group 의 컨슈머가 자기 파티션을 폴링해서 읽음 → 처리
SMS Consumer Group 의 컨슈머가 *같은 파티션* 을 자기 offset 으로 읽음 → 처리

→ 둘 다 처리해도 메시지는 토픽에 *계속 남음*
→ 일주일 후 분석팀이 *과거 데이터* 도 읽을 수 있음
```

### 새 알림 채널 (예: 슬랙) 추가 시

**RabbitMQ**: `comment.slack.queue` 생성 + Exchange 와 Binding 추가
**Kafka**: 슬랙 Consumer Group 만 추가하면 됨 (인프라 변경 없음)

→ 새 구독자 추가의 *코드 변경 범위* 가 다름.

## 7. 한 줄 요약 — 두 시스템의 정신

| | 한 줄 |
|---|---|
| **RabbitMQ** | *"메시지를 받는 사람에게 안전하게 전달하고 끝낸다"* — 큐는 *전달의 대기실* |
| **Kafka** | *"모든 메시지를 영구 기록하고, 원하는 사람은 언제든 읽어가게 한다"* — 파티션은 *영구 로그* |

## 학습자의 의문에 대한 최종 답

학습자가 *"래빗메시지큐를 상상하면서 카프카를 이해하려고 했더니 혼동이 왔다"* 라고 자가 진단한 것 — **정확히 그게 함정** 입니다.

핵심 마인드 셋 전환:

| 잘못된 사고 (RabbitMQ 식) | 정확한 사고 (Kafka 식) |
|---|---|
| *"각 메시지는 처리될 곳이 정해져 있다"* | *"메시지는 그저 토픽에 기록될 뿐, 누가 읽든 OK"* |
| *"컨슈머가 자기 메시지를 찾아간다"* | *"컨슈머는 할당된 파티션의 모든 메시지를 처리"* |
| *"메시지는 받으면 사라진다"* | *"메시지는 계속 남는다, 컨슈머는 자기 offset 만 관리"* |
| *"파티션 = 메시지 종류별 분류"* | *"파티션 = 같은 종류 메시지의 분할"* |

이 멘탈 모델 전환이 일어나면 Consumer Group, Offset, 파티션이 모두 자연스럽게 이해됩니다.

이번 학습은 *"같은 카테고리의 다른 시스템을 학습할 때, 첫 시스템의 멘탈 모델을 무의식적으로 강요하기 쉽다"* 는 메타 통찰까지 줍니다. Phase 9, 10 에서 다른 인프라 (Docker, Kubernetes, 다양한 DB) 학습할 때도 같은 함정에 주의하시면 좋습니다.
