# `@RabbitListener` + `@Async` 조합의 안티 패턴

## 개요

메시지 큐의 컨슈머에서 `@RabbitListener` 가 받은 메시지를 *`@Async` 메서드로 위임* 하여 처리하는 패턴은 직관적으로 자연스러워 보인다. 컨슈머 스레드를 빨리 비워서 더 많은 메시지를 받게 하는 발상.

그러나 이 조합은 **메시지 큐의 핵심 약속인 *안전한 전달* 을 깨뜨릴 수 있는 안티 패턴** 이다. 본 문서는 그 메커니즘과 실패 시나리오, 그리고 올바른 대안을 정리한다.

---

## 1. 메시지 큐의 ack 약속

메시지 큐(RabbitMQ 기준)는 *"메시지가 안전하게 처리되었음을 컨슈머가 확인(ack)할 때까지 큐에서 메시지를 삭제하지 않는다"* 는 약속을 제공한다.

```
[정상 흐름]
  1. 컨슈머가 메시지 수신
  2. 메시지 처리
  3. 처리 성공 → ack 전송
  4. 큐에서 메시지 삭제

[실패 흐름]
  1. 컨슈머가 메시지 수신
  2. 처리 중 예외 발생
  3. ack 전송하지 않음 (혹은 nack)
  4. RabbitMQ 가 메시지를 큐에 되돌리거나 DLQ 로 라우팅
  5. → 다음 컨슈머가 받아서 재처리
```

이 약속 덕분에 *"컨슈머가 죽어도 메시지는 잃지 않는다"* 는 보장이 가능하다. 메시지 큐를 도입하는 가장 큰 이유 중 하나.

### Spring AMQP 의 ack 처리

Spring AMQP 에서 `@RabbitListener` 메서드의 ack 시점은:

- **메서드가 정상 종료** → ack 전송 (메시지 처리 완료로 간주)
- **메서드에서 예외 throw** → nack 전송 (재처리 또는 DLQ)

즉, **메서드의 종료가 곧 "처리 완료" 신호** 다.

---

## 2. 안티 패턴의 메커니즘

`@RabbitListener` 메서드 안에서 `@Async` 서비스를 호출하면 다음과 같은 흐름이 된다.

```
[컨슈머 스레드]
  1. @RabbitListener 메서드 시작
  2. @Async 서비스 호출
  3. @Async 는 즉시 리턴 (Future 만 던지고 끝)
  4. @RabbitListener 메서드 종료
       ↓
  5. Spring AMQP: "메서드 정상 종료" → ack 전송
  6. RabbitMQ: 큐에서 메시지 제거
       ⚠️ 메시지가 "처리됐다" 고 큐에서 사라짐

[@Async 스레드 — 동시에]
  실제 작업 진행 중...
```

문제의 본질: **ack 시점과 실제 처리 완료 시점이 어긋난다**. ack 는 *컨슈머 메서드 종료* 에 묶여있지만, 실제 작업은 *별도 스레드* 에서 그 이후에 진행된다.

---

## 3. 실패 시나리오

이 어긋남이 실제로 손실을 일으키는 시나리오는 두 가지다.

### 시나리오 1: 비동기 작업 중 예외 발생

```
1. 컨슈머 메서드 종료 → ack 전송 → 큐에서 메시지 삭제
2. @Async 스레드에서 작업 실행
3. 외부 시스템 장애로 예외 발생 (예: SMTP 서버 응답 불가)
4. 메시지는 이미 ack 됐기 때문에 RabbitMQ 는 모름
5. → 재처리 불가능. 메시지 손실
```

`@Async` 안의 예외는 컨슈머 메서드 종료와 무관한 시점에 발생하므로 RabbitMQ 에 전달되지 않는다. 일반적인 try/catch 로도 잡히지 않는다 (Future 결과를 확인해야 알 수 있음).

### 시나리오 2: 비동기 작업 중 프로세스 종료

```
1. 컨슈머 메서드 종료 → ack 전송 → 큐에서 메시지 삭제
2. @Async 스레드에서 작업 시작 (예: 1.5초 걸리는 메일 발송)
3. 그 사이 프로세스가 OOM / 배포 / 장애로 종료
4. ThreadPoolTaskExecutor 의 작업 큐는 메모리에 있어서 휘발됨
5. → 진행 중이던 작업과 큐에 대기 중이던 작업 모두 손실
```

특히 **시나리오 2 (프로세스 종료) 가 실전에서 더 자주 발생**. 무중단 배포가 미흡한 환경, OOM, JVM 비상 종료 등에 항상 노출됨. 코드에 명시적인 예외가 없어도 발생할 수 있는 위험.

---

## 4. 깨지는 약속

이 안티 패턴이 위험한 진짜 이유는 *메시지 큐를 도입한 본래 목적* 이 무력화되기 때문이다.

| 메시지 큐의 약속 | 안티 패턴에서의 결과 |
|---|---|
| 메시지는 처리 완료까지 큐에 남는다 | ack 가 작업 완료보다 먼저 발생 → 깨짐 |
| 컨슈머가 죽어도 메시지는 잃지 않는다 | `@Async` 스레드의 큐는 휘발성 → 깨짐 |
| 실패 시 재처리 가능 | `@Async` 의 예외는 RabbitMQ 에 전달 안 됨 → 깨짐 |

**메시지 큐를 사용하면서 메시지 큐의 보장을 무력화시키는 구조**. 차라리 메시지 큐를 쓰지 않는 편이 명확할 정도.

---

## 5. 올바른 해결책 — 컨슈머 동시성 설정

처리량을 늘리고 싶다면 `@Async` 가 아니라 **컨슈머 스레드 자체를 여러 개로 늘리는 방식** 이 정답이다.

```java
@RabbitListener(
    queues = "comment.email.queue",
    concurrency = "3-10"   // 동시 처리 컨슈머 스레드 3~10개
)
public void handleCommentNotification(CommentCreatedEvent event) {
    // 동기적으로 끝까지 처리
    commentEmailNotificationService.notifyNewComment(...);
}
```

### 왜 이게 안전한가

```
[컨슈머 스레드 (여러 개 중 하나)]
  1. 메시지 수신
  2. 동기적으로 작업 끝까지 처리
  3. 작업 완료 → 메서드 종료 → ack 전송
       (실패 시 → 예외 → nack 전송 → 재처리/DLQ)
  4. 다음 메시지 받기
```

핵심:
- **ack 시점 = 실제 작업 완료 시점** → 약속 유지
- 예외 발생 시 RabbitMQ 가 인지하고 처리 → 재처리 가능
- 프로세스가 죽어도 메시지는 큐에 남음 → 다른 컨슈머가 받음

처리량은 `concurrency` 의 최대값을 늘리는 것으로 조절. 컨슈머 스레드를 추가로 띄우는 것은 메모리 비용이 약간 들지만, *안전성을 깨지 않는 방식의 처리량 증대* 다.

---

## 6. 처리량과 안전성의 비교

| 항목 | `@RabbitListener` + `@Async` (안티 패턴) | `@RabbitListener` + `concurrency` (권장) |
|---|---|---|
| 처리량 | ↑ (메시지를 빨리 받음) | ↑ (스레드 자체가 여러 개) |
| 안전성 | ↓ (메시지 손실 위험) | ✅ (메시지 큐 약속 유지) |
| 실패 처리 | 직접 구현해야 함 | RabbitMQ 가 자동 처리 |
| 프로세스 종료 시 | 진행 중 작업 손실 | 큐에 남아있어 다음 컨슈머가 받음 |
| 운영 모니터링 | 별도 추적 필요 | 큐 상태가 곧 진행 상태 |

처리량은 비슷하지만 **안전성에 압도적인 차이**.

---

## 7. 일반화 — 메시지 처리의 안전성 원칙

이 안티 패턴이 보여주는 더 큰 교훈:

> **메시지 처리의 ack 신호는 *실제 처리 완료* 와 일치해야 한다.**

이 원칙은 RabbitMQ 뿐 아니라 Kafka, SQS 등 모든 메시지 시스템에 동일하게 적용된다. ack(또는 commit, delete) 가 실제 처리보다 먼저 발생하는 모든 구조는 잠재적 손실 위험을 안고 있다.

### 비슷한 패턴

다른 영역에서도 동일한 원칙이 적용되는 사례:

- **데이터베이스 트랜잭션** — 트랜잭션 커밋이 실제 처리 완료를 보장해야 함. `@TransactionalEventListener(AFTER_COMMIT)` 이 사용되는 이유와 동일한 맥락
- **HTTP 요청** — 클라이언트에게 200 OK 를 응답하기 전에 실제 처리가 끝나야 함 (혹은 비동기 응답을 명시)
- **파일 시스템** — 파일 쓰기 후 fsync 가 실제 디스크 반영을 보장

*"완료를 알리는 신호와 실제 완료가 어긋나면 안 된다"* 는 분산 시스템의 보편 원칙.

## 8. 메모리 큐 vs 영속 큐의 차이

`@Async` 의 스레드 풀이 사용하는 작업 큐는 **메모리 기반의 휘발성 큐** 다. JVM 이 종료되면 사라진다.

반면 RabbitMQ 같은 메시지 브로커의 큐는 **영속성을 갖는 외부 저장소**. 컨슈머가 죽어도 메시지가 보존된다.

`@RabbitListener` + `@Async` 안티 패턴의 본질은 *"영속 큐에서 메시지를 빼서 휘발성 큐로 옮기는 것"* 과 같다. 안전성을 의도적으로 떨어뜨리는 셈.

```
[안티 패턴이 하는 일]

  [영속 큐 (RabbitMQ)]            [휘발성 큐 (ThreadPoolTaskExecutor)]
        │                                       ↑
        │  ack 후 영구히 사라짐                 │  메시지 옮김
        └───────────────────────────────────────┘
        
  → 안전한 곳에서 안전하지 않은 곳으로 메시지를 이동시킴
```

이 그림이 안티 패턴의 본질을 가장 명확하게 보여준다.

---

## 한 줄 요약

> `@RabbitListener` 안에서 `@Async` 를 호출하면 메서드 종료가 곧 ack 발송 시점이 되어, *실제 작업 완료* 와 *ack 시점* 이 어긋난다. 이는 메시지 큐의 안전 보장을 무력화하는 안티 패턴이며, 처리량을 늘리려면 `concurrency` 설정으로 컨슈머 스레드 자체를 늘려야 한다.
