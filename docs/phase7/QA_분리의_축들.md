# 분리의 세 가지 축 — 시간 / 결합도 / 프로세스

## 개요

소프트웨어에서 *"결합을 풀어준다"* 는 표현은 자주 쓰이지만 *어떤 결합을 푸는지* 가 흐릿한 채로 사용되기 쉽다. 결합은 한 종류가 아니라 **서로 다른 차원** 으로 존재하며, 각 차원을 풀어주는 도구도 다르다.

본 문서는 세 가지 분리 축 — **시간 축**, **결합도 축**, **프로세스 축** — 을 정리하고, 각 축의 도구와 한계, 그리고 *직교성* 의 의미를 설명한다.

---

## 1. 시간 축 (Temporal Axis)

### 정의

호출자가 피호출자의 처리가 끝날 때까지 *기다리는가* 의 차원. 푸는 도구: **`@Async`, Future, 별도 스레드 풀**.

### 동기와 비동기

```
[동기 — 시간 결합]
  호출자 → 피호출자 호출 → ··· 대기 ··· → 결과 받음 → 다음 작업

[비동기 — 시간 분리]
  호출자 → 피호출자 호출 (즉시 리턴) → 다음 작업
                          ↘ (별도 스레드에서 처리)
```

호출자의 *응답 시간* 에 피호출자의 *처리 시간* 이 더해지는가, 아닌가의 차이.

### 무엇을 해결하나

- 응답 지연: 메인 흐름이 무거운 작업을 기다리지 않게 됨
- 부분적 장애 전파: 피호출자가 느려도 호출자는 정상 응답

### 한계

- 같은 JVM 안에서의 분리일 뿐 — 호출자와 피호출자는 같은 프로세스 운명을 공유
- 호출자가 피호출자의 *타입을 직접 알아야* 한다 (강결합)
- 별도 스레드 풀의 큐는 휘발성 — 프로세스가 죽으면 큐에 있던 작업도 사라짐

---

## 2. 결합도 축 (Coupling Axis)

### 정의

호출자가 피호출자의 *존재나 타입을 알고 있는가* 의 차원. 푸는 도구: **이벤트 발행 / 구독 (`ApplicationEventPublisher`, `@EventListener`)**.

### 강결합과 약결합

```
[강결합 호출]
  class Caller {
      private final Callee callee;     // ← Caller가 Callee를 알아야 함
      void doWork() {
          callee.handle(data);          // ← 명시적 호출
      }
  }

[약결합 이벤트]
  class Publisher {
      private final ApplicationEventPublisher publisher;
      void doWork() {
          publisher.publishEvent(new SomeEvent(data));   // ← 누가 받는지 모름
      }
  }

  class Subscriber {
      @EventListener
      void on(SomeEvent event) { ... }   // ← 자기가 알아서 받음
  }
```

### 무엇을 해결하나

- **OCP (개방-폐쇄 원칙)**: 새로운 처리를 추가할 때 호출자 코드를 수정하지 않아도 됨
- **책임 비대화**: 호출자가 여러 피호출자를 알 필요가 없음 → 의존성이 적어지고 단일 책임 유지
- **확장성**: 같은 이벤트에 새로운 리스너를 자유롭게 추가

### 한계

- 같은 JVM 안에서의 결합도 분리일 뿐 — 같은 스레드, 같은 프로세스에서 동작
- 시간 축은 풀리지 않음: 기본 `@EventListener` 는 *동기적* 으로 호출됨 → 메인 흐름이 리스너 처리를 기다림

---

## 3. 프로세스 축 (Process Axis)

### 정의

호출자와 피호출자가 *같은 프로세스/JVM 에 있어야 하는가* 의 차원. 푸는 도구: **메시지 큐 (RabbitMQ, Kafka 등), HTTP 호출, RPC**.

### 같은 프로세스와 다른 프로세스

```
[같은 프로세스 — 프로세스 결합]
  하나의 JVM 안에서 모든 처리. 메모리 공유. JVM 운명 공유.

[다른 프로세스 — 프로세스 분리]
  네트워크를 통한 메시지 전달.
  각 프로세스는 자기 자원으로 독립적으로 동작.
```

### 무엇을 해결하나

- **장애 격리**: 한쪽이 죽어도 다른 쪽은 계속 동작
- **독립 배포**: 서로의 배포 일정에 영향받지 않음
- **자원 격리**: CPU, 메모리, 스레드 풀, GC 모두 독립
- **수평 확장**: 컨슈머/프로듀서를 각자 필요에 따라 늘림
- **메시지 영속성 (메시지 큐의 경우)**: 컨슈머가 죽어도 메시지가 큐에 남아있다 처리됨

### 메시지 큐의 추가 효과

메시지 큐는 *프로세스 분리* 와 함께 *시간 분리* 도 자동으로 포함한다. 메시지를 발행하는 행위 자체가 *"발행한 뒤로는 기다리지 않음"* 을 의미하기 때문.

→ 메시지 큐는 가장 강한 분리를 제공.

### 한계

- 인프라 복잡도 증가 (브로커 관리, 모니터링 필요)
- 디버깅 난이도: 메시지 흐름을 따라가기 위한 추적 도구 필요
- 메시지 순서 보장, 중복 처리 같은 새로운 문제 영역

## 4. 축의 직교성

세 축은 **직교(orthogonal)** 하다. 즉, 서로 독립적으로 선택 가능하다.

```
                동기                비동기
              ┌──────────────────┬─────────────────────┐
강결합 호출   │ 일반 메서드 호출 │ @Async 메서드 호출  │
              ├──────────────────┼─────────────────────┤
약결합 이벤트 │ @EventListener   │ @Async +            │
              │                  │ @EventListener      │
              └──────────────────┴─────────────────────┘

  + 위 모든 조합은 "같은 프로세스" 또는 "다른 프로세스(메시지 큐)" 위에서 동작 가능
```

### 직교성의 의미

- *"~를 풀려면 무조건 ~도 같이 써야 한다"* 같은 강제는 없음
- 어떤 축이 풀려있고 어떤 축이 풀려있지 않은지 *명확히 인식* 하는 것이 중요
- 각 축은 독립적인 *문제* 를 해결하므로, 직면한 문제에 따라 필요한 축만 골라 풀면 됨

### 도구 매칭 표

| 도구 | 시간 | 결합도 | 프로세스 |
|---|---|---|---|
| 일반 메서드 호출 | ✗ 동기 | ✗ 강결합 | ✗ 같은 프로세스 |
| `@Async` | ✅ 비동기 | ✗ 강결합 | ✗ 같은 프로세스 |
| `@EventListener` | ✗ 동기 | ✅ 약결합 | ✗ 같은 프로세스 |
| `@Async` + `@EventListener` | ✅ 비동기 | ✅ 약결합 | ✗ 같은 프로세스 |
| 메시지 큐 (RabbitMQ 등) | ✅ 자동 비동기 | ✅ 약결합 | ✅ 분리됨 |

메시지 큐가 가장 강한 분리지만, 그만큼 인프라 비용이 따른다. 직면한 문제에 따라 적절한 수준을 고르는 것이 설계.

## 5. 호출자 관점의 분리 완성 시점

분리의 효과를 측정할 때는 **호출자 관점** 에서 *"어디서부터 호출자의 관심이 끝나는가"* 를 따져야 한다.

### 예: 메시지 큐 발행

```
Producer 측:
  1. 비즈니스 로직 처리
  2. 메시지 큐에 발행
  ←─ 여기까지 ←─ 호출자(Producer)의 시간/관심 종료

Consumer 측:
  3. 메시지 수신
  4. 처리
  → Consumer 가 동기로 처리하든 비동기로 처리하든
    Producer 의 응답 시간과는 무관
```

메시지 큐로 분리된 이후, **컨슈머 내부의 동기/비동기 선택은 프로듀서와 무관한 *컨슈머 내부의 의사결정*** 이다. 컨슈머가 자기 자원을 어떻게 활용하느냐의 문제일 뿐, 분리의 효과를 추가로 더하지 않는다.

### 분리 완성 시점의 시사

같은 도구를 여러 위치에 쓴다고 *분리가 누적* 되는 것은 아니다. 호출자 관점에서 분리가 완성된 후에는 추가 도구가 더 이상 호출자에게 이득을 주지 않는다.

→ 도구를 *어디에* 쓰느냐가 *얼마나 많이 쓰느냐* 보다 중요.

## 6. 실전 설계의 사고 순서

분리를 도입할 때 권장되는 사고 흐름:

1. **어떤 종류의 결합이 문제인가** — 응답 지연인지, 의존성 폭발인지, 장애 전파인지를 명확히 한다
2. **해당 축을 푸는 가장 가벼운 도구는 무엇인가** — 응답 지연만 문제라면 `@Async` 하나로 충분할 수 있다
3. **풀리지 않은 다른 축은 받아들일 만한가** — `@Async` 만 도입하면 강결합과 프로세스 결합은 그대로다. 이게 허용 가능한지 판단
4. **인프라 비용 대비 효과가 정당한가** — 메시지 큐는 강력하지만 운영 비용이 따른다. 필요 이상의 분리는 오히려 복잡성을 키움

축을 명확히 인식하면 *"왜 이 도구를 도입하는가"* 의 답이 분명해진다.

## 7. 학습 코드와 실전 구조의 차이

학습 자료에서 여러 도구가 한 흐름에 등장한다고 해서 *실전에서도 모두 함께 쓰여야 한다* 는 의미는 아니다. 학습 코드는 **각 도구의 동작 메커니즘을 익히기 위한 인위적 흐름** 일 수 있고, 그 흐름을 그대로 실전 권장 패턴으로 받아들이면 함정이 된다.

특히 다음과 같은 경우를 구분해야 한다:

- **교육적 시연** — 학습한 도구를 한 번씩 다 써보기 위한 배치
- **시뮬레이션의 한계** — 분리된 구조를 하나의 JVM 에서 표현하기 위한 인위적 분리
- **실전 의사결정** — 진짜로 그 도구가 *해결하는 문제* 가 있어서 도입

세 가지를 구분하는 감각이 분리 축의 이해와 함께 따라가야 한다.

---

## 한 줄 요약

> **시간 축 (`@Async`), 결합도 축 (이벤트 발행), 프로세스 축 (메시지 큐)** 은 서로 다른 차원의 결합을 푸는 직교적 도구다. 분리의 효과는 *호출자 관점에서 완성되는 시점* 까지만 의미가 있으며, 그 이후의 추가 도구는 *별도 영역의 의사결정* 이다.
