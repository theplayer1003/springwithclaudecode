# Push vs Pull 모델 — 메시지 전달 방향의 차이

## 질문 배경

Phase 7 정리 C3 답변 중 RabbitMQ 의 구성요소를 정리하다가 다음 의문이 등장했다.

- *"Consumer 가 큐에서 빼가는 식인가? 아니면 메시지 큐 시스템 쪽에서 Consumer 에게 넣어주는 식인가? 혹은 자유롭게 설정 가능한가?"*

이 질문은 분산 메시징 시스템의 *핵심 통신 모델 차이* 를 짚는 의문이다. 같은 *메시지 큐* 카테고리지만 시스템마다 채택한 모델이 다르고, 그 선택이 시스템의 성격을 크게 좌우한다.

---

## 1. 두 모델의 정의

**Pull (Poll) 모델**. Consumer 가 *주기적으로 큐에 "메시지 있나?" 물어보고 가져오는* 방식. 우체국에 매일 가서 *"내 우편 있나요?"* 묻는 것과 같다.

**Push 모델**. 큐가 Consumer 에게 *"메시지 왔으니 받아라"* 적극적으로 전달하는 방식. 우체부가 *집까지 우편을 가져다 주는* 것과 같다.

이 둘은 *누가 통신의 주도권을 갖느냐* 의 차이다. Pull 은 Consumer 가 주도하고, Push 는 큐 (브로커) 가 주도한다.

---

## 2. RabbitMQ — Push 가 기본

RabbitMQ 는 *기본적으로 Push 모델* 을 채택한다. 동작은 다음과 같다.

Consumer 가 시작되면 RabbitMQ 에 *"구독" 을 등록* 한다 (`basic.consume` 명령). 이 등록은 *"이 큐에 메시지가 오면 나에게 보내주세요"* 의 의사 표현이다. 등록 후 RabbitMQ 는 Consumer 와의 *TCP 연결을 유지* 한다.

큐에 메시지가 들어오면 RabbitMQ 가 *알아서 Consumer 에게 전송* 한다. Consumer 는 받은 즉시 처리하고 ack 응답을 보낸다.

Spring AMQP 의 `@RabbitListener` 가 정확히 이 방식으로 동작한다. 본 학습 프로젝트에서 작성한 컨슈머 코드는 *한번 등록하고 가만히 있으면 RabbitMQ 가 알아서 호출해주는* 구조다. 폴링 루프를 작성하지 않아도 메시지가 알아서 들어온다.

RabbitMQ 도 *명시적으로 Pull 을 지원* 하긴 한다 (`basic.get` 명령). 다만 거의 사용되지 않는다. 일회성 디버깅이나 테스트 시나리오 정도에서나 쓰인다.

---

## 3. Kafka — Pull 이 기본

Kafka 는 정반대로 *Pull 모델이 기본* 이다. Consumer 가 Kafka 에 *"내가 마지막으로 읽은 다음 메시지부터 N 개 줘"* 라고 요청해서 가져온다.

이 모델은 Kafka 의 *재생 (replay) 능력* 의 기반이기도 하다. 메시지가 사라지지 않고 보관되기 때문에 Consumer 는 *언제든 자기가 원하는 offset 부터* 다시 읽어갈 수 있다. *"내가 어디까지 읽었다"* 의 offset 관리가 Consumer 책임이 된다.

Pull 모델의 또 다른 강점은 *배치 처리* 다. Consumer 가 자기 페이스로 *"한 번에 100 개 씩"* 가져가는 식의 결정을 스스로 할 수 있다. 네트워크 비용을 줄이는 데 효과적이다.

---

## 4. Push 가 일반적으로 권장되는 이유 (RabbitMQ 영역에서)

같은 *작업 큐* 시나리오에서는 Push 가 Pull 보다 거의 모든 면에서 우수하다.

**지연 면에서**, Push 는 메시지가 도착하면 *즉시* 전달한다. Pull 은 폴링 주기에 따라 *평균 0.5 × 주기* 만큼의 지연이 발생한다. 1 초마다 폴링한다면 평균 0.5 초 지연.

**자원 효율 면에서**, Push 는 *메시지가 있을 때만 통신* 이 발생한다. Pull 은 메시지가 없을 때도 *주기적으로 요청* 을 보내야 해서 의미 없는 트래픽이 누적된다.

**네트워크 면에서**, Push 는 *필요한 트래픽만 발생* 시킨다. Pull 은 *빈 응답을 받는 폴링* 이 다수 섞인다.

이런 특성 때문에 *실시간성과 자원 효율 모두 우수한* Push 가 RabbitMQ 같은 작업 큐 시스템에서 자연스러운 선택이다.

---

## 5. 그럼 Kafka 가 Pull 을 선택한 이유

Kafka 가 Pull 을 선택한 것은 *작업 큐* 가 아니라 *분산 로그* 라는 본질이 다르기 때문이다. 다음과 같은 요구사항을 만족시키려면 Pull 이 더 적합하다.

**Consumer 페이스의 자유** — Kafka 는 *대용량 스트리밍 데이터* 를 다루기 때문에 Consumer 가 처리 속도를 *스스로 조절* 할 수 있어야 한다. Push 라면 브로커가 보내는 속도에 Consumer 가 끌려가게 된다.

**재생 가능성** — 메시지가 영구 보관되므로 Consumer 가 *과거 메시지를 다시 읽는* 시나리오가 자연스럽다. 이는 *Consumer 가 자기 offset 을 관리* 한다는 전제와 맞물려 Pull 이 자연스럽다.

**배치 최적화** — Consumer 가 *"한 번에 얼마나 가져갈지"* 를 결정함으로써 네트워크 비용과 처리 비용을 최적화할 수 있다.

**다양한 Consumer Group 의 독립성** — 같은 토픽을 여러 그룹이 다른 페이스로 읽어가야 한다. Push 라면 브로커가 *모든 그룹에게 적합한 속도* 를 알 수 없다.

---

## 6. 두 모델의 트레이드오프 정리

Push 모델은 *실시간성, 자원 효율, 단순성* 에서 우위다. 다만 *Consumer 의 부하 제어* 가 어렵다. 브로커가 보내는 속도를 Consumer 가 따라가지 못하면 백프레셔 (backpressure) 문제가 생긴다.

Pull 모델은 *Consumer 의 자율성, 재생 가능성, 배치 처리* 에서 우위다. 다만 *폴링 지연과 빈 요청* 의 비용이 따른다. 실시간성이 떨어진다.

→ *작업 큐 시나리오에는 Push 가 자연스럽고*, *대용량 스트리밍이나 재생이 필요한 시나리오에는 Pull 이 자연스럽다*. 어느 것이 절대적으로 우수한 게 아니라 *시나리오 적합성* 의 문제다.

---

## 7. 학습자 질문에 대한 직답

> *"Consumer 가 큐에서 빼가는 식인가? 아니면 메시지 큐 시스템 쪽에서 Consumer 에게 넣어주는 식인가?"*

**RabbitMQ 는 후자 (Push)**. `@RabbitListener` 는 *"구독 등록"* 일 뿐, 실제 메시지 도착 시점에는 RabbitMQ 가 자동으로 호출한다.

> *"혹은 자유롭게 설정 가능한가?"*

**Pull 도 가능하지만 거의 안 쓴다**. 사실상 *어느 메시지 큐를 쓰느냐* 에 따라 권장 모델이 정해져 있다고 봐도 된다. RabbitMQ 는 Push, Kafka 는 Pull. 시스템의 본질과 통신 모델이 깊이 결합되어 있다.

---

## 한 줄 요약

Push 모델은 *브로커가 주도* 해서 메시지를 Consumer 에게 전달하는 방식이고 Pull 모델은 *Consumer 가 주도* 해서 메시지를 가져가는 방식이다. RabbitMQ 는 *실시간성과 자원 효율* 을 위해 Push 를, Kafka 는 *Consumer 자율성과 재생 가능성* 을 위해 Pull 을 기본으로 채택한다. 어느 것이 절대적으로 우수한 게 아니라 *시스템의 본질과 시나리오 적합성* 의 문제다.
